PROJECT 21: Find All Duplicate Numbers in Array

🎯 Cyclic Sort → Negative Marking / Placement Conflict → All-Duplicates Variant

🧩 THE HYPOTHETICAL PROBLEM

→ You are given an array of N integers
→ Each integer is in the range [1, N] inclusive
→ Some elements appear TWICE, others appear ONCE
→ Find ALL the elements that appear TWICE
→ You must do it in O(n) time and O(1) extra space
→ The output list does NOT count as extra space

Example 1:
  Input:  [4, 3, 2, 7, 8, 2, 3, 1]
  N = 8, values range [1, 8]
  
  Value 2 appears at index 2 and index 5 → DUPLICATE
  Value 3 appears at index 1 and index 6 → DUPLICATE
  Output: [2, 3]

Example 2:
  Input:  [1, 1, 2]
  N = 3, values range [1, 3]
  
  Value 1 appears twice → DUPLICATE
  Value 3 is missing
  Output: [1]

Example 3:
  Input:  [1, 2, 3, 4, 5]
  N = 5, all unique
  Output: []  (empty — no duplicates)

CONSTRAINTS:
  → O(n) time → no sorting allowed (that's O(n log n))
  → O(1) extra space → no HashSet/HashMap allowed
  → Must use the ARRAY ITSELF as the tracking mechanism

🔍 HOW TO APPROACH IT

Step 1 — Recall Project 8: Cyclic Sort and Missing Number

→ Project 8 taught us: when values are in range [1, N] or [0, N]
→ Each value has a "home index" → value V belongs at index V-1
→ We can PLACE each value at its correct position
→ After placement, any index where value ≠ index+1 reveals anomalies

→ Project 8 found ONE missing number
→ Project 21 finds ALL duplicate numbers
→ Same index-value mapping → different detection strategy

Step 2 — Why Standard Approaches Fail the Constraints

APPROACH 1: Sorting
  → Sort the array → duplicates become adjacent
  → Time: O(n log n) → VIOLATES O(n) requirement
  → Also modifies the array in a destructive way

APPROACH 2: HashSet
  → Traverse array, add to set, if already in set → duplicate
  → Time: O(n) ✓
  → Space: O(n) → VIOLATES O(1) space requirement

APPROACH 3: Brute Force
  → For each element, scan entire array for duplicates
  → Time: O(n²) → VIOLATES O(n) requirement

→ All standard approaches fail at least one constraint
→ We need a CLEVER trick that uses the array itself as storage

Step 3 — The Core Insight: Array as HashMap

→ Values are in range [1, N]
→ Array has indices [0, N-1]
→ There's a NATURAL MAPPING: value V → index (V-1)

→ If we could "mark" index (V-1) when we see value V:
     → First time we see V → mark index (V-1)
     → Second time we see V → index (V-1) is ALREADY marked → V is a duplicate

→ HOW to mark without extra space?
→ TWO techniques:

TECHNIQUE 1: Negative Marking
  → Negate the value at index (V-1) to indicate "V has been seen"
  → If arr[V-1] is already negative → V was already seen → DUPLICATE

TECHNIQUE 2: Cyclic Sort Placement
  → Try to place each value at its home index
  → If home already has the correct value → DUPLICATE
  → Same as Project 8 but detecting conflicts instead of gaps

Step 4 — Technique 1: Negative Marking (Detailed)

Array: [4, 3, 2, 7, 8, 2, 3, 1]
Index:  0  1  2  3  4  5  6  7

Process each element:
  → For value V, go to index |V|-1 (absolute value because we may have negated)
  → If arr[|V|-1] is POSITIVE → first time seeing V → negate it
  → If arr[|V|-1] is NEGATIVE → second time seeing V → V is a DUPLICATE

Step-by-step:
  i=0: val = |4| = 4 → check index 3 → arr[3] = 7 (positive) → negate → arr = [4,3,2,-7,8,2,3,1]
  i=1: val = |3| = 3 → check index 2 → arr[2] = 2 (positive) → negate → arr = [4,3,-2,-7,8,2,3,1]
  i=2: val = |-2| = 2 → check index 1 → arr[1] = 3 (positive) → negate → arr = [4,-3,-2,-7,8,2,3,1]
  i=3: val = |-7| = 7 → check index 6 → arr[6] = 3 (positive) → negate → arr = [4,-3,-2,-7,8,2,-3,1]
  i=4: val = |8| = 8 → check index 7 → arr[7] = 1 (positive) → negate → arr = [4,-3,-2,-7,8,2,-3,-1]
  i=5: val = |2| = 2 → check index 1 → arr[1] = -3 (NEGATIVE!) → 2 is DUPLICATE → add to result
  i=6: val = |-3| = 3 → check index 2 → arr[2] = -2 (NEGATIVE!) → 3 is DUPLICATE → add to result
  i=7: val = |-1| = 1 → check index 0 → arr[0] = 4 (positive) → negate → arr = [-4,-3,-2,-7,8,2,-3,-1]

Result: [2, 3] ✓

Step 5 — Technique 2: Cyclic Sort Placement (Detailed)

Array: [4, 3, 2, 7, 8, 2, 3, 1]
Index:  0  1  2  3  4  5  6  7

Goal: Place each value V at index V-1

CYCLIC SORT PROCESS:
  i=0: arr[0]=4, should be at index 3
       arr[3]=7 ≠ 4 → swap → arr = [7,3,2,4,8,2,3,1]
       arr[0]=7, should be at index 6
       arr[6]=3 ≠ 7 → swap → arr = [3,3,2,4,8,2,7,1]
       arr[0]=3, should be at index 2
       arr[2]=2 ≠ 3 → swap → arr = [2,3,3,4,8,2,7,1]
       arr[0]=2, should be at index 1
       arr[1]=3 ≠ 2 → swap → arr = [3,2,3,4,8,2,7,1]
       arr[0]=3, should be at index 2
       arr[2]=3 == 3 → ALREADY CORRECT → SKIP (would swap same values forever)
       → Move to i=1
       
  i=1: arr[1]=2, should be at index 1 → ALREADY CORRECT → move on
  i=2: arr[2]=3, should be at index 2 → ALREADY CORRECT → move on
  i=3: arr[3]=4, should be at index 3 → ALREADY CORRECT → move on
  i=4: arr[4]=8, should be at index 7
       arr[7]=1 ≠ 8 → swap → arr = [3,2,3,4,1,2,7,8]
       arr[4]=1, should be at index 0
       arr[0]=3 ≠ 1 → swap → arr = [1,2,3,4,3,2,7,8]
       arr[4]=3, should be at index 2
       arr[2]=3 == 3 → ALREADY CORRECT → SKIP
       → Move to i=5
       
  i=5: arr[5]=2, should be at index 1
       arr[1]=2 == 2 → ALREADY CORRECT → SKIP
       → Move to i=6
       
  i=6: arr[6]=7, should be at index 6 → ALREADY CORRECT → move on
  i=7: arr[7]=8, should be at index 7 → ALREADY CORRECT → move on

Final array: [1, 2, 3, 4, 3, 2, 7, 8]
                                         
Now scan for mismatches:
  i=0: arr[0]=1 == 0+1 ✓
  i=1: arr[1]=2 == 1+1 ✓
  i=2: arr[2]=3 == 2+1 ✓
  i=3: arr[3]=4 == 3+1 ✓
  i=4: arr[4]=3 ≠ 4+1 → 3 is DUPLICATE ← the value 3 couldn't go home
  i=5: arr[5]=2 ≠ 5+1 → 2 is DUPLICATE ← the value 2 couldn't go home
  i=6: arr[6]=7 == 6+1 ✓
  i=7: arr[7]=8 == 7+1 ✓

Result: [3, 2] → sorted: [2, 3] ✓

🛠️ HOW CAN WE SOLVE IT

Algorithm 1: Negative Marking — Precise Steps

findDuplicates_NegativeMarking(nums):

  1. CREATE empty result list
  
  2. FOR each index i from 0 to n-1:
     a. Get the actual value: val = |nums[i]|   (absolute value — may have been negated)
     b. Compute the "home" index: homeIdx = val - 1
     c. CHECK nums[homeIdx]:
        → IF nums[homeIdx] > 0:
             → First time seeing val → negate: nums[homeIdx] = -nums[homeIdx]
        → IF nums[homeIdx] < 0:
             → Second time seeing val → val is a DUPLICATE → add to result
  
  3. (Optional) RESTORE array: negate all negative values back to positive
  
  4. RETURN result

Algorithm 2: Cyclic Sort — Precise Steps

findDuplicates_CyclicSort(nums):

  1. CYCLIC SORT phase:
     → FOR each index i from 0 to n-1:
          → WHILE nums[i] != nums[nums[i] - 1]:
               → Swap nums[i] with nums[nums[i] - 1]
          → (Loop exits when value is at home OR home already has correct value)
  
  2. DETECTION phase:
     → FOR each index i from 0 to n-1:
          → IF nums[i] != i + 1:
               → nums[i] is a duplicate → add to result
  
  3. RETURN result

✅ WHAT SHOULD YOU DO

→ ALWAYS take absolute value before computing home index (negative marking)
     → Previous iterations may have negated the value
     → |nums[i]| gives the TRUE value regardless of marking state

→ ALWAYS check if value is in valid range [1, N]
     → If problem guarantees range → no check needed
     → If not guaranteed → out-of-range values would cause ArrayIndexOutOfBounds

→ ALWAYS understand WHY each technique works:
     → Negative marking: array[V-1] < 0 means "V has been seen"
     → Cyclic sort: value at wrong index means "correct value is a duplicate"

→ ALWAYS consider restoring the array if required
     → Negative marking modifies the array
     → Some problems require the array to remain unchanged
     → Restore by negating all negatives back to positive at the end

→ ALWAYS use the while loop (not if) in cyclic sort
     → One swap may bring a new displaced element to current index
     → Keep swapping until current index has its correct value
     → Or until we detect a conflict (duplicate blocking)

🚫 WHAT SHOULD YOU NOT DO

→ DO NOT forget absolute value in negative marking
     WRONG:
       int val = nums[i];        // could be negative from earlier marking
       int homeIdx = val - 1;     // negative index → crash!
     
     RIGHT:
       int val = Math.abs(nums[i]);
       int homeIdx = val - 1;

→ DO NOT create infinite loop in cyclic sort
     WRONG:
       while (nums[i] != i + 1) {
           swap(nums, i, nums[i] - 1);
       }
     → If nums[i]=3 and nums[2]=3 → swap gives same result → INFINITE LOOP
     
     RIGHT:
       while (nums[i] != nums[nums[i] - 1]) {
           swap(nums, i, nums[i] - 1);
       }
     → Check if TARGET position already has correct value
     → If yes → stop → duplicate detected later in scan phase

→ DO NOT use both techniques simultaneously
     → Pick ONE → either negative marking OR cyclic sort
     → Mixing them creates confusion and bugs
     → Negative marking is simpler → prefer it unless asked specifically

→ DO NOT add a value to result more than once
     → Negative marking naturally prevents this:
          → First encounter: positive → negate → don't add
          → Second encounter: negative → add to result
          → Third encounter? Can't happen → values appear at most twice

→ DO NOT confuse this with "find missing numbers"
     → Missing: which values in [1,N] are ABSENT from array?
     → Duplicates: which values in [1,N] appear MORE THAN ONCE?
     → Same technique → different detection logic:
          → Missing: after sort, arr[i] != i+1 → i+1 is missing
          → Duplicate: after sort, arr[i] != i+1 → arr[i] is duplicate

🔀 WHEN TO USE THIS PATTERN

✓ Values are bounded in range [1, N] or [0, N-1]
     → This range allows index-value mapping
     → Each value maps to exactly one index
     
✓ O(1) extra space is required
     → Can't use HashSet or HashMap
     → Must use the array itself as a tracking mechanism
     
✓ Finding duplicates, missing numbers, or both
     → Cyclic sort family handles all variants:
          → Project 8: find ONE missing number
          → Project 21: find ALL duplicates
          → Variant: find all missing numbers (same technique)
          → Variant: find the first missing positive
          
✓ Elements appear at most TWICE (for negative marking)
     → Negative marking works cleanly for at-most-twice
     → For more occurrences → cyclic sort is safer

✓ Array can be modified in-place
     → Both techniques alter the array during processing
     → If array must stay unchanged → need different approach

🚫 WHEN NOT TO USE THIS PATTERN

✗ Values are NOT bounded in a small range
     → Array [1000000, 5, 3] with N=3 → value 1000000 has no valid home index
     → Index-value mapping requires values ≈ indices
     → For unbounded values → use HashMap or sorting

✗ O(1) space is NOT required
     → HashSet is simpler and more readable
     → If space constraint is relaxed → prefer HashSet for clarity
     
✗ Array cannot be modified
     → Both techniques modify the array
     → If modification is forbidden AND O(1) space required → very constrained
     → May need XOR-based or math-based tricks
     
✗ Elements can appear more than twice
     → Negative marking breaks: third occurrence sees negative → adds again
     → Would need additional handling
     → Cyclic sort still works but detection changes

✗ Array contains zeros or negative values
     → Negative marking relies on sign as a flag → zeros break it
     → Cyclic sort relies on value-1 as index → negatives break it
     → Need value transformation or different technique

⚙️ HOW IT ONLY WORKS — Internal Mechanics

Negative Marking: The Sign-as-Flag Trick

CORE IDEA:
→ Each index position serves as a "bucket" for one value
→ Index 0 is the bucket for value 1
→ Index 1 is the bucket for value 2
→ Index k is the bucket for value k+1

→ The SIGN of arr[k] serves as a boolean flag:
     → Positive: value (k+1) has NOT been seen yet
     → Negative: value (k+1) HAS been seen

→ The MAGNITUDE of arr[k] preserves the original value
     → We need this because arr[k] itself may be processed later
     → |arr[k]| gives the true value regardless of flag state

WALK THROUGH THE STATE:
  Initial:  [4, 3, 2, 7, 8, 2, 3, 1]
  
  Process 4: flag index 3 → [4, 3, 2, -7, 8, 2, 3, 1]
                                       ^
                                    "4 seen"
  
  Process 3: flag index 2 → [4, 3, -2, -7, 8, 2, 3, 1]
                                    ^
                                 "3 seen"
  
  Process |-2|=2: flag index 1 → [4, -3, -2, -7, 8, 2, 3, 1]
                                      ^
                                   "2 seen"
  
  Process |-7|=7: flag index 6 → [4, -3, -2, -7, 8, 2, -3, 1]
                                                     ^
                                                  "7 seen"
  
  Process 8: flag index 7 → [4, -3, -2, -7, 8, 2, -3, -1]
                                                        ^
                                                     "8 seen"
  
  Process 2: check index 1 → arr[1] = -3 → ALREADY NEGATIVE
             → 2 was already seen → DUPLICATE! → add 2 to result
  
  Process |-3|=3: check index 2 → arr[2] = -2 → ALREADY NEGATIVE
                  → 3 was already seen → DUPLICATE! → add 3 to result
  
  Process |-1|=1: flag index 0 → [-4, -3, -2, -7, 8, 2, -3, -1]
                                    ^
                                 "1 seen"

FINAL STATE: [-4, -3, -2, -7, 8, 2, -3, -1]
→ Negative positions: 0,1,2,3,6,7 → values 1,2,3,4,7,8 were seen
→ Positive positions: 4,5 → values 5,6 were NOT seen (these are MISSING)
→ Duplicates found during process: 2, 3

BONUS INSIGHT:
→ After negative marking, you can find BOTH duplicates AND missing numbers!
→ Duplicates: detected during the process (second encounter of negative)
→ Missing: indices that remain POSITIVE after the process
→ This dual capability makes negative marking very powerful

Cyclic Sort: The Placement Approach

CORE IDEA:
→ Put each value at its "home" → value V goes to index V-1
→ After sorting, scan for misplaced values
→ Any value not at home is a duplicate (because its home is occupied by another copy)

WHY THE WHILE LOOP TERMINATES:
→ Each swap places AT LEAST ONE value at its correct home
→ A value at its correct home is NEVER moved again
→ Maximum N values → maximum N swaps total across ALL iterations
→ Total work across all i iterations: O(n) → amortized O(1) per element

PROOF OF O(n) TOTAL SWAPS:
→ Define "displaced count" = number of values NOT at their home
→ Initially: displaced count ≤ n
→ Each swap: at least one value reaches home → displaced count decreases by ≥ 1
→ Displaced count can never go below 0
→ Therefore: total swaps ≤ n
→ Each swap is O(1) → total sort phase: O(n) ✓

THE DUPLICATE DETECTION:
→ After cyclic sort: value V is at index V-1 for all non-duplicate values
→ Duplicate values: one copy is at home, other is "homeless"
→ Homeless copies end up at indices where the MISSING values should be
→ Scan: if arr[i] ≠ i+1 → arr[i] is homeless → arr[i] is a duplicate

Why Values Appear at Most Twice Matters

FOR NEGATIVE MARKING:
→ If value V appears 3 times:
     → First encounter: arr[V-1] positive → negate → correct
     → Second encounter: arr[V-1] negative → add V to result → correct
     → Third encounter: arr[V-1] is still negative → add V AGAIN → DUPLICATE IN RESULT
→ Need additional logic to prevent re-adding

FOR CYCLIC SORT:
→ If value V appears 3 times:
     → One copy goes to home index V-1
     → Two copies are homeless → both detected in scan
     → V appears twice in result → need deduplication
→ At-most-twice constraint simplifies both techniques significantly

Time and Space Complexity — Both Approaches

NEGATIVE MARKING:
  Time:  O(n) → single pass through array
  Space: O(1) extra → only loop variable and temporary values
  
CYCLIC SORT:
  Time:  O(n) → sort phase O(n) amortized + scan phase O(n)
  Space: O(1) extra → only loop variables and swap temp

Both are optimal for this problem.
Negative marking is simpler → fewer lines of code → less error-prone.
Cyclic sort is more general → works for more variants of the problem.

⚠️ COMMON PITFALLS AND FAILURE MODES

Pitfall 1: Forgetting Math.abs() — The #1 Bug

SCENARIO:
  Array: [3, 1, 3]
  
  i=0: val = 3, homeIdx = 2, arr[2] = 3 (positive) → negate → arr = [3, 1, -3]
  i=1: val = 1, homeIdx = 0, arr[0] = 3 (positive) → negate → arr = [-3, 1, -3]
  i=2: val = arr[2] = -3
  
  WITHOUT Math.abs():
    val = -3, homeIdx = -3-1 = -4 → ArrayIndexOutOfBoundsException!
  
  WITH Math.abs():
    val = |-3| = 3, homeIdx = 3-1 = 2, arr[2] = -3 (negative) → DUPLICATE → add 3 ✓

→ ALWAYS use Math.abs(nums[i]) when reading values
→ The negation from previous iterations WILL cause crashes without it

Pitfall 2: Infinite Loop in Cyclic Sort

SCENARIO:
  Array: [2, 2]
  
  i=0: nums[0] = 2, home = index 1
       nums[1] = 2 → nums[0] == nums[nums[0]-1] → 2 == 2 → STOP
  
  If we used the WRONG condition:
    while (nums[i] != i + 1):
      nums[0] = 2, i+1 = 1 → 2 ≠ 1 → swap
      swap(nums[0], nums[1]) → [2, 2] → SAME ARRAY → INFINITE LOOP

  RIGHT condition:
    while (nums[i] != nums[nums[i] - 1]):
      nums[0] = 2, nums[nums[0]-1] = nums[1] = 2 → 2 == 2 → STOP → no swap ✓

→ Compare value at CURRENT index with value at HOME index
→ NOT value with expected value at current index
→ This correctly handles duplicates without infinite loops

Pitfall 3: Index Off-by-One

→ Values range [1, N] → indices range [0, N-1]
→ Value V maps to index V-1 (NOT V)
→ Index i expects value i+1 (NOT i)

WRONG: homeIdx = val;      → off by one → crash at val=N
RIGHT: homeIdx = val - 1;  → correct mapping

WRONG: if (nums[i] != i)   → checks for [0,N-1] range values
RIGHT: if (nums[i] != i+1) → checks for [1,N] range values

Pitfall 4: Modifying Array When Not Allowed

→ Some problem variants say "do not modify the input array"
→ Negative marking modifies the array
→ Cyclic sort modifies the array

OPTIONS:
  a) Restore array after processing (negate back all negatives)
  b) Clone the array first → work on the clone
  c) Use a different approach entirely (Floyd's cycle detection on array)
     → Treat array as linked list: index → value → next index
     → Find duplicate = find cycle start (Project 20 technique!)
     → O(n) time, O(1) space, NO modification
     → Advanced technique — combines Projects 20 and 21

Pitfall 5: Result Order

→ Negative marking finds duplicates in ORDER OF SECOND OCCURRENCE
→ Cyclic sort finds duplicates in ORDER OF POSITION (left to right scan)
→ These may differ

Example: [4, 3, 2, 7, 8, 2, 3, 1]
  Negative marking: finds 2 first (index 5), then 3 (index 6) → [2, 3]
  Cyclic sort: finds 3 first (index 4), then 2 (index 5) → [3, 2]

→ If problem requires sorted output → sort the result list
→ If problem says "any order" → either technique is fine
→ READ THE PROBLEM for output order requirements

ALL rely on the same insight:
  Values in [1,N] → natural index mapping → array IS the hash table

🧠 KEY TAKEAWAY

WHAT WE LEARNED:
→ When values are bounded in [1,N], the array ITSELF becomes a hash table
→ Index-value mapping: value V → index V-1
→ Negative marking: sign of arr[V-1] = boolean flag for "V was seen"
→ Cyclic sort: place values at home → homeless values = duplicates
→ Both achieve O(n) time, O(1) space → optimal
→ Negative marking also reveals MISSING numbers as a bonus
→ Math.abs() is critical → previous negations corrupt raw values

THE CORE MENTAL MODEL:
→ "Values fit into index range → I can use the array as a boolean flags array"
→ "First visit → mark. Second visit → already marked → duplicate"
→ "The array IS the data structure — no extra space needed"

PATTERN SIGNATURE:
→ Values in range [1, N] ✓
→ Array of size N ✓
→ Find duplicates / missing ✓
→ O(1) extra space required ✓
→ = CYCLIC SORT FAMILY → NEGATIVE MARKING or PLACEMENT
