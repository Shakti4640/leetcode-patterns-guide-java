PROJECT 46: Minimum Meeting Rooms Required

🎯 Intervals → Min-Heap of End Times → Overlap Counting via Heap Size

🧩 THE HYPOTHETICAL PROBLEM

→ You are given an array of meeting time intervals
→ Each interval has a START time and an END time
→ intervals[i] = [start_i, end_i]
→ Find the MINIMUM number of conference rooms required
     so that ALL meetings can take place without conflict

→ A meeting [0, 30] and [5, 10] OVERLAP → need 2 rooms
→ A meeting [0, 10] and [10, 20] do NOT overlap → 1 room suffices
     (one ends exactly when the other starts)

Example 1:
  Input:  [[0,30], [5,10], [15,20]]
  Output: 2
  
  → [0,30] occupies Room 1 the entire time
  → [5,10] starts at 5 → Room 1 is busy → need Room 2
  → [15,20] starts at 15 → Room 2 freed at 10 → reuse Room 2
  → Peak simultaneous usage = 2 rooms

Example 2:
  Input:  [[7,10], [2,4]]
  Output: 1
  
  → [2,4] ends at 4 → [7,10] starts at 7 → no overlap → 1 room

Example 3:
  Input:  [[1,5], [2,6], [3,7], [4,8]]
  Output: 4
  
  → Every meeting overlaps with every other → need 4 rooms

Example 4:
  Input:  [[0,5], [5,10], [10,15], [15,20]]
  Output: 1
  
  → Chain of non-overlapping meetings → 1 room suffices

🔍 HOW TO APPROACH IT

Step 1 — Recognize the Difference from Project 14

Project 14 (Merge Overlapping Intervals):
  → QUESTION: Which intervals overlap? → MERGE them into one
  → OUTPUT: A reduced list of merged intervals
  → FOCUS: Combining overlapping ranges

THIS Problem:
  → QUESTION: At any moment, how many intervals overlap SIMULTANEOUSLY?
  → OUTPUT: The PEAK overlap count = rooms needed
  → FOCUS: Counting concurrent usage, not merging

KEY SHIFT:
  → Project 14 asked "WHAT overlaps with what?"
  → This asks "HOW MANY overlap at the SAME TIME?"
  → Same sorted-intervals foundation → completely different operation on top

Step 2 — Think About What "Needing a Room" Means

→ A meeting STARTS → it needs a room
→ A meeting ENDS → it frees a room
→ At any point in time:
     rooms_in_use = (meetings that have started) - (meetings that have ended)
→ The answer = MAXIMUM of rooms_in_use across all time points

→ But we don't need to check EVERY time point
→ Only EVENTS matter: starts and ends
→ The count only changes when a meeting starts or ends
→ This is an EVENT-DRIVEN problem

Step 3 — Consider Three Approaches

APPROACH A — Brute Force:
  → For each meeting, count how many OTHER meetings overlap with it
  → O(n²) time → too slow for large inputs
  → Also doesn't directly give the peak concurrent count

APPROACH B — Event Timeline (Chronological Ordering):
  → Treat each start as +1 event and each end as -1 event
  → Sort all events by time
  → Sweep through, maintaining a running count
  → Peak count = answer
  → O(n log n) time, O(n) space

APPROACH C — Min-Heap of End Times:
  → Sort meetings by start time
  → Use a min-heap to track end times of ACTIVE meetings
  → For each new meeting:
       → If it starts AFTER the earliest ending meeting → reuse that room
       → Otherwise → allocate a new room
  → Heap size at any point = rooms currently in use
  → O(n log n) time, O(n) space

→ APPROACH C is the INTERVIEW STANDARD for this problem
→ APPROACH B is equally valid and sometimes cleaner
→ We will implement BOTH

Step 4 — Visualize the Min-Heap Approach

Meetings: [[0,30], [5,10], [15,20]]
Sort by start: [[0,30], [5,10], [15,20]]  (already sorted)

HEAP tracks: end times of meetings currently using rooms

Step 1: Meeting [0, 30]
  → Heap is empty → no room to reuse
  → Allocate room → push end time 30
  → Heap: [30]
  → Rooms in use: 1

Step 2: Meeting [5, 10]
  → Heap top = 30 → earliest room frees at time 30
  → New meeting starts at 5 → 5 < 30 → CANNOT reuse
  → Allocate new room → push end time 10
  → Heap: [10, 30]
  → Rooms in use: 2  ← PEAK

Step 3: Meeting [15, 20]
  → Heap top = 10 → earliest room frees at time 10
  → New meeting starts at 15 → 15 ≥ 10 → CAN reuse!
  → Remove 10 from heap (room freed)
  → Push end time 20 (room now occupied until 20)
  → Heap: [20, 30]
  → Rooms in use: 2

Answer: max heap size seen = 2

Step 5 — Visualize the Event Timeline Approach

Meetings: [[0,30], [5,10], [15,20]]

Events:
  Time 0:  +1 (meeting starts)
  Time 5:  +1 (meeting starts)
  Time 10: -1 (meeting ends)
  Time 15: +1 (meeting starts)
  Time 20: -1 (meeting ends)
  Time 30: -1 (meeting ends)

Sort by time (ties: end before start → -1 before +1):
  Time 0:  +1 → count=1
  Time 5:  +1 → count=2  ← PEAK
  Time 10: -1 → count=1
  Time 15: +1 → count=2  ← ties PEAK
  Time 20: -1 → count=1
  Time 30: -1 → count=0

Answer: 2

🛠️ HOW CAN WE SOLVE IT

Algorithm A — Min-Heap of End Times (Primary)

1. Sort all meetings by START time ascending
2. Create a min-heap (priority queue) to store END times
3. For each meeting in sorted order:
     a. IF heap is NOT empty AND heap.peek() <= meeting.start:
          → The earliest-ending room is now free
          → POLL it from the heap (room reused)
     b. PUSH meeting.end into the heap (room occupied until this time)
4. Return heap.size() → this is the peak number of rooms needed

Why heap.peek() <= start (not strictly <)?

→ Meeting [0, 10] ends at time 10
→ Meeting [10, 20] starts at time 10
→ A room freed at 10 CAN be used by a meeting starting at 10
→ So: if earliest_end <= new_start → room is available → reuse
→ If earliest_end > new_start → room still occupied → need new room

Algorithm B — Event Timeline (Alternative)

1. Create two arrays: starts[] and ends[]
2. Sort both independently
3. Use two pointers: sPtr on starts, ePtr on ends
4. Sweep forward:
     → If starts[sPtr] < ends[ePtr]: a meeting starts before one ends
          → rooms_in_use++ → sPtr++
     → Else: a meeting ends before/at next start
          → rooms_in_use-- → ePtr++
5. Track maximum of rooms_in_use

✅ WHAT SHOULD YOU DO

→ ALWAYS sort by start time for the heap approach
     → Processing meetings in chronological start order is essential
     → Without sorting, heap comparisons become meaningless

→ ALWAYS use a MIN-heap (not max-heap)
     → You want the EARLIEST ending meeting on top
     → That's the room most likely to be freed soonest
     → This is GREEDY: always try to reuse the room that frees first

→ ALWAYS use <= for the reuse check (not <)
     → end_time == start_time means the room IS available
     → [0,10] and [10,20] → sequential, not overlapping

→ ALWAYS push the new meeting's end time AFTER the potential poll
     → Order matters: first check if a room is free, THEN occupy it

→ CONSIDER the event timeline approach as an alternative
     → Some interviewers prefer it for clarity
     → Both are O(n log n) → equally optimal

→ TRACK the maximum heap size if needed explicitly
     → In the min-heap approach, final heap size IS the answer
     → Because we only poll when reusing → heap never shrinks below peak
     
     → WAIT — this needs clarification:
     → Actually, heap CAN shrink when rooms are reused
     → But the FINAL heap size equals the peak
     → Because: once peak rooms are allocated, they're never ALL freed simultaneously
     → Actually NO — we need to track max(heap.size()) explicitly
     → OR: realize that heap.size() after processing ALL meetings = answer
     → Let me think again...
     
     → CORRECTION:
     → We poll AT MOST one room per meeting (the earliest ending one)
     → Then we push one room per meeting
     → Net effect: size stays same (reuse) or grows by 1 (new room)
     → Heap size NEVER decreases below any previous size after processing
     → Wait — it CAN: poll removes 1, push adds 1 → net 0 change
     → So heap only grows or stays same → final size = peak = answer ✓

🚫 WHAT SHOULD YOU NOT DO

→ DO NOT sort by end time for the heap approach
     → Activity Selection (Project 19) sorts by end time → DIFFERENT problem
     → Activity Selection asks "max non-overlapping meetings in ONE room"
     → This asks "min rooms for ALL meetings"
     → Different questions → different sort orders

→ DO NOT use a max-heap
     → Max-heap gives you the LATEST ending room
     → That's the room LEAST likely to be free
     → You'd never reuse any room → allocate n rooms always
     → WRONG answer

→ DO NOT poll ALL meetings that have ended before current start
     → Only poll ONE — the earliest ending
     → Why? Each poll represents ONE room being reused
     → Each new meeting uses exactly ONE room
     → You only need to free ONE room to accommodate ONE new meeting
     
     → ACTUALLY — you only NEED to poll one, but polling more won't hurt
     → Polling just the top (earliest) is sufficient and most efficient
     → If the earliest hasn't ended, no other has either (min-heap property)

→ DO NOT confuse this with Merge Intervals (Project 14)
     → Merge Intervals modifies the interval list
     → This counts concurrent usage
     → Merge would give you 1 merged interval for [[0,30],[5,10],[15,20]]
       → merged: [[0,30]] → tells you nothing about rooms needed

→ DO NOT forget edge cases
     → Empty input → 0 rooms
     → Single meeting → 1 room
     → All meetings at same time → n rooms
     → All meetings sequential → 1 room

🔀 WHEN TO USE THIS PATTERN

✓ You need to count MAXIMUM CONCURRENT / SIMULTANEOUS events
✓ Events have start and end times (intervals)
✓ You need to find PEAK resource usage
✓ Real-world analogs:
     → Meeting rooms needed
     → Server connections at peak
     → CPU cores needed for parallel tasks
     → Parking spots needed
     → Overlapping booking detection
     → Maximum bandwidth usage

✓ The "interval + heap" combination appears when:
     → Intervals need to be processed in order (sort by start)
     → You need to track what's CURRENTLY ACTIVE (heap of end times)
     → You need to make GREEDY reuse decisions (poll earliest ending)

✓ The "event timeline" alternative appears when:
     → You only care about the COUNT, not which specific rooms
     → The problem reduces to "max overlap depth"
     → You want a cleaner implementation

🚫 WHEN NOT TO USE THIS PATTERN

✗ You need to MERGE intervals into non-overlapping ranges
     → That's Project 14 → sort + linear scan → no heap needed

✗ You need to find the MAXIMUM number of non-overlapping intervals
     → That's Activity Selection (Project 19) → sort by END time → greedy pick

✗ Intervals are being inserted/deleted dynamically
     → Heap approach works for static input processed once
     → Dynamic interval management → segment trees or balanced BSTs

✗ You need to know WHICH meetings go in WHICH rooms
     → The min-heap approach tells you HOW MANY rooms
     → Not which meeting is assigned where
     → For assignment → need to track room IDs alongside end times

✗ The "resource" is not fungible
     → Min-heap assumes all rooms are identical
     → If rooms have different capacities/features → different problem

⚙️ HOW IT ONLY WORKS — Internal Mechanics

Why Sorting by Start Time is Essential

Meetings: [[5,10], [0,30], [15,20]]

Without sorting:
  Step 1: [5,10] → heap: [10] → 1 room
  Step 2: [0,30] → heap top=10, start=0 → 0 < 10 → new room
           → heap: [10, 30] → 2 rooms
  Step 3: [15,20] → heap top=10, start=15 → 15≥10 → reuse
           → poll 10, push 20 → heap: [20, 30] → 2 rooms
  Answer: 2 ✓ (happens to be correct here)

But consider: [[15,20], [0,30], [5,10]]
  Step 1: [15,20] → heap: [20] → 1 room
  Step 2: [0,30] → heap top=20, start=0 → 0 < 20 → new room
           → heap: [20, 30] → 2 rooms
  Step 3: [5,10] → heap top=20, start=5 → 5 < 20 → new room
           → heap: [10, 20, 30] → 3 rooms
  Answer: 3 ✗ WRONG!

WHY?
  → Without sorting, [5,10] arrives AFTER [15,20]
  → The heap already holds 20 (from [15,20])
  → [5,10] can't reuse a room that "ends at 20" because it starts at 5
  → But in reality, [5,10] happens BEFORE [15,20]
  → The room used by [5,10] would be FREE by time 15

→ Sorting by start time ensures we process events CHRONOLOGICALLY
→ This guarantees the heap always reflects the TRUE state of room usage
→ At the time we process meeting M:
     → All meetings starting before M are already in the heap
     → We can correctly check if any of them have ended

Why Min-Heap (Not Just Any Tracking)

When a new meeting starts at time S:
  → We need to check: "Has ANY active meeting ended by time S?"
  → The meeting MOST LIKELY to have ended is the one ending EARLIEST
  → If the earliest-ending meeting hasn't ended → NONE have ended
  → Min-heap gives us the earliest-ending meeting in O(1)

Alternative: Could we use a sorted list?
  → Yes, but insertion is O(n) → total O(n²)
  → Min-heap insertion is O(log n) → total O(n log n)
  → Heap is the efficient choice

Alternative: Could we check ALL heap entries?
  → Unnecessary — if min hasn't ended, nothing has
  → If min HAS ended, we only need to free ONE room
  → Because one new meeting needs exactly one room

The Greedy Correctness Proof

CLAIM: Always reusing the earliest-ending room gives optimal result.

PROOF BY EXCHANGE ARGUMENT:
  → Suppose optimal solution uses K rooms
  → Suppose our greedy uses K' rooms where K' > K
  → Then there exists a meeting M where greedy allocates a new room
    but optimal reuses an existing room
  → At time M.start, greedy's heap says ALL rooms are occupied
    (min end time > M.start)
  → This means there are K' meetings whose intervals all contain time M.start
  → K' intervals all overlap at one point → ANY solution needs ≥ K' rooms
  → Contradiction with K < K'
  → Therefore K' ≤ K → greedy is optimal ✓

INTUITION:
  → We never allocate a room unless FORCED to
  → "Forced" = every existing room is occupied
  → If every room is occupied when meeting M starts
  → Then M overlaps with every active meeting
  → Those active meetings + M all need separate rooms
  → No solution can avoid this allocation

Why Final Heap Size = Answer

→ For each meeting, we do ONE of:
     a. Poll one + Push one → heap size unchanged (room reused)
     b. Push one only → heap size grows by 1 (new room allocated)

→ Heap size NEVER decreases (we never poll without pushing)
→ Heap size only increases when a new room is NEEDED
→ Final heap size = total rooms allocated = answer

→ Alternatively: max(heap.size()) after each push = answer
→ But since size is monotonically non-decreasing → final = max

Time Complexity Breakdown

→ Sorting: O(n log n)
→ For each of n meetings:
     → heap.peek(): O(1)
     → heap.poll(): O(log n) — at most once per meeting
     → heap.offer(): O(log n) — exactly once per meeting
→ Total heap operations: O(n log n)
→ OVERALL: O(n log n)

Space Complexity Breakdown

→ Heap stores at most n end times (worst case: all meetings overlap)
→ Sorting may use O(log n) or O(n) depending on implementation
→ OVERALL: O(n)

⚠️ COMMON PITFALLS AND FAILURE MODES

Pitfall 1: Sorting by End Time Instead of Start Time

→ Activity Selection (Project 19): sort by END → maximize non-overlapping count
→ Meeting Rooms: sort by START → count peak overlap
→ These are OPPOSITE problems:
     → Activity Selection: fit MOST meetings in ONE room
     → Meeting Rooms: fit ALL meetings in FEWEST rooms
→ Wrong sort order → wrong answer guaranteed

Pitfall 2: Strict Inequality in Reuse Check

WRONG: if (heap.peek() < meeting.start)  → strictly less
  → Meeting [0,10] ends at 10, meeting [10,20] starts at 10
  → 10 < 10 is FALSE → won't reuse → allocates extra room
  → But the room IS free at time 10

RIGHT: if (heap.peek() <= meeting.start)  → less than or equal
  → 10 <= 10 is TRUE → reuses room → correct

→ HOWEVER: some problem definitions say [0,10] and [10,20] DO overlap
→ READ THE PROBLEM CAREFULLY
→ LeetCode 253 treats them as NON-overlapping (end == start → OK)

Pitfall 3: Polling Multiple Rooms

Some people try:
  while (!heap.isEmpty() && heap.peek() <= meeting.start) {
      heap.poll();  // free ALL rooms that have ended
  }

→ This is WRONG for counting rooms
→ If you free 3 rooms and then add 1 → heap shrinks by 2
→ But those 3 rooms still EXIST — they're just free
→ The total rooms ALLOCATED is what matters, not current heap size

→ ACTUALLY WAIT — let me reconsider:
→ If we poll multiple and then push one:
     → Heap size decreases → final size ≠ rooms allocated
     → We lose track of peak

→ CORRECT approach: poll AT MOST ONE
     → This maintains the invariant: heap size = rooms in use
     → Reuse one room → poll one, push one → size unchanged
     → New room → push one → size grows

→ OR: poll multiple BUT track max(heap.size()) separately
→ The single-poll approach is cleaner and standard

Pitfall 4: Empty Input

→ intervals = [] → answer is 0
→ Must handle before sorting or heap operations
→ Arrays.sort on empty array is fine (no-op)
→ But the heap will remain empty → heap.size() = 0 ✓

Pitfall 5: Confusing with "Can Attend All Meetings"

Related but DIFFERENT problem: "Can a person attend all meetings?"
  → Answer: Yes if no two meetings overlap
  → Solution: sort by start, check if any meeting starts before previous ends
  → This is a BOOLEAN question → much simpler
  → Meeting Rooms II asks HOW MANY rooms → quantitative answer

📈 SCALABILITY CONSIDERATIONS

n = number of meetings

→ n = 100 → O(n²) brute force = 10,000 → fine
→ n = 10,000 → O(n²) = 100M → slow
→ n = 100,000 → O(n log n) = ~1.7M → fast
→ n = 1,000,000 → O(n log n) = ~20M → still fast

→ Heap approach: O(n log n) time, O(n) space
→ Event timeline: O(n log n) time, O(n) space
→ Both scale identically

→ For STREAMING data (meetings arriving one by one):
     → Heap approach adapts naturally
     → Sort is already done if meetings arrive in start-time order
     → Event timeline needs all data upfront → less suitable for streaming
