PROJECT 5: Maximum Sum Subarray of Size K

🎯 Sliding Window → Fixed-Size Window → Running Sum Maintenance

🧩 THE HYPOTHETICAL PROBLEM

→ You are given an array of integers (positive numbers)
→ You are given a window size K
→ Find the MAXIMUM SUM of any contiguous subarray of size K
→ Return that maximum sum

Example 1:
  Input:  arr = [2, 1, 5, 1, 3, 2], K = 3
  Output: 9
  Why:    Subarray [5, 1, 3] has sum = 9 → maximum among all size-3 subarrays

  All windows of size 3:
    [2, 1, 5] → sum = 8
    [1, 5, 1] → sum = 7
    [5, 1, 3] → sum = 9  ← maximum
    [1, 3, 2] → sum = 6

Example 2:
  Input:  arr = [2, 3, 4, 1, 5], K = 2
  Output: 7
  Why:    Subarray [3, 4] has sum = 7

Example 3:
  Input:  arr = [1, 1, 1, 1, 1], K = 3
  Output: 3
  Why:    All windows have the same sum

Example 4:
  Input:  arr = [10], K = 1
  Output: 10

🔍 HOW TO APPROACH IT

Step 1 — Understand What "Contiguous Subarray of Size K" Means

→ A CONTIGUOUS subarray is a slice of consecutive elements
→ Size K means exactly K elements in a row
→ In an array of size N, there are exactly N - K + 1 such subarrays:
     Starting at index 0: arr[0..K-1]
     Starting at index 1: arr[1..K]
     Starting at index 2: arr[2..K+1]
     ...
     Starting at index N-K: arr[N-K..N-1]

→ We need to find which of these N-K+1 subarrays has the LARGEST sum

Step 2 — Think About the Brute Force First

→ Naive approach: For each starting position, sum K elements
     for i = 0 to N-K:
         sum = 0
         for j = i to i+K-1:
             sum += arr[j]
         maxSum = max(maxSum, sum)
     
→ Outer loop: N-K+1 iterations
→ Inner loop: K iterations each
→ Total: O(N × K) time

→ If N = 1,000,000 and K = 500,000
→ Operations ≈ 5 × 10^11 → WAY too slow

→ The problem: we're RECALCULATING the entire sum from scratch each time
→ But adjacent windows share K-1 elements!
→ Window [2, 1, 5] and window [1, 5, 1] share [1, 5]
→ We only LOST 2 and GAINED 1
→ Why recompute the shared part?

Step 3 — The Key Insight: Incremental Update

→ When the window slides one position to the right:
     → One element LEAVES (the leftmost element of the old window)
     → One element ENTERS (the rightmost element of the new window)
     → The rest STAYS the same

→ New sum = Old sum - element_that_left + element_that_entered

→ This is an O(1) update instead of O(K) recalculation
→ Total: O(N) time instead of O(N × K)

Visual:
  Array: [2, 1, 5, 1, 3, 2]   K = 3

  Window 1: [2, 1, 5]       sum = 8
                slide →
  Window 2: [_, 1, 5, 1]    sum = 8 - 2 + 1 = 7
                              (lost 2, gained 1)
                slide →
  Window 3: [_, _, 5, 1, 3]  sum = 7 - 1 + 3 = 9
                              (lost 1, gained 3)
                slide →
  Window 4: [_, _, _, 1, 3, 2] sum = 9 - 5 + 2 = 6
                              (lost 5, gained 2)

→ Maximum = 9 ✓
→ We computed each window's sum in O(1) after the first

Step 4 — How This Connects to Previous Projects

→ Projects 1-4: Two pointers moved based on CONDITIONS (sum too big/small)
→ Project 5: Two pointers move in LOCKSTEP — the gap between them is FIXED at K
→ This is NOT conditional movement — it's SYNCHRONIZED movement

→ Think of it as a PHYSICAL WINDOW sliding across the array:
     → The window has a fixed width (K)
     → It moves one step at a time
     → At each position, we measure something (sum)
     → We track the best measurement seen

→ The "two pointers" here are:
     → LEFT edge of the window (the element being removed)
     → RIGHT edge of the window (the element being added)
     → They always stay exactly K positions apart

Step 5 — Visualize the Sliding

Array: [2, 1, 5, 1, 3, 2]   K = 3

Phase 1: BUILD the first window (indices 0 to K-1)
  Add arr[0]=2 → runningSum = 2
  Add arr[1]=1 → runningSum = 3
  Add arr[2]=5 → runningSum = 8
  First window complete → maxSum = 8

Phase 2: SLIDE the window (index K to N-1)
  i=3: Add arr[3]=1, Remove arr[0]=2 → runningSum = 8-2+1 = 7 → max(8,7) = 8
  i=4: Add arr[4]=3, Remove arr[1]=1 → runningSum = 7-1+3 = 9 → max(8,9) = 9
  i=5: Add arr[5]=2, Remove arr[2]=5 → runningSum = 9-5+2 = 6 → max(9,6) = 9

Result: 9 ✓

🛠️ HOW CAN WE SOLVE IT

The Algorithm — Line by Line Logic

1. Handle edge cases: if array is empty or K > array length → return 0
2. PHASE 1 — Build the first window:
     runningSum = 0
     for i = 0 to K-1:
         runningSum += arr[i]
     maxSum = runningSum
3. PHASE 2 — Slide the window:
     for i = K to N-1:
         runningSum += arr[i]          ← new element enters (right edge)
         runningSum -= arr[i - K]      ← old element leaves (left edge)
         maxSum = max(maxSum, runningSum)
4. Return maxSum

Why Two Phases?

→ Phase 1 builds the initial window — no element to remove yet
→ Phase 2 slides — each step adds one and removes one
→ The first window is SPECIAL because it has no predecessor
→ After that, every window is derived from the previous one

→ Alternative: Single-loop approach (build and slide in one loop)
     → Use a conditional: if (i >= K) then remove arr[i-K]
     → Slightly more compact but same logic
     → Two-phase is CLEARER for understanding

✅ WHAT SHOULD YOU DO

→ ALWAYS compute the first window separately (or use a conditional)
     → The first K elements form the baseline
     → Every subsequent window is an incremental update

→ ALWAYS track the running sum as a single variable
     → Don't store all window sums — just the current one
     → Update it by subtracting the leaving element and adding the entering element
     → This is what makes it O(1) per step

→ ALWAYS update maxSum at EVERY window position
     → The maximum could be at any position
     → Must compare after every slide

→ ALWAYS verify K ≤ N
     → If K > N → no valid window exists
     → Return 0 or -1 or throw exception based on problem requirements

→ ALWAYS think about what "entering" and "leaving" mean:
     → Entering element: arr[i] (current index in the slide loop)
     → Leaving element: arr[i - K] (the element K positions behind)
     → The gap is ALWAYS exactly K

🚫 WHAT SHOULD YOU NOT DO

→ DO NOT recalculate the sum from scratch for each window
     BAD:
       for (int i = 0; i <= n - k; i++) {
           int sum = 0;
           for (int j = i; j < i + k; j++) sum += arr[j];
           maxSum = Math.max(maxSum, sum);
       }
     → O(N × K) — wastes the overlapping structure

     GOOD:
       Maintain running sum → update in O(1) per window
     → O(N) — exploits the overlap

→ DO NOT store all window sums in an array
     → Wastes O(N) space
     → You only need the CURRENT sum and the MAXIMUM so far
     → Two variables are sufficient

→ DO NOT confuse the entering/leaving indices
     → Entering: index i (the new right edge)
     → Leaving: index i - K (the old left edge)
     → If you subtract the wrong element → sum becomes incorrect
     → Draw it out if confused

→ DO NOT forget to initialize maxSum from the first window
     → If maxSum starts at 0 and all elements are negative → wrong answer
     → If maxSum starts at Integer.MIN_VALUE → works but first window
       initialization is cleaner and safer

→ DO NOT apply this to VARIABLE-size window problems
     → Fixed window: size K is given and constant
     → Variable window: size changes based on a condition
     → Variable window needs different logic (Project 6)
     → Using fixed-window logic on variable-size problems → wrong answers

🔀 WHEN TO USE THIS PATTERN

✓ The problem mentions "subarray of size K" or "window of size K"
✓ K is FIXED — given as input, doesn't change
✓ You need to compute something (sum, average, max, min) for EVERY window
✓ The metric can be updated INCREMENTALLY:
     → Sum: new_sum = old_sum - leaving + entering
     → Count: new_count = old_count - (leaving matches?) + (entering matches?)
     → Max/Min: more complex — may need a deque (advanced topic)

✓ Common variants:
     → Maximum sum subarray of size K (this project)
     → Average of all subarrays of size K
     → Maximum element in each window of size K (needs deque)
     → Count of distinct elements in each window of size K (needs HashMap)
     → Any "sliding metric" over fixed-size contiguous segments

🚫 WHEN NOT TO USE THIS PATTERN

✗ Window size is NOT fixed — it varies based on a condition
     → "Smallest subarray with sum ≥ target" → size varies
     → Need VARIABLE-size sliding window (Project 6)

✗ Elements are not contiguous
     → "Maximum sum of K elements" (not necessarily consecutive)
     → This is a selection problem, not a window problem
     → May need sorting or heap (Projects 31-32)

✗ The metric CANNOT be updated incrementally
     → "Median of each window" → adding/removing doesn't give new median easily
     → Needs special data structures (two heaps — Project 34)
     → Basic sliding window isn't enough

✗ The problem asks for non-overlapping windows
     → "Split array into chunks of size K" → different structure
     → No sliding — just partition and process each chunk

✗ K equals N (window covers entire array)
     → Only one window → just compute the metric once
     → No sliding needed

⚙️ HOW IT ONLY WORKS — Internal Mechanics

The Overlap Principle

Two adjacent windows of size K share exactly K-1 elements:

Window at position i:   [arr[i], arr[i+1], ..., arr[i+K-1]]
Window at position i+1: [arr[i+1], arr[i+2], ..., arr[i+K]]

Shared elements: arr[i+1] through arr[i+K-1] → that's K-1 elements
Removed element: arr[i]     → the old left edge
Added element:   arr[i+K]   → the new right edge

Sum update:
  sum(window_i+1) = sum(window_i) - arr[i] + arr[i+K]

→ This works because addition is COMMUTATIVE and ASSOCIATIVE:
     sum(A, B, C, D) - A + E = sum(B, C, D, E)
→ We don't need to know the ORDER of elements inside the window
→ We only need to know what ENTERS and what LEAVES

The Window as a State Machine

The sliding window can be modeled as a STATE MACHINE:

State: {runningSum, leftEdge, rightEdge}

Transitions:
  SLIDE_RIGHT: 
    runningSum += arr[rightEdge + 1]    ← add new element
    runningSum -= arr[leftEdge]         ← remove old element
    leftEdge++                          ← advance left boundary
    rightEdge++                         ← advance right boundary

  OBSERVE:
    maxSum = max(maxSum, runningSum)     ← record the metric

→ Each transition is O(1)
→ We make exactly N - K transitions
→ Total: O(N) observations, O(1) per observation

Why This is Different from Two Pointers

TWO POINTERS (Projects 1-4):
→ Pointers move INDEPENDENTLY
→ Movement is CONDITIONAL (based on comparison with target)
→ The gap between pointers CHANGES dynamically
→ Purpose: SEARCH for something or TRANSFORM the array

SLIDING WINDOW (Project 5):
→ Pointers move in LOCKSTEP
→ Movement is UNCONDITIONAL (slide every step)
→ The gap between pointers is FIXED at K
→ Purpose: MEASURE something across all windows

→ Sliding Window is a SPECIALIZATION of Two Pointers
→ It adds the constraint: window size is constant
→ This constraint enables incremental computation
→ The constraint SIMPLIFIES the movement logic
→ But the underlying structure (two indices defining a range) is the same

The Running Sum as an Accumulator

→ The running sum is NOT recalculated — it's MAINTAINED
→ Think of it like a bank balance:
     → Starting balance: sum of first K elements
     → Each "day" (slide): one deposit (entering element), one withdrawal (leaving element)
     → Balance = previous balance + deposit - withdrawal
     → We track the HIGHEST balance ever seen

→ This "accumulator" pattern appears in many contexts:
     → Running average: sum / K
     → Running count: increment/decrement based on entering/leaving
     → Running product: multiply by entering, divide by leaving (careful with zeros!)
     → The STRUCTURE is always the same — only the UPDATE OPERATION changes

Time Complexity Breakdown

→ Phase 1 (build first window): K additions → O(K)
→ Phase 2 (slide N-K times): each slide is O(1) → O(N-K)
→ Total: O(K) + O(N-K) = O(N)

→ Compare with brute force: O(N × K)
→ Improvement factor: K
→ If K = N/2 → brute force is O(N²) → sliding window is O(N)
→ Quadratic to linear — massive improvement

Space Complexity Breakdown

→ Two variables: runningSum, maxSum
→ No auxiliary arrays or data structures
→ TOTAL: O(1) space

→ This is optimal — can't do better than O(1) extra space
→ The input array is read-only — not modified

⚠️ COMMON PITFALLS AND FAILURE MODES

Pitfall 1: Off-by-One on Window Boundaries

→ The first window covers indices [0, K-1]
→ The slide loop starts at index K (not K-1 or K+1)
→ The leaving element is at index i - K (not i - K + 1)

COMMON MISTAKE:
  for (int i = K - 1; i < n; i++) {    // WRONG start
      runningSum += arr[i];
      if (i >= K) runningSum -= arr[i - K];  // conditional removal
  }
  → This WORKS but is error-prone — mixing build and slide

CLEANER:
  // Phase 1: Build
  for (int i = 0; i < K; i++) runningSum += arr[i];
  maxSum = runningSum;
  
  // Phase 2: Slide
  for (int i = K; i < n; i++) {
      runningSum += arr[i];        // always add
      runningSum -= arr[i - K];    // always remove
      maxSum = Math.max(maxSum, runningSum);
  }
  → Two separate phases — clear and bug-free

Pitfall 2: Not Handling K > N

Input: arr = [1, 2], K = 5
→ No window of size 5 exists in an array of size 2
→ Must return 0 or handle gracefully

→ ALWAYS check: if (arr.length < K) → handle edge case
→ Without this → Phase 1 loop reads beyond array bounds → crash

Pitfall 3: Negative Numbers in the Array

Input: arr = [-1, -2, -3, -4], K = 2
→ All sums are negative
→ Maximum sum = -3 (window [-1, -2])

→ If maxSum is initialized to 0 → returns 0 → WRONG
→ Must initialize maxSum from the FIRST WINDOW, not from 0
→ maxSum = runningSum (after Phase 1) → handles negatives correctly

Pitfall 4: Confusing Fixed vs Variable Window

→ Fixed window (this project): K is given → window size never changes
→ Variable window (Project 6): window size adapts to meet a condition

→ If you use fixed-window logic on a variable-window problem:
     → You'll check only windows of one size
     → Miss smaller or larger windows that satisfy the condition
     → Wrong answer

→ IDENTIFY which type the problem is BEFORE coding:
     → "Subarray of size K" → FIXED
     → "Smallest subarray with sum ≥ X" → VARIABLE
     → "Longest substring with at most K distinct" → VARIABLE

Pitfall 5: Integer Overflow on Running Sum

→ If array has large values and K is large:
     → arr = [1000000000, 1000000000, ...] with K = 100
     → sum = 10^11 → overflows int

→ Use long for runningSum and maxSum when values can be large
→ Or check constraints: if values ≤ 10^4 and K ≤ 10^5 → max sum ≤ 10^9 → int is safe

📈 SCALABILITY CONSIDERATIONS

→ N = 1,000, K = 100:
     → Brute force: 1,000 × 100 = 100,000 ops → fast
     → Sliding window: 1,000 ops → fast
     → Both work fine at this scale

→ N = 1,000,000, K = 500,000:
     → Brute force: 10^6 × 5×10^5 = 5×10^11 ops → IMPOSSIBLY slow
     → Sliding window: 10^6 ops → instant
     → Sliding window is the ONLY viable approach

→ N = 100,000,000, K = 50,000,000:
     → Sliding window: 10^8 ops → ~1 second → manageable
     → Brute force: 5×10^15 ops → would take YEARS

→ The key scalability advantage:
     → Brute force: O(N × K) → depends on BOTH N and K
     → Sliding window: O(N) → depends ONLY on N, independent of K
     → K could be 1 or K could be N/2 — same performance
     → This K-independence is the power of incremental computation

→ REAL-WORLD applications:
     → Moving average in stock prices (window = 50 days, data = decades)
     → Network traffic monitoring (window = 5 minutes, data = continuous)
     → Sensor data smoothing (window = 100 samples, data = millions)
     → Log analysis (window = 1 hour, events = billions)

🔍 INTERPRETABILITY CONSIDERATIONS

→ The physical metaphor is perfect:
     → Imagine a MAGNIFYING GLASS of fixed width K
     → Slide it across a row of numbers
     → At each position, add up what you see through the glass
     → Track the highest sum encountered

→ The incremental update is intuitive:
     → When the glass slides right by one:
          → The leftmost number EXITS the view
          → A new rightmost number ENTERS the view
          → Everything in between STAYS
     → New total = old total - exited + entered

→ Easy to debug:
     → Print the window contents, running sum, and max at each step
     → Verify manually for small inputs
     → Any sum discrepancy points to wrong entering/leaving index

→ Interview explanation:
     → "I maintain a running sum for a window of size K"
     → "Each slide: subtract the element leaving, add the element entering"
     → "Track the maximum sum across all windows"
     → "O(N) time, O(1) space"
     → Concise, clear, optimal

🧠 KEY TAKEAWAY

WHAT WE LEARNED:
→ Sliding Window is a SPECIALIZATION of Two Pointers
→ The window has FIXED size K — both edges move in lockstep
→ The key optimization: INCREMENTAL UPDATE instead of full recalculation
→ New sum = old sum - leaving element + entering element → O(1) per slide
→ Total: O(N) time, O(1) space — independent of K

THE SLIDING WINDOW TEMPLATE (Fixed Size):
  // Phase 1: Build first window
  for (int i = 0; i < k; i++) {
      accumulator += arr[i];  // or whatever metric
  }
  bestMetric = accumulator;

  // Phase 2: Slide
  for (int i = k; i < n; i++) {
      accumulator += arr[i];        // entering
      accumulator -= arr[i - k];    // leaving
      bestMetric = optimal(bestMetric, accumulator);
  }

→ Change the METRIC and you solve different problems:
     → Sum → maximum/minimum sum subarray
     → Count → windows with specific properties
     → Average → moving average calculation
     → The TEMPLATE stays the same

PATTERN EVOLUTION:
  Projects 1-4: Two Pointers → independent movement → search/transform
  Project 5:    Sliding Window → synchronized movement → measure across windows

NEW CONCEPT: INCREMENTAL COMPUTATION
→ Don't recompute from scratch when data changes slightly
→ Update the old result based on what changed
→ This principle appears EVERYWHERE in computer science:
     → Database incremental indexes
     → Streaming analytics
     → Game state updates
     → Machine learning gradient updates
