PROJECT 34: Longest Repeating Character Replacement

🎯 Sliding Window → Max-Frequency Trick → Replacement Budget

🧩 THE HYPOTHETICAL PROBLEM

→ You are given a string s containing only UPPERCASE English letters
→ You are given an integer k
→ You can replace AT MOST k characters in the string with any other character
→ Find the LENGTH of the LONGEST substring containing the same letter
   after performing at most k replacements
→ You want the longest window where all characters CAN BE made identical

Example 1:
  s = "AABABBA", k = 1
  Output: 4
  → Replace the one 'B' at index 3 → "AAAABBA" → "AAAA" has length 4
  → OR: Replace 'A' at index 4 → "AABBBBA" → "BBBB" has length 4

Example 2:
  s = "ABAB", k = 2
  Output: 4
  → Replace two 'A's → "BBBB" → length 4
  → OR replace two 'B's → "AAAA" → length 4

Example 3:
  s = "AAAA", k = 2
  Output: 4
  → Already all same → no replacements needed → length 4

Example 4:
  s = "ABCDE", k = 2
  Output: 3
  → Replace 'B','C' → "AAADE" → "AAA" length 3
  → OR replace 'A','C' → "BBBDE" → "BBB" length 3

🔍 HOW TO APPROACH IT

Step 1 — Understand What "Replacement" Really Means

→ We're NOT actually modifying the string
→ We're finding a WINDOW where:
     → If we COULD replace at most k characters
     → All characters in the window would become the same

→ REFRAME the problem:
     → Find the longest window where
     → (window size) - (count of most frequent character in window) ≤ k

→ WHY? Think about it:
     → Window "AABBA" has 5 characters
     → Most frequent = 'A' appearing 3 times
     → Characters that AREN'T the most frequent = 5 - 3 = 2
     → Those 2 characters need to be REPLACED to make everything 'A'
     → If 2 ≤ k → we can do it → window is VALID

→ This is the REPLACEMENT BUDGET concept:
     → Budget = k (maximum replacements allowed)
     → Cost = windowSize - maxFrequency (characters that need changing)
     → Valid window: cost ≤ budget
     → In other words: windowSize - maxFreq ≤ k

Step 2 — Why This is Different from All Previous Sliding Windows

Project 5 (Max Sum Size K):
  → Fixed window → track sum
  → Validity: window size == K (always valid, just slide)

Project 6 (Smallest Subarray ≥ Target):
  → Variable window → track running sum
  → Validity: sum ≥ target (external threshold)

Project 12 (Longest K Distinct):
  → Variable window → track distinct character count
  → Validity: distinct count ≤ K

Project 33 (Min Window Substring):
  → Variable window → track frequency match against target
  → Validity: all target characters present (match counter)

THIS PROJECT:
  → Variable window → track MAX FREQUENCY inside window
  → Validity: windowSize - maxFreq ≤ k
  → No external target to match against
  → The validity depends on an INTERNAL STATISTIC of the window
  → Specifically: the frequency of the MOST COMMON character
  → This is a completely new type of validity condition

Step 3 — The Key Insight: You Don't Need to Know WHICH Character to Keep

→ Naive thinking: "I should try keeping 'A', then try keeping 'B', ..."
     → 26 possible characters × O(n) per character = O(26n)
     → Works but misses the elegant insight

→ Clever thinking: "I don't care WHICH character dominates"
     → I only care about HOW MANY of the most frequent character exist
     → If maxFreq + k ≥ windowSize → I can make the whole window uniform
     → The identity of that character is irrelevant for determining validity

→ Example: window = "AABBC", k = 2
     → Freq: A=2, B=2, C=1 → maxFreq = 2
     → Cost = 5 - 2 = 3 → need 3 replacements
     → 3 > k=2 → INVALID (can't make all same with only 2 replacements)

→ Example: window = "AABB", k = 2
     → Freq: A=2, B=2 → maxFreq = 2
     → Cost = 4 - 2 = 2 → need 2 replacements
     → 2 ≤ k=2 → VALID (replace both B's with A, or both A's with B)

Step 4 — The Critical Optimization: maxFreq Never Needs to Decrease

→ This is the MOST SUBTLE and BRILLIANT part of this problem
→ When we shrink the window, maxFreq MIGHT decrease
→ But we DON'T NEED to update it downward

→ WHY? Because we want the LONGEST valid window
→ A valid window has: windowSize ≤ maxFreq + k
→ The longest valid window has the LARGEST windowSize
→ windowSize can only grow if maxFreq grows
→ If maxFreq doesn't grow → windowSize can't grow → we can't beat our current best
→ So an "outdated" (too-high) maxFreq just means:
     → The window won't pass the validity check
     → We won't record a new best
     → No harm done — we just slide forward

→ This means:
     → We ONLY update maxFreq when a character's frequency INCREASES
     → We NEVER scan all frequencies to recompute the true max
     → maxFreq is monotonically non-decreasing throughout the algorithm
     → This keeps the algorithm O(n) instead of O(26n)

→ Wait — does this produce WRONG results?
     → NO — here's why:
     → If the true maxFreq decreased → the window needs MORE replacements
     → Our stored maxFreq is ≥ true maxFreq → we think window needs FEWER
     → So we think the window MIGHT be valid when it's NOT
     → But we don't record a new best unless windowSize ACTUALLY increases
     → And windowSize only increases when maxFreq truly increases
     → So false "validity" from stale maxFreq never produces a wrong answer
     → It just means we don't shrink as aggressively → window slides forward

Step 5 — Two Approaches: Shrinking vs Non-Shrinking

APPROACH A — SHRINK when invalid (traditional):
  → Expand right
  → If windowSize - maxFreq > k → INVALID → shrink from left
  → While shrinking: update frequencies, recompute maxFreq
  → Track longest valid window

  → Issue: recomputing maxFreq after shrink is O(26) per shrink
  → Total: O(26n) → still acceptable but not as clean

APPROACH B — NEVER shrink more than 1 step (optimized):
  → Expand right
  → If windowSize - maxFreq > k → INVALID → shrink left by EXACTLY 1
  → Don't recompute maxFreq (let it stay stale)
  → Window size stays same or grows — never shrinks significantly
  → Track longest valid window via maxFreq + k

  → Insight: the window acts like a "high-water mark"
  → Once it reaches size L → it never shrinks below L
  → It only records a new best when it grows past L
  → This is O(n) with no inner loop at all

We will implement BOTH approaches

Wait — the answer is actually right - left at the widest valid point
Let's track differently:

The window NEVER shrinks below its maximum valid size
The answer = right - left + 1 at the end, 
  OR the maximum (right - left + 1) seen when valid

With approach B: answer = right - left + 1 when loop ends
  → right=6, left=3 → 6-3+1 = 4 ✓

🛠️ HOW CAN WE SOLVE IT

Approach A — Traditional Sliding Window (Shrink While Invalid)

1. Initialize frequency array int[26] for A-Z
2. Initialize left = 0, maxFreq = 0, result = 0
3. For right = 0 to s.length()-1:
     a. Add s[right] to frequency → freq[right_char]++
     b. Update maxFreq = max(maxFreq, freq[right_char])
     c. WHILE windowSize - maxFreq > k:
          → Window is INVALID → shrink
          → freq[s[left]]--
          → left++
          → Recompute maxFreq by scanning freq array (O(26))
     d. result = max(result, right - left + 1)
4. Return result

Approach B — Optimized (Never Truly Shrink — Single If)

1. Initialize frequency array int[26] for A-Z
2. Initialize left = 0, maxFreq = 0
3. For right = 0 to s.length()-1:
     a. Add s[right] to frequency → freq[right_char]++
     b. Update maxFreq = max(maxFreq, freq[right_char])
     c. IF (right - left + 1) - maxFreq > k:
          → Window is INVALID by exactly 1 → shrink by exactly 1
          → freq[s[left]]--
          → left++
          → Do NOT recompute maxFreq
     (No while loop — just single if)
4. Return right - left + 1 (which equals s.length() - left)
   → OR equivalently: return s.length() - left

Why Approach B Works — Detailed Proof

CLAIM: The window size (right - left + 1) only INCREASES or STAYS SAME

PROOF:
  → Each step: right increases by 1 → window grows by 1
  → If invalid: left increases by 1 → window shrinks by 1
  → Net change: +1 -1 = 0 (window stays same size)
  → If valid: left stays → window grows by 1

  → So window size is MONOTONICALLY NON-DECREASING

CLAIM: maxFreq is monotonically non-decreasing

PROOF:
  → We only do: maxFreq = max(maxFreq, freq[rightChar])
  → max(old, new) ≥ old → maxFreq never decreases

CLAIM: The answer equals the final window size

PROOF:
  → Window size only grows when a TRULY valid window is found
  → A valid window requires windowSize - trueMaxFreq ≤ k
  → Since maxFreq ≥ trueMaxFreq → our check is more LENIENT
  → When maxFreq == trueMaxFreq → check is exact → window grows only for valid windows
  → When maxFreq > trueMaxFreq → check is lenient → window MIGHT grow for invalid windows?
  
  → NO! Because:
       → maxFreq grew because some character's frequency increased
       → That character's frequency IS maxFreq → it's the true max
       → So when maxFreq updates → it's always the TRUE max at that moment
       → The only time maxFreq is "stale" is AFTER a shrink removes a character
       → But after a shrink → window didn't grow → no false best recorded

  → Therefore: the maximum window size seen = length of longest valid window ✓

✅ WHAT SHOULD YOU DO

→ ALWAYS use a frequency array int[26] for uppercase letters
     → Index: c - 'A' maps 'A'→0, 'B'→1, ..., 'Z'→25
     → Faster than HashMap for fixed character set

→ ALWAYS update maxFreq AFTER adding the new character
     → The new character might create a new maximum
     → Check immediately after incrementing its frequency

→ ALWAYS compute window validity as: (right - left + 1) - maxFreq > k
     → windowSize - maxFreq = number of characters to replace
     → If this exceeds k → window is invalid → must shrink

→ PREFER Approach B (single if) for interviews
     → Cleaner code → no inner while loop
     → O(n) guaranteed without the O(26) factor
     → Easier to write correctly under pressure

→ ALWAYS return (s.length() - left) for Approach B
     → After the loop: right = s.length()-1, window = [left, right]
     → Size = s.length() - left
     → This IS the answer because window size is monotonically non-decreasing

→ ALWAYS handle edge cases:
     → k ≥ s.length() → entire string is the answer → return s.length()
     → k = 0 → find longest run of same character → no replacements
     → s has only one distinct character → return s.length()
     → s is empty → return 0

🚫 WHAT SHOULD YOU NOT DO

→ DO NOT try to recompute maxFreq from scratch in Approach B
     → The whole point is that maxFreq stays stale intentionally
     → Recomputing adds O(26) per step → O(26n) total
     → Not wrong but defeats the optimization

→ DO NOT use while loop in Approach B
     → While loop means window can shrink MULTIPLE steps
     → Approach B shrinks AT MOST 1 step per expansion
     → If you use while → it works but it's Approach A, not B

→ DO NOT confuse "max frequency" with "frequency of specific character"
     → We track the GLOBAL maximum across ALL characters in the window
     → Not the frequency of one chosen character
     → The most frequent character can CHANGE as the window slides

→ DO NOT forget to decrement frequency when removing left character
     → Common bug in sliding window problems
     → If you only increment and never decrement → frequencies grow unbounded
     → Window validity check breaks

→ DO NOT try to track WHICH character has max frequency
     → You don't need it for the validity check
     → windowSize - maxFreq ≤ k works regardless of which character is dominant
     → Tracking the identity adds unnecessary complexity

→ DO NOT apply this to problems where the "budget" concept doesn't apply
     → This works because "number of replacements needed" is a clean metric
     → If the problem has a different cost model → different approach needed

🔀 WHEN TO USE THIS PATTERN

✓ "Longest substring with at most K changes/replacements"
     → This is the DIRECT application

✓ "Longest substring where all characters can be made identical"
     → Same problem, different wording

✓ "Maximum length window where cost ≤ budget"
     → Cost = windowSize - maxFreq
     → Budget = k
     → General "budget-based validity" pattern

✓ Problems where the validity condition involves:
     → windowSize minus some internal window statistic
     → A threshold on the number of "bad" elements in the window

✓ Similar problems:
     → Max consecutive ones with at most K flips (binary version)
     → Longest substring with at most K distinct characters (Project 12 variant)
     → Minimum swaps to group all 1s together
     → Maximum fruits in contiguous segment with at most K changes

✓ The "max-frequency trick" generalizes to:
     → Any problem where keeping the MAJORITY element fixed
     → And "fixing" the minority elements has bounded cost

🚫 WHEN NOT TO USE THIS PATTERN

✗ Replacements have DIFFERENT costs per character
     → Replacing 'A' with 'B' costs 1, but 'A' with 'C' costs 3
     → The uniform cost assumption breaks
     → Need more complex DP or greedy approaches

✗ Replacements must follow specific RULES
     → "Can only replace vowels" or "can only replace adjacent duplicates"
     → The "any character to any character" freedom is essential

✗ You need the SHORTEST window (not longest)
     → This technique finds the LONGEST valid window
     → For shortest: use the shrink-while-valid approach from Project 33

✗ The condition is NOT based on frequency
     → "Longest substring with balanced parentheses"
     → "Longest palindromic substring"
     → These have structural conditions, not frequency-based

✗ k = 0 AND you need more than "longest run"
     → k=0 makes the problem "find longest repeating character run"
     → Simple linear scan suffices → no sliding window needed

⚙️ HOW IT ONLY WORKS — Internal Mechanics

The Budget Model

CORE EQUATION:
  replacementsNeeded = windowSize - maxFrequencyInWindow
  isValid = (replacementsNeeded ≤ k)

INTUITION:
  → In any window, the OPTIMAL strategy is to keep the most frequent character
  → And replace ALL other characters to match it
  → The number of "other" characters = windowSize - maxFreq
  → If this number ≤ k → we have enough budget → valid

EXAMPLE:
  Window "AABBC", k = 2
  → Frequencies: A=2, B=2, C=1
  → maxFreq = 2 (either A or B)
  → replacementsNeeded = 5 - 2 = 3
  → 3 > 2 → INVALID

  Window "AABB", k = 2
  → maxFreq = 2
  → replacementsNeeded = 4 - 2 = 2
  → 2 ≤ 2 → VALID (replace both B's with A, or both A's with B)

The Stale maxFreq Proof (Approach B)

OBSERVATION: maxFreq can become "stale" (higher than true maximum)

WHEN does this happen?
  → When we remove the character that HAD the max frequency
  → Example: window "AAB" → maxFreq=2 (A has freq 2)
  → Remove 'A' → window "AB" → true maxFreq=1 → but stored maxFreq=2

DOES stale maxFreq cause wrong answers?

CASE 1: Window with stale maxFreq passes validity check
  → windowSize - staleMaxFreq ≤ k
  → But windowSize - trueMaxFreq > k (truly invalid)
  → We'd think it's valid when it's not
  
  → BUT: window size is at most staleMaxFreq + k
  → staleMaxFreq was set by a PREVIOUS valid window
  → That previous window had size = staleMaxFreq + k (or less)
  → Current window has SAME size (approach B never shrinks)
  → So current window size ≤ previous best
  → We DON'T record a new best → NO WRONG ANSWER

CASE 2: Window with stale maxFreq fails validity check
  → windowSize - staleMaxFreq > k
  → We shrink → correct behavior

CASE 3: maxFreq updates to TRUE new max
  → A character's frequency exceeds staleMaxFreq
  → maxFreq becomes accurate again
  → Now window CAN grow → potentially new best → correctly recorded

CONCLUSION:
  → Stale maxFreq never causes a FALSE POSITIVE for "new best"
  → It only causes the window to "coast" at its current size
  → When maxFreq truly increases → window can grow → real progress

⚠️ COMMON PITFALLS AND FAILURE MODES

Pitfall 1: Recomputing maxFreq Incorrectly in Approach A

WRONG:
  while (windowSize - maxFreq > k) {
      freq[s.charAt(left) - 'A']--;
      left++;
      // Forgot to recompute maxFreq!
  }

→ After removing a character, maxFreq might have decreased
→ But we're still using the old maxFreq
→ This makes the while loop shrink MORE than necessary
→ Result: valid window size is SMALLER than optimal → WRONG ANSWER

CORRECT (Approach A):
  while (windowSize - maxFreq > k) {
      freq[s.charAt(left) - 'A']--;
      left++;
      // Recompute maxFreq
      maxFreq = 0;
      for (int f : freq) {
          maxFreq = Math.max(maxFreq, f);
      }
  }

→ OR just use Approach B which avoids this entirely

Pitfall 2: Using maxFreq = max(maxFreq, freq[c]) BEFORE Incrementing

WRONG:
  maxFreq = Math.max(maxFreq, freq[rightChar - 'A']);  // Still old freq
  freq[rightChar - 'A']++;                              // Now incremented

→ You're comparing with the PREVIOUS frequency, not the updated one
→ maxFreq is always 1 behind → answer is wrong

CORRECT:
  freq[rightChar - 'A']++;                              // Increment FIRST
  maxFreq = Math.max(maxFreq, freq[rightChar - 'A']);  // THEN update max

Pitfall 3: Off-by-One in Window Size Calculation

WRONG:
  int windowSize = right - left;  // Missing +1

CORRECT:
  int windowSize = right - left + 1;

→ Window [2, 5] has elements at indices 2, 3, 4, 5 → size = 4 = 5-2+1
→ Not 5-2 = 3

Pitfall 4: Thinking Approach B Returns maxFreq + k

WRONG THINKING:
  → "The answer is maxFreq + k"
  → If maxFreq = 5 and k = 3 → answer = 8?

ISSUE:
  → maxFreq + k is the THEORETICAL maximum window size
  → But the actual answer is bounded by string length
  → And by the actual positions where characters appear

CORRECT:
  → Answer = s.length() - left (for Approach B)
  → Or answer = max(right - left + 1) tracked during iteration (for Approach A)
  → Both account for actual string constraints

Pitfall 5: Lowercase vs Uppercase

→ Problem says "uppercase English letters"
→ Use freq[c - 'A'] with array size 26
→ If problem says "lowercase" → use freq[c - 'a']
→ If problem says "any letter" → use freq[128] or freq[c - 'A'] with size 58
     → 'A'=65 to 'z'=122 → need array of size 58
     → Or just use 128 to cover all ASCII
→ READ THE PROBLEM for character constraints

📈 SCALABILITY CONSIDERATIONS

Time Complexity:
  → Approach A: O(26 × n) = O(n) — inner loop recomputes maxFreq over 26 chars
  → Approach B: O(n) strict — no inner loop, no recomputation
  → Both are effectively O(n) but B has smaller constant

Space Complexity:
  → O(26) = O(1) for the frequency array
  → Constant space regardless of input size

Practical performance:
  → n = 10^5: instant for both approaches
  → n = 10^7: Approach A ≈ 50ms, Approach B ≈ 20ms
  → n = 10^8: Approach A ≈ 500ms (26× factor matters), Approach B ≈ 200ms
  → For competitive programming: Approach B is safer for tight time limits

Comparison with brute force:
  → Brute force: O(n² × 26) → for n=10^5: ~2.6 × 10^11 → TIMEOUT
  → Sliding window: O(n) → for n=10^5: ~10^5 → instant

🔍 INTERPRETABILITY CONSIDERATIONS

→ The "budget" metaphor is very intuitive:
     → You have a BUDGET of k replacements
     → In any window, the COST is: characters that aren't the majority
     → If cost ≤ budget → you can afford to make the window uniform
     → Expand to increase potential → shrink when over-budget

→ The stale maxFreq argument requires practice to explain:
     → "maxFreq represents the best we've seen so far"
     → "If it decreases, we can't do better anyway"
     → "So we let the window coast at its current size"
     → "Only when maxFreq truly increases can we find a longer answer"

🧠 KEY TAKEAWAY

WHAT WE LEARNED:
→ The "replacement budget" model: cost = windowSize - maxFreq, budget = k
→ Validity is based on an INTERNAL window statistic (maxFreq)
→ Not matching against an external target (unlike Project 33)
→ maxFreq can stay stale (never decrease) without affecting correctness
→ This enables a true O(n) algorithm with no inner loop

THE MAX-FREQUENCY TRICK:
→ In any window, the optimal strategy is to KEEP the most frequent character
→ And REPLACE everything else
→ The number of replacements = windowSize - maxFreq
→ If replacements ≤ k → window is valid
→ You don't need to know WHICH character to keep — just how many of the most common

THE STALE maxFreq INSIGHT:
→ maxFreq only needs to INCREASE for us to find a LONGER valid window
→ If maxFreq decreases → we can't beat our current best anyway
→ So we let maxFreq stay high (stale) → window "coasts" at current size
→ Only when maxFreq truly increases → window can grow → new best possible
