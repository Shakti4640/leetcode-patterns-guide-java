🧩 THE HYPOTHETICAL PROBLEM

→ You are given a list of WORDS sorted in lexicographic order
  according to an ALIEN language's alphabet
→ Derive the ORDER of characters in the alien alphabet
→ Return a string of characters in the correct alien order
→ If the order is invalid (contradiction) → return ""
→ If multiple valid orderings exist → return ANY one

Example 1:
  Input: ["wrt", "wrf", "er", "ett", "rftt"]
  
  From the sorted order, we can deduce:
  → "wrt" before "wrf" → comparing char by char → t comes before f
  → "wrf" before "er"  → w comes before e
  → "er" before "ett"  → r comes before t
  → "ett" before "rftt"→ e comes before r
  
  Derived edges: t→f, w→e, r→t, e→r
  
  Topological sort: w → e → r → t → f
  
  Output: "wertf"

Example 2:
  Input: ["z", "x"]
  
  → "z" before "x" → z comes before x
  
  Output: "zx"

Example 3:
  Input: ["z", "x", "z"]
  
  → "z" before "x" → z comes before x
  → "x" before "z" → x comes before z
  → CONTRADICTION: z before x AND x before z → CYCLE
  
  Output: "" (invalid)

Example 4:
  Input: ["abc", "ab"]
  
  → "abc" comes before "ab"?
  → But "abc" is LONGER than "ab" and shares the same prefix
  → In any valid lexicographic order, "ab" should come before "abc"
  → This input is INVALID
  
  Output: "" (invalid)

🔍 HOW TO APPROACH IT

Step 1 — Understand What "Sorted Word List" Tells Us

→ The words are sorted in alien lexicographic order
→ Just like English dictionary: "apple" < "banana" < "cat"
→ Comparing two ADJACENT words reveals ONE ordering rule

How to extract a rule from two adjacent words:
→ Compare character by character from left to right
→ Find the FIRST position where they differ
→ The character in word1 comes BEFORE the character in word2
→ Characters after the first difference tell us NOTHING

Example: "wrt" vs "wrf"
  w == w → same, continue
  r == r → same, continue  
  t != f → FIRST DIFFERENCE → t comes before f → edge: t → f
  
Example: "er" vs "ett"
  e == e → same, continue
  r != t → FIRST DIFFERENCE → r comes before t → edge: r → t
  Characters after (tt) are IRRELEVANT

Step 2 — Map This to a Graph Problem

→ Each unique CHARACTER is a NODE
→ Each ordering rule is a DIRECTED EDGE
     → "a comes before b" → edge from a to b
→ Finding the full character order = TOPOLOGICAL SORT on this graph

THIS IS EXACTLY PROJECT 47 (Course Schedule):
→ Project 47: courses = nodes, prerequisites = edges
→ Project 56: characters = nodes, ordering rules = edges
→ Same algorithm (Kahn's BFS) — different graph CONSTRUCTION

THE KEY DIFFERENCE:
→ Project 47: graph was GIVEN to you (adjacency list of prerequisites)
→ Project 56: graph must be BUILT by YOU from comparing adjacent words
→ The hard part is not the topological sort — it's the graph construction

Step 3 — The Three Phases

Phase 1: EXTRACT all unique characters
→ Scan all words → collect every character → these are our nodes

Phase 2: BUILD the graph by comparing adjacent word pairs
→ For each pair (words[i], words[i+1]):
     → Find first differing character
     → Add directed edge: words[i][j] → words[i+1][j]
     → ALSO detect invalid case: longer word before shorter prefix

Phase 3: TOPOLOGICAL SORT using Kahn's BFS (from Project 47)
→ Compute in-degrees
→ Start with nodes having in-degree 0
→ Process BFS → append to result
→ If result length ≠ number of unique characters → CYCLE → return ""

Step 4 — Why Not Just Compare ALL Pairs?

→ We only need ADJACENT pairs in the sorted list
→ Why? Because if words are sorted: w1 < w2 < w3
→ Then w1 < w3 is already implied by w1 < w2 and w2 < w3
→ Comparing non-adjacent pairs gives REDUNDANT edges
→ Redundant edges don't break correctness but waste time
→ Adjacent pairs give us ALL the MINIMAL constraints needed

🛠️ HOW CAN WE SOLVE IT

Full Algorithm — Step by Step

STEP 1: Collect all unique characters
→ Iterate through every character in every word
→ Add to a Set<Character>
→ These become our graph nodes

STEP 2: Initialize graph structures
→ adjacencyList: Map<Character, List<Character>>
→ inDegree: Map<Character, Integer>
→ Initialize every character with empty adjacency list and in-degree 0

STEP 3: Compare adjacent words to build edges
→ For i = 0 to words.length - 2:
     word1 = words[i], word2 = words[i+1]
     
     Compare character by character:
     → Find first index j where word1[j] ≠ word2[j]
     → Add edge: word1[j] → word2[j]
     → Increment inDegree[word2[j]]
     → STOP comparing this pair (only first difference matters)
     
     Special case:
     → If all compared characters are same AND word1 is LONGER
     → Example: "abc" before "ab" → INVALID
     → Return ""

STEP 4: Kahn's BFS topological sort (exactly as Project 47)
→ Queue all characters with in-degree 0
→ While queue not empty:
     → Poll character c
     → Append c to result
     → For each neighbor of c:
          → Decrement neighbor's in-degree
          → If in-degree becomes 0 → add to queue
     
→ If result.length() == number of unique characters → return result
→ Otherwise → cycle exists → return ""

✅ WHAT SHOULD YOU DO

→ ALWAYS compare only ADJACENT word pairs
     → words[i] vs words[i+1] for all i
     → Non-adjacent pairs add redundant edges
     → Adjacent pairs capture all minimal ordering constraints

→ ALWAYS stop at the FIRST differing character
     → Only the first difference gives a valid ordering rule
     → Characters after the first difference are compared under
        a DIFFERENT prefix context → no valid deduction possible

→ ALWAYS check the "longer prefix" invalid case
     → If word1 is a prefix of word2 AND word1 is longer
     → Example: "abc" before "ab"
     → This violates lexicographic ordering → return ""
     → Easy to miss → interviewers specifically test this

→ ALWAYS include ALL unique characters in the result
     → Some characters may not appear in any edge
     → They still need to be in the output
     → They'll have in-degree 0 → processed first in BFS
     → If multiple characters have in-degree 0 → any order is valid

→ ALWAYS use Map-based structures (not array-based)
     → Characters may be any lowercase letter (a-z)
     → Not all 26 letters may be present
     → Map handles sparse character sets cleanly
     → (Array of size 26 also works but needs careful indexing)

→ ALWAYS verify result length equals number of unique characters
     → This is the cycle detection check (same as Project 47)
     → If cycle exists → some characters never reach in-degree 0
     → They never enter the queue → result is shorter → invalid

🚫 WHAT SHOULD YOU NOT DO

→ DO NOT compare all pairs of words
     → O(n²) pairs instead of O(n) adjacent pairs
     → Adds redundant edges → doesn't improve correctness
     → Wastes time and complicates duplicate edge handling

→ DO NOT extract multiple rules from one word pair
     → Only the FIRST differing character gives a rule
     → Example: "abc" vs "xyz" → only a→x is valid
     → b→y and c→z are NOT valid deductions
     → WHY? Because "a before x" is the reason "abc" < "xyz"
     → The comparison of b vs y only matters IF a==x

→ DO NOT forget characters that appear in words but not in edges
     → If character 'q' appears in words but never in a first-difference
     → It still exists in the alien alphabet
     → It must appear in the output
     → It will have in-degree 0 → placed freely

→ DO NOT add duplicate edges
     → Two different word pairs might give the same rule
     → Example: "ab" vs "ac" → b→c, and "xb" vs "xc" → b→c (same edge)
     → Adding duplicate edges inflates in-degree incorrectly
     → Use a Set to track existing edges OR check before adding

→ DO NOT use DFS-based topological sort without careful cycle detection
     → DFS topo sort works but cycle detection requires coloring (white/gray/black)
     → BFS (Kahn's) detects cycles naturally via result length check
     → For this problem, Kahn's BFS is cleaner (same as Project 47)

→ DO NOT assume all 26 letters are present
     → Only characters that appear in the input words exist
     → Building a graph with 26 nodes when only 5 are used → wastes space
     → And you'd need to figure out which characters to include in output

🚫 WHEN NOT TO USE THIS PATTERN

✗ The ordering is BIDIRECTIONAL (A relates to B AND B relates to A)
     → Topological sort requires DIRECTED ACYCLIC graph
     → Bidirectional relationships → use Union Find (Project 57)

✗ You need the SHORTEST/LONGEST path, not just an ordering
     → Topological sort gives A valid ordering
     → Not the optimal path → use DP on DAG or Dijkstra

✗ The graph is UNDIRECTED
     → Topological sort is meaningless on undirected graphs
     → Use BFS/DFS for traversal, Union Find for components

✗ You need ALL possible orderings
     → Topological sort (Kahn's) gives ONE valid ordering
     → Finding ALL valid orderings is exponential
     → Different problem entirely

✗ The constraints don't form a DAG (guaranteed to have cycles)
     → If cycles are expected and valid → different problem structure
     → Topological sort explicitly REJECTS cycles

⚙️ HOW IT ONLY WORKS — Internal Mechanics

Why Adjacent Word Comparison is Sufficient

Given sorted words: w1 < w2 < w3 < ... < wn

Comparing w1 vs w2 gives edge: a → b
Comparing w2 vs w3 gives edge: c → d
...

Do we need to compare w1 vs w3?
→ We know w1 < w2 and w2 < w3
→ So w1 < w3 is already implied by TRANSITIVITY
→ Any edge from w1 vs w3 is either:
     → The SAME as an edge we already have (redundant)
     → IMPLIED by the chain of edges (redundant)
→ Adjacent comparisons capture ALL necessary constraints

Mathematical proof:
→ The sorted list defines a total ordering on words
→ Adjacent differences give the GENERATORS of this ordering
→ All other relationships are compositions of generators
→ In graph terms: transitive closure = same reachability

Why Only the First Difference Matters

Compare "abc" vs "axz"

Position 0: a == a → same → continue
Position 1: b ≠ x → FIRST DIFFERENCE → b comes before x

Can we also say c comes before z?
→ NO! Here's why:
→ The reason "abc" < "axz" is because b < x (in alien order)
→ The characters at position 2 (c and z) are NEVER compared
→ In lexicographic ordering, you stop at the first difference
→ Characters after that are irrelevant to the ordering decision

Counterexample showing why position 2 is meaningless:
→ Alien order might be: z comes before c
→ "abc" < "axz" is still valid because b < x
→ So we CANNOT deduce c < z from this comparison

Why the "Longer Prefix" Case is Invalid

If word1 = "abc" and word2 = "ab":
→ Compare: a==a, b==b, then word2 is exhausted
→ word1 has remaining characters, word2 does not
→ In ANY lexicographic ordering, the shorter prefix comes first
→ "ab" should come before "abc"
→ But we're told "abc" comes before "ab"
→ CONTRADICTION → the input is invalid

This is like saying "apple" comes before "app" in English
→ That's wrong → "app" < "apple" always
→ The longer string can never come before its own prefix

Kahn's BFS — Why It Detects Cycles (Recap from Project 47)

→ Characters in a cycle ALL have in-degree ≥ 1
→ None of them ever reaches in-degree 0
→ None of them ever enters the BFS queue
→ They are NEVER added to the result
→ Result length < number of unique characters
→ This is our cycle detection signal

Example cycle: a → b → c → a
→ in-degree: a=1, b=1, c=1
→ Queue starts empty (assuming no other nodes)
→ BFS processes nothing → result = ""
→ 0 ≠ 3 → cycle detected → return ""

Handling Multiple Valid Orderings

If multiple characters have in-degree 0 simultaneously:
→ BFS processes them in queue order (FIFO)
→ Different queue orders → different valid results
→ Problem says "return ANY valid ordering" → all are acceptable

Example: edges a→c, b→c
→ In-degrees: a=0, b=0, c=2
→ Queue: [a, b] or [b, a] depending on insertion order
→ Valid results: "abc" or "bac"
→ Both are correct

If problem asks "is the ordering UNIQUE?":
→ Check if the queue ever has MORE than one element simultaneously
→ If yes → multiple valid orderings exist → not unique
→ If queue always has exactly one element → ordering is unique

⚠️ COMMON PITFALLS AND FAILURE MODES

Pitfall 1: Duplicate Edges Inflating In-Degree

Input: ["ab", "ac", "bb", "bc"]

Pair 1: "ab" vs "ac" → b≠c → edge b→c → inDegree[c] = 1
Pair 2: "ac" vs "bb" → a≠b → edge a→b → inDegree[b] = 1
Pair 3: "bb" vs "bc" → b==b, b≠c → edge b→c → DUPLICATE edge!

If we add the duplicate:
→ inDegree[c] = 2 (should be 1)
→ When we process b and decrement c's in-degree by 1
→ c still has in-degree 1 → never enters queue
→ FALSE cycle detection!

FIX: Check if edge already exists before adding
→ Use a Set<String> of edges: "b->c" already exists → skip
→ OR use adjacencyList as Set<Character> instead of List<Character>

Pitfall 2: Forgetting the Prefix Length Check

Input: ["abc", "ab"]

Naive comparison:
→ a==a, b==b → all shared characters match
→ No differing character found → no edge added
→ Algorithm proceeds normally → returns some ordering
→ BUT THE INPUT IS INVALID

Must check: if all shared characters match AND word1.length > word2.length
→ Return "" immediately

This is a SPECIFIC test case that interviewers love
→ Easy to forget → instant rejection in interviews

Pitfall 3: Characters Not in Any Edge

Input: ["z", "z"]

Pair 1: "z" vs "z" → identical → no edge
→ Unique characters: {z}
→ Graph: no edges
→ In-degree: {z: 0}
→ BFS: process z → result = "z"
→ CORRECT

Input: ["ab", "cd"]

Pair 1: "ab" vs "cd" → a≠c → edge a→c
→ Unique characters: {a, b, c, d}
→ Edges: a→c
→ In-degrees: {a:0, b:0, c:1, d:0}
→ BFS queue starts with: [a, b, d] (three characters with in-degree 0)
→ Result includes all 4 characters in some valid order
→ b and d can go anywhere (no constraints on them)

If you only tracked characters that appeared in edges:
→ You'd miss b and d → result = "ac" → WRONG (missing characters)
→ ALWAYS collect ALL characters from ALL words

Pitfall 4: Single Word Input

Input: ["hello"]

→ Only one word → no adjacent pairs to compare
→ Zero edges in graph
→ All characters have in-degree 0
→ Result: any permutation of unique characters {h, e, l, o}
→ Valid output: "helo" or "ehlo" etc.

Must handle: words.length == 1 → no comparisons → just return unique chars

Pitfall 5: Empty Words or Empty Input

Input: []
→ No words → no characters → return ""

Input: [""]
→ One empty word → no characters → return ""

Input: ["", "a"]
→ Word1 is empty → word2 starts with 'a'
→ No first difference (word1 exhausted at position 0)
→ word1 is shorter → valid (empty string comes before everything)
→ No edge extracted → just return unique characters

Must handle all these edge cases explicitly

📈 SCALABILITY CONSIDERATIONS

Let:
  N = number of words
  L = total length of all words combined
  C = number of unique characters

TIME COMPLEXITY:
→ Step 1 (collect characters): O(L) → scan all characters
→ Step 2 (build graph): O(N × max_word_length) → compare adjacent pairs
     → But total comparisons across all pairs ≤ L
     → So effectively O(L)
→ Step 3 (topological sort): O(C + E) where E = number of edges
     → E ≤ N-1 (one edge per adjacent pair at most)
     → C ≤ 26 (lowercase English letters)
     → So O(26 + N) = O(N)
→ TOTAL: O(L + N) which simplifies to O(L) since L ≥ N

SPACE COMPLEXITY:
→ Graph adjacency list: O(C + E) = O(26 + N) = O(N)
→ In-degree map: O(C) = O(26) = O(1)
→ Queue: O(C) = O(26) = O(1)
→ TOTAL: O(N) in practice, O(1) if we consider C as constant (≤26)

PRACTICAL PERFORMANCE:
→ N = 100 words → instant
→ N = 100,000 words → O(L) ≈ O(1,000,000) → < 100ms
→ C is bounded by 26 → topological sort is always fast
→ The bottleneck is graph construction, not the sort
