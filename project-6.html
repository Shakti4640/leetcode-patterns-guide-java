PROJECT 6: Smallest Subarray with Sum ≥ Target

🎯 Sliding Window → Variable/Dynamic-Size Window → Shrink Logic

🧩 THE HYPOTHETICAL PROBLEM

→ You are given an array of POSITIVE integers
→ You are given a TARGET sum S
→ Find the length of the SMALLEST contiguous subarray
  whose sum is GREATER THAN OR EQUAL TO S
→ If no such subarray exists, return 0

Example 1:
  Input:  arr = [2, 1, 5, 2, 3, 2], S = 7
  Output: 2
  Why:    Subarray [5, 2] has sum = 7 ≥ 7, length = 2
          No single element ≥ 7 → length 1 is impossible
          [5, 2] is the shortest → answer is 2

Example 2:
  Input:  arr = [2, 1, 5, 2, 8], S = 7
  Output: 1
  Why:    Element [8] alone has sum = 8 ≥ 7, length = 1

Example 3:
  Input:  arr = [3, 4, 1, 1, 6], S = 8
  Output: 3
  Why:    [3, 4, 1] → sum = 8 ≥ 8, length = 3
          [1, 1, 6] → sum = 8 ≥ 8, length = 3
          No length-2 subarray sums to ≥ 8 → answer is 3

Example 4:
  Input:  arr = [1, 1, 1, 1, 1], S = 100
  Output: 0
  Why:    Total array sum = 5 < 100 → impossible

Example 5:
  Input:  arr = [1, 4, 4], S = 4
  Output: 1
  Why:    [4] alone ≥ 4 → length 1

🔍 HOW TO APPROACH IT

Step 1 — Why Fixed-Size Window (Project 5) Doesn't Work Here

→ Project 5: window size K was GIVEN → slide a fixed window
→ Here: we DON'T KNOW the window size → it's what we're LOOKING FOR
→ The optimal window could be size 1, size 2, ... or size N
→ We need to DISCOVER the minimum window size that satisfies the condition

→ If we tried every possible window size:
     Try K=1: check all windows of size 1 → O(N)
     Try K=2: check all windows of size 2 → O(N)
     ...
     Try K=N: check the one window of size N → O(N)
     Total: O(N²) → too slow

→ We need a smarter approach that ADAPTS the window size dynamically

Step 2 — Think About the Brute Force First

→ Naive approach: Check every possible subarray
     for i = 0 to N-1:
         sum = 0
         for j = i to N-1:
             sum += arr[j]
             if sum >= S:
                 minLength = min(minLength, j - i + 1)
                 break  // no need to extend further from this start
     
→ This is O(N²) in the worst case
→ The inner loop restarts the sum from scratch for each starting position
→ But adjacent starting positions share a LOT of overlap
→ We're wasting computation — just like in Project 5 before sliding window

Step 3 — The Key Insight: Expand Then Shrink

→ In Project 5: the window SLID — both edges moved together
→ Here: the window BREATHES — it EXPANDS and CONTRACTS

→ Two distinct phases in each cycle:
     EXPAND: Move the RIGHT pointer to ADD elements until sum ≥ S
     SHRINK: Move the LEFT pointer to REMOVE elements while sum still ≥ S

→ The EXPAND phase makes the window VALID (sum ≥ S)
→ The SHRINK phase makes the window MINIMAL (smallest valid window)

→ Think of it like filling a glass:
     → Pour water (expand) until the glass is full (sum ≥ S)
     → Then pour OUT water (shrink) until it's JUST barely full
     → The "barely full" state is the minimum needed
     → Record that amount
     → Continue pouring more in (expand again)

Step 4 — Visualize the Process

Array: [2, 1, 5, 2, 3, 2]   S = 7

Phase: EXPAND right until sum ≥ 7
  right=0: sum = 2           (2 < 7 → keep expanding)
  right=1: sum = 2+1 = 3     (3 < 7 → keep expanding)
  right=2: sum = 3+5 = 8     (8 ≥ 7 → VALID! window = [2,1,5] length=3)

Phase: SHRINK left while still ≥ 7
  Remove arr[0]=2: sum = 8-2 = 6  (6 < 7 → can't shrink → STOP)
  → Minimum so far: 3

Phase: EXPAND right again
  right=3: sum = 6+2 = 8     (8 ≥ 7 → VALID! window = [1,5,2] length=3)

Phase: SHRINK left while still ≥ 7
  Remove arr[1]=1: sum = 8-1 = 7  (7 ≥ 7 → still valid! window = [5,2] length=2)
  Remove arr[2]=5: sum = 7-5 = 2  (2 < 7 → can't shrink → STOP)
  → Minimum so far: min(3, 2) = 2

Phase: EXPAND right again
  right=4: sum = 2+3 = 5     (5 < 7 → keep expanding)
  right=5: sum = 5+2 = 7     (7 ≥ 7 → VALID! window = [2,3,2] length=3)

Phase: SHRINK left while still ≥ 7
  Remove arr[3]=2: sum = 7-2 = 5  (5 < 7 → can't shrink → STOP)
  → Minimum so far: min(2, 3) = 2

Right pointer exhausted → DONE
Answer: 2 → subarray [5, 2]

Step 5 — The Two Pointers Realization

→ LEFT pointer = left edge of window (shrink boundary)
→ RIGHT pointer = right edge of window (expand boundary)
→ But unlike Projects 1-4:
     → They DON'T move in lockstep (Project 5)
     → They DON'T converge toward each other (Projects 1-2)
     → They DON'T move at the same speed (Project 3)
→ They move INDEPENDENTLY based on the window's SUM condition:
     → Right moves when sum < target (need MORE)
     → Left moves when sum ≥ target (can REDUCE)
→ This is CONDITIONAL movement — the most flexible two-pointer variant

🛠️ HOW CAN WE SOLVE IT

The Algorithm — Line by Line Logic

1. Initialize: left = 0, runningSum = 0, minLength = Integer.MAX_VALUE
2. FOR right = 0 to N-1:
     a. EXPAND: runningSum += arr[right]    ← add entering element
     b. SHRINK: WHILE runningSum >= S:
          → Record: minLength = min(minLength, right - left + 1)
          → Remove: runningSum -= arr[left]
          → Advance: left++
3. IF minLength is still Integer.MAX_VALUE → return 0 (no valid window)
4. ELSE return minLength

Why a WHILE Loop for Shrinking (Not IF)?

→ After expanding, the sum might be MUCH larger than S
→ We might need to remove MULTIPLE elements from the left to minimize
→ Example: arr = [1, 1, 1, 10], S = 7
     → After adding 10: sum = 1+1+1+10 = 13 → window [1,1,1,10] length=4
     → Remove 1: sum = 12 ≥ 7 → still valid → window [1,1,10] length=3
     → Remove 1: sum = 11 ≥ 7 → still valid → window [1,10] length=2
     → Remove 1: sum = 10 ≥ 7 → still valid → window [10] length=1
     → Remove 10: sum = 0 < 7 → STOP

→ With IF instead of WHILE: we'd only shrink once → miss length=1
→ WHILE ensures we shrink to the ABSOLUTE minimum
→ This is the GREEDY shrinking — remove as much as possible while staying valid

✅ WHAT SHOULD YOU DO

→ ALWAYS use WHILE (not IF) for the shrink loop
     → Multiple shrinks may be needed per expansion
     → IF catches only the first valid shrink
     → WHILE catches the MINIMUM valid window

→ ALWAYS record the window length BEFORE shrinking
     → The current window IS valid → record it
     → Then try to shrink → if still valid → record the smaller one
     → The while loop handles this: record → shrink → check → record → shrink ...

→ ALWAYS check if a valid window was ever found
     → If minLength remains Integer.MAX_VALUE → no subarray sums to ≥ S
     → Return 0 in this case

→ ALWAYS understand the pointer roles:
     → RIGHT pointer: controlled by the FOR loop → always advances → EAGER
     → LEFT pointer: controlled by the WHILE loop → advances only when valid → GREEDY
     → Right EXPANDS the window to SATISFY the condition
     → Left SHRINKS the window to MINIMIZE the length

→ ALWAYS use this only when ALL elements are POSITIVE
     → The shrink logic assumes: removing an element DECREASES the sum
     → If elements can be negative: removing might INCREASE the sum
     → The greedy shrink breaks with negatives (more on this below)

🚫 WHAT SHOULD YOU NOT DO

→ DO NOT use a fixed-size window approach
     → You don't know K → trying all K values is O(N²)
     → Variable window finds the answer in O(N)

→ DO NOT use IF instead of WHILE for shrinking
     BAD:
       if (runningSum >= S) {
           minLength = Math.min(minLength, right - left + 1);
           runningSum -= arr[left];
           left++;
       }
     → Only shrinks ONCE per expansion
     → Misses further shrink opportunities
     → Returns a length that's LARGER than optimal

     GOOD:
       while (runningSum >= S) {
           minLength = Math.min(minLength, right - left + 1);
           runningSum -= arr[left];
           left++;
       }
     → Shrinks to ABSOLUTE minimum

→ DO NOT record the length AFTER the while loop
     BAD:
       while (runningSum >= S) {
           runningSum -= arr[left];
           left++;
       }
       minLength = Math.min(minLength, right - left + 1);  // WRONG
     → After the while loop, sum < S → window is INVALID
     → Recording an invalid window's length → wrong answer

     GOOD: record INSIDE the while loop BEFORE shrinking

→ DO NOT apply this when elements can be NEGATIVE
     → Negative elements break the shrink assumption
     → Removing a negative element INCREASES the sum
     → The while loop would shrink PAST the optimal point
     → For arrays with negatives → use prefix sum + binary search or deque

→ DO NOT forget that left can never exceed right + 1
     → The while loop has runningSum >= S as condition
     → As left approaches right, sum approaches arr[right] (single element)
     → If arr[right] < S → loop stops → left ≤ right
     → If arr[right] ≥ S → left goes past right → sum becomes 0 → loop stops
     → The algorithm naturally prevents left from going out of bounds

🔀 WHEN TO USE THIS PATTERN

✓ Finding the SMALLEST/SHORTEST subarray satisfying a condition
✓ Finding the LONGEST/LARGEST subarray satisfying a condition
✓ The condition is about a RUNNING METRIC (sum, count, etc.)
✓ ALL elements are POSITIVE (or the metric is MONOTONICALLY affected by expansion)
✓ The metric INCREASES when we expand and DECREASES when we shrink
     → This monotonicity is what makes greedy shrinking correct

✓ Common variants:
     → Smallest subarray with sum ≥ S (this project)
     → Longest subarray with sum ≤ S (reverse condition)
     → Longest substring with at most K distinct characters (Project 12)
     → Minimum window substring (Project 33)
     → Longest substring without repeating characters
     → Any "find optimal window size satisfying condition X"

🚫 WHEN NOT TO USE THIS PATTERN

✗ Elements can be NEGATIVE
     → Sum doesn't monotonically increase with window expansion
     → Adding a negative element can DECREASE the sum
     → Removing a negative element can INCREASE the sum
     → The expand-shrink logic assumes monotonic behavior → breaks

✗ The condition is NOT about a contiguous subarray
     → "Find K elements (not necessarily adjacent) with sum ≥ S"
     → This is a selection problem → sorting or heap → not sliding window

✗ The window size is FIXED
     → Use fixed-size sliding window (Project 5) instead
     → Variable window adds unnecessary complexity for fixed-K problems

✗ The metric is NOT incrementally updatable
     → "Smallest subarray where median ≥ X"
     → Median can't be updated by simple add/subtract
     → Needs more complex data structures

✗ You need ALL subarrays satisfying the condition (not just the smallest)
     → Sliding window finds the OPTIMAL one
     → Finding ALL valid windows may need different enumeration

⚙️ HOW IT ONLY WORKS — Internal Mechanics

The Monotonicity Requirement

WHY POSITIVE ELEMENTS ARE REQUIRED:

→ The expand-shrink strategy relies on this guarantee:
     → EXPANDING (adding element) → sum INCREASES (or stays same)
     → SHRINKING (removing element) → sum DECREASES (or stays same)

→ This creates a MONOTONIC relationship:
     → Larger window → larger (or equal) sum
     → Smaller window → smaller (or equal) sum

→ Because of this monotonicity:
     → If window [left, right] has sum ≥ S
     → Then window [left, right+1] ALSO has sum ≥ S (bigger window, bigger sum)
     → So we NEVER need to re-expand a valid window
     → We can SAFELY shrink and know that expanding again will restore validity

→ If elements could be negative:
     → Adding arr[right+1] = -5 might DECREASE the sum below S
     → Removing arr[left] = -3 might INCREASE the sum above S
     → The shrink/expand decisions become unreliable
     → Greedy approach breaks → need different algorithm

The Amortized O(N) Time Proof

→ At first glance: outer FOR loop + inner WHILE loop → O(N²)?
→ NO — this is O(N) due to AMORTIZED analysis

→ KEY OBSERVATION: Each element is added AT MOST once and removed AT MOST once
     → Right pointer visits each element once: N additions total
     → Left pointer visits each element AT MOST once: at most N removals total
     → Total operations: N additions + N removals = 2N = O(N)

→ The WHILE loop doesn't restart from the beginning each time
     → Left pointer NEVER moves backward
     → It only moves forward
     → Across ALL iterations of the outer loop, left moves at most N times total

→ Think of it like two people walking along the same road:
     → Person R (right) walks the entire road: N steps
     → Person L (left) also walks the same road but may pause: ≤ N steps
     → Total steps by both: ≤ 2N
     → Even though L sometimes takes multiple steps while R takes one
     → L's TOTAL steps across the entire algorithm ≤ N

→ VISUAL PROOF:
     left:  0 → 0 → 0 → 1 → 2 → 3 → 3 → 3 → 4 → 5 → 6
     right: 0 → 1 → 2 → 2 → 2 → 2 → 3 → 4 → 4 → 4 → 4
     
     left moved:  0,0,0,1,1,1,0,0,1,1,1 → total = 6 forward moves
     right moved: 1,1,1,0,0,0,1,1,0,0,0 → total = 5 forward moves (in FOR loop)
     
     → Combined: 11 operations for array of size ~7
     → Always ≤ 2N

Comparing Fixed vs Variable Sliding Window

FIXED WINDOW (Project 5):
  → Right pointer: advances EVERY iteration (FOR loop)
  → Left pointer: advances EVERY iteration (i - K, lockstep with right)
  → Window size: CONSTANT at K
  → Movement: SYNCHRONIZED
  → Decision: NONE — just slide
  → Metric update: subtract leaving, add entering

VARIABLE WINDOW (Project 6):
  → Right pointer: advances EVERY iteration (FOR loop — eager)
  → Left pointer: advances CONDITIONALLY (WHILE loop — greedy)
  → Window size: VARIES from 1 to N
  → Movement: INDEPENDENT
  → Decision: expand vs shrink based on condition
  → Metric update: add on expand, subtract on shrink

SHARED FOUNDATION:
  → Both maintain a RUNNING METRIC
  → Both use LEFT and RIGHT boundaries
  → Both UPDATE incrementally (not recalculate)
  → The variable window is a GENERALIZATION of the fixed window
  → Fixed window is a special case where the shrink condition is "window > K"

The Expand-Shrink Cycle

The algorithm follows a repeating pattern:

  EXPAND → EXPAND → EXPAND → [condition met] → SHRINK → SHRINK → [condition broken]
  → EXPAND → [condition met] → SHRINK → [condition broken]
  → EXPAND → EXPAND → [condition met] → SHRINK → SHRINK → SHRINK → [condition broken]
  → ...

→ The expansions GROW the window to meet the condition
→ The shrinks TRIM the window to find the minimum
→ Each shrink records a potential answer
→ The overall minimum across all recorded answers is the final answer

→ The algorithm NEVER backtracks:
     → Right never moves left
     → Left never moves right (backward)
     → Both only move FORWARD
     → This forward-only movement is what guarantees O(N)

Space Complexity

→ Variables: left, right (implicit in FOR), runningSum, minLength
→ No additional data structures
→ TOTAL: O(1) space
→ Identical to fixed-window — variable window doesn't need more memory

⚠️ COMMON PITFALLS AND FAILURE MODES

Pitfall 1: Using IF Instead of WHILE (The #1 Mistake)

This is the MOST COMMON error with variable sliding windows.

Example: arr = [1, 1, 1, 1, 10], S = 7

With IF:
  right=4: sum = 1+1+1+1+10 = 14 ≥ 7
  IF triggers once: record length=5, remove arr[0]=1, sum=13, left=1
  → Move to next right... but right is already at end
  → Returns 5 → WRONG (correct answer is 1, since 10 ≥ 7)

With WHILE:
  right=4: sum = 14 ≥ 7
  WHILE iteration 1: record length=5, remove 1, sum=13, left=1
  WHILE iteration 2: record length=4, remove 1, sum=12, left=2
  WHILE iteration 3: record length=3, remove 1, sum=11, left=3
  WHILE iteration 4: record length=2, remove 1, sum=10, left=4
  WHILE iteration 5: record length=1, remove 10, sum=0, left=5
  → sum=0 < 7 → WHILE stops
  → Minimum recorded = 1 → CORRECT

Pitfall 2: Recording Length AFTER the While Loop

BAD:
  while (runningSum >= S) {
      runningSum -= arr[left];
      left++;
  }
  // At this point: sum < S → window is INVALID
  minLength = Math.min(minLength, right - left + 1);  // WRONG

→ After the while loop, the window is INVALID (sum < S)
→ Recording its length captures an invalid window

GOOD:
  while (runningSum >= S) {
      minLength = Math.min(minLength, right - left + 1);  // record VALID window
      runningSum -= arr[left];  // THEN shrink
      left++;
  }

→ Record BEFORE shrinking → window is still valid at recording time

Pitfall 3: Not Handling "No Valid Window" Case

Input: arr = [1, 1, 1], S = 100
→ Total sum = 3 < 100
→ No subarray sums to ≥ 100
→ minLength stays at Integer.MAX_VALUE

→ Must check: if (minLength == Integer.MAX_VALUE) return 0;
→ Returning Integer.MAX_VALUE directly → nonsensical answer

Pitfall 4: Applying to Arrays with Negative Numbers

Input: arr = [2, -1, 5, -2, 3], S = 5
→ Subarray [2, -1, 5] has sum = 6 ≥ 5, length = 3
→ Subarray [5] has sum = 5 ≥ 5, length = 1
→ Correct answer: 1

But with our algorithm:
  right=0: sum=2, sum < 5
  right=1: sum=2+(-1)=1, sum < 5
  right=2: sum=1+5=6, sum ≥ 5
    WHILE: record length=3, remove arr[0]=2, sum=4, left=1
    4 < 5 → STOP
  right=3: sum=4+(-2)=2, sum < 5
  right=4: sum=2+3=5, sum ≥ 5
    WHILE: record length=4, remove arr[1]=-1, sum=5-(-1)=6, left=2
    → sum INCREASED by removing negative! → wrong behavior
    WHILE: record length=3, remove arr[2]=5, sum=1, left=3
    → Missed the window [5] of length 1!

→ The algorithm returned 3 instead of 1 → WRONG
→ Negative numbers break the monotonicity assumption

Pitfall 5: Empty Array or S = 0

→ Empty array: no elements → no valid window → return 0
→ S = 0: every non-empty subarray has sum ≥ 0 → minimum length = 1
→ S < 0: same logic → minimum length = 1 (if array is non-empty)
→ Handle these edge cases explicitly

📈 SCALABILITY CONSIDERATIONS

→ N = 100, S = 50:
     → Brute force O(N²) = 10,000 ops → fast
     → Variable window O(N) = 100 ops → fast
     → Both fine at this scale

→ N = 1,000,000, S = 100:
     → Brute force O(N²) = 10^12 ops → hours to complete
     → Variable window O(N) = 10^6 ops → milliseconds
     → 1,000,000x faster

→ N = 100,000,000:
     → Brute force: completely infeasible
     → Variable window: ~1 second → perfectly fine

→ Memory: O(1) extra space regardless of input size
→ The algorithm processes the array in a SINGLE FORWARD PASS
→ This means it works on STREAMING data:
     → Don't need the entire array in memory
     → Process elements as they arrive
     → Maintain left pointer position and running sum
     → Perfect for real-time applications

→ REAL-WORLD applications:
     → Network bandwidth: "smallest time window where traffic ≥ threshold"
     → Financial: "shortest trading period where profit ≥ target"
     → Sensor monitoring: "minimum samples needed to detect anomaly"
     → Video streaming: "minimum buffer size for smooth playback"

🔍 INTERPRETABILITY CONSIDERATIONS

→ The expand-shrink metaphor is highly intuitive:
     → Imagine a RUBBER BAND on a number line
     → Stretch the right end until the covered sum is enough
     → Then pull in the left end to make it as tight as possible
     → Record the tightest fit
     → Stretch right again → tighten left → record → repeat

→ The algorithm has a clear NARRATIVE:
     → "I grow the window until it's valid"
     → "Then I shrink it as much as possible while keeping it valid"
     → "I record every valid shrink as a candidate answer"
     → "The smallest candidate wins"

→ Debugging is straightforward:
     → At every step, print: left, right, sum, window contents, minLength
     → Expand steps: sum increases, window grows
     → Shrink steps: sum decreases, window tightens, candidates recorded
     → Any error is immediately visible in the trace

→ Interview explanation:
     → "I use a variable-size sliding window"
     → "Right pointer expands to make the sum valid"
     → "Left pointer shrinks to minimize the window"
     → "Each element is added and removed at most once → O(N)"
     → "Only works for positive elements due to monotonicity"

🧠 KEY TAKEAWAY

WHAT WE LEARNED:
→ Variable-size sliding window ADAPTS the window dynamically
→ Right pointer EXPANDS (eager — always moves forward in FOR loop)
→ Left pointer SHRINKS (greedy — moves forward in WHILE loop when condition met)
→ The WHILE loop (not IF) is critical for finding the true minimum
→ Only works with POSITIVE elements (monotonicity requirement)
→ Amortized O(N) time because each element is added/removed at most once

THE COMPLETE SLIDING WINDOW FAMILY:
  Project 5 (Fixed):    "What is the best metric for windows of EXACT size K?"
  Project 6 (Variable): "What is the SMALLEST window satisfying condition X?"

COMPARISON TABLE:

  ├─────────────────────┼───────────────────┼───────────────────────┤
  │ Window size         │ Constant K        │ Dynamic (1 to N)      │
  │ Right pointer       │ FOR loop          │ FOR loop              │
  │ Left pointer        │ Lockstep (i - K)  │ WHILE loop (greedy)   │

THE EXPAND-SHRINK TEMPLATE:
  int left = 0;
  int metric = 0;
  int bestAnswer = ...;

  for (int right = 0; right < n; right++) {
      metric = UPDATE(metric, arr[right]);       // expand

      while (CONDITION(metric)) {                // while valid
          bestAnswer = OPTIMIZE(bestAnswer, ...); // record
          metric = UNDO(metric, arr[left]);       // shrink
          left++;
      }
  }

PROJECT 7 (Detect Cycle in a Linked List) will introduce:

→ Two pointers moving at DIFFERENT SPEEDS

The connection to what we learned:
→ Projects 1-4: Two pointers on ARRAYS (random access)
→ Projects 5-6: Two pointers defining a WINDOW on arrays
→ Project 7:    Two pointers on LINKED LISTS (sequential access only)

Why linked lists need special treatment:
→ Arrays: can access any index in O(1) → easy to use left/right pointers
→ Linked lists: can ONLY follow .next → can't go backward, can't jump ahead
→ The only tool we have: follow pointers forward
→ Two pointers at different SPEEDS exploit this limitation brilliantly

The Tortoise & Hare technique:
→ Slow pointer: moves 1 node per step → "tortoise"
→ Fast pointer: moves 2 nodes per step → "hare"
→ If there's a cycle → fast will eventually LAP slow → they meet
→ If there's no cycle → fast reaches null → they never meet
→ Simple yet mathematically profound
