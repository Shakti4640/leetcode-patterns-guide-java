PROJECT 35: Find Median from a Data Stream

🎯 Two Heaps → Max-Heap + Min-Heap Balancing → Running Median Maintenance

🧩 THE HYPOTHETICAL PROBLEM

→ You are designing a system that receives STREAMING numbers
→ Numbers arrive ONE AT A TIME in arbitrary order
→ At ANY point, you must be able to report the MEDIAN of all numbers received so far
→ Median definition:
     → If count is ODD: the middle element when sorted
     → If count is EVEN: average of the two middle elements
→ Design a data structure that supports:
     → addNum(int num): Add a number from the stream
     → findMedian(): Return the median of all numbers added so far

Example:
  addNum(1)    → stream: [1]           → median = 1.0
  addNum(3)    → stream: [1, 3]        → median = (1+3)/2 = 2.0
  addNum(5)    → stream: [1, 3, 5]     → median = 3.0
  addNum(2)    → stream: [1, 2, 3, 5]  → median = (2+3)/2 = 2.5
  addNum(4)    → stream: [1,2,3,4,5]   → median = 3.0
  addNum(0)    → stream: [0,1,2,3,4,5] → median = (2+3)/2 = 2.5

Another example:
  addNum(41)   → median = 41.0
  addNum(35)   → median = (35+41)/2 = 38.0
  addNum(62)   → median = 41.0
  addNum(4)    → median = (35+41)/2 = 38.0
  addNum(97)   → median = 41.0
  addNum(108)  → median = (41+62)/2 = 51.5

🔍 HOW TO APPROACH IT

Step 1 — Why Naive Approaches Fail

APPROACH 1 — Unsorted List + Sort Each Time:
  → addNum: O(1) — just append
  → findMedian: O(n log n) — sort, then pick middle
  → For n queries: O(n² log n) total → TOO SLOW

APPROACH 2 — Sorted List (Insertion Sort):
  → addNum: O(n) — find position via binary search O(log n), 
                    but SHIFTING elements is O(n)
  → findMedian: O(1) — just index into middle
  → For n queries: O(n²) total → still slow for large n

APPROACH 3 — Balanced BST (TreeMap):
  → addNum: O(log n) — balanced tree insertion
  → findMedian: O(log n) — but finding median in BST is tricky
  → Handling duplicates adds complexity
  → Works but implementation is complex

WHAT WE NEED:
  → addNum: O(log n)
  → findMedian: O(1)
  → Simple to implement
  → This is what TWO HEAPS gives us

Step 2 — The Two-Heap Insight

→ Imagine all numbers sorted: [1, 2, 3, 4, 5, 6, 7, 8]
→ Split them into TWO HALVES at the median:
     Left half:  [1, 2, 3, 4]  → the SMALLER half
     Right half: [5, 6, 7, 8]  → the LARGER half

→ The MEDIAN is:
     → If odd total: the top of whichever half has one extra
     → If even total: average of the TOP of each half

→ What do we need from each half?
     → Left half: we need the MAXIMUM (the largest of the small numbers)
     → Right half: we need the MINIMUM (the smallest of the large numbers)

→ Data structure that gives the MAXIMUM efficiently? → MAX-HEAP
→ Data structure that gives the MINIMUM efficiently? → MIN-HEAP

→ So:
     → Left half → MAX-HEAP (gives us the largest small number)
     → Right half → MIN-HEAP (gives us the smallest large number)
     → Median = peek at the tops of both heaps

This is the TWO-HEAP pattern:
  ┌─────────────┐     ┌─────────────┐
  │  MAX-HEAP   │     │  MIN-HEAP   │
  │ (left half) │     │(right half) │
  │             │     │             │
  │  smaller    │     │   larger    │
  │  numbers    │     │  numbers    │
  │             │     │             │
  │   TOP = 4   │     │   TOP = 5   │
  │  [1,2,3,4]  │     │  [5,6,7,8]  │
  └─────────────┘     └─────────────┘
       ↑                    ↑
  largest of small    smallest of large
       ↑                    ↑
       └──── MEDIAN ────────┘
           (4+5)/2 = 4.5

Step 3 — The Balancing Rules

RULE 1: Every element in max-heap ≤ every element in min-heap
  → All "small" numbers are in the left heap
  → All "large" numbers are in the right heap
  → max-heap's top ≤ min-heap's top (always)

RULE 2: Heaps differ in size by AT MOST 1
  → |maxHeap.size() - minHeap.size()| ≤ 1
  → This ensures the median is always at the top(s)

RULE 3: We choose max-heap to be the "extra" one
  → If odd count: max-heap has one more element
  → maxHeap.size() == minHeap.size()  OR  maxHeap.size() == minHeap.size() + 1
  → Convention: max-heap is always ≥ min-heap in size
  → (Some implementations use the reverse — both work)

MEDIAN EXTRACTION:
  → If maxHeap.size() > minHeap.size():
       → Odd count → median = maxHeap.peek()
  → If maxHeap.size() == minHeap.size():
       → Even count → median = (maxHeap.peek() + minHeap.peek()) / 2.0

Step 4 — The Add Algorithm

When a new number arrives, we need to:
  1. Place it in the correct heap
  2. Rebalance if needed

STRATEGY — Always add to max-heap first, then rebalance:

Step A: Add to max-heap
  → maxHeap.add(num)
  → This might violate Rule 1 (max-heap's top > min-heap's top)

Step B: Move max-heap's top to min-heap
  → minHeap.add(maxHeap.poll())
  → Now Rule 1 is guaranteed (we moved the largest small to the large side)
  → But Rule 2 might be violated (min-heap might be too big)

Step C: Rebalance sizes
  → If minHeap.size() > maxHeap.size():
       → maxHeap.add(minHeap.poll())
       → Now sizes are balanced

WHY THIS THREE-STEP DANCE WORKS:
  → Step A: ensures new number enters the system
  → Step B: ensures the split point is correct (no small number in large heap)
  → Step C: ensures size balance (max-heap is always ≥ min-heap)

ALTERNATIVE STRATEGY — Direct placement + rebalance:
  → If num ≤ maxHeap.peek(): add to maxHeap
  → Else: add to minHeap
  → Then rebalance sizes
  → Both approaches work — first is simpler to code

🛠️ HOW CAN WE SOLVE IT

Java Heap Fundamentals (Must Know Before Coding)

JAVA PriorityQueue:
  → Default: MIN-HEAP (smallest element at top)
  → PriorityQueue<Integer> minHeap = new PriorityQueue<>();
  → minHeap.add(5), minHeap.add(3), minHeap.add(7)
  → minHeap.peek() → 3 (smallest)
  → minHeap.poll() → 3 (removes smallest)

MAX-HEAP in Java:
  → Use Collections.reverseOrder() comparator
  → PriorityQueue<Integer> maxHeap = new PriorityQueue<>(Collections.reverseOrder());
  → maxHeap.add(5), maxHeap.add(3), maxHeap.add(7)
  → maxHeap.peek() → 7 (largest)
  → maxHeap.poll() → 7 (removes largest)

OPERATIONS COMPLEXITY:
  → add(): O(log n) — insert + bubble up
  → poll(): O(log n) — remove top + bubble down
  → peek(): O(1) — just look at top
  → size(): O(1) — tracked internally

The Algorithm — Precise Steps

DATA STRUCTURES:
  → maxHeap: stores the SMALLER half (left side)
  → minHeap: stores the LARGER half (right side)

addNum(int num):
  1. maxHeap.add(num)                    // Always add to max-heap first
  2. minHeap.add(maxHeap.poll())         // Move largest-of-small to min-heap
  3. if (minHeap.size() > maxHeap.size()): // Rebalance if needed
        maxHeap.add(minHeap.poll())       // Move smallest-of-large back

findMedian():
  1. if (maxHeap.size() > minHeap.size()):
        return maxHeap.peek()             // Odd count → max-heap has extra
  2. else:
        return (maxHeap.peek() + minHeap.peek()) / 2.0  // Even count → average

Why the Three-Step Add Works — Invariant Proof

INVARIANT 1: Every element in maxHeap ≤ every element in minHeap

PROOF:
  → Step 1: We add num to maxHeap
       → maxHeap might now have an element > some element in minHeap → VIOLATION possible

  → Step 2: We move maxHeap.poll() (the LARGEST in maxHeap) to minHeap
       → After this: maxHeap's new top ≤ old top ≤ element just moved to minHeap
       → And the moved element is ≤ all existing elements in minHeap?
       → Not necessarily! But it doesn't matter because:
       → The moved element IS the largest from maxHeap
       → So everything remaining in maxHeap < moved element
       → And minHeap now contains this element (correctly on the "large" side)
       → INVARIANT 1 HOLDS ✓

INVARIANT 2: maxHeap.size() == minHeap.size() OR maxHeap.size() == minHeap.size() + 1

PROOF:
  → Before adding: let maxHeap.size() = M, minHeap.size() = N
  → After Step 1: M+1, N
  → After Step 2: M, N+1
  → Step 3 check: if N+1 > M → move one back → M+1, N
  → Cases:
       → If M == N (was balanced):
            → After Step 2: M, N+1 = M, M+1 → minHeap bigger → Step 3 fires
            → After Step 3: M+1, M → maxHeap has one more ✓
       → If M == N+1 (maxHeap had one more):
            → After Step 2: M, N+2 = M, M+1 → but wait, N+1 = M, so N+2 = M+1
            → Actually: after Step 2: M, N+1
            → N+1 vs M: since M = N+1, N+1 = M → sizes equal → no Step 3
            → After: M, M → balanced ✓
  → INVARIANT 2 HOLDS ✓

✅ WHAT SHOULD YOU DO

→ ALWAYS use Collections.reverseOrder() for max-heap in Java
     → PriorityQueue<Integer> maxHeap = new PriorityQueue<>(Collections.reverseOrder());
     → This is the standard Java idiom
     → Alternative: new PriorityQueue<>((a, b) -> b - a)
          → WARNING: (b - a) can OVERFLOW for extreme values
          → Use (a, b) -> Integer.compare(b, a) for safety

→ ALWAYS add to maxHeap first, then rebalance
     → This ensures the "flow" goes: maxHeap → minHeap → back if needed
     → The flow guarantees both invariants are maintained
     → Alternative flows work too, but this one is simplest

→ ALWAYS check size balance AFTER the transfer step
     → Don't check before — the transfer might have fixed it already
     → Don't skip the check — an imbalance > 1 breaks median extraction

→ ALWAYS use double for median return type
     → Even count: (a + b) / 2.0 — must use 2.0 not 2
     → Integer division truncates: (3 + 4) / 2 = 3 → WRONG
     → Double division: (3 + 4) / 2.0 = 3.5 → CORRECT

→ ALWAYS handle the empty heap edge case
     → findMedian() when no numbers added → both heaps empty → NPE on peek()
     → Guard with: if (maxHeap.isEmpty()) throw or return 0

→ ALWAYS consider integer overflow in median calculation
     → maxHeap.peek() + minHeap.peek() could overflow int
     → Use: maxHeap.peek() / 2.0 + minHeap.peek() / 2.0
     → Or cast to long first: ((long)a + (long)b) / 2.0

🚫 WHAT SHOULD YOU NOT DO

→ DO NOT use (b - a) as max-heap comparator
     → (b - a) overflows when b = Integer.MIN_VALUE, a = positive
     → Use Integer.compare(b, a) or Collections.reverseOrder()

→ DO NOT maintain a sorted list alongside heaps
     → Defeats the purpose — heaps ARE the efficient structure
     → Adding a sorted list means O(n) insertion → worse than heaps

→ DO NOT try to find median by merging heaps
     → Merging heaps to sort = O(n log n) → same as naive approach
     → The whole point is that median is at the TOPS → O(1) extraction

→ DO NOT forget to handle the odd/even case in findMedian
     → Odd: return maxHeap.peek() (single middle element)
     → Even: return average of both tops
     → Returning wrong value for odd/even = wrong answer

→ DO NOT allow maxHeap to be smaller than minHeap
     → Our convention: maxHeap.size() ≥ minHeap.size()
     → If reversed: the odd-case median would come from minHeap
     → Inconsistent convention leads to bugs

→ DO NOT use add() to both heaps independently without rebalancing
     → "If num < median, add to maxHeap; else add to minHeap"
     → This works for PLACEMENT but doesn't guarantee SIZE BALANCE
     → Must still rebalance after each add

→ DO NOT use this pattern for finding Kth element (not median)
     → Two heaps finds the MEDIAN (middle element)
     → For Kth largest → use single heap of size K (Project 38)
     → Two heaps is overkill and wrong for arbitrary K

🚫 WHEN NOT TO USE THIS PATTERN

✗ You need the Kth element (not specifically the median)
     → K = 1 (maximum): just track max
     → K = n (minimum): just track min
     → K = arbitrary: use single heap of size K (Project 38)
     → K = n/2 (median): use two heaps (this project)

✗ You need to find median of a STATIC array
     → QuickSelect: O(n) average → better than building heaps O(n log n)
     → Sorting: O(n log n) → simpler for one-time computation
     → Two heaps are for REPEATED median queries on changing data

✗ Data is removed from arbitrary positions (not just the window edges)
     → Heaps don't support efficient arbitrary removal
     → Standard heap: remove by value is O(n)
     → Need indexed priority queue or balanced BST instead

✗ You need all elements sorted, not just the median
     → Heaps give you the top element, not the full sorted order
     → For full sorting: use a different structure

✗ Memory is extremely constrained
     → Two heaps store ALL elements → O(n) space
     → For approximate median with less space → use reservoir sampling or t-digest

⚙️ HOW IT ONLY WORKS — Internal Mechanics

Heap Internals (Brief Review)

A HEAP is a complete binary tree stored as an ARRAY:
  → Parent of index i: (i - 1) / 2
  → Left child of i: 2*i + 1
  → Right child of i: 2*i + 2

MAX-HEAP property: parent ≥ both children
  → Root (index 0) is the MAXIMUM

MIN-HEAP property: parent ≤ both children
  → Root (index 0) is the MINIMUM

ADD operation (O(log n)):
  → Place new element at the END of array
  → "Bubble up": swap with parent while parent < child (max-heap)
  → At most log₂(n) swaps (height of tree)

POLL operation (O(log n)):
  → Remove root (the max/min)
  → Move LAST element to root position
  → "Bubble down": swap with larger child while violating heap property
  → At most log₂(n) swaps

PEEK operation (O(1)):
  → Just return array[0] — the root

Complexity Analysis

addNum():
  → maxHeap.add(): O(log n)
  → maxHeap.poll(): O(log n)
  → minHeap.add(): O(log n)
  → Possible minHeap.poll() + maxHeap.add(): O(log n) each
  → TOTAL: O(log n) per call

findMedian():
  → maxHeap.peek(): O(1)
  → minHeap.peek(): O(1)
  → Arithmetic: O(1)
  → TOTAL: O(1) per call

Space: O(n)
  → All n elements stored across two heaps

COMPARISON:
  ┌──────────────────────┬──────────────┬──────────────┬─────────────┐
  │ Approach             │ addNum()     │ findMedian() │ Space       │
  ├──────────────────────┼──────────────┼──────────────┼─────────────┤
  │ Unsorted list + sort │ O(1)         │ O(n log n)   │ O(n)        │
  │ Sorted list (insert) │ O(n)         │ O(1)         │ O(n)        │
  │ Balanced BST         │ O(log n)     │ O(log n)     │ O(n)        │
  │ TWO HEAPS            │ O(log n)     │ O(1)         │ O(n)        │
  └──────────────────────┴──────────────┴──────────────┴─────────────┘

  → Two heaps gives the BEST combination: O(log n) add + O(1) median

The Alternative Add Strategy (Direct Placement)

STRATEGY B — Place directly in correct heap:

addNum(int num):
  1. if maxHeap is empty OR num ≤ maxHeap.peek():
        maxHeap.add(num)
     else:
        minHeap.add(num)
  
  2. Rebalance:
     if maxHeap.size() > minHeap.size() + 1:
        minHeap.add(maxHeap.poll())
     else if minHeap.size() > maxHeap.size():
        maxHeap.add(minHeap.poll())

WHY BOTH STRATEGIES WORK:
  Strategy A (add to max first, flow to min, rebalance):
    → Simpler code — no conditional on which heap to add to
    → Always 2-3 heap operations
    → Invariant maintained by the "flow" pattern

  Strategy B (direct placement + rebalance):
    → More intuitive — place where it belongs
    → Needs conditional for initial placement
    → Then rebalance for size

  Both produce IDENTICAL final states
  Both are O(log n) per add
  Strategy A is preferred for cleaner code

⚠️ COMMON PITFALLS AND FAILURE MODES

Pitfall 1: Integer Overflow in Average

WRONG:
  return (maxHeap.peek() + minHeap.peek()) / 2.0;

  → If maxHeap.peek() = 2,000,000,000 and minHeap.peek() = 2,000,000,000
  → Sum = 4,000,000,000 → OVERFLOWS int (max is ~2.1 billion)
  → Result is WRONG negative number

CORRECT (Option A — divide first):
  return maxHeap.peek() / 2.0 + minHeap.peek() / 2.0;

CORRECT (Option B — cast to long):
  return ((long) maxHeap.peek() + (long) minHeap.peek()) / 2.0;

→ Both avoid integer overflow
→ Option A is cleaner
→ Option B is more explicit

Pitfall 2: Max-Heap Comparator Overflow

WRONG:
  PriorityQueue<Integer> maxHeap = new PriorityQueue<>((a, b) -> b - a);

  → If b = Integer.MIN_VALUE and a = 1
  → b - a = -2147483648 - 1 = 2147483647 (overflow → positive!)
  → Comparator returns positive → thinks b > a → WRONG ordering

CORRECT:
  PriorityQueue<Integer> maxHeap = new PriorityQueue<>(Collections.reverseOrder());
  
  OR:
  PriorityQueue<Integer> maxHeap = new PriorityQueue<>((a, b) -> Integer.compare(b, a));

  → Integer.compare handles overflow correctly
  → Collections.reverseOrder() uses Integer.compare internally

Pitfall 3: Empty Heap Peek

WRONG:
  public double findMedian() {
      if (maxHeap.size() > minHeap.size()) {
          return maxHeap.peek();
      }
      return (maxHeap.peek() + minHeap.peek()) / 2.0;
  }

  → What if BOTH heaps are empty? → maxHeap.peek() → NullPointerException
  → What if only maxHeap has elements? → minHeap.peek() → NPE

CORRECT:
  → Either guard with size checks
  → Or document that findMedian() should only be called after addNum()
  → In interviews: mention the edge case even if you don't fully handle it

Pitfall 4: Wrong Size Convention

→ Convention A: maxHeap.size() ≥ minHeap.size()
     → Odd count: median = maxHeap.peek()
     → Even count: average of both tops

→ Convention B: minHeap.size() ≥ maxHeap.size()
     → Odd count: median = minHeap.peek()
     → Even count: average of both tops

→ BOTH work — but you MUST be consistent
→ If you mix conventions → findMedian returns wrong value
→ Pick one and stick with it throughout
→ Our code uses Convention A

Pitfall 5: Forgetting Rebalance Step

WRONG:
  void addNum(int num) {
      maxHeap.add(num);
      minHeap.add(maxHeap.poll());
      // Forgot to rebalance!
  }

→ After n additions:
     → maxHeap: 0 elements (everything flows to minHeap)
     → minHeap: n elements
     → maxHeap.peek() → NPE in findMedian()

→ The rebalance step is ESSENTIAL:
  if (minHeap.size() > maxHeap.size()) {
      maxHeap.add(minHeap.poll());
  }

Pitfall 6: Using poll() When You Mean peek()

WRONG:
  public double findMedian() {
      return maxHeap.poll();  // ← REMOVES the element!
  }

→ poll() removes the top → next findMedian() call returns different value
→ The data structure is DESTROYED by querying it

CORRECT:
  return maxHeap.peek();  // ← Just looks, doesn't remove

📈 SCALABILITY CONSIDERATIONS

Time Complexity:
  → n addNum calls: O(n log n) total
  → n findMedian calls: O(n) total
  → Combined: O(n log n)

Space Complexity:
  → O(n) — all elements stored in heaps

Practical Performance:
  → n = 10^5: 10^5 × log(10^5) ≈ 10^5 × 17 ≈ 1.7M operations → instant
  → n = 10^6: 10^6 × 20 ≈ 20M operations → ~100ms
  → n = 10^7: 10^7 × 23 ≈ 230M operations → ~1-2 seconds

Memory Considerations:
  → Java PriorityQueue stores Integer objects (boxed)
  → Each Integer object: ~16 bytes overhead
  → For n = 10^7: ~160MB → might be tight
  → For extreme cases: consider int-based heap implementation

Comparison with Sorted List:
  → Sorted list: O(n) per add, O(1) per median → O(n²) for n adds
  → Two heaps: O(log n) per add, O(1) per median → O(n log n) for n adds
  → For n = 10^6: sorted list ≈ 10^12 ops → TIMEOUT
  → For n = 10^6: two heaps ≈ 2 × 10^7 ops → fast

🔍 INTERPRETABILITY CONSIDERATIONS

→ The "two halves" metaphor is very intuitive:
     → Imagine a sorted number line
     → Draw a vertical line at the median
     → Left half goes into max-heap (we care about the rightmost = maximum)
     → Right half goes into min-heap (we care about the leftmost = minimum)
     → The median is where the two halves meet

     → Imagine a seesaw balanced at the median
     → Left side (max-heap) holds smaller numbers
     → Right side (min-heap) holds larger numbers
     → Seesaw stays balanced when sizes differ by at most 1
     → Tilts when a side gets too heavy → rebalance by moving one element

🧠 KEY TAKEAWAY

WHAT WE LEARNED:
→ Two heaps split streaming data into two halves at the median
→ Max-heap holds the smaller half → its top is the largest small number
→ Min-heap holds the larger half → its top is the smallest large number
→ The median lives at the TOPS of the two heaps → O(1) access
→ Adding a number requires O(log n) heap operations + rebalancing
→ The three-step flow (add → transfer → rebalance) maintains both invariants

THE TWO INVARIANTS:
  1. ORDER: maxHeap.peek() ≤ minHeap.peek() (all small ≤ all large)
  2. SIZE: maxHeap.size() ≥ minHeap.size() ≥ maxHeap.size() - 1

THE MEDIAN EXTRACTION:
  → Odd count: maxHeap.peek() (the extra middle element)
  → Even count: (maxHeap.peek() + minHeap.peek()) / 2.0

WHY TWO HEAPS IS THE OPTIMAL PATTERN:
  → Sorted list: O(n) add, O(1) median → O(n²) total
  → Two heaps: O(log n) add, O(1) median → O(n log n) total
  → For streaming data: heaps are unbeatable

THE HEAP PATTERN BEGINS:
  → This is the FIRST heap-based project
  → It introduces PriorityQueue in Java
  → It introduces the "maintain-a-property-dynamically" paradigm
  → Future heap projects build directly on this foundation

PATTERN SIGNATURE:
  → "Streaming data" + "median" ✓
  → "Dynamic dataset" + "middle element" ✓
  → "Split into two halves" + "efficient access to split point" ✓
  → = TWO HEAPS

PROJECT 36 (Next Greater Element) shifts to a completely DIFFERENT pattern:
→ MONOTONIC STACK
→ The problem: for each element in an array, find the NEXT element that is GREATER
→ This seems simple but brute force is O(n²)
→ A monotonic stack solves it in O(n)

→ What is a monotonic stack?
     → A stack where elements are maintained in SORTED ORDER (increasing or decreasing)
     → When a new element arrives that VIOLATES the order
     → We POP elements until the order is restored
     → Each popped element has FOUND its "next greater" (or "next smaller")
     → The new element then pushes onto the stack

→ Why it connects to what we've learned:
     → Project 9 (Binary Search) introduced the idea of ELIMINATING candidates
     → A monotonic stack eliminates candidates differently:
          → Binary search eliminates HALF at each step → O(log n)
          → Monotonic stack eliminates elements as they find their answer → amortized O(n)
     → Project 35 (Two Heaps) introduced priority-based data structures
          → Heaps maintain a PARTIAL ORDER (parent ≥ children)
          → Monotonic stacks maintain a TOTAL ORDER (strictly increasing/decreasing)
          → Both are structures that maintain ORDER INVARIANTS

→ Why it matters for later projects:
     → Project 44 (Largest Rectangle in Histogram):
          → Uses monotonic stack to find LEFT and RIGHT boundaries
          → Each element's "next smaller" on both sides defines the rectangle width
     → Project 45 (Trapping Rain Water):
          → Uses monotonic stack to find bounded containers
          → Stack-based approach processes water layer by layer
     → Understanding monotonic stack HERE unlocks BOTH of those hard problems

→ The key mental shift:
     → Heaps: "What is the current best element?" → QUERY-oriented
     → Monotonic stacks: "When does each element find its match?" → EVENT-oriented
     → In the stack: elements WAIT until their answer arrives
     → When the answer arrives: they POP and record the result
     → This "waiting + resolution" pattern is unique to monotonic stacks

The progression into stack-based patterns:
  Project 36: Monotonic stack → next greater element → basic pattern
  Project 44: Monotonic stack → left/right boundaries → area maximization
  Project 45: Monotonic stack → bounded containers → water trapping

→ Project 36 is the GATEWAY to all monotonic stack problems
→ Once you understand "elements wait on the stack for their answer"
→ Every variant (next smaller, previous greater, bounded regions) follows naturally
