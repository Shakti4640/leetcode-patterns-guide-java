PROJECT 7: Detect Cycle in a Linked List

🎯 Fast & Slow Pointers → Tortoise & Hare → Linked List Traversal

🧩 THE HYPOTHETICAL PROBLEM

→ You are given the HEAD of a singly linked list
→ Determine if the linked list has a CYCLE in it
→ A cycle exists if some node's next pointer points BACK to a previous node
→ This creates an infinite loop — traversal never ends
→ Return TRUE if cycle exists, FALSE otherwise
→ Do NOT modify the linked list
→ Use O(1) extra space

Example 1 (Cycle exists):
  3 → 2 → 0 → -4 → (back to 2)
       ↑              |
       └──────────────┘
  Output: true
  Why: Node -4 points back to node 2 → infinite loop

Example 2 (Cycle exists):
  1 → 2 → (back to 1)
  ↑       |
  └───────┘
  Output: true

Example 3 (No cycle):
  1 → 2 → 3 → 4 → null
  Output: false
  Why: Last node points to null → traversal ends naturally

Example 4 (Single node, no cycle):
  1 → null
  Output: false

Example 5 (Single node, self-loop):
  1 → (back to 1)
  ↑    |
  └────┘
  Output: true

🔍 HOW TO APPROACH IT

Step 1 — Understand Linked Lists (New Data Structure)

→ Projects 1-6 worked on ARRAYS:
     → Fixed size, random access (arr[i] in O(1))
     → Can jump to any position instantly
     → Two pointers could be placed at ANY indices

→ Linked Lists are FUNDAMENTALLY different:
     → Each node contains: a VALUE and a NEXT pointer
     → You can ONLY access the next node (no random access)
     → You CANNOT go backward (singly linked)
     → You CANNOT know the size without traversing the entire list
     → The ONLY operation: follow .next to the next node

→ Structure of a node:
     class ListNode {
         int val;
         ListNode next;
     }

→ Traversal: start at head, follow .next until null
     current = head
     while (current != null):
         process(current)
         current = current.next

→ If there's a CYCLE: current.next never becomes null → INFINITE LOOP
→ The challenge: how do we DETECT this without getting stuck?

Step 2 — Think About the Brute Force First

→ Naive approach: Track every node we've visited
     Create a HashSet
     current = head
     while (current != null):
         if current is in HashSet:
             return true    // we've visited this node before → cycle!
         add current to HashSet
         current = current.next
     return false           // reached null → no cycle

→ This WORKS:
     → Time: O(N) — visit each node once
     → Space: O(N) — store every node in the HashSet

→ But the problem asks for O(1) space
→ We can't store visited nodes
→ How can we detect revisiting without memory?

Step 3 — The Key Insight: Different Speeds

→ Imagine two runners on a CIRCULAR track:
     → Runner A (slow): runs at 1 mph
     → Runner B (fast): runs at 2 mph
     → If the track is circular → B will eventually LAP A → they MEET
     → If the track is straight (no loop) → B reaches the end first → they NEVER meet

→ This is EXACTLY our linked list situation:
     → Slow pointer: moves 1 node per step
     → Fast pointer: moves 2 nodes per step
     → If there's a cycle: fast laps slow → they meet at some node
     → If there's no cycle: fast reaches null → they never meet

→ This is called the TORTOISE AND HARE algorithm (Floyd's algorithm)
     → Tortoise = slow pointer
     → Hare = fast pointer
     → Named after Aesop's fable, but here the hare ALWAYS wins (catches up)

Step 4 — Why Do They Definitely Meet?

→ Suppose the cycle has length C
→ When slow enters the cycle, fast is already somewhere inside it
→ The RELATIVE SPEED of fast vs slow = 2 - 1 = 1 node per step
→ So from slow's perspective, fast approaches by 1 node per step
→ The distance between them decreases by 1 each step
→ Eventually the distance becomes 0 → they are at the SAME node → MEETING

→ More precisely:
     → Let's say when slow enters the cycle, fast is D nodes ahead
     → Each step: the gap between them (within the cycle) decreases by 1
     → After D steps: gap = 0 → they meet
     → D ≤ C (cycle length) → meeting happens within C steps of slow entering
     → Total steps ≤ N → O(N) time guaranteed

→ KEY: They MUST meet. It's not "might" — it's mathematically certain.
     → The relative speed of 1 means the gap shrinks by exactly 1 each step
     → There's no way to "skip over" each other
     → They converge with certainty

Step 5 — Visualize the Meeting

List: 3 → 2 → 0 → -4 → (back to 2)

Nodes indexed: 
  Node 0: val=3, next→Node 1
  Node 1: val=2, next→Node 2
  Node 2: val=0, next→Node 3
  Node 3: val=-4, next→Node 1 (CYCLE)

Step 0: slow=Node 0(3), fast=Node 0(3)
Step 1: slow=Node 1(2), fast=Node 2(0)     [slow +1, fast +2]
Step 2: slow=Node 2(0), fast=Node 1(2)     [fast went 2→3→1, wrapping around]
Step 3: slow=Node 3(-4), fast=Node 3(-4)   [fast went 1→2→3]
→ slow == fast → CYCLE DETECTED at Node 3 ✓

No cycle list: 1 → 2 → 3 → 4 → null

Step 0: slow=Node 0(1), fast=Node 0(1)
Step 1: slow=Node 1(2), fast=Node 2(3)
Step 2: slow=Node 2(3), fast=null          [fast went 3→4→null... actually fast=Node 4, fast.next=null]
→ Actually: fast=Node 2(3), fast.next=Node 3(4), fast.next.next=null
→ Step 2: slow=Node 1(2), fast reaches beyond → fast or fast.next is null → NO CYCLE

🛠️ HOW CAN WE SOLVE IT

The Algorithm — Line by Line Logic

1. Handle edge case: if head is null → return false (empty list)
2. Initialize slow pointer at head
3. Initialize fast pointer at head
4. WHILE fast is not null AND fast.next is not null:
     a. slow = slow.next           ← move 1 step
     b. fast = fast.next.next      ← move 2 steps
     c. IF slow == fast:
          → CYCLE DETECTED → return true
5. Loop ended (fast reached null) → return false (no cycle)

Why Check fast != null AND fast.next != null?

→ fast moves 2 steps: fast = fast.next.next
→ If fast is null → fast.next throws NullPointerException
→ If fast.next is null → fast.next.next throws NullPointerException
→ Both must be non-null for the 2-step move to be safe

→ We DON'T check slow for null because:
     → slow moves 1 step per iteration
     → fast moves 2 steps per iteration
     → fast always reaches null BEFORE slow
     → If fast is non-null → slow is definitely non-null
     → fast's null check PROTECTS slow automatically

✅ WHAT SHOULD YOU DO

→ ALWAYS check fast AND fast.next before moving fast
     → Two-step move requires both to be non-null
     → Missing either check → NullPointerException

→ ALWAYS start both pointers at head
     → Some implementations start slow at head, fast at head.next
     → This works too but changes the loop structure slightly
     → Starting both at head is cleaner and more standard

→ ALWAYS compare pointers (references), not values
     → slow == fast checks if they point to the SAME NODE
     → slow.val == fast.val checks if values are equal → WRONG
     → Two different nodes can have the same value
     → We need REFERENCE equality, not value equality

→ ALWAYS handle edge cases:
     → null head → no list → no cycle → return false
     → Single node with no self-loop → no cycle → return false
     → Single node with self-loop → cycle → return true

→ ALWAYS remember: this detects cycle EXISTENCE only
     → It tells you YES or NO
     → It does NOT tell you WHERE the cycle starts (that's Project 20)
     → It does NOT tell you the cycle LENGTH (that's an extension)

🚫 WHAT SHOULD YOU NOT DO

→ DO NOT use a HashSet (unless O(1) space isn't required)
     → HashSet works but uses O(N) space
     → Fast/slow pointers achieve O(1) space
     → When the problem says "constant space" → use fast/slow

→ DO NOT modify the linked list
     → Some solutions mark visited nodes by changing values
     → This destroys the original data
     → Problem typically says "do not modify the list"
     → Fast/slow is non-destructive — read-only traversal

→ DO NOT compare values instead of references
     BAD:  if (slow.val == fast.val) → could match on different nodes
     GOOD: if (slow == fast) → same exact node in memory

→ DO NOT check only fast != null (forgetting fast.next != null)
     BAD:  while (fast != null) {
               fast = fast.next.next;  // crashes if fast.next is null
           }
     GOOD: while (fast != null && fast.next != null) {
               fast = fast.next.next;  // safe
           }

→ DO NOT move pointers before comparing (in same-start variant)
     → If both start at head and you compare BEFORE moving:
          → slow == fast is ALWAYS true initially → returns true for all lists
     → Move FIRST, then compare

→ DO NOT try to count nodes to detect cycle
     → "If I've visited more than N nodes, there must be a cycle"
     → But you don't KNOW N without traversing (which may be infinite)
     → Chicken-and-egg problem → doesn't work

🔀 WHEN TO USE THIS PATTERN

✓ Detecting CYCLES in linked lists
✓ Finding the MIDDLE node of a linked list (Project 15)
     → When fast reaches the end, slow is at the middle
✓ Finding the START of a cycle (Project 20)
     → Phase 1: detect meeting point
     → Phase 2: find entry point
✓ Finding the Kth node from the end
     → Move fast K steps ahead, then move both at same speed
     → When fast reaches end, slow is K from end
✓ Determining if a linked list is a palindrome
     → Find middle, reverse second half, compare
✓ Any problem where TWO SPEEDS reveal structural properties
✓ Detecting cycles in sequences/functions (not just linked lists)
     → Any repeated function application: x → f(x) → f(f(x)) → ...
     → If values start repeating → cycle in the function

🚫 WHEN NOT TO USE THIS PATTERN

✗ The data structure is an ARRAY (not a linked list)
     → Arrays have random access → use index-based two pointers
     → Fast/slow on arrays is possible but usually unnecessary
     → Exception: cyclic sort (Project 8) uses array indices as "pointers"

✗ You need to detect cycles in a GRAPH (not a linked list)
     → Graphs have multiple edges per node → fast/slow doesn't generalize
     → Use DFS with visited set or coloring (Project 47)

✗ You need to find ALL cycles (not just detect one)
     → Fast/slow finds ONE meeting point in ONE cycle
     → For all cycles → need graph traversal algorithms

✗ The linked list is DOUBLY linked
     → Can traverse both directions → different techniques available
     → Fast/slow still works but may not be the best approach

✗ You need random access to list elements
     → Linked lists don't support O(1) indexing
     → If you need arr[i]-style access → convert to array first
     → Or use a different data structure

⚙️ HOW IT ONLY WORKS — Internal Mechanics

The Relative Speed Proof

SETUP:
→ Let the list have a "tail" of length T (nodes before the cycle)
→ Let the cycle have length C

WHEN SLOW ENTERS THE CYCLE:
→ Slow has taken T steps (one per step)
→ Fast has taken 2T steps (two per step)
→ Fast is already inside the cycle
→ Fast's position in cycle = (2T - T) mod C = T mod C

DISTANCE BETWEEN THEM (within cycle):
→ Slow is at position 0 (just entered)
→ Fast is at position T mod C
→ Distance from slow to fast (going forward) = T mod C
→ Distance from fast to slow (going forward) = C - (T mod C)

CLOSING THE GAP:
→ Each step: slow moves +1, fast moves +2
→ Relative speed of fast vs slow = +1 (within cycle)
→ Fast "chases" slow at relative speed 1
→ Distance from fast to slow decreases by 1 each step
→ After C - (T mod C) steps: distance = 0 → THEY MEET

GUARANTEED MEETING:
→ Meeting happens within C steps after slow enters the cycle
→ Total steps ≤ T + C ≤ N → O(N) time
→ They MUST meet — there's no scenario where they don't
→ The math is deterministic, not probabilistic

What If Fast Moved at Speed 3?

→ If fast moved 3 steps per iteration:
     → Relative speed = 3 - 1 = 2
     → Gap decreases by 2 each step
     → If initial gap is ODD → gap goes: 5, 3, 1, -1 (wraps around)
     → They might SKIP past each other!

→ Speed 2 is special because:
     → Relative speed = 1
     → Gap decreases by exactly 1 each step
     → Gap goes: D, D-1, D-2, ..., 2, 1, 0
     → They MUST reach gap=0 → guaranteed meeting
     → No skipping possible

→ This is why fast MUST move at exactly 2x slow's speed
→ Higher speeds can work but require more complex analysis
→ Speed 2 gives the simplest, most reliable algorithm

Why O(1) Space

→ Only two pointer variables: slow and fast
→ No HashSet, no visited array, no extra data structures
→ The nodes themselves are not modified
→ The pointers just traverse the existing structure

→ Compare with HashSet approach:
     → HashSet stores references to up to N nodes
     → Each reference uses memory
     → Total: O(N) space

→ Fast/slow: O(1) space regardless of list size
→ List could have 1 billion nodes → still just 2 pointers

Connection to Previous Patterns

PROJECT 1-2 (Convergence):
→ Two pointers start at opposite ends, move toward each other
→ They meet in the MIDDLE

PROJECT 3 (Read/Write):
→ Two pointers start at same end, move in same direction
→ One is faster (read) and one is slower (write)

PROJECT 5-6 (Sliding Window):
→ Two pointers define window boundaries
→ They move forward with fixed or variable gap

PROJECT 7 (Fast/Slow):
→ Two pointers start at same position, move in same direction
→ One moves FASTER than the other
→ If there's a cycle → they MEET (like convergence, but in a loop)
→ If there's no cycle → fast reaches the end (like read/write, fast is ahead)

→ Fast/slow is a HYBRID concept:
     → Same-direction movement (like read/write)
     → Meeting detection (like convergence)
     → Speed differential creates the detection mechanism

⚠️ COMMON PITFALLS AND FAILURE MODES

Pitfall 1: Comparing Before Moving

BAD (both start at head):
  slow = head; fast = head;
  while (fast != null && fast.next != null) {
      if (slow == fast) return true;  // ALWAYS true on first check!
      slow = slow.next;
      fast = fast.next.next;
  }

→ First iteration: slow == fast == head → returns true for ALL lists

GOOD:
  slow = head; fast = head;
  while (fast != null && fast.next != null) {
      slow = slow.next;              // move FIRST
      fast = fast.next.next;
      if (slow == fast) return true;  // THEN compare
  }

→ After first move: slow and fast are at different positions (unless cycle)

Pitfall 2: Forgetting fast.next Check

List: 1 → 2 → 3 → null

Step 1: slow=2, fast=3
Step 2: fast.next = null → fast.next.next throws NullPointerException!

→ Must check: fast != null && fast.next != null
→ The && short-circuits: if fast is null, fast.next is never evaluated
→ This is safe in Java due to short-circuit evaluation

Pitfall 3: Confusing Reference Equality with Value Equality

List: 1 → 2 → 1 → null (no cycle, but values repeat)

BAD: if (slow.val == fast.val) → returns true when slow is at first 1 
     and fast is at last 1 → WRONG (no cycle, just same value)

GOOD: if (slow == fast) → checks if they point to the SAME NODE object
     → Only true if they're literally at the same node in memory

Pitfall 4: Single Node Edge Cases

Case 1: head → null (no cycle)
→ fast = head, fast.next = null
→ While condition: fast != null (true) && fast.next != null (false)
→ Loop doesn't execute → return false ✓

Case 2: head → head (self-loop)
→ fast = head, fast.next = head
→ While condition: both non-null → enter loop
→ slow = head.next = head, fast = head.next.next = head
→ slow == fast → return true ✓

→ Both cases handled correctly by the algorithm

Pitfall 5: Assuming Meeting Point is Cycle Start

→ Where slow and fast MEET is NOT necessarily where the cycle STARTS
→ The meeting point is somewhere inside the cycle
→ To find the cycle START → need additional logic (Project 20)
→ This project only detects EXISTENCE, not entry point

📈 SCALABILITY CONSIDERATIONS

→ List size N = 100 → both HashSet and fast/slow are fast
→ List size N = 1,000,000:
     → HashSet: 1M entries × ~40 bytes each ≈ 40 MB extra memory
     → Fast/slow: 2 pointers ≈ 16 bytes extra memory
     → 2,500,000x less memory!

→ List size N = 100,000,000:
     → HashSet: 4 GB+ → may exceed available memory
     → Fast/slow: still 16 bytes → runs perfectly
     → This is not a theoretical advantage — it's a PRACTICAL necessity

→ Time: both approaches are O(N)
→ Space: HashSet is O(N), fast/slow is O(1)
→ For memory-constrained environments → fast/slow is the ONLY option

→ REAL-WORLD applications:
     → Detecting infinite loops in programs (state = node, transition = next)
     → Detecting cycles in garbage collection (reference chains)
     → Network routing loop detection (packet path = linked list)
     → Database deadlock detection (wait-for chains)
     → Pseudorandom number generator cycle detection

🔍 INTERPRETABILITY CONSIDERATIONS

→ The running track analogy makes this immediately intuitive:
     → Two runners on a track
     → One runs twice as fast
     → If the track loops → faster runner catches slower one
     → If the track is straight → faster runner finishes first

→ The algorithm is EXTREMELY simple:
     → Move slow by 1, fast by 2
     → If they meet → cycle
     → If fast reaches null → no cycle
     → Three lines of logic inside the loop

→ The mathematical proof is elegant:
     → Relative speed = 1 → gap closes by 1 each step
     → Gap eventually reaches 0 → guaranteed meeting
     → No probability, no approximation → deterministic certainty

→ Debugging is straightforward:
     → Print slow position, fast position at each step
     → If they meet → cycle found
     → If fast reaches null → no cycle
     → The trace is short and clear

→ Interview explanation:
     → "I use Floyd's cycle detection with two pointers"
     → "Slow moves 1 step, fast moves 2 steps"
     → "If there's a cycle, fast catches slow"
     → "O(N) time, O(1) space"
     → One of the most elegant algorithms to explain

THE FAST/SLOW POINTER TEMPLATE:
  ListNode slow = head;
  ListNode fast = head;
  
  while (fast != null && fast.next != null) {
      slow = slow.next;           // 1 step
      fast = fast.next.next;      // 2 steps
      
      if (slow == fast) {
          // CYCLE DETECTED — or meeting point found
      }
  }
  // If loop exits: NO CYCLE — fast reached the end
