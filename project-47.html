PROJECT 47: Course Schedule — Cycle Detection in DAG

🎯 Topological Sort → In-Degree Counting → Kahn's BFS Algorithm

🧩 THE HYPOTHETICAL PROBLEM

→ You are given N courses labeled 0 to N-1
→ You are given a list of PREREQUISITE pairs
→ prerequisites[i] = [courseA, courseB]
     → Meaning: to take courseA, you MUST first complete courseB
     → Directed edge: courseB → courseA

→ QUESTION 1 (Course Schedule I — LeetCode 207):
     → Can you finish ALL courses?
     → Return TRUE or FALSE
     → FALSE if there's a CIRCULAR dependency (cycle)

→ QUESTION 2 (Course Schedule II — LeetCode 210):
     → Return the ORDER in which to take courses
     → Return empty array if impossible (cycle exists)

Example 1:
  numCourses = 4
  prerequisites = [[1,0], [2,0], [3,1], [3,2]]
  
  Meaning:
    → Course 1 requires Course 0
    → Course 2 requires Course 0
    → Course 3 requires Course 1
    → Course 3 requires Course 2
  
  Graph:
    0 → 1 → 3
    0 → 2 → 3
  
  Valid order: [0, 1, 2, 3] or [0, 2, 1, 3]
  Answer (can finish?): TRUE

Example 2:
  numCourses = 2
  prerequisites = [[1,0], [0,1]]
  
  Meaning:
    → Course 1 requires Course 0
    → Course 0 requires Course 1
    → CIRCULAR → impossible
  
  Graph:
    0 → 1 → 0 → 1 → ... (cycle!)
  
  Answer (can finish?): FALSE

Example 3:
  numCourses = 3
  prerequisites = [[1,0]]
  
  Meaning:
    → Course 1 requires Course 0
    → Course 2 has NO prerequisites
  
  Valid order: [0, 2, 1] or [2, 0, 1] or [0, 1, 2]
  Answer: TRUE

Example 4:
  numCourses = 1
  prerequisites = []
  
  → One course, no prerequisites → trivially possible
  Answer: TRUE

🔍 HOW TO APPROACH IT

Step 1 — Recognize This is a GRAPH Problem

→ Courses = NODES in a graph
→ Prerequisites = DIRECTED EDGES
→ "courseB must come before courseA" = edge from B to A
→ The entire problem is about ORDERING nodes in a DIRECTED GRAPH

→ This is NOT an interval problem (Project 46)
→ This is NOT a tree problem (Projects 17, 18)
→ This is a GENERAL DIRECTED GRAPH problem
→ Trees have one parent per node → graphs can have MANY
→ Trees have no cycles → graphs CAN have cycles

Step 2 — Understand What "Topological Sort" Means

DEFINITION:
  → A topological ordering of a directed graph is a LINEAR ordering
    of its vertices such that for every directed edge u → v,
    vertex u comes BEFORE vertex v in the ordering

EXAMPLE:
  Edges: 0→1, 0→2, 1→3, 2→3
  
  Topological orders (multiple valid):
    [0, 1, 2, 3] ✓  (0 before 1, 0 before 2, 1 before 3, 2 before 3)
    [0, 2, 1, 3] ✓  (same constraints satisfied)
    [1, 0, 2, 3] ✗  (1 before 0, but edge says 0 must come before 1)

KEY PROPERTY:
  → Topological sort exists IF AND ONLY IF the graph has NO CYCLES
  → A directed graph with no cycles = DAG (Directed Acyclic Graph)
  → If a cycle exists → no valid ordering → return FALSE / empty array

Step 3 — Why Cycles Make It Impossible

Cycle: A → B → C → A

→ A must come before B (edge A→B)
→ B must come before C (edge B→C)
→ C must come before A (edge C→A)
→ So A must come before B, before C, before A
→ A must come before ITSELF → IMPOSSIBLE
→ No linear ordering can satisfy circular dependencies

Step 4 — Connect to Previous Knowledge

Project 17 (Tree BFS):
  → We processed nodes LEVEL BY LEVEL using a queue
  → Level 0 first, then level 1, then level 2...
  → Each level's nodes were "ready" because their parent was processed

THIS Problem:
  → Same idea but on a GENERAL DIRECTED GRAPH
  → Instead of "parent processed" → "ALL prerequisites completed"
  → Instead of "level" → "wave of courses that become available"
  → Instead of "single root" → "multiple starting nodes" (no prerequisites)
  
  → BFS on trees = special case of topological sort on DAGs
  → Trees guarantee no cycles → topological sort always exists
  → Graphs might have cycles → must DETECT them

Project 19 (Activity Selection / Greedy):
  → We processed events in a specific ORDER to make greedy choices
  → The ORDER was determined by sorting (by end time)
  
THIS Problem:
  → We process courses in a specific ORDER determined by DEPENDENCIES
  → The ORDER is not from sorting by a value
  → It's from the STRUCTURE of the dependency graph
  → Topological sort IS the ordering algorithm

Step 5 — The Key Concept: In-Degree

IN-DEGREE of a node = number of INCOMING edges
  → How many prerequisites does this course have?
  → How many courses must be completed before this one?

IN-DEGREE = 0 means:
  → This course has NO prerequisites
  → It can be taken IMMEDIATELY
  → It's a STARTING POINT (like a root in a tree)

THE INSIGHT:
  → Start with ALL nodes that have in-degree 0
  → Process them (take those courses)
  → For each processed node, REMOVE its outgoing edges
  → This REDUCES the in-degree of its neighbors
  → Some neighbors may now have in-degree 0 → they become available
  → Repeat until no more nodes can be processed
  → If ALL nodes processed → valid ordering exists → return TRUE
  → If some nodes remain → they're in a cycle → return FALSE

  → Remove edges 0→2 and 0→3
  → in-degree[2] = 1→0 → enqueue 2
  → in-degree[3] = 1→0 → enqueue 3

  → Order so far: [0]

  → Remove edge 1→4
  → in-degree[4] = 2→1 → not 0 yet
  → Queue: [2, 3]
  → Order: [0, 1]

STEP 4: Process node 2
  → Remove edge 2→4
  → in-degree[4] = 1→0 → enqueue 4
  → Queue: [3, 4]
  → Order: [0, 1, 2]

  → Remove edge 3→5
  → in-degree[5] = 2→1 → not 0 yet
  → Queue: [4]

  → in-degree[5] = 1→0 → enqueue 5

Step 7 — Visualize Cycle Detection

Courses: 0, 1, 2
Prerequisites: [1,0], [2,1], [0,2]

Graph:
  0 → 1 → 2 → 0  (CYCLE!)

In-degrees:
  0: 1 (needs 2)
  1: 1 (needs 0)
  2: 1 (needs 1)

STEP 1: Enqueue all nodes with in-degree 0
  → NONE! Every node has in-degree ≥ 1
  → Queue: []

STEP 2: Queue is empty → nothing to process
  → Processed count = 0
  → Total nodes = 3
  → 0 ≠ 3 → CYCLE DETECTED → return FALSE

WHY IT WORKS:
  → In a cycle, every node depends on at least one other node in the cycle
  → No node in the cycle ever reaches in-degree 0
  → They are NEVER enqueued
  → They are NEVER processed
  → Unprocessed nodes = nodes trapped in cycles

🛠️ HOW CAN WE SOLVE IT

Kahn's Algorithm — Step by Step

1. BUILD the graph:
     → Adjacency list: for each node, list of nodes it points TO
     → In-degree array: for each node, count of incoming edges

2. INITIALIZE the queue:
     → Find all nodes with in-degree 0
     → Add them to a queue (BFS starting points)

3. BFS PROCESSING:
     → While queue is not empty:
          a. Dequeue a node → add to result order
          b. For each neighbor of this node:
               → Decrement neighbor's in-degree by 1
               → If neighbor's in-degree becomes 0 → enqueue it
          c. Increment processedCount

4. CYCLE CHECK:
     → If processedCount == numCourses → all courses can be taken → TRUE
     → If processedCount < numCourses → cycle exists → FALSE

5. RETURN:
     → For Course Schedule I: return boolean
     → For Course Schedule II: return the order array (or empty if cycle)

✅ WHAT SHOULD YOU DO

→ ALWAYS build BOTH the adjacency list AND the in-degree array
     → Adjacency list: tells you WHERE to go from each node
     → In-degree array: tells you WHEN a node becomes available
     → Both are essential — one without the other is incomplete

→ ALWAYS use a QUEUE (FIFO) for Kahn's algorithm
     → BFS order → processes nodes in "waves" of availability
     → Could technically use a stack → gives valid topological order too
     → But queue gives the more intuitive level-by-level processing

→ ALWAYS count processed nodes and compare with total
     → This is the CYCLE DETECTION mechanism
     → If count < total → some nodes never reached in-degree 0 → cycle

→ ALWAYS handle disconnected components
     → Multiple courses with no prerequisites → multiple starting points
     → All get enqueued initially → all components processed
     → Kahn's handles this naturally

→ ALWAYS handle edge cases
     → 0 courses → return true / empty array
     → No prerequisites → all courses independent → any order works
     → Self-loop [0, 0] → "course 0 requires course 0" → cycle

🚫 WHAT SHOULD YOU NOT DO

→ DO NOT confuse edge direction
     → prerequisites[i] = [a, b] means b → a (b must come before a)
     → NOT a → b
     → Getting this backward = wrong graph = wrong answer
     → READ THE PROBLEM: "to take course a, you must first complete course b"
     → So the DEPENDENCY flows: b must precede a → edge from b to a

→ DO NOT use DFS without proper state tracking
     → DFS-based topological sort exists but needs 3-state marking:
          → UNVISITED, IN_PROGRESS, COMPLETED
     → If you encounter an IN_PROGRESS node → cycle detected
     → Missing the IN_PROGRESS state → cycles go undetected
     → Kahn's BFS is simpler and less error-prone

→ DO NOT try to sort prerequisites by some numeric property
     → This is NOT an interval sorting problem
     → The ordering comes from GRAPH STRUCTURE, not numeric values
     → Project 14 sorts by start time → this problem sorts by TOPOLOGY

→ DO NOT assume the graph is connected
     → Courses can form multiple disconnected groups
     → Course 0→1 and Course 2→3 → two separate components
     → Both must be processed → Kahn's handles this automatically

→ DO NOT modify the in-degree array and then try to reuse it
     → After Kahn's, in-degree values are modified
     → If you need original in-degrees later → make a copy first

🔀 WHEN TO USE TOPOLOGICAL SORT

✓ You have DIRECTED dependencies (A must come before B)
✓ You need to find a VALID ORDERING that respects all dependencies
✓ You need to DETECT CYCLES in a directed graph
✓ The problem involves SCHEDULING with prerequisites
✓ The problem involves BUILD ORDER (compile A before B)
✓ The problem involves TASK SEQUENCING with constraints

REAL-WORLD ANALOGS:
  → University course prerequisites
  → Software build dependencies (Makefile, Gradle)
  → Package installation order (npm, pip, apt)
  → Spreadsheet cell dependency evaluation
  → Job scheduling in pipelines
  → Compilation order in multi-file projects
  → Recipe step ordering (chop before fry before serve)

🚫 WHEN NOT TO USE TOPOLOGICAL SORT

✗ Graph is UNDIRECTED
     → Topological sort is defined only for DIRECTED graphs
     → Undirected graphs → use BFS/DFS for traversal, Union-Find for components

✗ Graph has no ordering constraint
     → If you just need to VISIT all nodes → plain BFS/DFS suffices
     → Topological sort adds unnecessary complexity

✗ You need the SHORTEST PATH
     → Topological sort gives ORDERING, not shortest path
     → Use Dijkstra or BFS (unweighted) for shortest paths

✗ Edges are BIDIRECTIONAL dependencies
     → A depends on B AND B depends on A → always a cycle → impossible
     → Topological sort would immediately detect this

✗ You need to find STRONGLY CONNECTED COMPONENTS
     → Topological sort detects cycles exist
     → To find WHICH nodes are in cycles → use Tarjan's or Kosaraju's algorithm
     → Different (more advanced) technique

⚙️ HOW IT ONLY WORKS — Internal Mechanics

The In-Degree Invariant

AT EVERY STEP of Kahn's algorithm:

→ Nodes in the queue have in-degree 0 (in the REMAINING graph)
→ In-degree 0 = all their prerequisites have been PROCESSED
→ They are SAFE to take now — no unresolved dependencies

WHEN WE PROCESS A NODE:
→ We "remove" it from the graph conceptually
→ All edges FROM this node are removed
→ Neighbors' in-degrees decrease
→ Some neighbors may reach in-degree 0 → they join the queue

THIS IS IDENTICAL TO:
→ Peeling off the "outermost layer" of the graph
→ Like peeling an onion — layer by layer
→ Each layer = nodes with no remaining dependencies
→ Process layer → reveal next layer → repeat

Why This Detects Cycles

In a cycle: A → B → C → A

→ in-degree(A) ≥ 1 (C points to A)
→ in-degree(B) ≥ 1 (A points to B)
→ in-degree(C) ≥ 1 (B points to C)

→ None of them start with in-degree 0
→ None of them are ever enqueued
→ Processing other (non-cycle) nodes doesn't help:
     → Removing edges FROM non-cycle nodes doesn't affect in-degrees
       of nodes IN the cycle (because no non-cycle node points into the cycle
       ... wait, a non-cycle node CAN point into a cycle)

CORRECTION — More precise:
→ Even if external nodes point INTO the cycle, those edges are
  in ADDITION to the cycle edges
→ Removing external edges reduces in-degrees but cycle edges remain
→ In-degree of cycle nodes never reaches 0 because:
     → Each cycle node has at least one in-cycle predecessor
     → That predecessor is also in the cycle → never processed
     → So that edge is never removed
     → So in-degree never reaches 0

FORMAL:
→ In a cycle, node X has predecessor Y (also in cycle)
→ Y is never processed (same argument applies to Y)
→ Edge Y→X is never removed
→ in-degree(X) never reaches 0
→ X is never enqueued → never processed
→ processedCount < totalNodes → cycle detected

Kahn's vs DFS-Based Topological Sort

KAHN'S (BFS-based):
  → Start from nodes with no dependencies
  → Process outward layer by layer
  → Naturally produces FORWARD ordering
  → Cycle detection: processedCount < totalNodes
  → Easier to implement correctly
  → Better for "level-by-level" processing

DFS-BASED:
  → Start from any node, go as deep as possible
  → When backtracking, add node to result (REVERSE postorder)
  → Result is in REVERSE → must reverse at end
  → Cycle detection: encountering IN_PROGRESS node
  → Three-state marking required (more error-prone)
  → Better for "path-based" reasoning

FOR INTERVIEWS:
  → Kahn's BFS is recommended — cleaner, less error-prone
  → Directly builds on BFS knowledge from Project 17
  → We implement BOTH for completeness

Time Complexity Breakdown

→ Building adjacency list: O(E) where E = number of edges (prerequisites)
→ Computing in-degrees: O(E)
→ Initial queue population: O(V) where V = number of nodes (courses)
→ BFS processing:
     → Each node dequeued exactly once: O(V)
     → Each edge examined exactly once (when source node processed): O(E)
→ TOTAL: O(V + E)

→ For this problem: V = numCourses, E = prerequisites.length
→ O(numCourses + prerequisites.length)

Space Complexity Breakdown

→ Adjacency list: O(V + E)
→ In-degree array: O(V)
→ Queue: at most O(V) nodes
→ Result array: O(V)
→ TOTAL: O(V + E)

⚠️ COMMON PITFALLS AND FAILURE MODES

Pitfall 1: Edge Direction Reversal

prerequisites = [[1, 0]]
→ "To take course 1, complete course 0 first"
→ Dependency: 0 must come before 1
→ Edge: 0 → 1

COMMON MISTAKE:
  → Reading [1, 0] as "edge from 1 to 0"
  → This reverses the direction
  → 1 would have in-degree 0 and be processed first
  → But 1 REQUIRES 0 → should be processed AFTER 0

FIX:
  → For each [a, b]: edge goes FROM b TO a
  → adjacencyList[b].add(a)
  → inDegree[a]++

Pitfall 2: Self-Loops

prerequisites = [[0, 0]]
→ "Course 0 requires course 0"
→ This is a self-loop → a cycle of length 1
→ in-degree[0] = 1 → never reaches 0 → never processed
→ Kahn's algorithm handles this correctly automatically
→ No special case needed

Pitfall 3: Duplicate Prerequisites

prerequisites = [[1, 0], [1, 0]]
→ Same edge listed twice
→ If you add both: in-degree[1] = 2
→ When processing node 0: in-degree[1] decremented twice → becomes 0 ✓
→ OR: in-degree[1] decremented once (if adjacency list has duplicate) → becomes 1 ✗

FIX:
  → Either deduplicate prerequisites first
  → OR ensure adjacency list stores duplicates AND in-degree counts duplicates consistently
  → Standard approach: just add duplicates to adjacency list
     → If edge 0→1 appears twice, adjacencyList[0] has 1 twice
     → in-degree[1] = 2
     → When processing 0, decrement for EACH entry → 2 decrements → 0 ✓

Pitfall 4: Disconnected Nodes

numCourses = 5
prerequisites = [[1, 0]]

→ Only courses 0 and 1 are connected
→ Courses 2, 3, 4 are ISOLATED (no prerequisites, no dependents)
→ in-degree: [0, 1, 0, 0, 0]
→ Initial queue: [0, 2, 3, 4] → all processed immediately
→ Then 1 becomes available → processed
→ Total processed = 5 = numCourses → TRUE ✓

→ Kahn's handles disconnected nodes naturally
→ They have in-degree 0 → enqueued immediately → processed

Pitfall 5: Confusing Course Schedule I and II

Course Schedule I (LeetCode 207):
  → Return BOOLEAN: can you finish all courses?
  → Only need processedCount == numCourses

Course Schedule II (LeetCode 210):
  → Return the ORDER array
  → Return empty array if impossible
  → Need to actually BUILD the order during BFS

📈 SCALABILITY CONSIDERATIONS

V = numCourses, E = number of prerequisites

→ V = 100, E = 200 → O(300) → instant
→ V = 10,000, E = 50,000 → O(60,000) → instant
→ V = 100,000, E = 500,000 → O(600,000) → fast
→ V = 1,000,000, E = 5,000,000 → O(6,000,000) → still manageable

→ Kahn's algorithm is LINEAR in graph size → scales excellently
→ Bottleneck is usually graph CONSTRUCTION, not the BFS itself

→ For very large graphs:
     → Adjacency list representation is essential (not adjacency matrix)
     → Adjacency matrix would be O(V²) space → prohibitive for V > 10,000
     → Our implementation uses adjacency list → O(V + E) space → optimal

→ For dense graphs (E close to V²):
     → O(V + E) ≈ O(V²) → still manageable for V up to ~10,000
     → But rarely encountered in practice — most dependency graphs are sparse
