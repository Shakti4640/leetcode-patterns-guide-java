PROJECT 13: Fruits into Baskets

🎯 Sliding Window → At-Most-K-Distinct Constraint → Disguised Problem Recognition

🔗 DEPENDENCY CHAIN

Project 5  → Sliding Window → fixed-size → running sum
Project 6  → Sliding Window → variable-size → shrink when condition violated
Project 12 → Sliding Window + HashMap → character frequency tracking → at most K distinct
Project 13 → SAME algorithm as Project 12 with K=2
             → applied to INTEGER ARRAY instead of string
             → teaches the CRITICAL skill: recognizing disguised problems
             → the algorithm is NOT new — the RECOGNITION is new

🧩 THE HYPOTHETICAL PROBLEM

→ You are visiting a row of fruit trees arranged in a line
→ Each tree produces exactly ONE type of fruit
→ The trees are represented as an integer array: fruits[i] = type of fruit at tree i
→ You have TWO baskets
→ Each basket can hold ONLY ONE type of fruit (unlimited quantity of that type)
→ You must pick fruits from a CONTIGUOUS section of trees
→ You start at any tree and pick from every tree moving RIGHT until you must stop
→ You stop when you encounter a THIRD type of fruit (your baskets can't hold it)
→ Find the MAXIMUM number of fruits you can collect

Example 1:
  Input:  fruits = [1, 2, 1]
  Output: 3
  → Pick all three → types {1, 2} → 2 baskets → fits ✓
  → Total fruits: 3

Example 2:
  Input:  fruits = [0, 1, 2, 2]
  Output: 3
  → [1, 2, 2] → types {1, 2} → 2 baskets → 3 fruits ✓
  → [0, 1, 2] → types {0, 1, 2} → 3 types → doesn't fit ✗

Example 3:
  Input:  fruits = [1, 2, 3, 2, 2]
  Output: 4
  → [2, 3, 2, 2] → types {2, 3} → 2 baskets → 4 fruits ✓

Example 4:
  Input:  fruits = [3, 3, 3, 1, 2, 1, 1, 2, 3, 3, 4]
  Output: 5
  → [1, 2, 1, 1, 2] → types {1, 2} → 2 baskets → 5 fruits ✓

🔍 HOW TO APPROACH IT

Step 1 — Strip Away the Story → Find the Core Problem

THE STORY SAYS:
  → "Row of fruit trees" → array of integers
  → "Type of fruit" → integer value at each index
  → "Two baskets, one type each" → at most 2 DISTINCT values
  → "Contiguous section" → subarray (not subsequence)
  → "Maximum fruits" → longest valid subarray

THE CORE PROBLEM IS:
  → Find the longest contiguous subarray with at most 2 distinct elements

COMPARE WITH PROJECT 12:
  → Project 12: longest SUBSTRING with at most K distinct CHARACTERS
  → Project 13: longest SUBARRAY with at most 2 distinct INTEGERS
  
  Mapping:
    substring → subarray          (same concept, different data type)
    characters → integers          (same concept, different data type)
    K → 2                          (specific value of K)
    string → integer array         (same concept, different data type)

→ This is Project 12 with K hardcoded to 2 and applied to integers
→ The algorithm is IDENTICAL
→ The ONLY new skill here is RECOGNIZING this equivalence

Step 2 — Why Recognition Is the Hardest Part

IN AN INTERVIEW:
  → Nobody says "solve this sliding window problem with K=2 distinct"
  → They say "you have baskets and fruit trees and..."
  → Your job is to TRANSLATE the story into a known pattern

THE TRANSLATION PROCESS:
  1. Identify the DATA STRUCTURE: "row of trees" → array
  2. Identify the CONSTRAINT: "two baskets, one type each" → at most 2 distinct
  3. Identify the OBJECTIVE: "maximum fruits" → longest valid subarray
  4. Identify the CONTIGUITY: "contiguous section" → sliding window applicable
  5. MAP to known pattern: this is "Longest Subarray At Most K Distinct, K=2"

COMMON DISGUISES FOR THE SAME PROBLEM:
  → "Maximum length subarray with at most K types"
  → "Longest sequence with at most 2 unique elements"
  → "Maximum consecutive elements with limited variety"
  → "Fruits into baskets" (this problem)
  → "Pick candies with 2 bags"
  → "Maximum cards of at most K colors in a row"
  
→ ALL the same algorithm → different costumes

Step 3 — What Changes from Project 12

WHAT STAYS THE SAME (refer to Project 12 for details):
  → Sliding window expand/shrink skeleton
  → HashMap frequency tracking
  → Remove zero-count entries from map
  → Amortized O(n) time complexity
  → maxLength = max(maxLength, windowEnd - windowStart + 1)

WHAT CHANGES:
  1. Data type: char → int
     → HashMap<Character, Integer> → HashMap<Integer, Integer>
     → s.charAt(windowEnd) → fruits[windowEnd]
     
  2. K value: variable K → fixed 2
     → charFrequency.size() > k → fruitFrequency.size() > 2
     
  3. Input: String → int[]
     → s.length() → fruits.length
     → Iteration over string indices → iteration over array indices

→ The STRUCTURE is identical → only the TYPES change
→ This is why learning PATTERNS beats learning SOLUTIONS
→ One pattern → many problems

🛠️ HOW CAN WE SOLVE IT

The Algorithm

→ Refer to Project 12 for the full expand/shrink mechanics
→ Here we apply the SAME template with:
     → int[] instead of String
     → K = 2 (hardcoded)
     → Integer keys in HashMap instead of Character keys

1. Initialize HashMap<Integer, Integer>, windowStart=0, maxLength=0
2. FOR windowEnd = 0 to fruits.length - 1:
     a. EXPAND: fruitMap.put(fruits[windowEnd], count + 1)
     b. SHRINK: while fruitMap.size() > 2 → remove from left, windowStart++
     c. UPDATE: maxLength = max(maxLength, windowEnd - windowStart + 1)
3. RETURN maxLength

⚙️ HOW IT ONLY WORKS — Internal Mechanics

Mechanic 1: The Disguise Taxonomy

Problems come in DISGUISES. Learning to strip disguises is a META-SKILL.

LEVEL 1 DISGUISE (easy to recognize):
  → "Find longest substring with at most K distinct characters"
  → Almost verbatim description of the algorithm → direct mapping

LEVEL 2 DISGUISE (moderate — this problem):
  → "Fruits into baskets" / "Pick candies with bags"
  → Story wrapping around the same math
  → Need to TRANSLATE: baskets=distinct limit, contiguous=subarray, maximum=longest

LEVEL 3 DISGUISE (hard):
  → "Replace at most K characters to make longest uniform substring"
  → Doesn't mention "distinct" at all
  → But equivalent: window is valid if (window_size - max_freq_char ≤ K)
  → Still sliding window, but the VALIDITY CONDITION is disguised
  → (This is Project 34 — Longest Repeating Character Replacement)

THE RECOGNITION PROCESS:
  1. What is the INPUT? → array or string → candidate for sliding window
  2. Is CONTIGUITY required? → yes → sliding window confirmed
  3. What is the CONSTRAINT? → limited diversity → "at most K distinct"
  4. What is the OBJECTIVE? → maximize/minimize → longest/shortest window
  5. MAP to pattern → solve with known template

Mechanic 2: Why K=2 Doesn't Simplify the Algorithm

You COULD solve K=2 without a HashMap:
  → Track just two variables: type1, type2
  → Track their last-seen positions
  → When a third type appears, jump windowStart to min(lastSeen) + 1

This "optimized K=2" approach:
  → Avoids HashMap overhead
  → Is slightly faster in practice
  → BUT is specific to K=2 → can't extend to K=3

The HashMap approach:
  → Works for ANY K
  → Is the GENERAL PATTERN
  → Slight overhead but same O(n) complexity
  → Is what you should code in interviews (shows pattern mastery)

EXCEPTION: if the interviewer specifically asks to optimize for K=2
  → Then show the specialized approach as a follow-up
  → But start with the general approach first

Mechanic 3: Array vs String — Implementation Differences

STRING (Project 12):
  → s.charAt(windowEnd) → returns char
  → HashMap<Character, Integer>
  → s.length() for bounds
  → Characters have known small domain (26, 128, or 256)

ARRAY (Project 13):
  → fruits[windowEnd] → returns int
  → HashMap<Integer, Integer>
  → fruits.length for bounds
  → Integers have potentially LARGE domain (up to 10^9 distinct values)

WHY THIS MATTERS FOR OPTIMIZATION:
  → String: can use int[26] or int[128] array instead of HashMap
    → O(1) lookup guaranteed (no hash collisions)
    → Significant speedup (2-5x)
  
  → Array with large values: CANNOT use array-based optimization
    → Values like 100000 would need int[100001] → wastes memory
    → Values could be negative → no valid array index
    → MUST use HashMap

→ The HashMap approach is UNIVERSAL → works for both
→ The array optimization is a STRING-SPECIFIC bonus

Time and Space Complexity

→ Identical to Project 12 (refer there for full proof)
→ TIME: O(n) — each element added once, removed at most once
→ SPACE: O(K) = O(2) = O(1) — at most 3 entries before shrink triggers
→ For K=2 specifically: HashMap holds at most 3 entries at any time
→ Effectively O(1) space (excluding input and output)

⚠️ COMMON PITFALLS AND FAILURE MODES

Pitfall 1: Thinking It's a Different Problem

"Fruits? Baskets? This doesn't look like sliding window..."
→ STOP → translate the story → recognize the pattern

Developers who memorize solutions instead of patterns:
  → See "fruits" → never encountered this exact problem → stuck
  
Developers who learn patterns:
  → See "contiguous + limited types + maximize" → sliding window + K distinct
  → Solved in 2 minutes

→ This pitfall is about MINDSET, not technique

Pitfall 2: Using the Wrong K

"Two baskets" → K = 2 ✓
"Two baskets, each holding one type" → still K = 2 ✓

But watch for variations:
  "Three baskets" → K = 3
  "One basket" → K = 1
  "N baskets" → K = N

→ Always confirm K from the problem statement
→ Don't assume K=2 just because this problem uses it

Pitfall 3: Assuming Fruits Start from 0 or Are Sequential

fruits = [5, 100, 5, 100, 200]
→ Fruit types: 5, 100, 200 → NOT sequential, NOT starting from 0
→ HashMap handles this perfectly (keys are arbitrary integers)
→ Array-based optimization would need int[201] → wasteful

→ Don't assume anything about the value range
→ HashMap is the safe choice for integer arrays

Pitfall 4: Returning the Subarray Instead of the Length

→ Problem asks for MAXIMUM NUMBER of fruits → an integer (length)
→ Not the subarray itself
→ Don't waste time building the result subarray unless asked

→ If asked for the subarray: track maxStart alongside maxLength
→ Same technique as Project 12's "return actual substring" variant
