PROJECT 23: Reverse a Sub-portion of a Linked List

🎯 In-place Reversal → Positional Reversal → Boundary Stitching

🧩 THE HYPOTHETICAL PROBLEM

→ You are given the HEAD of a singly linked list
→ You are given two integers LEFT and RIGHT (1-based positions)
→ Reverse the nodes from position LEFT to position RIGHT
→ Return the modified list
→ Do it in ONE PASS and O(1) extra space

Example 1:
  Input:  1 → 2 → 3 → 4 → 5, left=2, right=4
  
  Before: 1 → [2 → 3 → 4] → 5
                ↑ reverse ↑
  After:  1 → [4 → 3 → 2] → 5
  
  Output: 1 → 4 → 3 → 2 → 5

Example 2:
  Input:  1 → 2 → 3 → 4 → 5, left=1, right=5
  
  Reverse entire list
  Output: 5 → 4 → 3 → 2 → 1

Example 3:
  Input:  1 → 2 → 3 → 4 → 5, left=3, right=3
  
  Reverse single node → no change
  Output: 1 → 2 → 3 → 4 → 5

Example 4:
  Input:  1 → 2 → 3 → 4 → 5, left=1, right=3
  
  Before: [1 → 2 → 3] → 4 → 5
  After:  [3 → 2 → 1] → 4 → 5
  
  Output: 3 → 2 → 1 → 4 → 5
  (Head changes! → need to return new head)

Example 5:
  Input:  5, left=1, right=1
  
  Single node → no change
  Output: 5

🔍 HOW TO APPROACH IT

Step 1 — Recall Project 16: Full Linked List Reversal

→ Project 16 taught us the prev/curr/next pointer dance:
     prev = null
     curr = head
     while curr != null:
         next = curr.next
         curr.next = prev
         prev = curr
         curr = next
     return prev

→ That reverses the ENTIRE list
→ Project 23 reverses only positions LEFT to RIGHT
→ We need to:
     1. NAVIGATE to the reversal start point
     2. REVERSE only the sub-portion
     3. STITCH the reversed portion back into the surrounding list
     
→ Steps 1 and 3 are NEW → Step 2 reuses Project 16 knowledge

Step 2 — Connect to Project 22: Three-Region Thinking

→ Project 22 taught us three-region decomposition on intervals:
     → BEFORE: unchanged, pass through
     → MIDDLE: process (merge / reverse)
     → AFTER: unchanged, pass through

→ Same structure applies to this linked list:
     → BEFORE: nodes 1 to left-1 → unchanged
     → MIDDLE: nodes left to right → REVERSE
     → AFTER: nodes right+1 to end → unchanged

  1 → 2 → 3 → 4 → 5    (left=2, right=4)
  [B]  [── MIDDLE ──]  [A]
  
  Before: [1]
  Middle: [2 → 3 → 4] → reverse → [4 → 3 → 2]
  After:  [5]
  
  Stitch: [1] → [4 → 3 → 2] → [5]

Step 3 — Identify the Critical Nodes

For left=2, right=4 on list: 1 → 2 → 3 → 4 → 5

We need FOUR critical reference points:

  1. beforeLeft: the node JUST BEFORE the reversal region
     → Node at position left-1 = position 1 = node(1)
     → After reversal, this node must point to the NEW first node of reversed portion
     
  2. leftNode: the first node of the reversal region
     → Node at position left = position 2 = node(2)
     → After reversal, this becomes the LAST node of the reversed portion
     → It must point to the AFTER region
     
  3. rightNode: the last node of the reversal region
     → Node at position right = position 4 = node(4)
     → After reversal, this becomes the FIRST node of the reversed portion
     → beforeLeft must point to this
     
  4. afterRight: the node JUST AFTER the reversal region
     → Node at position right+1 = position 5 = node(5)
     → leftNode (now last in reversed portion) must point to this

BEFORE reversal:
  beforeLeft(1) → leftNode(2) → 3 → rightNode(4) → afterRight(5)

AFTER reversal:
  beforeLeft(1) → rightNode(4) → 3 → leftNode(2) → afterRight(5)

STITCHING:
  beforeLeft.next = rightNode    (connect before-region to reversed-region)
  leftNode.next = afterRight     (connect reversed-region to after-region)

Step 4 — The Special Case: left = 1

When left = 1:
  → There is NO node before the reversal region
  → beforeLeft does not exist
  → The HEAD of the list changes after reversal!

  Input:  [1 → 2 → 3] → 4 → 5   (left=1, right=3)
  After:  [3 → 2 → 1] → 4 → 5
  New head = node(3) = rightNode

SOLUTION: Use a DUMMY NODE (sentinel)
  → Create a fake node that points to head
  → dummy.next = head
  → Now beforeLeft = dummy (for left=1)
  → After reversal: new head = dummy.next
  → The dummy node elegantly handles the edge case

  dummy → 1 → 2 → 3 → 4 → 5   (left=1, right=3)
  beforeLeft = dummy
  
  After reversal:
  dummy → 3 → 2 → 1 → 4 → 5
  
  Return dummy.next = node(3) = new head ✓

Step 5 — Visualize the Complete Process

List: 1 → 2 → 3 → 4 → 5, left=2, right=4

STEP 1: Add dummy node
  dummy(0) → 1 → 2 → 3 → 4 → 5

STEP 2: Navigate to beforeLeft (position left-1 = 1)
  Move pointer 1 time from dummy
  beforeLeft = node(1)

STEP 3: Identify leftNode
  leftNode = beforeLeft.next = node(2)

STEP 4: Reverse the sub-portion (left to right)
  We need to reverse 3 nodes: 2, 3, 4 (right - left + 1 = 3)
  
  Using Project 16's technique on the sub-portion:
  
  Initial state:
    beforeLeft(1) → 2 → 3 → 4 → 5
    
  We use a DIFFERENT reversal technique here:
  "Repeatedly move the next node to the front of the reversed portion"
  
  Iteration 1: Move node(3) to front
    beforeLeft(1) → 3 → 2 → 4 → 5
    
  Iteration 2: Move node(4) to front
    beforeLeft(1) → 4 → 3 → 2 → 5

STEP 5: Stitching happened automatically during the process!
  beforeLeft.next now points to 4 (front of reversed)
  leftNode(2).next now points to 5 (after region)
  
  Result: dummy(0) → 1 → 4 → 3 → 2 → 5
  Return dummy.next = node(1) → 1 → 4 → 3 → 2 → 5 ✓

Step 6 — The "Move to Front" Reversal Technique

This is a KEY technique for partial reversal.

Instead of the classic prev/curr/next approach:
→ We repeatedly take the NEXT node after leftNode
→ And move it to the FRONT of the reversed portion
→ beforeLeft.next always points to the current front

WHY this technique?
→ Classic reversal needs to know the END point in advance
→ Classic reversal disconnects nodes → hard to stitch back
→ "Move to front" keeps the list CONNECTED throughout
→ Stitching happens AUTOMATICALLY as we go
→ No separate stitching step needed

DETAILED MECHANICS:

Starting: beforeLeft(1) → leftNode(2) → 3 → 4 → 5

Variables:
  beforeLeft = node(1)
  leftNode = node(2)    (this stays fixed — it becomes the LAST node)
  
Iteration 1: (move node after leftNode to front)
  nodeToMove = leftNode.next = node(3)
  
  Step A: leftNode.next = nodeToMove.next
          node(2).next = node(4)
          State: 1 → 2 → 4 → 5,  node(3) detached
  
  Step B: nodeToMove.next = beforeLeft.next
          node(3).next = node(2)
          State: 1 → 2 → 4 → 5,  node(3) → node(2) → 4 → 5
  
  Step C: beforeLeft.next = nodeToMove
          node(1).next = node(3)
          State: 1 → 3 → 2 → 4 → 5 ✓
          
Iteration 2: (move node after leftNode to front)
  nodeToMove = leftNode.next = node(4)
  
  Step A: leftNode.next = nodeToMove.next
          node(2).next = node(5)
  
  Step B: nodeToMove.next = beforeLeft.next
          node(4).next = node(3)
  
  Step C: beforeLeft.next = nodeToMove
          node(1).next = node(4)
          
  State: 1 → 4 → 3 → 2 → 5 ✓

DONE! (right - left = 4 - 2 = 2 iterations)

🛠️ HOW CAN WE SOLVE IT

The Algorithm — Precise Steps

reverseBetween(head, left, right):

  1. IF head is null OR left == right → return head (nothing to reverse)

  2. CREATE dummy node: dummy.next = head
     → Handles the left=1 edge case elegantly

  3. NAVIGATE to beforeLeft:
     → Start at dummy
     → Move forward (left - 1) times
     → beforeLeft = current position

  4. IDENTIFY leftNode:
     → leftNode = beforeLeft.next
     → This node will become the LAST node of the reversed portion

  5. REVERSE using "move to front" technique:
     → Repeat (right - left) times:
          a. nodeToMove = leftNode.next
          b. leftNode.next = nodeToMove.next       (skip over nodeToMove)
          c. nodeToMove.next = beforeLeft.next      (nodeToMove points to current front)
          d. beforeLeft.next = nodeToMove            (nodeToMove becomes new front)

  6. RETURN dummy.next (the real head, which may have changed)

Why (right - left) Iterations?

→ We have (right - left + 1) nodes to reverse
→ The first node (leftNode) stays in place and shifts to the end naturally
→ We need to move each of the remaining (right - left) nodes to the front
→ Each iteration moves exactly ONE node
→ Total iterations = right - left

Example: left=2, right=4 → reverse nodes at positions 2,3,4
  → 3 nodes to reverse
  → leftNode (position 2) stays → shifts to end
  → Move position 3 to front → 1 iteration
  → Move position 4 to front → 2 iterations
  → Total: right - left = 4 - 2 = 2 iterations ✓

✅ WHAT SHOULD YOU DO

→ ALWAYS use a dummy node
     → Eliminates special case for left=1
     → dummy.next = head → return dummy.next at the end
     → Makes code cleaner and less error-prone
     → O(1) extra space → acceptable

→ ALWAYS keep leftNode reference fixed
     → leftNode is set ONCE and never reassigned
     → It naturally migrates to the end of the reversed portion
     → Its .next pointer changes each iteration but the reference stays

→ ALWAYS use the "move to front" technique for partial reversal
     → Keeps list connected throughout
     → Automatic stitching
     → No separate stitching step needed
     → Fewer edge cases than classic prev/curr/next

→ ALWAYS verify with small examples:
     → left=1, right=1 → no change
     → left=1, right=n → full reversal → head changes
     → left=right → no change
     → left=1, right=2 → swap first two nodes

→ ALWAYS count iterations carefully
     → (right - left) iterations, NOT (right - left + 1)
     → Off-by-one here reverses one too many or too few nodes

🚫 WHAT SHOULD YOU NOT DO

→ DO NOT use classic prev/curr/next reversal without modification
     → Classic reversal disconnects nodes
     → Hard to stitch back into surrounding list
     → Need to save beforeLeft and afterRight references
     → "Move to front" is cleaner for partial reversal

→ DO NOT forget the dummy node
     → Without dummy: special case for left=1
     → Code branches into two paths → doubled complexity
     → Dummy eliminates the branch entirely

→ DO NOT modify leftNode reference during reversal
     WRONG:
       leftNode = leftNode.next;  // inside reversal loop
     → leftNode must stay fixed → it's our anchor
     → Its .next changes → but the reference doesn't

→ DO NOT navigate using 1-based counting from head
     WRONG:
       for (int i = 1; i < left; i++) curr = curr.next;
       // curr is now at position left → NOT beforeLeft!
     
     RIGHT:
       // Start from dummy (position 0)
       for (int i = 0; i < left - 1; i++) curr = curr.next;
       // curr is now beforeLeft ✓

→ DO NOT reverse then stitch separately
     → Reversing disconnects the sub-list from the main list
     → You lose the afterRight reference
     → Need to save it first → more variables → more bugs
     → "Move to front" avoids all this

→ DO NOT use extra data structures
     → No arrays, no stacks, no queues
     → Pure pointer manipulation
     → O(1) space is the goal

→ DO NOT ignore the return value
     → When left=1, the head changes
     → Always return dummy.next, not original head
     → Returning original head when left=1 → wrong answer

🔀 WHEN TO USE THIS PATTERN

✓ Reversing a PORTION of a linked list (not the whole thing)
✓ Positions are given (left, right) rather than node references
✓ In-place modification required (O(1) space)
✓ Single-pass solution needed
✓ Variants:
     → Reverse every k-group of nodes
     → Reverse alternating k-groups
     → Reverse second half of list
     → All build on this sub-portion reversal technique
✓ The "move to front" technique works whenever:
     → You need to reverse a contiguous segment
     → You have a reference to the node BEFORE the segment
     → You want automatic stitching

🚫 WHEN NOT TO USE THIS PATTERN

✗ Need to reverse the ENTIRE list
     → Simpler: use Project 16's classic prev/curr/next
     → No navigation or stitching needed

✗ Doubly linked list
     → Reversal is simpler with double links
     → Just swap prev and next pointers for each node in range
     → Then fix boundary connections
     → No "move to front" needed

✗ Need to reverse based on VALUES, not positions
     → "Reverse all nodes with value > 5" → different problem
     → Position-based technique doesn't apply
     → Need value-based filtering + reversal

✗ Random access is available (array-based list)
     → Arrays: just use two-pointer swap from both ends
     → Much simpler than linked list pointer manipulation
     → This technique is specifically for singly linked lists

✗ Multiple non-contiguous segments to reverse
     → Each segment needs its own navigation + reversal
     → Consider if a different approach (stack, array conversion) is cleaner

⚙️ HOW IT ONLY WORKS — Internal Mechanics

The Dummy Node Technique — Deep Dive

WHY a dummy node works:

Without dummy (left=1):
  head → 1 → 2 → 3 → 4 → 5
  → beforeLeft = ??? → no node before position 1
  → Must handle separately: "if left == 1, then..."
  → After reversal: new head = rightNode
  → Special return logic needed

With dummy:
  dummy(0) → 1 → 2 → 3 → 4 → 5
  → beforeLeft = dummy → always exists!
  → Same algorithm works for all cases
  → After reversal: dummy.next = correct new head
  → Return dummy.next → always correct

The dummy node is a SENTINEL:
→ It provides a uniform "before" reference
→ It absorbs the edge case into the general case
→ Cost: one extra object → negligible
→ Benefit: eliminates branching → fewer bugs

This technique appears in MANY linked list problems:
→ Merge two sorted lists
→ Remove nth node from end
→ Partition list around value
→ Any problem where the head might change

The "Move to Front" Technique — Step by Step Mechanics

The three pointer operations per iteration:

BEFORE iteration:
  beforeLeft → [front] → ... → leftNode → nodeToMove → rest
  
OPERATION A: leftNode.next = nodeToMove.next
  → Detaches nodeToMove from its current position
  → leftNode now skips over nodeToMove → points to rest
  
  beforeLeft → [front] → ... → leftNode → rest
                                              nodeToMove (floating)

OPERATION B: nodeToMove.next = beforeLeft.next
  → nodeToMove now points to current front of reversed portion
  
  beforeLeft → [front] → ... → leftNode → rest
               ↑
  nodeToMove ──┘

OPERATION C: beforeLeft.next = nodeToMove
  → nodeToMove becomes the new front
  
  beforeLeft → nodeToMove → [old front] → ... → leftNode → rest

AFTER iteration:
  beforeLeft → nodeToMove → [old front] → ... → leftNode → rest
  → nodeToMove is now the FIRST node after beforeLeft
  → leftNode is still the LAST node of the reversed portion
  → leftNode.next points to rest (the unreversed remainder)

KEY OBSERVATIONS:
  → leftNode NEVER MOVES in the list structure
  → leftNode gradually becomes the last reversed node
  → Each new node jumps to the front
  → beforeLeft always points to the newest front
  → The list stays FULLY CONNECTED at every step

Why the List Stays Connected

This is the CRITICAL advantage over classic reversal.

Classic reversal (Project 16):
  prev=null, curr=head
  → curr.next = prev → DISCONNECTS curr from the rest!
  → The rest of the list is only reachable through "next" variable
  → If you lose "next", you lose the rest of the list
  → Must be very careful with pointer order

"Move to front":
  → leftNode.next = nodeToMove.next → RECONNECTS leftNode to rest
  → nodeToMove.next = beforeLeft.next → CONNECTS nodeToMove to front
  → beforeLeft.next = nodeToMove → CONNECTS beforeLeft to nodeToMove
  → At NO point is any part of the list unreachable
  → The list is always fully traversable from dummy

This makes the technique:
  → Safer (no dangling pointers)
  → Easier to debug (can print list at any step)
  → Less error-prone (fewer ways to lose references)

Time and Space Complexity

TIME: O(n)
→ Navigate to beforeLeft: O(left) steps
→ Reverse sub-portion: O(right - left) iterations
→ Total: O(left) + O(right - left) = O(right) ≤ O(n)
→ Single pass → true O(n)

SPACE: O(1)
→ Variables: dummy, beforeLeft, leftNode, nodeToMove
→ Fixed number of pointers regardless of list size
→ No recursion → no call stack growth
→ Dummy node: 1 extra object → O(1)

COMPARISON WITH ALTERNATIVES:
  Stack-based: O(n) time, O(right-left) space → violates O(1) space
  Array conversion: O(n) time, O(n) space → violates O(1) space
  Recursive: O(n) time, O(n) stack space → violates O(1) space
  "Move to front": O(n) time, O(1) space → OPTIMAL ✓

The Three Operations Order Matters

The three operations MUST happen in exactly this order:

CORRECT ORDER: A → B → C
  A: leftNode.next = nodeToMove.next      (detach)
  B: nodeToMove.next = beforeLeft.next     (point to front)
  C: beforeLeft.next = nodeToMove          (become new front)

WHY this specific order?

If we do C first (beforeLeft.next = nodeToMove):
  → We lose reference to old front (beforeLeft used to point there)
  → nodeToMove.next hasn't been set yet → dangling
  → BROKEN

If we do B first (nodeToMove.next = beforeLeft.next):
  → This is actually OK → reads beforeLeft.next before modifying it
  → But then A must happen before C
  → A detaches nodeToMove → then C sets new front

If we do A first (leftNode.next = nodeToMove.next):
  → nodeToMove is detached from its position
  → But we still have nodeToMove reference → not lost
  → Then B and C connect it at the front
  → CORRECT

Actually, A-B-C and B-A-C both work.
The key constraint: C must be LAST (otherwise we lose beforeLeft.next reference for B).

Safest order: A → B → C (detach → repoint → connect)

⚠️ COMMON PITFALLS AND FAILURE MODES

Pitfall 1: Wrong Number of Navigation Steps

SCENARIO: left=2, list starts at dummy

How many steps from dummy to beforeLeft?
  dummy(pos 0) → node(pos 1) → node(pos 2) → ...
  
  beforeLeft is at position left-1 = 1
  Steps from dummy to position 1 = 1 step
  
  FORMULA: (left - 1) steps from dummy to reach beforeLeft

WRONG:
  for (int i = 0; i < left; i++) curr = curr.next;
  → This reaches position LEFT, not beforeLeft
  → One step too many!

RIGHT:
  for (int i = 0; i < left - 1; i++) curr = curr.next;
  → Reaches position left-1 = beforeLeft ✓

VERIFICATION:
  left=1: steps = 0 → stay at dummy → beforeLeft = dummy ✓
  left=2: steps = 1 → move once → beforeLeft = first node ✓
  left=5: steps = 4 → move four times → beforeLeft = fourth node ✓

Pitfall 2: Wrong Number of Reversal Iterations

SCENARIO: left=2, right=4 → reverse 3 nodes (positions 2,3,4)

Number of "move to front" iterations needed?
  → Node at position 2 (leftNode) stays → ends up last
  → Node at position 3 → move to front → iteration 1
  → Node at position 4 → move to front → iteration 2
  → Total: 2 iterations = right - left

WRONG:
  for (int i = 0; i < right - left + 1; i++) ...
  → right - left + 1 = 3 → one too many iterations
  → Moves a node from the AFTER region into the reversed portion!

RIGHT:
  for (int i = 0; i < right - left; i++) ...
  → right - left = 2 → exactly correct ✓

Pitfall 3: Not Returning dummy.next

SCENARIO: left=1, right=3 on list 1 → 2 → 3 → 4 → 5

After reversal: dummy → 3 → 2 → 1 → 4 → 5
Original head was node(1), now head should be node(3)

WRONG: return head;
  → Returns node(1) → but node(1) is now in position 3
  → Caller gets: 1 → 4 → 5 (missing the reversed front)

RIGHT: return dummy.next;
  → Returns node(3) → the true new head ✓

Pitfall 4: left = right (No Reversal Needed)

left=3, right=3 → reverse single node → no change

Algorithm: right - left = 0 → zero iterations → no reversal
→ Handled automatically → no special case needed

But some code has bugs here:
→ If you check "if left == right return head" at the start → correct but unnecessary
→ The algorithm handles it naturally → zero iterations → unchanged list

Pitfall 5: Reversing When List is Shorter Than Expected

What if right > list length?
→ Problem typically guarantees valid left and right
→ If not guaranteed → nodeToMove could be null → NullPointerException

DEFENSIVE CODE:
  for (int i = 0; i < right - left; i++) {
      ListNode nodeToMove = leftNode.next;
      if (nodeToMove == null) break;  // safety net
      ...
  }

→ Add null check if problem doesn't guarantee valid bounds

Pitfall 6: Confusing Node References After Reversal

After reversal, leftNode and beforeLeft still point to the same NODES
But those nodes are now in different POSITIONS:

BEFORE: dummy → 1 → 2 → 3 → 4 → 5
  beforeLeft = node(1) at position 1
  leftNode = node(2) at position 2

AFTER: dummy → 1 → 4 → 3 → 2 → 5
  beforeLeft = node(1) still at position 1 ✓
  leftNode = node(2) now at position 4 (end of reversed portion)

→ The NODES don't move in memory → their LINKS change
→ leftNode is the same object → but its position in the list changed
→ This is expected and correct

🧠 KEY TAKEAWAY

WHAT WE LEARNED:
→ Partial linked list reversal needs THREE components:
     1. Navigation to the boundary (beforeLeft)
     2. Reversal of the sub-portion (move-to-front)
     3. Stitching (automatic with move-to-front)
→ Dummy node eliminates the left=1 edge case
→ "Move to front" keeps the list connected throughout
→ leftNode stays fixed → naturally becomes the last reversed node
→ (right - left) iterations → NOT (right - left + 1)
→ Always return dummy.next → head may have changed

THE CORE MENTAL MODEL:
→ "Navigate to the node before the reversal zone"
→ "Repeatedly move the next node to the front of the zone"
→ "After all moves, the zone is reversed and connected"
→ "Dummy node ensures there's always a 'before' node"

PATTERN SIGNATURE:
→ Singly linked list ✓
→ Reverse between positions L and R ✓
→ In-place O(1) space ✓
→ = MOVE-TO-FRONT PARTIAL REVERSAL + DUMMY NODE
