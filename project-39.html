PROJECT 39: K Closest Points to Origin

🎯 Top K Elements → Max-Heap with Distance Comparator → Bounded Heap

🧩 THE HYPOTHETICAL PROBLEM

→ You are given an array of POINTS on a 2D plane
→ Each point is represented as [x, y]
→ You are given a value K
→ Find the K points CLOSEST to the origin (0, 0)
→ Distance = Euclidean distance = √(x² + y²)
→ Return the K points in ANY order

Example 1:
  Input:  points = [[1,3], [-2,2]], k = 1
  Distances: [1,3] → √(1+9) = √10 ≈ 3.16
             [-2,2] → √(4+4) = √8 ≈ 2.83
  Output: [[-2,2]]  (closest single point)

Example 2:
  Input:  points = [[3,3], [5,-1], [-2,4]], k = 2
  Distances: [3,3] → √(9+9) = √18
             [5,-1] → √(25+1) = √26
             [-2,4] → √(4+16) = √20
  Output: [[3,3], [-2,4]]  (two closest)

Example 3:
  Input:  points = [[0,1], [1,0], [0,-1], [-1,0], [2,2]], k = 3
  Distances: √1, √1, √1, √1, √8
  Output: Any 3 of the first 4 points (all distance √1)

🔍 HOW TO APPROACH IT

Step 1 — Map This to the Top K Framework from Project 38

Project 38:
  → Given: array of VALUES
  → Find: K LARGEST values
  → Tool: MIN-heap of size K (root = gatekeeper = Kth largest)

Project 39:
  → Given: array of POINTS
  → Find: K CLOSEST points (K SMALLEST distances)
  → Tool: ???-heap of size K

KEY SHIFT:
  → Project 38 wanted K LARGEST → used MIN-heap (opposite!)
  → Project 39 wants K SMALLEST (closest) → use MAX-heap (opposite!)
  
  → The "opposite heap" rule from Project 38 applies directly:
       K largest  → MIN-heap → root = smallest of top K = gatekeeper
       K smallest → MAX-heap → root = largest of bottom K = gatekeeper

Step 2 — Understand the Comparison Metric Change

Project 38:
  → Compare elements by their RAW VALUE
  → num > heap.peek() → swap in
  → Straightforward integer comparison

Project 39:
  → Compare elements by their COMPUTED DISTANCE
  → Elements are int[] (point coordinates)
  → Cannot compare points directly — need a DERIVED metric
  → distance(point) = x² + y²  (we skip the √ — explained below)

THIS IS THE CORE NEW CONCEPT:
  → The heap doesn't store distances — it stores POINTS
  → But the heap ORDERS by distance
  → This requires a CUSTOM COMPARATOR

Step 3 — Why We Skip the Square Root

Euclidean distance = √(x² + y²)

We need to COMPARE distances, not compute exact values:
  → √(x₁² + y₁²)  vs  √(x₂² + y₂²)
  → Since √ is a MONOTONICALLY INCREASING function:
       √a < √b  ⟺  a < b
  → So: x₁² + y₁²  vs  x₂² + y₂²  gives the SAME ordering

BENEFITS OF SKIPPING √:
  → Avoids floating-point precision issues
  → Avoids expensive Math.sqrt() computation
  → Integer arithmetic only → faster and exact
  → Comparison results are IDENTICAL

SQUARED DISTANCE: distSq(x, y) = x*x + y*y
  → This is our comparison metric throughout

Step 4 — Visualize the Max-Heap Gatekeeper for "K Closest"

points = [[3,3], [5,-1], [-2,4], [1,1], [0,2]], k = 3
Squared distances:  18,   26,    20,    2,    4

Goal: 3 closest = points with distances 2, 4, 18 → [1,1], [0,2], [3,3]

Process with MAX-heap of size K=3 (ordered by distance):

Step 1: Process [3,3] dist=18 → heap not full → ADD
        heap: {[3,3](18)}  root=18

Step 2: Process [5,-1] dist=26 → heap not full → ADD
        heap: {[5,-1](26), [3,3](18)}  root=26

Step 3: Process [-2,4] dist=20 → heap not full → ADD
        heap: {[5,-1](26), [-2,4](20), [3,3](18)}  root=26

Step 4: Process [1,1] dist=2 → 2 < root(26) → SWAP
        remove [5,-1](26), add [1,1](2)
        heap: {[-2,4](20), [3,3](18), [1,1](2)}  root=20

Step 5: Process [0,2] dist=4 → 4 < root(20) → SWAP
        remove [-2,4](20), add [0,2](4)
        heap: {[3,3](18), [0,2](4), [1,1](2)}  root=18

Answer: all 3 points in heap = [[3,3], [0,2], [1,1]] ✓

THE GATEKEEPER LOGIC:
  → Root = FARTHEST of the K closest = the Kth closest
  → New point closer than root? → it BELONGS in the K closest → swap in
  → New point farther than root? → it's NOT in the K closest → reject
  → IDENTICAL logic to Project 38 — just flipped direction

🛠️ HOW CAN WE SOLVE IT

The Algorithm

1. Create a MAX-heap of points, ordered by SQUARED DISTANCE
     → In Java: PriorityQueue with custom Comparator
     → Comparator compares (b[0]*b[0]+b[1]*b[1]) vs (a[0]*a[0]+a[1]*a[1])
     → Reversed order = max-heap behavior
2. For each point in the array:
     a. If heap size < K → add point directly
     b. Else if point's distance < root's distance:
          → poll root (farthest of K closest)
          → offer new point (closer one takes its place)
     c. Else → skip (point is farther than all K closest)
3. Drain heap into result array → return

✅ WHAT SHOULD YOU DO

→ ALWAYS use squared distance for comparison
     → Avoids floating-point errors
     → Faster computation
     → Same ordering guarantee

→ ALWAYS use MAX-heap for "K smallest/closest" (opposite rule from Project 38)
     → Root = farthest of K closest = gatekeeper
     → Only closer points pass the gate

→ ALWAYS use custom Comparator for non-primitive heap elements
     → Points are int[] — no natural ordering for "distance"
     → Comparator encodes the distance logic INTO the heap

→ ALWAYS consider integer overflow for squared distances
     → x can be up to 10^4 → x² = 10^8
     → x² + y² = 2 * 10^8 → fits in int (max ~2.1 * 10^9)
     → BUT if coordinates are larger → use long
     → Safer to always cast to long in comparator

→ ALWAYS verify: does the problem want K points or just the Kth point?
     → K points → drain entire heap into result
     → Kth point → peek at root (like Project 38)

🚫 WHAT SHOULD YOU NOT DO

→ DO NOT compute Math.sqrt() for comparisons
     → Unnecessary computation
     → Introduces floating-point imprecision
     → Squared distance preserves ordering perfectly

→ DO NOT use MIN-heap for "K closest"
     → Min-heap root = closest point = 1st closest
     → Cannot efficiently identify Kth closest
     → Would need to store ALL points → O(n) space
     → Project 38 rule: K smallest → MAX-heap (opposite!)

→ DO NOT compare distances using subtraction in Comparator
     → (a_dist - b_dist) can overflow for large coordinates
     → Use Integer.compare() or explicit if/else instead
     → Safe: Integer.compare(distB, distA) for max-heap order

→ DO NOT forget that the result order doesn't matter
     → Problem says "return in any order"
     → No need to sort the K results
     → Just drain heap directly

→ DO NOT create Point objects unnecessarily
     → int[] works perfectly → avoids object overhead
     → In competitive programming → raw arrays are faster
     → In production code → Point class is fine for readability

🔀 WHEN TO USE THIS PATTERN

✓ "Find K closest / nearest / smallest" → MAX-heap of size K
✓ "Find K farthest / largest / biggest" → MIN-heap of size K
✓ Comparison metric is DERIVED (distance, frequency, score, etc.)
✓ The metric can be computed per element independently
✓ K << n → bounded heap gives space advantage
✓ Result order doesn't matter (or you sort K elements at end)

REAL-WORLD APPLICATIONS:
  → Nearest neighbor search (K closest data points)
  → Recommendation systems (K most similar items)
  → GPS: K nearest restaurants/gas stations
  → Machine learning: K-nearest neighbors (KNN) algorithm
  → Game dev: K closest enemies to player

🚫 WHEN NOT TO USE THIS PATTERN

✗ K is close to n → sorting is simpler and equally fast
✗ You need results in SORTED order by distance
     → Heap gives K elements but not sorted
     → Would need additional O(K log K) sort
     → If K ≈ n → just sort original array

✗ Distance computation is EXPENSIVE and elements are compared multiple times
     → Heap may re-compare during sift operations
     → Pre-compute distances, store as (distance, point) pairs
     → Then use simple numeric heap

✗ Coordinates change over time (dynamic points)
     → Heap doesn't support updating existing elements efficiently
     → Need more advanced spatial data structures (KD-tree, R-tree)

✗ You need the EXACT Kth closest (not all K)
     → Just peek at root — same as Project 38
     → But if you ONLY need the Kth distance value → even simpler

⚙️ HOW IT ONLY WORKS — Internal Mechanics

Custom Comparator Deep Dive

Java PriorityQueue is a MIN-heap by default:
  → new PriorityQueue<>() → smallest at root

To make it a MAX-heap for points by distance:

OPTION A: Explicit Comparator (clearest)
  new PriorityQueue<>((a, b) -> {
      int distA = a[0]*a[0] + a[1]*a[1];
      int distB = b[0]*b[0] + b[1]*b[1];
      return Integer.compare(distB, distA);  // REVERSED = max-heap
  });

OPTION B: Negate distance (trick)
  new PriorityQueue<>((a, b) -> {
      int distA = a[0]*a[0] + a[1]*a[1];
      int distB = b[0]*b[0] + b[1]*b[1];
      return Integer.compare(distA, distB);  // normal = min-heap
  });
  → This gives MIN-heap by distance → WRONG for our gatekeeper approach
  → We NEED max-heap → OPTION A is correct

OPTION C: Using Comparator.comparingInt (concise)
  new PriorityQueue<>(
      Comparator.comparingInt((int[] p) -> p[0]*p[0] + p[1]*p[1]).reversed()
  );
  → .reversed() flips to max-heap

HOW THE COMPARATOR FLOWS THROUGH THE HEAP:
  → When offer(point) is called:
       → Point placed at bottom of heap
       → SIFT UP: compare with parent using our comparator
       → Larger distance floats UP (max-heap behavior)
       → Root always has LARGEST distance
  → When poll() is called:
       → Root removed (largest distance point)
       → Last element moved to root
       → SIFT DOWN: compare with children using our comparator
       → Maintains max-heap property

The Gatekeeper Invariant (Extension of Project 38)

CLAIM: After processing i points, the heap contains the K closest
       among those i points (or all i if i < K).

The proof is IDENTICAL to Project 38 — just with direction flipped:

→ New point CLOSER than root? 
     → Root (farthest of K closest) gets kicked out
     → New closer point takes its place
     → Heap still holds K closest ✓

→ New point FARTHER than root?
     → It's farther than even the farthest of K closest
     → Cannot possibly be in K closest
     → Skip it → heap unchanged ✓

→ After all n points: heap = K closest overall
→ Root = farthest of K closest = Kth closest point

Time Complexity

→ For each of n points:
     → Compute squared distance: O(1)
     → Compare with root: O(1)
     → Possible offer + poll: O(log K) each
→ Total: O(n log K)
→ Final drain of heap: O(K log K)
→ Overall: O(n log K)

vs Sorting approach: O(n log n)
When K = 10, n = 10,000,000:
  → Heap: 10^7 * 3.3 ≈ 33M ops
  → Sort: 10^7 * 23 ≈ 230M ops → 7x slower

Space Complexity

→ Heap stores K points: O(K)
→ Result array: O(K)
→ Total: O(K)

vs Sorting: O(1) extra but modifies input (or O(n) if copy needed)
When K = 10, n = 10,000,000 → heap uses 10 points vs 10M points

Why the "Opposite Heap" Rule is Universal

PROJECT 38: K LARGEST → MIN-heap
  → Root = SMALLEST of top K = Kth LARGEST = LOWER BOUNDARY
  → Only elements ABOVE the boundary enter
  → Boundary naturally RISES as better (larger) elements arrive

PROJECT 39: K CLOSEST (smallest distance) → MAX-heap  
  → Root = FARTHEST of K closest = Kth CLOSEST = UPPER BOUNDARY
  → Only elements BELOW the boundary enter
  → Boundary naturally DROPS as better (closer) elements arrive

UNIVERSAL PATTERN:
  → You always want the root to be the WEAKEST member of the elite group
  → "Weakest" = the one most likely to be replaced
  → For K largest: weakest = smallest → min-heap root
  → For K smallest: weakest = largest → max-heap root
  → The root is always the THRESHOLD for admission

⚠️ COMMON PITFALLS AND FAILURE MODES

Pitfall 1: Wrong Heap Direction

THE MOST COMMON MISTAKE (same as Project 38, but trickier here):

"I want K closest (smallest distances) → I'll use min-heap!"

WRONG:
  → Min-heap root = closest point = 1st closest
  → To find K closest → need all n points in heap → O(n) space
  → Defeats the purpose of bounded heap

CORRECT:
  → Max-heap of size K → root = farthest of K closest = gatekeeper
  → Only closer points pass → O(K) space

MEMORY AID (repeat from Project 38):
  → K largest → MIN-heap (opposite)
  → K smallest/closest → MAX-heap (opposite)
  → ROOT = GATEKEEPER = WEAKEST ELITE MEMBER

Pitfall 2: Comparator Overflow

Coordinates: x = 40000, y = 40000
Squared distance = 40000² + 40000² = 1,600,000,000 + 1,600,000,000 
                 = 3,200,000,000

int max = 2,147,483,647 → OVERFLOW!

FIX: Cast to long in comparator
  (a, b) -> {
      long distA = (long)a[0]*a[0] + (long)a[1]*a[1];
      long distB = (long)b[0]*b[0] + (long)b[1]*b[1];
      return Long.compare(distB, distA);
  }

WHEN IS THIS NEEDED?
  → LeetCode constraint: -10^4 ≤ x, y ≤ 10^4
  → Max squared distance = 10^8 + 10^8 = 2 * 10^8 → fits in int
  → Safe for THIS problem but NOT safe in general
  → Always consider the constraint range

Pitfall 3: Comparator Using Subtraction

WRONG:
  (a, b) -> (b[0]*b[0] + b[1]*b[1]) - (a[0]*a[0] + a[1]*a[1])

This uses subtraction for comparison → CAN OVERFLOW:
  distB = 2,000,000,000
  distA = -2,000,000,000  (if allowed)
  distB - distA = 4,000,000,000 → overflows int → returns NEGATIVE
  → Heap order is CORRUPTED

CORRECT:
  (a, b) -> Integer.compare(distB, distA)
  → Integer.compare uses conditional logic, NOT subtraction
  → NEVER overflows

Pitfall 4: Ties in Distance

points = [[1,0], [0,1], [-1,0], [0,-1], [2,0]], k = 3
Distances: 1, 1, 1, 1, 4

Four points at distance 1, but K = 3 → which 3?
→ ANY 3 of the 4 equidistant points are valid
→ The heap handles this AUTOMATICALLY
→ First 3 enter → 4th has same distance as root
→ 4th is NOT strictly closer → gatekeeper rejects it
→ Result: whichever 3 entered first

If problem requires DETERMINISTIC tie-breaking:
  → Add secondary sort to comparator (e.g., by x then y)
  → Most LeetCode problems say "any order" → don't overcomplicate

Pitfall 5: Distance to a Point OTHER Than Origin

Problem variant: "K closest to point P = (px, py)"

Distance formula changes:
  → distSq = (x - px)² + (y - py)²

Everything else stays IDENTICAL:
  → Max-heap of size K
  → Comparator uses (x-px)² + (y-py)² instead of x² + y²
  → Same gatekeeper logic

Common mistake: forgetting to subtract px, py
  → If origin (0,0) → subtraction has no effect → works by accident
  → If P ≠ origin → MUST subtract → test with non-origin reference point

📈 SCALABILITY CONSIDERATIONS

SCENARIO 1: Small K, large n
  → K = 10, n = 10,000,000
  → Heap: O(10^7 * log 10) = O(10^7 * 3.3) ≈ 33M ops, O(10) space
  → Sorting: O(10^7 * log 10^7) = O(10^7 * 23) ≈ 230M ops
  → Heap wins decisively

SCENARIO 2: K close to n
  → K = 9,000,000, n = 10,000,000
  → Heap: O(10^7 * log 9*10^6) ≈ O(10^7 * 23) ≈ 230M ops
  → Sorting: same ≈ 230M ops
  → No advantage → sorting is simpler

SCENARIO 3: Streaming points (GPS, sensors)
  → Points arrive one at a time
  → Heap processes each in O(log K)
  → Don't need to store ALL points — only K at a time
  → Perfect for real-time "K nearest" tracking

SCENARIO 4: 3D or higher dimensions
  → Distance formula: x² + y² + z² + ...
  → Comparator changes → everything else identical
  → Heap approach scales to ANY dimension
  → For very high dimensions → consider approximate methods (LSH, KD-tree)

SCENARIO 5: Repeated queries with same points but different K
  → Rebuild heap for each K → O(n log K) per query
  → Better: sort once by distance → O(n log n) → answer any K in O(K)
  → Pre-sort when multiple queries expected
