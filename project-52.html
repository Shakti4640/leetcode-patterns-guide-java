PROJECT 52: Word Break — Can String Be Segmented?

🎯 Dynamic Programming → String Partitioning → Dictionary-Driven DP Transitions

🧩 THE HYPOTHETICAL PROBLEM

→ You are given a STRING s
→ You are given a DICTIONARY of words (list of strings)
→ Determine if s can be segmented into a space-separated sequence
  of one or more dictionary words
→ The SAME dictionary word may be reused multiple times
→ Return TRUE or FALSE

Example 1:
  s = "leetcode"
  wordDict = ["leet", "code"]
  Output: true
  
  → "leetcode" = "leet" + "code"
  → Both "leet" and "code" are in the dictionary ✓

Example 2:
  s = "applepenapple"
  wordDict = ["apple", "pen"]
  Output: true
  
  → "applepenapple" = "apple" + "pen" + "apple"
  → "apple" is reused → allowed ✓

Example 3:
  s = "catsandog"
  wordDict = ["cats", "dog", "sand", "and", "cat"]
  Output: false
  
  → "cats" + "andog" → "andog" not in dict
  → "cat" + "sandog" → "sandog" not in dict
  → "cat" + "sand" + "og" → "og" not in dict
  → "cats" + "and" + "og" → "og" not in dict
  → No valid segmentation exists

Example 4:
  s = "aaaaaaa"
  wordDict = ["aaa", "aaaa"]
  Output: true
  
  → "aaaa" + "aaa" = 7 characters ✓

Example 5:
  s = "bb"
  wordDict = ["a", "b", "bbb", "bbbb"]
  Output: true
  
  → "b" + "b" ✓

Example 6:
  s = "goalspecial"
  wordDict = ["go", "goal", "goals", "special"]
  Output: true
  
  → "goal" + "special" ✓
  → Note: "go" matches the start but leaves "alspecial" which can't be segmented
  → Choosing "goal" over "go" is critical → greedy fails

🔍 HOW TO APPROACH IT

Step 1 — Why Greedy Fails (Again)

s = "goalspecial"
wordDict = ["go", "goal", "goals", "special"]

GREEDY (match longest first):
  → "goals" matches s[0..4] → remaining "pecial" → NOT in dict → FAIL
  → Backtrack? Greedy doesn't backtrack

GREEDY (match shortest first):
  → "go" matches s[0..1] → remaining "alspecial" → can't segment → FAIL

CORRECT:
  → "goal" matches s[0..3] → remaining "special" → IN dict → TRUE ✓

→ Neither longest-first nor shortest-first greedy works
→ The CHOICE of which word to match affects ALL subsequent segmentation
→ We need to explore ALL possibilities → DP or recursion with memoization

SAME LESSON AS:
  → Project 50 (LCS): greedy matching misses better alignments
  → Project 51 (Edit Distance): greedy operations miss cheaper paths
  → Project 52 (Word Break): greedy word choice blocks valid segmentations

Step 2 — Recognize the Problem Structure

OBSERVATION:
  → If s[0..j-1] can be segmented (dp[j] = true)
  → AND s[j..i-1] is a word in the dictionary
  → THEN s[0..i-1] can be segmented (dp[i] = true)

THIS IS:
  → A PARTITIONING problem
  → We're partitioning string s into consecutive pieces
  → Each piece must be a dictionary word
  → We need to check if a COMPLETE partition exists

DP STATE:
  → dp[i] = can s[0..i-1] be segmented into dictionary words?
  → dp[0] = true → empty string is trivially "segmented" (base case)
  → dp[n] = our answer → can the entire string be segmented?

THIS IS 1D DP (not 2D):
  → Unlike LCS/Edit Distance which compare TWO strings
  → Here we have ONE string being decomposed
  → The dictionary is a LOOKUP resource, not a second sequence
  → Similar dimension to Climbing Stairs (Project 43) but with string operations

Step 3 — Connect to Previous DP Knowledge

Project 43 (Climbing Stairs):
  → dp[i] = number of ways to reach step i
  → dp[i] = dp[i-1] + dp[i-2]
  → Fixed transitions: step 1 or step 2 back

Project 44 (0/1 Knapsack):
  → dp[i][w] = max value with first i items and capacity w
  → Fixed items, fixed capacity increments

Project 50 (LCS) / Project 51 (Edit Distance):
  → dp[i][j] = comparing two string prefixes
  → Fixed transitions: diagonal, top, left

THIS Problem (Word Break):
  → dp[i] = can s[0..i-1] be segmented?
  → VARIABLE transitions: for EACH j < i, check if s[j..i-1] is a word
  → The number of transitions depends on dictionary words
  → This is like Climbing Stairs but with VARIABLE step sizes
  → Step sizes = lengths of dictionary words that match at position j

ANALOGY:
  Climbing Stairs: can you reach step n using steps of size 1 or 2?
  Word Break: can you reach position n using "steps" that are dictionary words?

Step 4 — Define the Recurrence

STATE:
  dp[i] = true if s[0..i-1] can be segmented into dictionary words
  dp[i] = false otherwise

BASE CASE:
  dp[0] = true → empty prefix is trivially segmentable

RECURRENCE:
  dp[i] = true if there EXISTS any j (0 ≤ j < i) such that:
    → dp[j] == true (s[0..j-1] is segmentable)
    → AND s[j..i-1] is a word in the dictionary

  In other words:
  dp[i] = OR over all j from 0 to i-1 of (dp[j] AND s.substring(j,i) in dict)

ANSWER:
  dp[n] where n = s.length()

Step 5 — Visualize the Process

wordDict = {"leet", "code"} → stored in a HashSet for O(1) lookup

dp[0] = true (base case — empty string)

dp[1]: check all j < 1
  j=0: dp[0]=true AND s[0..0]="l" in dict? NO
  dp[1] = false

dp[2]: check all j < 2
  j=0: dp[0]=true AND s[0..1]="le" in dict? NO
  j=1: dp[1]=false → skip
  dp[2] = false

dp[3]: check all j < 3
  j=0: dp[0]=true AND s[0..2]="lee" in dict? NO
  j=1: dp[1]=false → skip
  j=2: dp[2]=false → skip
  dp[3] = false

dp[4]: check all j < 4
  j=0: dp[0]=true AND s[0..3]="leet" in dict? YES! ✓
  dp[4] = true

dp[5]: check all j < 5
  j=0: dp[0]=true AND s[0..4]="leetc" in dict? NO
  j=1-3: dp[1-3]=false → skip
  j=4: dp[4]=true AND s[4..4]="c" in dict? NO
  dp[5] = false

  j=0: dp[0]=true AND s[0..7]="leetcode" in dict? NO
  j=1-3: dp[1-3]=false → skip

  dp[8] = true

ANSWER: dp[8] = true → "leetcode" = "leet" + "code" ✓

Step 6 — Two Implementation Strategies

STRATEGY A — Check all splits (j from 0 to i-1):
  for each i from 1 to n:
    for each j from 0 to i-1:
      if dp[j] AND s[j..i-1] in dict:
        dp[i] = true; break;

  → For each i, we try ALL possible split points
  → Substring check: O(i-j) per check (or O(1) with hashing)
  → Time: O(n² × L) where L = average substring length

STRATEGY B — Check all dictionary words at each position:
  for each i from 1 to n:
    for each word in dictionary:
      int len = word.length()
      if i >= len AND dp[i-len] AND s[i-len..i-1] == word:
        dp[i] = true; break;

  → For each i, we try ALL dictionary words
  → Check if the word fits ending at position i
  → Time: O(n × W × L) where W = dict size, L = max word length

WHICH IS BETTER?
  → Strategy A: better when dictionary is large but string is short
  → Strategy B: better when string is long but dictionary is small
  → Both are valid → choose based on constraints
  → We implement both

🛠️ HOW CAN WE SOLVE IT

Algorithm — Bottom-Up DP

1. Convert dictionary to a HashSet for O(1) lookups
2. Create boolean array dp of size n+1
3. dp[0] = true (empty string base case)
4. FOR i = 1 to n:
     FOR j = 0 to i-1:
       IF dp[j] == true AND s.substring(j, i) is in dictionary:
         dp[i] = true
         BREAK (no need to check more j values)
5. RETURN dp[n]

Optimization — Limit j Range by Max Word Length

→ If the longest word in dictionary has length maxLen
→ Then s[j..i-1] can only be a dictionary word if i-j ≤ maxLen
→ So j only needs to range from max(0, i-maxLen) to i-1
→ This PRUNES the inner loop significantly

OPTIMIZED:
  int maxLen = max word length in dictionary
  FOR i = 1 to n:
    FOR j = max(0, i-maxLen) to i-1:
      IF dp[j] AND s.substring(j, i) in dict:
        dp[i] = true; break;

→ Inner loop runs at most maxLen iterations instead of i
→ Time: O(n × maxLen × maxLen) for substring creation
→ Much better when maxLen << n

✅ WHAT SHOULD YOU DO

→ ALWAYS convert dictionary to a HashSet
     → List lookup: O(W) per check where W = dictionary size
     → HashSet lookup: O(L) per check where L = word length (for hashing)
     → Massive speedup for large dictionaries

→ ALWAYS set dp[0] = true
     → Empty string is the starting point
     → Without this, no word can match starting at index 0
     → dp[0]=false → dp[i]=false for all i → always returns false → WRONG

→ ALWAYS break early when dp[i] becomes true
     → Once we find ONE valid segmentation ending at i, that's enough
     → No need to check other j values for the same i
     → This is a boolean problem: one valid path suffices

→ CONSIDER limiting the inner loop by max word length
     → If maxLen = 10 and n = 1000 → inner loop is 10 instead of up to 1000
     → Dramatic speedup for long strings with short dictionary words

→ CONSIDER both directions for the inner loop
     → Forward: j from 0 to i-1 → checks longer prefixes first
     → Backward: j from i-1 down to 0 → checks shorter words first
     → Backward often finds matches faster (short words are more common)
     → With maxLen pruning: backward from i-1 to max(0, i-maxLen)

🚫 WHAT SHOULD YOU NOT DO

→ DO NOT use greedy word matching
     → Matching the longest word first can fail (goalspecial example)
     → Matching the shortest word first can also fail
     → Must consider ALL possibilities → DP is required

→ DO NOT forget dp[0] = true
     → This is the ANCHOR that makes everything work
     → dp[4] = true because dp[0]=true AND "leet" in dict
     → Without dp[0]=true, the chain never starts

→ DO NOT use a List for dictionary lookups
     → list.contains(word) is O(W) where W = dictionary size
     → set.contains(word) is O(L) where L = word length
     → For large dictionaries → orders of magnitude difference

→ DO NOT create substrings unnecessarily
     → s.substring(j, i) creates a new String object in Java
     → For very long strings → excessive memory allocation and GC pressure
     → Alternative: use s.startsWith(word, j) or character-by-character comparison
     → Or precompute hash values for O(1) substring comparison

→ DO NOT use naive recursion without memoization
     → Recursive approach: "can s[i..n-1] be segmented?"
     → Without memoization: exponential time
       → s = "aaaa...a", dict = ["a", "aa", "aaa", ...]
       → Each position has multiple choices → branching tree
     → With memoization: O(n²) → same as bottom-up DP

→ DO NOT confuse this with subsequence problems
     → Word Break: CONTIGUOUS substrings must match dictionary words
     → LCS (Project 50): non-contiguous subsequences
     → The partition must cover the ENTIRE string with no gaps
     → Every character must belong to exactly one word

🔀 WHEN TO USE THIS PATTERN

✓ You need to determine if a sequence can be PARTITIONED into valid pieces
✓ Each piece must satisfy some MEMBERSHIP criterion (in a set/dictionary)
✓ Pieces are CONTIGUOUS (substrings, not subsequences)
✓ Pieces must COVER the entire sequence (no gaps, no overlaps)
✓ The same piece can be reused (or not, depending on variant)

VARIANTS:
  → Word Break I: can the string be segmented? (boolean — this project)
  → Word Break II: return ALL valid segmentations (backtracking + DP)
  → Palindrome Partitioning: segment into palindromes
  → Minimum Cost Partitioning: segment with minimum total cost
  → Rod Cutting: partition a rod into pieces for maximum profit

REAL-WORLD ANALOGS:
  → Natural Language Processing: word segmentation for languages without spaces
     → Chinese, Japanese, Thai → no spaces between words
     → Must determine where word boundaries are
  → DNA analysis: finding gene patterns in sequences
  → URL parsing: segmenting concatenated words in URLs
     → "expertsexchange" → "experts exchange" or... other segmentations
  → Compound word detection: "basketball" → "basket" + "ball"
  → Text decompression: reconstructing original text from compressed form

🚫 WHEN NOT TO USE THIS PATTERN

✗ Pieces can be non-contiguous (subsequences)
     → That's LCS-style problems (Project 50) → 2D DP
     → Word Break requires contiguous substrings

✗ You need to TRANSFORM the string (not partition it)
     → That's Edit Distance (Project 51) → 2D DP
     → Word Break doesn't modify characters

✗ The dictionary changes dynamically
     → Rebuilding HashSet is expensive if dictionary changes often
     → For dynamic dictionaries → Trie (Project 58) is better
     → Trie supports efficient prefix matching and dynamic updates

✗ You need to find the LONGEST dictionary word in the string
     → That's a different problem → simple iteration over dictionary
     → Word Break asks about COMPLETE partitioning, not single word matching

✗ Pieces can overlap
     → Word Break assumes non-overlapping, contiguous, complete coverage
     → Overlapping patterns → different technique (suffix trees, Aho-Corasick)

⚙️ HOW IT ONLY WORKS — Internal Mechanics

The Partition Chain Invariant

INVARIANT:
  dp[i] = true means there exists a sequence of positions
  0 = p₀ < p₁ < p₂ < ... < pₖ = i
  such that s[pⱼ..pⱼ₊₁-1] is a dictionary word for all j

PROOF BY INDUCTION:

Base: dp[0] = true → trivially, the empty string is segmented (k=0)

Inductive: If dp[i] = true, it was set because:
  → dp[j] = true for some j < i
  → AND s[j..i-1] is a dictionary word
  → By induction, dp[j] has a valid partition: 0 = p₀ < ... < pₘ = j
  → Appending i: 0 = p₀ < ... < pₘ = j < pₘ₊₁ = i
  → s[j..i-1] is a valid word → the extended partition is valid ✓

CONSEQUENCE:
  → dp[n] = true guarantees a COMPLETE valid segmentation exists
  → dp[n] = false guarantees NO valid segmentation exists
  → The DP correctly captures ALL possible partition chains

Why the Dictionary as HashSet is Critical

WITHOUT HashSet (using List):
  → For each (i, j) pair: check if s[j..i-1] is in dictionary
  → List.contains() compares with EVERY word: O(W) per check
  → Total: O(n² × W × L) where L = comparison cost
  → For n=1000, W=1000, L=10: 10 billion operations → TOO SLOW

WITH HashSet:
  → HashSet.contains() computes hash and checks bucket: O(L) average
  → Total: O(n² × L) 
  → For n=1000, L=10: 10 million operations → fast

WITH HashSet + maxLen pruning:
  → Inner loop limited to maxLen iterations
  → Total: O(n × maxLen × L)
  → For n=1000, maxLen=20, L=20: 400,000 operations → instant

The Relationship to BFS/DFS on a Graph

ALTERNATIVE VIEW: Word Break as a Graph Problem

→ Nodes: positions 0, 1, 2, ..., n in the string
→ Edge from j to i: exists if s[j..i-1] is a dictionary word
→ Question: is there a PATH from node 0 to node n?

EXAMPLE:
  s = "leetcode", dict = {"leet", "code"}
  
  Edges:
    0 → 4 (s[0..3] = "leet" ✓)
    4 → 8 (s[4..7] = "code" ✓)
  
  Path: 0 → 4 → 8 → EXISTS → return true

THIS IS EXACTLY what the DP computes:
  → dp[i] = true if node i is REACHABLE from node 0
  → The DP is essentially BFS/DFS on this implicit graph
  → But structured as DP for cleaner implementation

CONNECTION TO PROJECT 49 (Jump Game):
  → Jump Game: from index i, can jump to i+1..i+nums[i]
  → Word Break: from position j, can "jump" to j+len(word) if word matches
  → Both are reachability problems on implicit graphs
  → Jump Game used greedy (contiguous reachable set)
  → Word Break can't use greedy (reachable set is NOT contiguous)
  → Word Break needs full DP

Why Greedy Fails — Formal Argument

CLAIM: No greedy strategy (longest-first, shortest-first, etc.) 
       is guaranteed to find a valid segmentation.

COUNTEREXAMPLE for longest-first:
  s = "aab", dict = {"a", "aab"}
  → Longest match at position 0: "aab" (length 3) → covers entire string → TRUE
  → But what about: s = "aaab", dict = {"a", "aab"}
  → Longest match at 0: "aab" (length 3) → remaining "b" → NOT in dict → FAIL
  → But correct: "a" + "aab" → TRUE
  → Longest-first FAILS

COUNTEREXAMPLE for shortest-first:
  s = "goalspecial", dict = {"go", "goal", "special"}
  → Shortest match at 0: "go" (length 2) → remaining "alspecial" → can't segment
  → Correct: "goal" (length 4) + "special" → TRUE
  → Shortest-first FAILS

ROOT CAUSE:
  → The choice at position 0 affects what's available at later positions
  → No local criterion can guarantee global optimality
  → Must consider ALL possibilities → DP is necessary

Time Complexity Analysis

BASIC DP (Strategy A):
  → Outer loop: n iterations
  → Inner loop: up to i iterations (0 to i-1)
  → Substring creation: O(i-j) per check
  → HashSet lookup: O(i-j) for hashing
  → Total: O(n³) worst case (sum of i² for substring creation)

OPTIMIZED with maxLen pruning:
  → Outer loop: n iterations
  → Inner loop: at most maxLen iterations
  → Substring creation: O(maxLen) per check
  → HashSet lookup: O(maxLen)
  → Total: O(n × maxLen²)
  → If maxLen is constant → O(n) → linear!

STRATEGY B (iterate over dictionary):
  → Outer loop: n iterations
  → Inner loop: W dictionary words
  → String comparison: O(L) per word
  → Total: O(n × W × L)

Space Complexity

→ dp array: O(n)
→ HashSet of dictionary: O(W × L) where W = words, L = average length
→ TOTAL: O(n + W × L)
→ Much less than 2D DP problems (LCS, Edit Distance)

⚠️ COMMON PITFALLS AND FAILURE MODES

Pitfall 1: Forgetting dp[0] = true

s = "leet", dict = {"leet"}

WITHOUT dp[0] = true:
  dp[1]: j=0 → dp[0]=false → skip → dp[1]=false
  dp[2]: all false → dp[2]=false
  dp[3]: all false → dp[3]=false
  dp[4]: j=0 → dp[0]=false → skip → dp[4]=false
  ANSWER: false → WRONG!

WITH dp[0] = true:
  dp[4]: j=0 → dp[0]=true AND "leet" in dict → dp[4]=true
  ANSWER: true → CORRECT ✓

→ dp[0]=true is the SEED that starts the chain
→ Without it, nothing can ever become true

Pitfall 2: Substring Index Off-by-One

In Java: s.substring(j, i) returns characters from index j to i-1 (inclusive)
  → s = "leetcode"
  → s.substring(0, 4) = "leet" (indices 0,1,2,3)
  → s.substring(4, 8) = "code" (indices 4,5,6,7)

COMMON MISTAKE:
  → s.substring(j, i-1) → misses last character
  → s.substring(j, i+1) → includes extra character
  → Always verify with a concrete example

Pitfall 3: TLE on Long Strings with Repeated Characters

s = "aaaa...a" (n = 10000)
dict = ["a", "aa", "aaa", ..., "a×100"]

→ Every position has multiple valid words
→ Without maxLen optimization: inner loop runs up to i for each i
→ O(n²) substring checks → can be slow for n > 10000

FIX: Use maxLen to limit inner loop
  → maxLen = 100 → inner loop is at most 100 iterations
  → O(n × 100) = O(n) → fast

FIX: Use Trie instead of HashSet (connects to Project 58)
  → Trie can check ALL matching prefixes in one traversal
  → Avoids creating multiple substrings

Pitfall 4: Not Breaking Early When dp[i] = true

WRONG (inefficient):
  for (int j = 0; j < i; j++) {
      if (dp[j] && dict.contains(s.substring(j, i))) {
          dp[i] = true;
          // continues checking other j values → wasted work
      }
  }

RIGHT (efficient):
  for (int j = 0; j < i; j++) {
      if (dp[j] && dict.contains(s.substring(j, i))) {
          dp[i] = true;
          break;  // ← found one valid segmentation → stop
      }
  }

→ Boolean problem: one valid path is sufficient
→ Break saves significant time in practice

Pitfall 5: Dictionary Words Longer Than String

s = "ab", dict = {"abc", "a", "b"}

→ "abc" has length 3 > length of s (2) → can never match
→ Should be harmlessly skipped in the inner loop
→ Strategy A: j ranges from 0 to i-1 → substring is at most length i ≤ n → safe
→ Strategy B: check word.length() ≤ n before processing → optimization

📈 SCALABILITY CONSIDERATIONS

n = string length, W = dictionary size, L = max word length

BASIC DP: O(n² × L)
  → n=300 → 300² × 20 ≈ 1.8M → instant
  → n=1000 → 1M × 20 ≈ 20M → fast
  → n=10,000 → 100M × 20 ≈ 2B → slow
  → n=100,000 → too slow

OPTIMIZED DP (maxLen pruning): O(n × L²)
  → n=10,000, L=20 → 10K × 400 = 4M → instant
  → n=100,000, L=20 → 100K × 400 = 40M → fast
  → n=1,000,000, L=20 → 1M × 400 = 400M → manageable

TRIE-BASED (Project 58 preview): O(n × L)
  → Even better for large dictionaries
  → Single traversal per position checks all matching prefixes
  → n=1,000,000, L=20 → 20M → fast

🧠 KEY TAKEAWAY

WHAT WE LEARNED:
→ String partitioning is a 1D DP problem where transitions are VARIABLE
→ Dictionary words define which "jumps" are possible from each position
→ dp[i] = true if ANY split point j makes both dp[j] and s[j..i-1] valid
→ dp[0] = true is the CRITICAL base case that seeds the entire chain
→ HashSet gives O(1) dictionary lookups → essential for performance
→ Max word length pruning limits the inner loop → dramatic speedup
→ Greedy FAILS because word choice at position j affects all later positions
→ Word Break is like Jump Game but with non-contiguous reachable positions

THE CORE MENTAL MODEL:
→ "Can I hop from position 0 to position n?"
→ "At each position, check: can I land here? (dp[j] = true)"
→ "If yes, what dictionary words start here? (s[j..i-1] in dict)"
→ "Each valid word creates a new landing pad at position i"
→ "If position n is a landing pad → the string can be segmented"

PATTERN SIGNATURE:
→ Single sequence to partition ✓
→ Each piece must satisfy a membership criterion ✓
→ Pieces are contiguous and non-overlapping ✓
→ Complete coverage required ✓
→ = 1D DP WITH DICTIONARY-DRIVEN TRANSITIONS

THE GREEDY-DP SPECTRUM (Completed):
  Project 19: Greedy works → Activity Selection (sort by end time)
  Project 49: Greedy works → Jump Game (contiguous reachable set)
  Project 50: Greedy fails → LCS (early match blocks better alignment)
  Project 51: Greedy fails → Edit Distance (early operation blocks cheaper path)
  Project 52: Greedy fails → Word Break (early word blocks valid partition)
  
  → WHEN greedy works: problem has monotonic/contiguous structure
  → WHEN greedy fails: choices have non-local consequences → use DP
  → Recognizing which regime you're in = the KEY interview skill
