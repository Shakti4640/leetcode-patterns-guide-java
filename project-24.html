PROJECT 24: Binary Tree Zigzag Level Order Traversal

🎯 Tree BFS → Alternating Direction Flag → Level-Aware Queue Processing

🧩 THE HYPOTHETICAL PROBLEM

→ You are given the ROOT of a binary tree
→ Return the ZIGZAG level order traversal of its nodes
→ Level 0: left to right
→ Level 1: right to left
→ Level 2: left to right
→ Level 3: right to left
→ ... alternating at every level

Example 1:
              3
             / \
            9   20
               / \
              15   7

  Level 0 (L→R): [3]
  Level 1 (R→L): [20, 9]
  Level 2 (L→R): [15, 7]
  
  Output: [[3], [20, 9], [15, 7]]

Example 2:
              1
             / \
            2   3
           / \   \
          4   5   6

  Level 0 (L→R): [1]
  Level 1 (R→L): [3, 2]
  Level 2 (L→R): [4, 5, 6]
  
  Output: [[1], [3, 2], [4, 5, 6]]

Example 3:
              1
             / \
            2   3
           / \ / \
          4  5 6  7
         /        \
        8          9

  Level 0 (L→R): [1]
  Level 1 (R→L): [3, 2]
  Level 2 (L→R): [4, 5, 6, 7]
  Level 3 (R→L): [9, 8]
  
  Output: [[1], [3, 2], [4, 5, 6, 7], [9, 8]]

Example 4:
  Input: null (empty tree)
  Output: [] (empty list)

Example 5:
  Input: single node [1]
  Output: [[1]]

🔍 HOW TO APPROACH IT

Step 1 — Recall Project 17: Standard Level Order Traversal (BFS)

→ Project 17 taught us the BFS template:
     1. Put root in queue
     2. While queue is not empty:
          a. Get level size = queue.size()
          b. Process exactly levelSize nodes
          c. For each node: add value to current level list, enqueue children
     3. Add current level list to result

→ This produces: [[3], [9, 20], [15, 7]] → always left to right

→ Project 24 needs: [[3], [20, 9], [15, 7]] → alternating direction
→ The BFS TRAVERSAL stays the same → we always process left child before right
→ Only the ORDER OF VALUES in the level list changes
→ On odd levels, we REVERSE the collected values (or insert them differently)

Step 2 — Why Not Change Traversal Order?

WRONG APPROACH: "On odd levels, process right child before left"

→ This seems intuitive but BREAKS the next level's order
→ If we process level 1 as [20, 9] by enqueueing right first:
     → Queue after level 1: [7, 15, ...] (20's children first)
     → Level 2 would be: [7, 15] → but we want [15, 7]!
     → The traversal order of level N affects level N+1's queue order
     → Changing traversal order creates a CASCADE of problems

→ CORRECT APPROACH: Always traverse left-to-right
     → Collect values normally
     → On odd levels, REVERSE the collected values
     → Or use a deque to insert at front on odd levels
     → Traversal order stays consistent → queue order is correct for all levels

Step 3 — Three Strategies for Zigzag

STRATEGY A: Reverse on odd levels
  → Collect values left-to-right (always)
  → After collecting a level: if level is odd → reverse the list
  → Simple, clear, slight extra work for reversal

STRATEGY B: LinkedList with direction-aware insertion
  → Use a LinkedList for each level
  → Even levels: addLast() (append to end)
  → Odd levels: addFirst() (insert at beginning)
  → No separate reversal step needed
  → Values end up in correct order as we go

STRATEGY C: Index-based placement
  → Use an array of known size (levelSize)
  → Even levels: fill left to right (index 0, 1, 2, ...)
  → Odd levels: fill right to left (index size-1, size-2, ...)
  → Direct placement → no reversal needed

→ Strategy B is the cleanest → we'll implement it as primary
→ Strategy A is the simplest to understand → we'll implement for comparison

  Process node(7): addLast(7) → levelList = [4, 5, 6, 7]
Queue after: [8, ..., 9] (if children exist)
Result: [[1], [3, 2], [4, 5, 6, 7]]

Toggle direction after each level: true → false → true → false → ...

Step 5 — Why addFirst Works for Reverse Order

On an odd level, nodes come out of queue in LEFT-TO-RIGHT order:
  → Queue: [2, 3]
  → First out: 2 (left child)
  → Second out: 3 (right child)

We want RIGHT-TO-LEFT order: [3, 2]

Using addFirst:
  → addFirst(2) → list = [2]
  → addFirst(3) → list = [3, 2]  ← 3 pushed 2 to the right

Result: [3, 2] → RIGHT-TO-LEFT ✓

It works because:
  → Each new value is inserted at position 0
  → Previous values shift right
  → First processed node ends up LAST
  → Last processed node ends up FIRST
  → This is exactly reversal without explicit reverse step

🛠️ HOW CAN WE SOLVE IT

The Algorithm — Precise Steps

zigzagLevelOrder(root):

  1. IF root is null → return empty list

  2. CREATE result list
  3. CREATE queue, add root
  4. SET leftToRight = true

  5. WHILE queue is not empty:
       a. levelSize = queue.size()
       b. CREATE levelList (LinkedList for O(1) addFirst)
       
       c. FOR i = 0 to levelSize - 1:
            → node = queue.poll()
            
            → IF leftToRight:
                 levelList.addLast(node.val)
            → ELSE:
                 levelList.addFirst(node.val)
            
            → IF node.left != null: queue.offer(node.left)
            → IF node.right != null: queue.offer(node.right)
       
       d. result.add(levelList)
       e. leftToRight = !leftToRight   ← TOGGLE direction

  6. RETURN result

✅ WHAT SHOULD YOU DO

→ ALWAYS keep the BFS traversal order consistent (left then right)
     → Never change the enqueueing order
     → Only change how VALUES are placed in the level list
     → Traversal order affects future levels → don't touch it

→ ALWAYS use LinkedList for the level list (not ArrayList)
     → addFirst on ArrayList is O(n) → shifts all elements
     → addFirst on LinkedList is O(1) → just pointer manipulation
     → For large levels, this makes a real difference

→ ALWAYS toggle the flag AFTER processing the entire level
     → Not inside the for loop
     → Not before the for loop
     → After adding levelList to result, then toggle
     → One toggle per level

→ ALWAYS start with leftToRight = true
     → Level 0 (root level) is left-to-right by convention
     → If problem says level 0 is right-to-left → start with false
     → READ THE PROBLEM for starting direction

→ ALWAYS use the level-size snapshot technique from Project 17
     → int size = queue.size() BEFORE the for loop
     → Queue grows during the loop → size must be captured first
     → This is the SAME technique → Project 17 knowledge

🚫 WHAT SHOULD YOU NOT DO

→ DO NOT change the order of child enqueueing
     WRONG:
       if (!leftToRight) {
           queue.offer(node.right);  // right first on odd levels
           queue.offer(node.left);
       }
     → This corrupts the queue order for the NEXT level
     → Zigzag on level N would break level N+1

→ DO NOT use ArrayList with addFirst-style insertion
     WRONG:
       levelList.add(0, node.val);  // O(n) per insertion!
     → ArrayList.add(0, val) shifts all elements → O(n) per call
     → For a level with k nodes → O(k²) per level
     → Use LinkedList.addFirst → O(1) per call

→ DO NOT forget to handle null root
     → null root → return empty list
     → Enqueueing null → NullPointerException when polling

→ DO NOT use a Deque (ArrayDeque) as both queue AND insertion tool
     → The queue manages BFS traversal
     → The level list manages value collection
     → These are SEPARATE concerns → separate data structures
     → Mixing them creates confusion

→ DO NOT reverse the entire result at the end
     → Some think: "collect all levels normally, then reverse alternate levels"
     → This works but is an extra O(n) pass over all nodes
     → addFirst handles it during collection → no extra pass

→ DO NOT use DFS for this problem
     → DFS CAN solve it (with level tracking) but it's unnatural
     → BFS processes level-by-level → natural fit for level-order problems
     → Project 17 established BFS as the tool for level-order → use it

⚙️ HOW IT ONLY WORKS — Internal Mechanics

The Direction Flag as a Toggle

leftToRight is a BOOLEAN that flips every level:

Level 0: leftToRight = true  → addLast  → [1]
Level 1: leftToRight = false → addFirst → [3, 2]
Level 2: leftToRight = true  → addLast  → [4, 5, 6, 7]
Level 3: leftToRight = false → addFirst → [9, 8]

Toggle: leftToRight = !leftToRight

This is the simplest possible state machine:
  → Two states: true / false
  → One transition: toggle after each level
  → No complex logic needed

Alternative: use level number
  → if (level % 2 == 0) → left to right
  → if (level % 2 == 1) → right to left
  → Same effect, slightly more computation
  → Boolean toggle is cleaner

Why addFirst Produces Reverse Order — Formal Proof

CLAIM: If nodes arrive in order [a, b, c, d] and we addFirst each:
  → Result is [d, c, b, a]

PROOF by induction on insertion:
  After addFirst(a): [a]
  After addFirst(b): [b, a]
  After addFirst(c): [c, b, a]
  After addFirst(d): [d, c, b, a]
  
  Each new element goes to position 0
  All previous elements shift right by 1
  First inserted element ends up at position n-1 (last)
  Last inserted element ends up at position 0 (first)
  → This IS the reversal of the insertion order ∎

For our BFS:
  → Nodes come from queue in left-to-right order
  → addFirst produces right-to-left order
  → Exactly what zigzag needs on odd levels ✓

Why Traversal Order Must Stay Left-to-Right

CRITICAL INSIGHT:
  The queue order for level N+1 depends on the PROCESSING ORDER of level N

Example tree:
        1
       / \
      2   3
     / \   \
    4   5   6

Level 0: Queue = [1]
  Process 1: enqueue left(2), right(3)
  Queue after: [2, 3]

Level 1: Queue = [2, 3]
  Process 2: enqueue left(4), right(5)
  Process 3: enqueue right(6)
  Queue after: [4, 5, 6]

Level 2: Queue = [4, 5, 6]
  → Correct left-to-right order for this level ✓

NOW IMAGINE we processed level 1 in reverse order [3, 2]:
  Process 3: enqueue right(6)
  Process 2: enqueue left(4), right(5)
  Queue after: [6, 4, 5]  ← WRONG ORDER for level 2!
  
  Level 2 would be: [6, 4, 5] → but correct is [4, 5, 6]
  → Even after reversal: [5, 4, 6] → STILL WRONG

CONCLUSION:
  → NEVER change the queue processing order
  → Always process left-to-right
  → Handle zigzag ONLY in the value collection step
  → The queue is for TRAVERSAL, the level list is for PRESENTATION
  → Keep these concerns SEPARATE

Time and Space Complexity

TIME: O(n)
→ Every node is enqueued once and dequeued once → O(n)
→ addLast is O(1) for LinkedList
→ addFirst is O(1) for LinkedList
→ Direction toggle is O(1) per level
→ Total: O(n) — identical to standard BFS

SPACE: O(n)
→ Queue: at most one full level → O(w) where w = max width
→ For balanced tree: max width ≈ n/2 → O(n)
→ For skewed tree: max width = 1 → O(1)
→ Result list: stores all n node values → O(n)
→ Total: O(n) — same as Project 17

COMPARISON WITH ALTERNATIVE STRATEGIES:
  Strategy A (reverse on odd levels):
    Time: O(n) + O(n/2) for reversals = O(n) → same
    Space: O(n) → same
    But: extra pass for reversal → slightly more work
    
  Strategy B (addFirst/addLast):
    Time: O(n) → each insertion O(1)
    Space: O(n)
    Cleanest → no extra passes
    
  Strategy C (index-based placement):
    Time: O(n)
    Space: O(n) → need pre-allocated array per level
    More complex → less readable

LinkedList vs ArrayList Performance Detail

For a level with K nodes:

ArrayList:
  addLast (even level): O(1) amortized → total O(K) ✓
  add(0, val) (odd level): O(K) per insertion → total O(K²) ✗
  
  K=1000 → even: 1000 ops → odd: 1,000,000 ops → 1000x slower!

LinkedList:
  addLast (even level): O(1) → total O(K) ✓
  addFirst (odd level): O(1) → total O(K) ✓
  
  K=1000 → even: 1000 ops → odd: 1000 ops → same speed ✓

For small K (< 100): difference is negligible
For large K (> 1000): LinkedList is significantly faster on odd levels
For interview: mentioning this shows awareness of data structure performance

⚠️ COMMON PITFALLS AND FAILURE MODES

Pitfall 1: Using ArrayList.add(0, val) Instead of LinkedList.addFirst

SYMPTOM: Correct output but Time Limit Exceeded on large trees

CAUSE:
  ArrayList.add(0, val) → O(n) per insertion → O(n²) per level
  
  For a complete binary tree with 20 levels:
    Level 19 has 2^19 = 524,288 nodes
    ArrayList: 524,288² / 2 ≈ 137 billion operations → TLE
    LinkedList: 524,288 operations → fast

FIX: Use LinkedList for level lists
  LinkedList<Integer> levelList = new LinkedList<>();
  levelList.addFirst(val);  // O(1) ✓

Pitfall 2: Toggling Direction Inside the Loop

WRONG:
  for (int i = 0; i < size; i++) {
      node = queue.poll();
      if (leftToRight) levelList.addLast(node.val);
      else levelList.addFirst(node.val);
      leftToRight = !leftToRight;  // ← WRONG! Toggles per NODE, not per LEVEL
  }

→ Direction flips for EVERY node → bizarre mixed order
→ Node 1: addLast, Node 2: addFirst, Node 3: addLast, ...
→ Completely wrong output

RIGHT: Toggle AFTER the entire level is processed
  for (int i = 0; i < size; i++) { ... }
  result.add(levelList);
  leftToRight = !leftToRight;  // ← Toggle once per LEVEL ✓

Pitfall 3: Forgetting to Snapshot Queue Size

WRONG:
  while (!queue.isEmpty()) {
      for (int i = 0; i < queue.size(); i++) {  // queue.size() changes during loop!
          ...
          queue.offer(node.left);   // size increases here
          queue.offer(node.right);  // and here
      }
  }

→ queue.size() is re-evaluated each iteration of the for loop
→ As we add children, size grows → loop processes nodes from NEXT level
→ Level boundaries are lost

RIGHT:
  int size = queue.size();  // snapshot BEFORE the loop
  for (int i = 0; i < size; i++) { ... }

→ This is the SAME pitfall from Project 17
→ Always snapshot → never use dynamic queue.size() in loop condition

Pitfall 4: Starting Direction Wrong

→ Most problems: level 0 = left to right → start with true
→ Some problems: level 0 = right to left → start with false
→ Some problems: don't specify → read examples carefully

→ Test with the root level:
     → If output[0] = [root.val] → direction doesn't matter (single element)
     → Check output[1] → is it left-to-right or right-to-left?
     → Adjust starting flag accordingly

Pitfall 5: Null Children Creating Problems

→ Tree nodes may have one child or no children
→ Always check: if (node.left != null) before enqueueing
→ Enqueueing null → next poll() returns null → NullPointerException on .val

→ This is the SAME null check from Project 17
→ Standard BFS practice → always guard child enqueueing

Pitfall 6: Empty Tree

→ root == null → return empty list
→ Don't try to enqueue null root
→ Check at the very beginning

  if (root == null) return new ArrayList<>();
