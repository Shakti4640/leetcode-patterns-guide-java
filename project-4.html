PROJECT 4: Squares of a Sorted Array

🎯 Two Pointers → Merge from Both Ends → Absolute Value Comparison

🧩 THE HYPOTHETICAL PROBLEM

→ You are given a SORTED array of integers (ascending order)
→ The array may contain NEGATIVE numbers
→ Square every element
→ Return the result in SORTED (ascending) order
→ Must run in O(n) time

Example 1:
  Input:  [-4, -1, 0, 3, 10]
  Output: [0, 1, 9, 16, 100]

Example 2:
  Input:  [-7, -3, 2, 3, 11]
  Output: [4, 9, 9, 49, 121]

Example 3:
  Input:  [-5, -4, -3, -2, -1]
  Output: [1, 4, 9, 16, 25]

Example 4:
  Input:  [1, 2, 3, 4, 5]
  Output: [1, 4, 9, 16, 25]

Example 5:
  Input:  [0]
  Output: [0]

🔍 HOW TO APPROACH IT

Step 1 — Why Is This Problem Non-Trivial?

→ If the array had ONLY positive numbers → square each → still sorted → done
     [1, 2, 3, 4, 5] → [1, 4, 9, 16, 25] → already sorted ✓

→ If the array had ONLY negative numbers → square each → REVERSED order
     [-5, -4, -3, -2, -1] → [25, 16, 9, 4, 1] → sorted in DESCENDING ✗
     → Need to reverse → [1, 4, 9, 16, 25] ✓

→ If the array has BOTH negatives and positives → squaring BREAKS the order
     [-4, -1, 0, 3, 10] → [16, 1, 0, 9, 100] → completely unsorted ✗

→ The PROBLEM: negative numbers, when squared, become positive
→ The most negative number produces the LARGEST square on the left side
→ The most positive number produces the LARGEST square on the right side
→ The LARGEST squares live at BOTH ENDS of the original array
→ The SMALLEST squares live near the MIDDLE (around zero)

Step 2 — Think About the Brute Force First

→ Naive approach: Square everything, then sort
     Step 1: Square each element → O(n)
     Step 2: Sort the result → O(n log n)
     Total: O(n log n)

→ This WORKS but the problem asks for O(n)
→ We are ignoring the fact that the input is ALREADY sorted
→ The sorted structure gives us enough information to avoid re-sorting

Step 3 — The Key Insight: Largest Squares at Both Ends

→ Original sorted array: [-4, -1, 0, 3, 10]
→ Absolute values:       [ 4,  1, 0, 3, 10]
→ The LARGEST absolute value is at one of the TWO ENDS:
     → Left end: |-4| = 4
     → Right end: |10| = 10
     → The larger is 10 → its square (100) is the LARGEST in the result

→ After placing the largest → remove it from consideration
→ The NEXT largest is again at one of the two ends of what remains
→ This is a REPEATED process of picking from two ends

→ This is EXACTLY the convergence pattern from Project 1!
→ Two pointers at opposite ends → compare → pick the larger → advance that pointer

→ But unlike Project 1 (which searched for a pair):
     → We are BUILDING an output array (like Project 3's write concept)
     → We fill the output from the BACK (largest first) to the FRONT (smallest last)

Step 4 — Visualize the Process

Array: [-4, -1, 0, 3, 10]    Result: [_, _, _, _, _]

Step 1: left=0 (-4), right=4 (10)
  |-4|=4 vs |10|=10 → right is bigger
  result[4] = 10² = 100    → result: [_, _, _, _, 100]
  right--

Step 2: left=0 (-4), right=3 (3)
  |-4|=4 vs |3|=3 → left is bigger
  result[3] = (-4)² = 16   → result: [_, _, _, 16, 100]
  left++

Step 3: left=1 (-1), right=3 (3)
  |-1|=1 vs |3|=3 → right is bigger
  result[2] = 3² = 9       → result: [_, _, 9, 16, 100]
  right--

Step 4: left=1 (-1), right=2 (0)
  |-1|=1 vs |0|=0 → left is bigger
  result[1] = (-1)² = 1    → result: [_, 1, 9, 16, 100]
  left++

Step 5: left=2 (0), right=2 (0)
  left == right → only one element left
  result[0] = 0² = 0       → result: [0, 1, 9, 16, 100]

DONE → [0, 1, 9, 16, 100] ✓

🛠️ HOW CAN WE SOLVE IT

The Algorithm — Line by Line Logic

1. Create a result array of same size as input
2. Place LEFT pointer at index 0 (most negative / smallest)
3. Place RIGHT pointer at index n-1 (most positive / largest)
4. Place WRITE pointer at index n-1 of result (filling from the BACK)
5. WHILE left <= right:
     a. Compute leftSquare = nums[left] * nums[left]
     b. Compute rightSquare = nums[right] * nums[right]
     c. IF leftSquare >= rightSquare:
          → Left end has larger (or equal) square
          → result[write] = leftSquare
          → left++
     d. ELSE:
          → Right end has larger square
          → result[write] = rightSquare
          → right--
     e. write--
6. Return result array

Why left <= right (Not left < right)?

→ In Project 1: left < right → we needed TWO different elements for a pair
→ Here: left <= right → when left == right, ONE element remains unplaced
→ That last element must still be squared and placed in the result
→ If we used left < right → we'd miss the final element

Example: [5] → left=0, right=0 → left == right
  → Must still place 5² = 25 into result[0]
  → left < right would skip this → result would be empty → WRONG

✅ WHAT SHOULD YOU DO

→ ALWAYS compare ABSOLUTE VALUES (or equivalently, compare squares)
     → Don't compare raw values: -4 < 3 but |-4| > |3|
     → The square of a number IS its absolute value squared
     → Comparing squares directly avoids needing Math.abs()

→ ALWAYS fill the result from the BACK (index n-1 → 0)
     → We find the LARGEST squares first (from the ends)
     → Largest goes at the back of the sorted result
     → Write pointer decrements from n-1 to 0

→ ALWAYS use left <= right (not left < right)
     → Must process the final remaining element
     → Covered in detail above

→ ALWAYS create a NEW result array
     → Unlike Project 3 which modified in-place
     → Here we CANNOT modify in-place because:
          → We read from both ends simultaneously
          → Writing to the array would corrupt unread data
     → O(n) space for the result is REQUIRED and acceptable

→ ALWAYS decrement write pointer every iteration
     → Every step places exactly ONE element
     → Write moves backward unconditionally

🚫 WHAT SHOULD YOU NOT DO

→ DO NOT square first, then sort
     → O(n log n) when O(n) is achievable
     → Wastes the sorted-input information

→ DO NOT compare raw values instead of absolute values
     BAD:  if (nums[left] > nums[right])
     → -10 is NOT greater than 5, but |-10| > |5|
     → Raw comparison gives wrong decision

     GOOD: if (nums[left] * nums[left] >= nums[right] * nums[right])
     → Compares magnitudes correctly

→ DO NOT fill the result from the FRONT (index 0 → n-1)
     → We discover LARGEST values first
     → If we fill from the front → result is in DESCENDING order → wrong
     → Must fill from the back to get ascending order

→ DO NOT try to do this in-place on the original array
     → Reading from both ends while writing → corrupts data
     → Project 3's in-place worked because read was always AHEAD of write
     → Here read and write positions are UNRELATED → in-place is unsafe

→ DO NOT forget integer overflow when squaring
     → nums[i] can be up to 10^4 or 10^5 depending on constraints
     → (10^5)² = 10^10 → overflows int (max ~2.1 × 10^9)
     → Use long for squares if values can be large
     → Or check constraints: if |nums[i]| ≤ 46340 → int is safe

🔀 WHEN TO USE THIS PATTERN

✓ Input is sorted but a TRANSFORMATION breaks the sort order
✓ The transformation creates a "U-shape" or "V-shape" relationship:
     → Values are large at both ends, small in the middle
     → OR values are small at both ends, large in the middle
✓ You need to MERGE two sorted sequences into one:
     → Left half (reversed) and right half → merge from ends
✓ You need O(n) time and the input gives directional information
✓ Variants:
     → Squares of sorted array (this project)
     → Merge two sorted arrays into one (similar end-merge technique)
     → Absolute value sorting of a sorted array
     → Any "two sorted subsequences hidden in one array" scenario

🚫 WHEN NOT TO USE THIS PATTERN

✗ The transformation does NOT create a predictable value distribution
     → If squaring doesn't create largest-at-ends pattern
     → Example: cubing preserves sort order → no merge needed
     → [-3, -1, 2, 4] → [-27, -1, 8, 64] → already sorted

✗ The input is NOT sorted
     → Without sorted input → no directional guarantees
     → Can't know that ends have largest values
     → Must fall back to O(n log n) sort

✗ The problem requires IN-PLACE modification with O(1) space
     → This technique needs O(n) space for the result array
     → Can't build from the back in-place without corrupting reads
     → Different approach needed for strict O(1) space

✗ Multiple transformations need to be composed
     → Squaring then cubing then modular arithmetic
     → The simple end-merge only works for one predictable transformation

⚙️ HOW IT ONLY WORKS — Internal Mechanics

The V-Shape Property

When you square elements of a sorted array with negatives:

Original (sorted):  [-7, -3, -1, 0, 2, 5, 8]
Squares:            [49,  9,  1, 0, 4, 25, 64]

Plot the squares:
  49 ·                                    · 64
     ·                               · 
     ·                          · 25
      · 9                  · 
         ·            · 4
           · 1    · 
              · 0

→ This is a V-SHAPE (or U-shape)
→ Largest values at the EXTREMES
→ Smallest values near ZERO (the valley)

→ The left half [49, 9, 1, 0] is sorted DESCENDING
→ The right half [0, 4, 25, 64] is sorted ASCENDING
→ We have TWO sorted sequences (one reversed) meeting in the middle

→ MERGING two sorted sequences is a classic technique
→ But instead of merging from the FRONT (smallest first)
→ We merge from the BACK (largest first) because the largest are at the ENDS
→ This is a REVERSE MERGE using two pointers

How This Synthesizes Projects 1 and 3

FROM PROJECT 1 (Convergence):
→ Two pointers at opposite ends
→ Compare values at both ends
→ Move the pointer that "wins" the comparison
→ Pointers converge toward the middle

FROM PROJECT 3 (Write Pointer):
→ A separate write position that tracks where to place the next element
→ Write pointer advances (here: decrements) with each placement

SYNTHESIS (Project 4):
→ LEFT and RIGHT pointers from Project 1 → read from both ends
→ WRITE pointer from Project 3 → write to result array
→ The DECISION from Project 1 → which end has the larger square
→ The PLACEMENT from Project 3 → put it at the write position
→ This is a MERGE operation powered by two-pointer convergence

Why Filling from the Back is Necessary

→ We discover elements in DESCENDING order of magnitude
→ First: the largest square (from one end)
→ Second: the next largest (from the other end or same end)
→ ...
→ Last: the smallest square (near zero)

→ If we fill from index 0 → [100, 16, 9, 1, 0] → DESCENDING → wrong
→ If we fill from index n-1 → [0, 1, 9, 16, 100] → ASCENDING → correct

→ The write pointer starts at n-1 and decrements
→ This automatically produces ascending order

Time Complexity Breakdown

→ left starts at 0, moves right → at most n moves
→ right starts at n-1, moves left → at most n moves
→ TOTAL left + right moves = exactly n (each element is processed once)
→ Each step: O(1) work (one comparison, one multiplication, one assignment)
→ TOTAL: O(n) time

→ Compare with brute force: O(n) square + O(n log n) sort = O(n log n)
→ Two-pointer merge: O(n) → better by a factor of log n

Space Complexity Breakdown

→ Result array: O(n) — REQUIRED (can't return in-place)
→ Three integer variables: left, right, write
→ TOTAL: O(n) space

→ This is NOT wasteful:
     → The problem asks us to RETURN a new array
     → We MUST allocate O(n) for the output
     → No additional space beyond what's required
     → So the "extra" space is O(1) — only the pointers

⚠️ COMMON PITFALLS AND FAILURE MODES

Pitfall 1: Comparing Raw Values Instead of Magnitudes

BAD:
  if (nums[left] > nums[right])  →  -10 > 5?  → false → WRONG decision

→ -10 is "less than" 5 as a raw value
→ But (-10)² = 100 > 25 = 5²
→ The left side should win, but raw comparison says right wins

GOOD:
  if (Math.abs(nums[left]) > Math.abs(nums[right]))
  OR equivalently:
  if (nums[left] * nums[left] > nums[right] * nums[right])

→ Comparing squares or absolute values gives correct magnitude ordering

Pitfall 2: Using left < right Instead of left <= right

Array: [3]  → left=0, right=0

With left < right:
  → Loop doesn't execute
  → result[0] is never set → returns [0] → WRONG (should be [9])

With left <= right:
  → Loop executes once
  → result[0] = 9 → CORRECT

→ The <= handles the final remaining element
→ Without it → one element is always missed

Pitfall 3: Forgetting to Decrement Write Pointer

→ Every iteration MUST decrement write
→ If you forget: all squares get placed at the same index
→ result[n-1] gets overwritten repeatedly → only last square survives

→ write-- must happen UNCONDITIONALLY after every placement
→ It's outside the if/else — it happens regardless of which pointer moved

Pitfall 4: Integer Overflow on Squaring

→ If nums[i] = 100,000 → nums[i] * nums[i] = 10,000,000,000
→ This overflows Java's int (max: 2,147,483,647)

FIX 1: Use long for computation
  long leftSq = (long) nums[left] * nums[left];

FIX 2: Check problem constraints
  → LeetCode's version: |nums[i]| ≤ 10^4 → max square = 10^8 → fits in int ✓
  → But if constraints say |nums[i]| ≤ 10^5 → need long

→ ALWAYS check constraints before deciding int vs long

Pitfall 5: All Positives or All Negatives Edge Cases

All positives: [1, 2, 3, 4, 5]
→ Squares: [1, 4, 9, 16, 25] → already sorted
→ Algorithm still works: right pointer always wins
→ left never moves → right decrements from n-1 to 0
→ Result fills correctly from back to front

All negatives: [-5, -4, -3, -2, -1]
→ Squares: [25, 16, 9, 4, 1] → reverse sorted
→ Algorithm still works: left pointer always wins
→ right never moves → left increments from 0 to n-1
→ Result fills correctly from back to front

→ No special handling needed for these cases
→ The algorithm NATURALLY handles them

📈 SCALABILITY CONSIDERATIONS

→ n = 100 → brute force O(n log n) ≈ 700 ops → fast
→ n = 1,000,000 → brute force O(n log n) ≈ 20,000,000 ops → ok
→ n = 1,000,000 → two pointers O(n) = 1,000,000 ops → faster
→ n = 100,000,000 → brute force ≈ 2.6 billion ops → slow
→ n = 100,000,000 → two pointers = 100M ops → manageable

→ The log n factor matters at scale:
     → For n = 10^8: log n ≈ 27
     → Brute force does 27x more work than two pointers
     → That's the difference between 1 second and 27 seconds

→ Memory: both approaches need O(n) for the output
→ But the two-pointer approach has better CACHE BEHAVIOR:
     → Reads sequentially from both ends → predictable access pattern
     → Sort-based approach has random access during sorting → more cache misses

🔍 INTERPRETABILITY CONSIDERATIONS

→ The V-shape mental model makes the algorithm intuitive:
     → "Squares form a valley near zero"
     → "Peaks are at both ends"
     → "I pick the larger peak each time and fill from the back"

→ The merge-from-ends concept is universally understood:
     → Like merging two sorted decks of cards
     → One deck is the left half (reversed)
     → One deck is the right half (normal)
     → Pick the larger top card each time

→ Easy to trace: at each step, exactly ONE element is placed
     → The result array fills from right to left
     → Each position is set exactly once → no overwrites → clean

→ Interview explanation:
     → "Squares of a sorted array form a V-shape"
     → "Largest squares are at the ends"
     → "I use two pointers from both ends, compare squares"
     → "Place the larger square at the back of the result"
     → "Decrement the write position each time"
     → "O(n) time, O(n) space for the result"

🧠 KEY TAKEAWAY

WHAT WE LEARNED:
→ Two-pointer convergence can BUILD output, not just SEARCH
→ When a transformation creates a V-shape (largest at ends, smallest in middle)
→ Merge from both ends into the result, filling from the BACK
→ This achieves O(n) time by exploiting the hidden sorted structure

PATTERN EVOLUTION — THE COMPLETE TWO POINTERS FAMILY:
  Project 1: Opposite ends → converge → SEARCH for a pair (sum)
  Project 2: Opposite ends → converge + skip → CHECK symmetry (palindrome)
  Project 3: Same direction → read/write → TRANSFORM in-place (dedup)
  Project 4: Opposite ends → converge → BUILD sorted output (merge)

FOUR VARIANTS OF TWO POINTERS — NOW MASTERED:
  1. Left/Right Convergence for SEARCHING     (Project 1)
  2. Left/Right Convergence with FILTERING     (Project 2)
  3. Same-Direction Read/Write for TRANSFORMING (Project 3)
  4. Left/Right Convergence for MERGING/BUILDING (Project 4)

THE META-INSIGHT:
→ Two Pointers is NOT one technique — it's a FAMILY of techniques
→ The SKELETON is always the same: two indices, movement rules, decisions
→ The VARIATION is in:
     → Direction of movement (same vs opposite)
     → Purpose (search, filter, transform, merge)
     → Decision logic (sum comparison, equality, magnitude, etc.)
→ Recognizing WHICH variant to use is the real skill
