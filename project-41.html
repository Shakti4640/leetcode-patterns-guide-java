PROJECT 41: Sliding Window Median

🎯 Two Heaps → Heap Rebalancing on Window Slide → Lazy Deletion

🧩 THE HYPOTHETICAL PROBLEM

→ You are given an array of integers
→ You are given a window size K
→ The window slides from left to right, one position at a time
→ For EACH window position, find the MEDIAN of the K elements in the window
→ Return an array of all medians

Median definition:
  → If K is odd: the middle element when sorted
  → If K is even: average of the two middle elements

Example 1:
  Input:  nums = [1, 3, -1, -3, 5, 3, 6, 7], k = 3
  
  Window positions:
    [1,  3, -1] → sorted: [-1, 1, 3]    → median = 1
    [3, -1, -3] → sorted: [-3, -1, 3]   → median = -1
    [-1, -3, 5] → sorted: [-3, -1, 5]   → median = -1
    [-3,  5, 3] → sorted: [-3, 3, 5]    → median = 3
    [5,  3,  6] → sorted: [3, 5, 6]     → median = 5
    [3,  6,  7] → sorted: [3, 6, 7]     → median = 6
  
  Output: [1.0, -1.0, -1.0, 3.0, 5.0, 6.0]

Example 2:
  Input:  nums = [1, 2, 3, 4, 2, 3, 1, 4, 2], k = 3
  Output: [2.0, 3.0, 3.0, 3.0, 2.0, 3.0, 2.0]

Example 3 (even K):
  Input:  nums = [1, 3, -1, -3, 5, 3], k = 4
  
  Window positions:
    [1, 3, -1, -3] → sorted: [-3, -1, 1, 3] → median = (-1+1)/2 = 0.0
    [3, -1, -3, 5] → sorted: [-3, -1, 3, 5] → median = (-1+3)/2 = 1.0
    [-1, -3, 5, 3] → sorted: [-3, -1, 3, 5] → median = (-1+3)/2 = 1.0
  
  Output: [0.0, 1.0, 1.0]

🔍 HOW TO APPROACH IT

Step 1 — Identify the Component Patterns

This problem is a FUSION of two patterns we've mastered:

PATTERN 1: Sliding Window (Projects 5, 6, 12)
  → Fixed-size window of K elements
  → Window slides right: add one element, remove one element
  → Compute something for each window position

PATTERN 2: Two Heaps for Median (Project 35)
  → Max-heap for lower half, min-heap for upper half
  → Roots of two heaps = the middle elements
  → Median = root of max-heap (odd) or average of both roots (even)

THE FUSION CHALLENGE:
  → Project 35: Elements only ENTER the heaps → easy
  → Project 41: Elements ENTER and LEAVE the heaps → hard
  → When the window slides:
       → One new element ENTERS (right side of window)
       → One old element LEAVES (left side of window)
  → The element leaving might be DEEP inside a heap
  → Heaps only support efficient removal of the ROOT
  → Removing arbitrary elements is O(K) → too slow
  → THIS is where LAZY DELETION comes in

Step 2 — Why Direct Removal Fails

Standard heap operations:
  → offer(x):  O(log n) — add to heap
  → poll():    O(log n) — remove ROOT
  → remove(x): O(n)     — remove ARBITRARY element (linear scan!)

For each window slide:
  → We need to remove the element leaving the window
  → That element could be ANYWHERE in the heap
  → Java's PriorityQueue.remove(Object): O(n) linear scan
  → With n = K and (n-K+1) window positions:
       → Total: O(n * K) → too slow for large inputs

WE NEED A BETTER STRATEGY.

Step 3 — The Lazy Deletion Insight

CORE IDEA: Don't remove elements immediately.
           Instead, MARK them as "deleted" and clean up LATER.

HOW IT WORKS:
  → Maintain a HashMap<Integer, Integer> counting "pending deletions"
  → When an element should leave:
       → Increment its count in the deletion map
       → DON'T actually remove from heap yet
  → When we peek or poll from a heap:
       → Check if the root is "marked for deletion"
       → If yes → poll it (discard) → check next root
       → Repeat until root is NOT marked for deletion
  → This is called "lazy" because deletion happens only when
    the element surfaces to the root

WHY THIS WORKS:
  → Elements marked for deletion are GHOSTS in the heap
  → They occupy space but don't affect correctness
  → When they eventually reach the root (through other removals)
       → We discard them instead of using them
  → The heap's SIZE tracking must account for ghosts
       → We maintain a LOGICAL size (actual elements minus ghosts)
       → Separate from the heap's physical size

EFFICIENCY:
  → Marking for deletion: O(1) — HashMap put
  → Lazy cleanup at root: O(log K) per cleanup — standard poll
  → Each element is lazily removed AT MOST ONCE
  → Amortized: O(log K) per window slide

Step 4 — Two Heaps + Lazy Deletion Architecture

DATA STRUCTURES:
  → maxHeap: stores LOWER half (larger values near root)
  → minHeap: stores UPPER half (smaller values near root)
  → delayedMap: HashMap<Integer, Integer> — counts pending deletions
  → maxHeapSize: logical size of maxHeap (excluding ghosts)
  → minHeapSize: logical size of minHeap (excluding ghosts)

INVARIANTS (same as Project 35, but with logical sizes):
  → maxHeapSize == minHeapSize     (K even)
  → maxHeapSize == minHeapSize + 1 (K odd)
  → All elements in maxHeap ≤ all elements in minHeap
  → Median = maxHeap.peek() (odd K)
            or (maxHeap.peek() + minHeap.peek()) / 2.0 (even K)

WINDOW SLIDE PROCEDURE:
  1. ADD new element (entering window from right)
  2. REMOVE old element (leaving window from left) — lazy
  3. REBALANCE heaps (adjust logical sizes)
  4. PRUNE ghost elements from roots (lazy cleanup)
  5. COMPUTE median from clean roots

Step 5 — Visualize a Complete Window Slide

nums = [1, 3, -1, -3, 5], k = 3

─── INITIAL WINDOW [1, 3, -1] ───
Build two heaps with first K elements:
  Sorted: [-1, 1, 3]
  maxHeap (lower): [-1, 1]  → root = 1
  minHeap (upper): [3]      → root = 3
  Sizes: max=2, min=1 (odd K → max has one extra)
  Median = maxHeap.peek() = 1 ✓

─── SLIDE: Remove 1, Add -3 → Window [3, -1, -3] ───

Step 1 — ADD -3:
  → -3 ≤ maxHeap.peek(1) → add to maxHeap
  → maxHeap: [-1, 1, -3] root=1   maxSize=3
  → minHeap: [3]                    minSize=1

Step 2 — REMOVE 1 (lazy):
  → 1 is in maxHeap (1 ≤ maxHeap.peek=1)
  → Mark in delayedMap: {1: 1}
  → Decrement maxHeapSize: maxSize=2
  → DON'T actually remove from heap yet

Step 3 — REBALANCE:
  → maxSize=2, minSize=1 → already balanced for odd K ✓

Step 4 — PRUNE roots:
  → maxHeap.peek() = 1 → check delayedMap → 1 is marked!
  → Poll 1 from maxHeap → decrement delayedMap: {1: 0} → remove key
  → maxHeap.peek() = -1 → not marked → CLEAN ✓
  → minHeap.peek() = 3 → not marked → CLEAN ✓

Step 5 — MEDIAN:
  → maxHeap.peek() = -1 → Median = -1 ✓
  
  Verify: window [3, -1, -3] → sorted [-3, -1, 3] → median = -1 ✓

🛠️ HOW CAN WE SOLVE IT

The Complete Algorithm

INITIALIZATION:
  1. Create maxHeap (lower half) — reversed comparator
  2. Create minHeap (upper half) — default comparator
  3. Create delayedMap — HashMap<Integer, Integer>
  4. Initialize maxHeapSize = 0, minHeapSize = 0

PROCESS FIRST WINDOW:
  5. Sort first K elements
  6. Lower half → maxHeap, upper half → minHeap
  7. Set logical sizes accordingly
  8. Record first median

SLIDE WINDOW (for each new position):
  9.  outgoing = nums[i - k]  (element leaving)
  10. incoming = nums[i]       (element entering)
  
  11. STEP A — Record lazy deletion for outgoing:
        delayedMap[outgoing]++
        if outgoing ≤ maxHeap.peek() → maxHeapSize--
        else → minHeapSize--
  
  12. STEP B — Add incoming to appropriate heap:
        if incoming ≤ maxHeap.peek() → maxHeap.offer(incoming); maxHeapSize++
        else → minHeap.offer(incoming); minHeapSize++
  
  13. STEP C — Rebalance:
        if maxHeapSize > minHeapSize + 1:
            → move maxHeap root to minHeap
            → maxHeapSize--, minHeapSize++
        if minHeapSize > maxHeapSize:
            → move minHeap root to maxHeap
            → minHeapSize--, maxHeapSize++
  
  14. STEP D — Prune ghost elements from BOTH roots:
        while maxHeap not empty AND maxHeap.peek() is in delayedMap:
            → poll from maxHeap → decrement delayedMap
        while minHeap not empty AND minHeap.peek() is in delayedMap:
            → poll from minHeap → decrement delayedMap
  
  15. STEP E — Compute median:
        if K is odd → median = maxHeap.peek()
        if K is even → median = (maxHeap.peek() + minHeap.peek()) / 2.0

✅ WHAT SHOULD YOU DO

→ ALWAYS track LOGICAL sizes separate from physical heap sizes
     → Physical size includes ghost elements
     → Logical size = actual live elements
     → Balance decisions use LOGICAL sizes
     → heap.size() is UNRELIABLE (includes ghosts)

→ ALWAYS prune BOTH heap roots after rebalancing
     → Ghosts can end up at root of EITHER heap
     → After rebalancing, elements move between heaps
     → A ghost might become the new root
     → Must clean both roots before computing median

→ ALWAYS determine which heap an element belongs to before lazy-deleting
     → outgoing ≤ maxHeap.peek() → it's in maxHeap → decrement maxHeapSize
     → outgoing > maxHeap.peek() → it's in minHeap → decrement minHeapSize
     → Getting this wrong corrupts the logical size tracking

→ ALWAYS handle the order: delete → add → rebalance → prune → median
     → This specific order maintains invariants correctly
     → Changing the order can create temporary invalid states

→ ALWAYS use long or double for median computation with even K
     → (maxHeap.peek() + minHeap.peek()) can overflow int
     → Cast to double BEFORE dividing: ((double)a + (double)b) / 2.0

→ ALWAYS initialize heaps from sorted first-K elements
     → Guarantees correct split from the start
     → Avoids complex incremental initialization logic

🚫 WHAT SHOULD YOU NOT DO

→ DO NOT use PriorityQueue.remove(Object) for deletion
     → O(K) per removal → O(nK) total → too slow
     → Lazy deletion achieves amortized O(log K) per slide

→ DO NOT use physical heap.size() for balance decisions
     → Physical size includes ghost elements
     → Ghost elements don't participate in the median
     → Using physical size → wrong balance → wrong median

→ DO NOT prune roots BEFORE rebalancing
     → Rebalancing moves elements between heaps
     → The moved element might BECOME a ghost at the new root
     → Prune AFTER rebalancing catches all cases

→ DO NOT forget to handle the case where delayedMap count reaches 0
     → When count reaches 0 → REMOVE the key from map
     → Otherwise → future checks see stale entries
     → delayedMap.containsKey(x) would incorrectly return true

→ DO NOT move ghost elements during rebalancing
     → When transferring root from one heap to another:
       → Poll from source → check if it's a ghost → if ghost, discard and retry
       → Or: prune source root first, THEN transfer
     → Moving a ghost between heaps corrupts both heaps

→ DO NOT sort the window for each position
     → Naive approach: sort K elements per window → O(K log K) per position
     → Total: O(n * K log K) → much slower than O(n log K)

🔀 WHEN TO USE THIS PATTERN

✓ Sliding window + need MEDIAN (or any order-statistic) per window
✓ Window size K is large enough that per-window sorting is expensive
✓ Need O(n log K) instead of O(n * K log K)
✓ Elements enter AND leave the data structure over time
✓ Heap-based structure is needed but arbitrary deletion is required

GENERALIZATION — Lazy Deletion applies whenever:
  → You have a heap-based algorithm
  → Elements need to be removed from ARBITRARY positions
  → The heap's root is what matters for correctness
  → Ghost elements deep in the heap don't affect current answers
  → They only matter when they surface to the root

🚫 WHEN NOT TO USE THIS PATTERN

✗ Window size K is very small (K ≤ 10)
     → Just sort K elements per window → O(n * K log K) is fine
     → Simpler code, fewer edge cases

✗ You only need the MAXIMUM or MINIMUM of each window
     → Use Monotonic Deque → O(n) total → much simpler
     → Two heaps is overkill for just max or min

✗ You need the Kth element (not median)
     → Augmented BST (like TreeMap) with count tracking
     → Or sorted multiset with index access

✗ Elements are from a small range (e.g., 0 to 255)
     → Use counting/bucket approach → O(n) per window slide
     → Simpler and faster for bounded integer ranges

✗ Window size equals array size (K = n)
     → Single median computation → sort once → O(n log n)
     → No sliding needed

⚙️ HOW IT ONLY WORKS — Internal Mechanics

Lazy Deletion Deep Dive

THE PROBLEM WITH HEAPS:
  → Heaps are optimized for ROOT operations: peek O(1), poll O(log n)
  → Finding an arbitrary element: O(n) — must scan entire array
  → Removing an arbitrary element: O(n) — find + restructure
  → This is a FUNDAMENTAL limitation of the heap data structure

LAZY DELETION CIRCUMVENTS THIS:
  → Instead of finding and removing element X deep in the heap
  → We RECORD that X should be deleted (HashMap entry)
  → X stays in the heap as a "ghost"
  → When X eventually becomes the root (through natural heap operations)
  → We recognize it as a ghost and discard it
  → The ghost is cleaned up WITHOUT ever searching for it

CORRECTNESS GUARANTEE:
  → Ghosts only affect us when they're at the ROOT
  → We ALWAYS prune ghost roots before reading the root value
  → Therefore: root is ALWAYS a live (non-ghost) element
  → Median computation reads ONLY roots → always correct

SPACE COST OF GHOSTS:
  → At any time, at most K ghosts exist (one per window slide)
  → Actually: ghosts are cleaned up over time
  → Worst case: heap physical size = K + (some ghosts)
  → Practical overhead is small

The Rebalancing Protocol — Step by Step

After adding one element and lazily deleting another:
  → Logical sizes may be off by 1 from the ideal

IDEAL BALANCE (invariant we maintain):
  → K odd:  maxHeapSize = (K+1)/2, minHeapSize = K/2
  → K even: maxHeapSize = K/2, minHeapSize = K/2

REBALANCE RULES:
  Rule 1: maxHeapSize > minHeapSize + 1 → max has too many
    → Transfer: poll maxHeap root → offer to minHeap
    → Adjust sizes: maxHeapSize--, minHeapSize++
    → BUT: the polled root might be a ghost!
    → Must prune maxHeap root before transferring

  Rule 2: minHeapSize > maxHeapSize → min has too many
    → Transfer: poll minHeap root → offer to maxHeap
    → Adjust sizes: minHeapSize--, maxHeapSize++
    → Same ghost concern → prune first

PRUNE-THEN-TRANSFER:
  → Before transferring from heap X:
       while X.peek() is in delayedMap:
           poll and discard (it's a ghost)
           decrement delayedMap count
       → Now X.peek() is a live element → safe to transfer

Time Complexity — Precise Analysis

INITIALIZATION:
  → Sort first K elements: O(K log K)
  → Build heaps: O(K)
  → Total init: O(K log K)

PER WINDOW SLIDE:
  → Lazy delete (mark): O(1)
  → Add to heap: O(log K)
  → Rebalance (at most 1 transfer): O(log K)
  → Prune roots: AMORTIZED O(log K)
       → Each element is pruned exactly once across all slides
       → Total pruning across all slides: O(n log K)
       → Per slide amortized: O(log K)
  → Total per slide: O(log K)

TOTAL: O(K log K + n log K) = O(n log K)

COMPARISON WITH ALTERNATIVES:
  ┌─────────────────────────────┬──────────────────┐
  │ Approach                    │ Time             │
  ├─────────────────────────────┼──────────────────┤
  │ Sort each window            │ O(n * K log K)   │
  │ Insertion sort maintenance  │ O(n * K)         │
  │ Two heaps + direct remove   │ O(n * K)         │
  │ Two heaps + lazy deletion   │ O(n log K)       │
  │ Balanced BST (TreeMap)      │ O(n log K)       │
  └─────────────────────────────┴──────────────────┘

Space Complexity

→ maxHeap: up to K + ghosts ≈ O(K) physical elements
→ minHeap: up to K + ghosts ≈ O(K) physical elements
→ delayedMap: up to K entries (bounded by window slides)
→ Result array: O(n - K + 1)
→ Total: O(n) for result + O(K) for heaps/map

Why TreeMap is an Alternative

Java's TreeMap is a balanced BST → supports:
  → add: O(log K)
  → remove arbitrary: O(log K)  ← THIS is what heaps can't do!
  → first/last: O(log K)

BUT:
  → TreeMap doesn't handle duplicates natively
  → Need TreeMap<Integer, Integer> for counts → extra complexity
  → OR use index-augmented entries → even more complex
  → Constant factors are larger than heap operations
  → Lazy deletion on heaps is FASTER in practice

INTERVIEW PREFERENCE:
  → Two heaps + lazy deletion → shows deeper understanding
  → TreeMap → acceptable but less impressive
  → Both are O(n log K) → choose based on comfort

⚠️ COMMON PITFALLS AND FAILURE MODES

Pitfall 1: Wrong Heap Assignment for Outgoing Element

When marking an element for lazy deletion, you must know
WHICH heap it logically belongs to → affects logical size tracking.

RULE: If outgoing ≤ maxHeap.peek() → it's in the lower half (maxHeap)
      If outgoing > maxHeap.peek() → it's in the upper half (minHeap)

WRONG:
  delayedMap.merge(outgoing, 1, Integer::sum);
  maxHeapSize--;  // ← ALWAYS decrementing maxHeap → WRONG

CORRECT:
  delayedMap.merge(outgoing, 1, Integer::sum);
  if (outgoing <= maxHeap.peek()) {
      maxHeapSize--;
  } else {
      minHeapSize--;
  }

EDGE CASE: What if maxHeap is empty when checking?
  → Only happens if all elements are in minHeap
  → Shouldn't happen with correct balance maintenance
  → Guard with: if (!maxHeap.isEmpty() && outgoing <= maxHeap.peek())

Pitfall 2: Ghost Elements Transferred During Rebalance

When rebalancing, you poll from one heap and offer to another.
If the polled element is a GHOST → you just moved garbage.

WRONG:
  // maxHeap too big → transfer root to minHeap
  int transferred = maxHeap.poll();
  minHeap.offer(transferred);
  maxHeapSize--;
  minHeapSize++;
  // BUT: transferred might be a ghost → corrupts minHeap

CORRECT:
  // Prune maxHeap root first
  pruneHeap(maxHeap);
  // NOW root is guaranteed live
  int transferred = maxHeap.poll();
  minHeap.offer(transferred);
  maxHeapSize--;
  minHeapSize++;

Pitfall 3: Integer Overflow in Median Calculation

For even K: median = (maxHeap.peek() + minHeap.peek()) / 2.0

If both values are large:
  maxHeap.peek() = 2,000,000,000
  minHeap.peek() = 2,000,000,000
  Sum = 4,000,000,000 → OVERFLOW int!

FIX:
  double median = ((double) maxHeap.peek() + (double) minHeap.peek()) / 2.0;
  OR:
  double median = maxHeap.peek() / 2.0 + minHeap.peek() / 2.0;
  (second form avoids intermediate overflow entirely)

Pitfall 4: Delayed Map Stale Entries

When decrementing a count in delayedMap:
  → If count reaches 0 → MUST remove the key

WRONG:
  delayedMap.put(val, delayedMap.get(val) - 1);
  // Count is now 0 but key still exists
  // Future delayedMap.containsKey(val) returns TRUE
  // Ghost check incorrectly fires on a live element!

CORRECT:
  int newCount = delayedMap.get(val) - 1;
  if (newCount == 0) {
      delayedMap.remove(val);
  } else {
      delayedMap.put(val, newCount);
  }

Pitfall 5: Duplicate Values Across Heaps

nums = [1, 1, 1, 1, 1], k = 3
All values are 1 → both heaps contain 1's.

When removing a 1 → which heap does it belong to?
  → Check: 1 ≤ maxHeap.peek(1) → YES → it's in maxHeap
  → This is correct even with duplicates
  → The ≤ comparison handles equality correctly

BUT: delayedMap tracks by VALUE, not by heap membership.
  → If we mark "1" for deletion and it's in maxHeap
  → But the ghost-check fires on minHeap's root (also 1)
  → We might accidentally prune a LIVE element from minHeap!

SOLUTION: The pruneHeap function only fires on the SPECIFIC heap
being pruned. Both heaps share the delayedMap, but:
  → We only prune a value from a heap when it's at THAT heap's root
  → The logical size tracking ensures correct balance
  → Even if the "wrong" heap's ghost is pruned, sizes stay correct
     because the ghost was already counted in the right heap's size

ACTUALLY → this IS a subtle issue. The safest approach:
  → delayedMap is shared → a ghost value at ANY root gets cleaned
  → Logical sizes were already adjusted when we MARKED the deletion
  → Physical removal during pruning doesn't change logical sizes
  → Therefore: it doesn't matter WHICH heap physically holds the ghost
  → The logical accounting was done correctly at mark time

Pitfall 6: Empty Heap During Pruning

Pruning loop:
  while (!heap.isEmpty() && delayedMap.containsKey(heap.peek())) {
      int ghost = heap.poll();
      // decrement delayed count
  }

WITHOUT the isEmpty() check:
  → If heap becomes empty during pruning → peek() throws exception
  → ALWAYS check emptiness before peeking

📈 SCALABILITY CONSIDERATIONS

SCENARIO 1: Large array, moderate window
  → n = 10,000,000, K = 1,000
  → Two heaps + lazy deletion: O(10^7 * log 1000) ≈ O(10^7 * 10) = 10^8
  → Sort each window: O(10^7 * 1000 * 10) = 10^11 → way too slow
  → Lazy deletion wins by 1000x

SCENARIO 2: Large array, small window
  → n = 10,000,000, K = 5
  → Two heaps: O(10^7 * log 5) ≈ O(10^7 * 2.3) = 23M ops
  → Sort each window: O(10^7 * 5 * 2.3) ≈ 115M ops
  → Heap wins but sort is acceptable for tiny K

SCENARIO 3: Large array, large window
  → n = 10,000,000, K = 5,000,000
  → Two heaps: O(10^7 * log 5*10^6) ≈ O(10^7 * 22.3) = 223M ops
  → Only ~5 window positions → fast regardless
  → Initialization dominates: O(K log K) ≈ O(5*10^6 * 22.3) = 112M ops

MEMORY SCALING:
  → Heap physical size ≤ K + number of active ghosts
  → Ghosts are bounded: at most O(K) at any time
  → Total memory: O(K)
  → For K = 1,000,000 → ~4MB for int heaps → manageable

WHY LAZY:
  → Digging through the pile to remove a specific score is SLOW
  → Marking it and waiting for it to surface is EFFICIENT
  → Scores deep in the pile don't affect the median anyway
  → Only the TOP matters → only clean the top

WHAT WE LEARNED:

→ Mark elements for deletion in a HashMap → clean up when they surface
→ Maintain LOGICAL sizes separate from PHYSICAL heap sizes
→ Balance decisions use logical sizes → median reads clean roots
→ Order matters: delete → add → rebalance → prune → read

THE LAZY DELETION PRINCIPLE:
→ "Don't search for what you can't efficiently find"
→ "Mark it, forget it, clean it when it comes to you"
→ This principle appears in many advanced data structures:
     → Lazy propagation in segment trees
     → Tombstones in hash tables
     → Soft deletes in databases
     → Mark-and-sweep in garbage collection

COMPLEXITY ACHIEVEMENT:
