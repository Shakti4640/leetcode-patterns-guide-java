PROJECT 15: Middle of a Linked List

🎯 Fast & Slow Pointers → Half-Speed Traversal → Single-Pass Midpoint

🔗 DEPENDENCY CHAIN

Project 1  → Two Pointers → two variables scanning data simultaneously
Project 7  → Fast & Slow Pointers → speed differential on linked list → cycle detection
Project 14 → Merge Intervals → sort-then-scan → sequential processing
Project 15 → Fast & Slow Pointers REVISITED → midpoint finding
             → same speed differential technique from Project 7
             → different PURPOSE: not detecting cycles, but finding POSITION
             → the simplest non-trivial linked list operation
             → foundation for merge sort on linked lists, palindrome checks, etc.

🧩 THE HYPOTHETICAL PROBLEM

→ You are given a SINGLY LINKED LIST
→ Find the MIDDLE node
→ Return the middle node (not just its value)
→ If the list has EVEN number of nodes → return the SECOND middle node

Example 1 (Odd length):
  Input:  1 → 2 → 3 → 4 → 5
  Output: Node with value 3
  → 5 nodes → middle is the 3rd → value 3

Example 2 (Even length):
  Input:  1 → 2 → 3 → 4 → 5 → 6
  Output: Node with value 4
  → 6 nodes → two middles: 3 and 4 → return the SECOND middle → value 4

Example 3 (Single node):
  Input:  1
  Output: Node with value 1
  → Only one node → it IS the middle

Example 4 (Two nodes):
  Input:  1 → 2
  Output: Node with value 2
  → Two nodes → second middle → value 2

WHY THIS MATTERS:
  → Finding the middle is a BUILDING BLOCK for many algorithms:
    → Merge sort on linked list → split list at middle → sort each half
    → Palindrome check → reverse second half → compare with first half
    → Balanced binary tree from sorted list → middle becomes root
  → It's also the simplest application of fast/slow pointers for POSITION

🔍 HOW TO APPROACH IT

Step 1 — The Linked List Constraint

ARRAY:
  → Length is known → arr.length
  → Middle index = length / 2
  → Direct access: arr[length / 2]
  → O(1) time → trivial

LINKED LIST:
  → Length is NOT known upfront
  → NO random access → can't jump to index i
  → Must TRAVERSE from head to reach any node
  → To find length: traverse entire list → O(n)
  → To find middle: traverse to length/2 → O(n)
  → Two-pass approach: first count, then traverse to middle → 2 passes

CAN WE DO IT IN ONE PASS?
  → Yes → Fast & Slow Pointers
  → The speed differential automatically finds the middle
  → No counting needed → no second pass needed

Step 2 — Revisiting Fast & Slow from Project 7

PROJECT 7 (Cycle Detection):
  → Fast moves 2 steps, Slow moves 1 step
  → If cycle exists → they MEET inside the cycle
  → If no cycle → fast reaches the end (null)
  → The MEETING was the signal → "cycle exists"

PROJECT 15 (Midpoint Finding):
  → Fast moves 2 steps, Slow moves 1 step
  → When fast reaches the END → slow is at the MIDDLE
  → The POSITION of slow when fast finishes is the answer
  → No cycle involved → fast always reaches the end

SAME TECHNIQUE, DIFFERENT QUESTION:
  Project 7:  "Do they meet?" → boolean answer (cycle or not)
  Project 15: "Where is slow when fast finishes?" → positional answer (middle node)

Step 3 — Why Fast=2x Speed Gives the Middle

INTUITION:
  → Fast pointer covers TWICE the distance of slow pointer
  → When fast reaches the END (distance = n)
  → Slow has covered n/2 → exactly the MIDDLE

FORMAL:
  → List has n nodes (indexed 0 to n-1)
  → Each step: fast advances 2, slow advances 1
  → After k steps: fast is at position 2k, slow is at position k
  → Fast reaches end when 2k ≈ n → k ≈ n/2
  → At that moment: slow is at position k ≈ n/2 → THE MIDDLE

VISUAL (Odd length — 5 nodes):
  Step 0: slow=0, fast=0
    [S/F] → 2 → 3 → 4 → 5

  Step 1: slow=1, fast=2
    1 → [S] → [F] → 4 → 5

  Step 2: slow=2, fast=4
    1 → 2 → [S] → 4 → [F]

  Fast is at last node → STOP
  Slow is at node 2 (value 3) → MIDDLE ✓

VISUAL (Even length — 6 nodes):
  Step 0: slow=0, fast=0
    [S/F] → 2 → 3 → 4 → 5 → 6

  Step 1: slow=1, fast=2
    1 → [S] → [F] → 4 → 5 → 6

  Step 2: slow=2, fast=4
    1 → 2 → [S] → 4 → [F] → 6

  Step 3: slow=3, fast=6 (past end → null)
    1 → 2 → 3 → [S] → 5 → 6 → [F=null]

  Fast is null → STOP
  Slow is at node 3 (value 4) → SECOND MIDDLE ✓

Step 4 — The Termination Condition

WHEN DOES FAST STOP?

Case 1: ODD number of nodes (fast lands ON the last node)
  1 → 2 → 3 → 4 → 5
  
  Fast positions: 0, 2, 4 (last node)
  fast.next = null → can't move 2 more → STOP
  
  Condition: fast.next == null

Case 2: EVEN number of nodes (fast goes PAST the last node)
  1 → 2 → 3 → 4 → 5 → 6
  
  Fast positions: 0, 2, 4, then tries to go to 6 → position 6 is null
  fast = null → STOP
  
  Condition: fast == null

COMBINED LOOP CONDITION:
  while (fast != null && fast.next != null)
  
  → fast != null: handles even-length case (fast went past end)
  → fast.next != null: handles odd-length case (fast is on last node)
  → BOTH checks are necessary
  → Order matters: fast != null must be checked FIRST (short-circuit)
    → If fast is null, fast.next would be NullPointerException
    → Java's && short-circuits → if fast is null, fast.next is never evaluated ✓

Step 5 — First Middle vs Second Middle

For EVEN-length lists, there are TWO middle nodes:
  1 → 2 → [3] → [4] → 5 → 6
              ↑       ↑
         1st middle  2nd middle

THE STANDARD ALGORITHM (as described) returns the SECOND middle:
  → After 3 steps: slow is at node 3 (value 4) → second middle

TO GET THE FIRST MIDDLE INSTEAD:
  → Start fast ONE STEP AHEAD: fast = head.next
  → OR change the loop condition slightly
  → OR use a "previous slow" pointer

  Method: Start fast at head.next
    Step 0: slow=0 (head), fast=1 (head.next)
      [S] → [F] → 3 → 4 → 5 → 6

    Step 1: slow=1, fast=3
      1 → [S] → 3 → [F] → 5 → 6

    Step 2: slow=2, fast=5
      1 → 2 → [S] → 4 → 5 → [F]

    fast.next = null → STOP
    Slow is at node 2 (value 3) → FIRST MIDDLE ✓

WHY THE PROBLEM USUALLY ASKS FOR SECOND MIDDLE:
  → More useful for splitting the list into two halves
  → First half: head to node before middle
  → Second half: middle to end
  → Second middle gives a clean split for merge sort

🛠️ HOW CAN WE SOLVE IT

The Algorithm

1. Initialize:
     → slow = head
     → fast = head

2. WHILE fast is not null AND fast.next is not null:
     → slow = slow.next        (move 1 step)
     → fast = fast.next.next   (move 2 steps)

3. RETURN slow → this is the middle node

The Two-Pass Alternative (for comparison)

1. FIRST PASS: Count the total number of nodes
     → Traverse entire list → count = n

2. SECOND PASS: Traverse to position n/2
     → Move a pointer n/2 steps from head

3. RETURN the node at position n/2

→ This is correct but requires TWO passes
→ Fast/slow does it in ONE pass
→ Both are O(n) time, but one pass is more elegant
→ In streaming scenarios (data arriving one node at a time), one pass is NECESSARY

✅ WHAT SHOULD YOU DO

→ ALWAYS check BOTH conditions: fast != null && fast.next != null
     → fast != null → handles even-length lists
     → fast.next != null → handles odd-length lists
     → Missing either → NullPointerException for certain list lengths

→ ALWAYS check fast != null BEFORE fast.next != null
     → Java short-circuit evaluation: if fast is null, second condition is skipped
     → Reversed order: fast.next != null checked first → NPE when fast is null

→ ALWAYS clarify which middle is needed for even-length lists
     → "Second middle" (standard): both pointers start at head
     → "First middle": fast starts at head.next (or use prev pointer)
     → The problem statement will specify → read carefully

→ ALWAYS handle the edge case of single-node list
     → head.next is null → fast.next is null → loop doesn't execute
     → Return head → correct (single node is its own middle)

→ ALWAYS remember: this is a BUILDING BLOCK, not just a standalone problem
     → Finding the middle enables: merge sort, palindrome check, reorder list
     → Master this → unlock multiple advanced problems

🚫 WHAT SHOULD YOU NOT DO

→ DO NOT traverse the list to count length first (unless required)
     → Two-pass approach works but wastes a traversal
     → Fast/slow is more elegant and equally fast
     → In interviews, fast/slow shows you know the technique

→ DO NOT use a counter variable alongside a single pointer
     WRONG APPROACH:
       int count = 0;
       ListNode current = head;
       while (current != null) { count++; current = current.next; }
       current = head;
       for (int i = 0; i < count / 2; i++) current = current.next;
       return current;
     
     → Works but doesn't demonstrate pattern knowledge
     → Also requires two traversals → less efficient in practice

→ DO NOT convert to an array then find middle
     → Tempting: put all nodes in ArrayList → return list.get(size/2)
     → Works but uses O(n) extra space
     → Fast/slow uses O(1) extra space
     → Converting to array defeats the purpose of linked list operations

→ DO NOT forget that linked list traversal is ONE-DIRECTIONAL
     → Singly linked list: can only move FORWARD
     → Can't go backward to "previous node" without storing it
     → If you need the node BEFORE middle → use a "prev" variable

→ DO NOT modify the list structure while finding the middle
     → Finding middle is a READ operation → don't break links
     → The list should be intact after finding the middle
     → Modification comes in separate operations (reverse, split, etc.)

🔀 WHEN TO USE THIS PATTERN

✓ Finding the MIDDLE of a linked list
     → The direct application

✓ SPLITTING a linked list into two halves
     → Find middle → cut at middle → two separate lists
     → Used in merge sort for linked lists

✓ Checking if a linked list is a PALINDROME
     → Find middle → reverse second half → compare with first half
     → Fast/slow finds the split point

✓ REORDERING a linked list (L0→Ln→L1→Ln-1→...)
     → Find middle → reverse second half → interleave both halves

✓ Any problem requiring the HALFWAY POINT of a sequential structure
     → Where random access is not available
     → Where length is unknown upfront

✓ STREAMING data where you need the median position
     → Process elements one at a time → maintain middle pointer
     → (Though for running median, two heaps are better → Project 35)

🚫 WHEN NOT TO USE THIS PATTERN

✗ Data is in an ARRAY (not linked list)
     → Array has random access → middle = arr[arr.length / 2] → O(1)
     → Fast/slow on arrays is unnecessary overhead

✗ You need the middle VALUE of a sorted sequence (median)
     → Median finding is a different problem
     → For unsorted arrays: quickselect algorithm → O(n) average
     → For streaming data: two heaps → Project 35
     → Fast/slow finds the positional middle, not the value-based median

✗ Linked list is DOUBLY linked
     → Can traverse from both ends simultaneously → regular two pointers
     → No need for speed differential
     → Start one pointer at head, one at tail → move inward

✗ You need to find a position OTHER than the middle
     → 1/3 point: fast moves 3x, slow moves 1x
     → 1/4 point: fast moves 4x, slow moves 1x
     → Generalizable but unusual → specific problems may need this

✗ The list has a CYCLE
     → Fast/slow for midpoint assumes no cycle
     → With a cycle, fast never reaches null → infinite loop
     → Must detect cycle first (Project 7) → handle separately

⚙️ HOW IT ONLY WORKS — Internal Mechanics

Mechanic 1: The Speed Ratio Mathematics

SETUP:
  → List of n nodes: node_0, node_1, ..., node_(n-1)
  → slow starts at node_0, moves 1 step per iteration
  → fast starts at node_0, moves 2 steps per iteration

AFTER k ITERATIONS:
  → slow is at node_k
  → fast is at node_(2k)

TERMINATION (odd n, e.g., n=5):
  → fast reaches node_(n-1) → the last node
  → fast.next == null → stop
  → At this point: 2k = n-1 → k = (n-1)/2
  → slow is at node_((n-1)/2) = node_2 → THE MIDDLE ✓

  Verify: n=5 → k=(5-1)/2=2 → slow at node_2 → 3rd node → middle ✓

TERMINATION (even n, e.g., n=6):
  → fast reaches node_(n-2), then moves 2 → goes past node_(n-1) → null
  → fast == null → stop
  → At this point: 2k = n → k = n/2
  → slow is at node_(n/2) = node_3 → THE SECOND MIDDLE ✓

  Verify: n=6 → k=6/2=3 → slow at node_3 → 4th node → second middle ✓

THE MATH GUARANTEES:
  → Odd n: slow ends at (n-1)/2 → exact middle
  → Even n: slow ends at n/2 → second of two middles
  → Both are correct per the problem definition

Mechanic 2: Why Both Null Checks Are Necessary

SCENARIO: Even length (n=4)
  1 → 2 → 3 → 4

  Step 0: slow=node_0(1), fast=node_0(1)
  Step 1: slow=node_1(2), fast=node_2(3)
  Step 2: slow=node_2(3), fast=node_4(null)

  At step 2: fast = null
  → If we only checked fast.next != null:
    → fast.next would be null.next → NullPointerException!
  → The fast != null check CATCHES this case

SCENARIO: Odd length (n=5)
  1 → 2 → 3 → 4 → 5

  Step 0: slow=node_0(1), fast=node_0(1)
  Step 1: slow=node_1(2), fast=node_2(3)
  Step 2: slow=node_2(3), fast=node_4(5)

  At step 2: fast = node_4, fast.next = null
  → If we only checked fast != null:
    → fast is not null → loop continues
    → fast.next.next → null.next → NullPointerException!
  → The fast.next != null check CATCHES this case

BOTH CHECKS TOGETHER:
  while (fast != null && fast.next != null)
  → Handles ALL list lengths correctly
  → Short-circuit evaluation prevents NPE

Mechanic 3: One Pass vs Two Pass — Practical Difference

TWO-PASS APPROACH:
  Pass 1: head → ... → null (count = n)        → touches n nodes
  Pass 2: head → ... → middle                  → touches n/2 nodes
  Total nodes touched: n + n/2 = 3n/2

ONE-PASS (Fast/Slow):
  Slow: head → ... → middle                    → touches n/2 nodes
  Fast: head → ... → end                       → touches n nodes
  Total nodes touched: n + n/2 = 3n/2

SAME TOTAL WORK! Both are O(n).

BUT:
  → Two-pass requires the list to be TRAVERSABLE TWICE
  → In a streaming scenario → data comes once → can't re-traverse
  → One-pass works on streaming data
  → One-pass also has better CACHE behavior (data accessed in one sweep)
  → One-pass is preferred in interviews (shows technique knowledge)

Mechanic 4: The ListNode Structure in Java

Definition (standard LeetCode):

  class ListNode {
      int val;
      ListNode next;
      
      ListNode() {}
      ListNode(int val) { this.val = val; }
      ListNode(int val, ListNode next) { 
          this.val = val; 
          this.next = next; 
      }
  }

KEY PROPERTIES:
  → Each node stores ONE value and ONE pointer
  → The pointer goes to the NEXT node only (singly linked)
  → The last node's next is null → signals end of list
  → The head pointer is the entry point → no access without it
  → No "previous" pointer → can't go backward
  → No "length" property → must count by traversing

MEMORY MODEL:
  head → [val:1|next:→] → [val:2|next:→] → [val:3|next:null]
  
  → Each node is a separate object on the heap
  → Pointers (references) connect them
  → Nodes are NOT contiguous in memory (unlike arrays)
  → This means NO random access → must follow pointers sequentially

Time and Space Complexity

TIME: O(n)
  → Fast pointer traverses the entire list (n nodes)
  → Slow pointer traverses half the list (n/2 nodes)
  → Total: O(n) + O(n/2) = O(n)

SPACE: O(1)
  → Two pointer variables: slow and fast
  → No additional data structures
  → No recursion → no stack frames
  → Truly constant extra space

⚠️ COMMON PITFALLS AND FAILURE MODES

Pitfall 1: NullPointerException on Fast Pointer

WRONG:
  while (fast.next != null) {    // crashes when fast IS null (even length)
      slow = slow.next;
      fast = fast.next.next;
  }

ALSO WRONG:
  while (fast != null) {         // crashes when fast.next is null (odd length)
      slow = slow.next;
      fast = fast.next.next;     // fast.next.next when fast.next is null → NPE
  }

CORRECT:
  while (fast != null && fast.next != null) {
      slow = slow.next;
      fast = fast.next.next;
  }

→ BOTH conditions needed
→ Order matters (fast != null FIRST due to short-circuit)

Pitfall 2: Wrong Middle for Even-Length Lists

List: 1 → 2 → 3 → 4

Standard algorithm (both start at head):
  Step 1: slow=2, fast=3
  Step 2: slow=3, fast=null → STOP
  Returns node 3 (value 3) → SECOND MIDDLE ✓

If problem asks for FIRST middle (node 2):
  → Need to start fast at head.next
  
  Step 0: slow=1(head), fast=2(head.next)
  Step 1: slow=2, fast=4
  fast.next = null → STOP
  Returns node 2 (value 2) → FIRST MIDDLE ✓

→ Clarify WHICH middle the problem wants
→ LeetCode 876 asks for second middle → standard algorithm
→ Merge sort split typically wants first middle → start fast ahead

Pitfall 3: Empty List

Input: head = null

  while (fast != null && fast.next != null)
  → fast is null → condition is false → loop doesn't execute
  → return slow → slow is null

→ Returning null for empty list is CORRECT
→ But add a guard if the caller expects non-null:

  if (head == null) return null;
  // or throw new IllegalArgumentException("List is empty");

Pitfall 4: List With Cycle

If the list has a CYCLE:
  → Fast pointer NEVER reaches null
  → Loop runs forever → infinite loop → program hangs

→ Fast/slow for MIDPOINT assumes NO CYCLE
→ If cycle is possible → detect cycle first (Project 7)
→ Then handle accordingly (break the cycle, or return error)

→ In most problems, the input is guaranteed cycle-free
→ But be aware of this assumption

Pitfall 5: Confusing Node with Value

WRONG: return slow.val;
→ Returns the VALUE at the middle node (an integer)
→ But the problem asks for the NODE itself

CORRECT: return slow;
→ Returns the actual ListNode object
→ Caller can access slow.val, slow.next, etc.

→ Read the problem: "return the middle node" → return the node
→ "return the value of the middle node" → return slow.val
→ Usually the node is returned (more useful for chaining operations)

📈 SCALABILITY CONSIDERATIONS

LIST LENGTH SCALING:
→ n = 100         → 150 pointer moves → instant
→ n = 10,000      → 15,000 pointer moves → instant
→ n = 1,000,000   → 1,500,000 pointer moves → ~2ms
→ n = 100,000,000 → 150,000,000 pointer moves → ~300ms

COMPARISON WITH ALTERNATIVES:
  → Two-pass (count + traverse): same O(n) → but 50% more node accesses
  → Array conversion: O(n) time + O(n) space → much worse on space
  → Fast/slow: O(n) time + O(1) space → optimal on both dimensions

CACHE BEHAVIOR:
  → Linked list nodes are scattered in memory (not contiguous)
  → Each node access is potentially a cache miss
  → Minimizing node accesses matters more for linked lists than arrays
  → Fast/slow touches 3n/2 nodes → same as two-pass
  → But in one sweep → slightly better cache prediction

PRACTICAL NOTE:
  → For very large lists (millions of nodes), memory allocation/GC is the bottleneck
  → The algorithm itself is never the performance issue
  → Linked lists are inherently slower than arrays due to pointer chasing
  → Use arrays when possible → linked lists only when insertion/deletion is frequent
