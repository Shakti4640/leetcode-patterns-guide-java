PROJECT 45: Longest Increasing Subsequence (LIS)

🎯 Dynamic Programming → DP + Binary Search Hybrid → Patience Sorting

🧩 THE HYPOTHETICAL PROBLEM

→ You are given an UNSORTED array of integers
→ Find the length of the LONGEST STRICTLY INCREASING SUBSEQUENCE
→ A subsequence is derived by deleting some (or no) elements
     WITHOUT changing the order of remaining elements
→ Elements need NOT be contiguous (subsequence ≠ subarray)

Example 1:
  Input:  nums = [10, 9, 2, 5, 3, 7, 101, 18]
  
  Some increasing subsequences:
    [2, 5, 7, 101]  → length 4 ✓ (longest)
    [2, 5, 7, 18]   → length 4 ✓ (also longest)
    [2, 3, 7, 101]  → length 4 ✓ (also longest)
    [10, 101]        → length 2
    [9, 101]         → length 2
    [2, 5, 101]      → length 3
    
  Output: 4

Example 2:
  Input:  nums = [0, 1, 0, 3, 2, 3]
  
  Longest: [0, 1, 2, 3] → length 4
  Output: 4

Example 3:
  Input:  nums = [7, 7, 7, 7, 7]
  
  STRICTLY increasing → no two same elements
  Longest: [7] → length 1
  Output: 1

Example 4:
  Input:  nums = [1, 2, 3, 4, 5]
  
  Already sorted → entire array is the LIS
  Output: 5

Example 5:
  Input:  nums = [5, 4, 3, 2, 1]
  
  Strictly decreasing → every single element is a LIS of length 1
  Output: 1

🔍 HOW TO APPROACH IT

Step 1 — Understand "Subsequence" vs "Subarray"

SUBARRAY: contiguous elements
  [10, 9, 2, 5, 3, 7, 101, 18]
  Subarray [2, 5, 3] → indices 2,3,4 → contiguous ✓
  
SUBSEQUENCE: maintain order but can skip elements
  [10, 9, 2, 5, 3, 7, 101, 18]
  Subsequence [2, 5, 7, 101] → indices 2,3,5,6 → order preserved ✓
  
  [2, 7, 5] is NOT a valid subsequence → 7 comes before 5 in the result
  but 7 is at index 5 and 5 is at index 3 → order VIOLATED ✗

WHY THIS MATTERS:
  → Subarrays = sliding window problems (Projects 5, 6, 12)
  → Subsequences = DP problems (you choose which elements to keep)
  → The CHOICE of which elements to include is what creates overlapping subproblems

Step 2 — Try the Classic DP Approach (O(n²))

DEFINE STATE:
  dp[i] = length of the longest increasing subsequence ENDING at index i

WHY "ENDING AT i"?
  → If we defined dp[i] = "LIS of first i elements" (like knapsack)
  → We can't easily build the recurrence
  → We don't know WHERE the LIS ends → can't extend it
  → By defining "ending at i" → we know the LAST element is nums[i]
  → We can check: can we EXTEND some previous LIS by appending nums[i]?

RECURRENCE:
  dp[i] = 1 + max(dp[j]) for all j < i where nums[j] < nums[i]
  
  In words:
  → Look at ALL elements before index i
  → Find those that are SMALLER than nums[i] (can extend)
  → Take the LONGEST such subsequence
  → Append nums[i] → length increases by 1

BASE CASE:
  dp[i] = 1 for all i (every element alone is a subsequence of length 1)

ANSWER:
  max(dp[0], dp[1], ..., dp[n-1]) — LIS could end at ANY index

EXAMPLE: nums = [10, 9, 2, 5, 3, 7, 101, 18]

  dp[0] = 1                          → LIS ending at 10: [10]
  dp[1] = 1                          → LIS ending at 9: [9] (no prev < 9? 10>9)
  dp[2] = 1                          → LIS ending at 2: [2]
  dp[3] = 1 + max(dp[2]) = 2         → LIS ending at 5: [2,5]
  dp[4] = 1 + max(dp[2]) = 2         → LIS ending at 3: [2,3]
  dp[5] = 1 + max(dp[3],dp[4]) = 3   → LIS ending at 7: [2,5,7] or [2,3,7]
  dp[6] = 1 + max(dp[5]) = 4         → LIS ending at 101: [2,5,7,101]
  dp[7] = 1 + max(dp[5]) = 4         → LIS ending at 18: [2,5,7,18]

  Answer = max(1,1,1,2,2,3,4,4) = 4 ✓

TIME: O(n²) — for each i, scan all j < i
SPACE: O(n) — dp array

Step 3 — Why O(n²) Can Be Too Slow

For n = 10,000:  O(n²) = 10^8 → borderline (1-2 seconds)
For n = 100,000: O(n²) = 10^10 → WAY too slow
For n = 1,000,000: forget it

WE NEED A FASTER APPROACH.

THE BOTTLENECK:
  → For each i, we scan ALL j < i to find the best predecessor
  → This "scan all previous" step is O(n) per element → O(n²) total
  → Can we find the best predecessor FASTER?
  → YES — using BINARY SEARCH on a cleverly maintained structure

Step 4 — The Patience Sorting Insight

ANALOGY: Patience card game (a form of solitaire)

RULES:
  → Cards arrive one by one (array elements)
  → You place each card on PILES:
       → If the card is LARGER than ALL pile tops → start a NEW pile
       → Otherwise → place it on the LEFTMOST pile whose top is ≥ the card
  → The NUMBER OF PILES = length of LIS

EXAMPLE: nums = [10, 9, 2, 5, 3, 7, 101, 18]

  Card 10:  Piles: [10]
  Card 9:   9 < 10 → replace 10 with 9
            Piles: [9]
  Card 2:   2 < 9 → replace 9 with 2
            Piles: [2]
  Card 5:   5 > 2 → new pile
            Piles: [2] [5]
  Card 3:   3 > 2, 3 < 5 → replace 5 with 3
            Piles: [2] [3]
  Card 7:   7 > 3 → new pile
            Piles: [2] [3] [7]
  Card 101: 101 > 7 → new pile
            Piles: [2] [3] [7] [101]
  Card 18:  18 < 101 → replace 101 with 18
            Piles: [2] [3] [7] [18]

  Number of piles = 4 = LIS length ✓

KEY OBSERVATION:
  → The pile tops are ALWAYS in SORTED order
  → "Leftmost pile whose top ≥ card" = BINARY SEARCH for insertion point
  → Each element: O(log L) where L = current number of piles
  → Total: O(n log n)

THIS IS THE BREAKTHROUGH:
  O(n²) → O(n log n) by using binary search on pile tops

Step 5 — Why This Works (The Proof Intuition)

CLAIM: The number of piles = LIS length

WHY?

OBSERVATION 1: Pile tops are always sorted (ascending)
  → When we place a card on a pile → that pile's top decreases or stays
  → When we start a new pile → its top > all existing tops
  → Therefore: tops are always in increasing order left to right

OBSERVATION 2: Each pile represents a "slot" in potential LIS
  → Pile k's top = the SMALLEST possible ending value for an increasing
    subsequence of length k+1
  → By keeping the smallest possible ending values, we MAXIMIZE our
    ability to extend subsequences in the future

OBSERVATION 3: If we have L piles, an LIS of length L exists
  → Each pile has a card that's part of a valid increasing subsequence
  → The "new pile" events correspond to genuine extensions of the LIS
  → Replacements optimize for FUTURE extensions

INTUITION:
  → tails[k] = smallest value that can END an increasing subsequence of length k+1
  → Maintaining this array with binary search → O(n log n)
  → The LENGTH of this array = LIS length
  → The VALUES in this array are NOT the actual LIS elements
       (they represent optimal "tail" values for each length)

Step 6 — The tails Array Mechanics

tails[k] = smallest ending value of any increasing subsequence of length k+1

PROCESSING EACH ELEMENT nums[i]:

CASE 1: nums[i] > tails[last]
  → nums[i] is larger than the largest tail
  → We can EXTEND the longest subsequence by 1
  → Append nums[i] to tails → tails length increases by 1

CASE 2: nums[i] ≤ tails[last]
  → nums[i] can't extend the longest subsequence
  → But it might IMPROVE a shorter subsequence's tail
  → Find the LEFTMOST position in tails where tails[pos] >= nums[i]
  → Replace tails[pos] with nums[i]
  → This doesn't change LIS length but optimizes for future extensions

FINDING THE POSITION: Binary search for "first element ≥ nums[i]"
  → tails is sorted → binary search works → O(log L)
  → This is the "lower bound" search (from Project 9's binary search)

EXAMPLE WALKTHROUGH: nums = [10, 9, 2, 5, 3, 7, 101, 18]

  tails = []

  Process 10: tails empty → append
    tails = [10]

  Process 9:  9 < 10 → find first ≥ 9 in tails → index 0 (value 10)
              replace tails[0] with 9
    tails = [9]

  Process 2:  2 < 9 → find first ≥ 2 in tails → index 0 (value 9)
              replace tails[0] with 2
    tails = [2]

  Process 5:  5 > 2 (last element) → append
    tails = [2, 5]

  Process 3:  3 ≤ 5 → find first ≥ 3 in tails → index 1 (value 5)
              replace tails[1] with 3
    tails = [2, 3]

  Process 7:  7 > 3 (last element) → append
    tails = [2, 3, 7]

  Process 101: 101 > 7 → append
    tails = [2, 3, 7, 101]

  Process 18: 18 ≤ 101 → find first ≥ 18 in tails → index 3 (value 101)
              replace tails[3] with 18
    tails = [2, 3, 7, 18]

  LIS length = tails.length = 4 ✓

NOTE: tails = [2, 3, 7, 18] is NOT necessarily the actual LIS
  → It's the optimal tail values for each subsequence length
  → An actual LIS is [2, 5, 7, 101] or [2, 3, 7, 18] or [2, 5, 7, 18]
  → To recover the actual LIS → need additional tracking (covered below)

🛠️ HOW CAN WE SOLVE IT

The Algorithm — Both Approaches

APPROACH 1: Classic DP — O(n²)
  1. Create dp[n] initialized to 1 (each element is LIS of length 1)
  2. For i = 1 to n-1:
       For j = 0 to i-1:
         if nums[j] < nums[i]:
             dp[i] = max(dp[i], dp[j] + 1)
  3. Return max(dp[0..n-1])

APPROACH 2: Binary Search on tails — O(n log n)
  1. Create tails list (initially empty)
  2. For each num in nums:
       if tails is empty OR num > tails[last]:
           tails.append(num)    → extend LIS
       else:
           pos = binary search for first element ≥ num in tails
           tails[pos] = num     → optimize tail value
  3. Return tails.length

✅ WHAT SHOULD YOU DO

→ ALWAYS clarify: STRICTLY increasing or non-decreasing?
     → Strictly increasing: nums[j] < nums[i] → use "first ≥" (lower_bound)
     → Non-decreasing: nums[j] ≤ nums[i] → use "first >" (upper_bound)
     → Different binary search target → different results!

→ ALWAYS start with the O(n²) DP to understand the problem
     → It's correct and easy to verify
     → Then optimize to O(n log n) with binary search
     → In interviews: explain O(n²) first, then optimize

→ ALWAYS remember: tails array gives LENGTH but not the actual LIS
     → Length = tails.size() → always correct
     → Values in tails → NOT necessarily the actual subsequence
     → To recover actual LIS → need parent tracking (covered below)

→ ALWAYS use Arrays.binarySearch or custom binary search correctly
     → Java's Arrays.binarySearch returns:
          → index if found
          → -(insertion point) - 1 if not found
     → Need careful handling for "first element ≥ target"
     → Custom binary search is often cleaner for this specific case

→ ALWAYS handle the empty tails case
     → First element always starts a new subsequence
     → Either check explicitly or handle via binary search logic

→ ALWAYS consider whether to use int[] or ArrayList for tails
     → int[] with size counter: faster (no boxing/unboxing)
     → ArrayList<Integer>: simpler code but slower
     → For competitive programming: int[] preferred
     → For interviews: either is fine

🚫 WHAT SHOULD YOU NOT DO

→ DO NOT confuse subsequence with subarray
     → Subarray = contiguous → sliding window
     → Subsequence = can skip → DP
     → "Longest increasing subarray" is a different (easier) problem

→ DO NOT assume tails array IS the LIS
     → tails = [2, 3, 7, 18] → this is NOT necessarily a valid subsequence
     → It represents optimal tail values for each possible length
     → The actual LIS might be [2, 5, 7, 101]
     → Only the LENGTH of tails is guaranteed correct

→ DO NOT use "first element >" when problem says "strictly increasing"
     → Strictly increasing: each element > previous
     → Binary search for "first ≥" (lower_bound) → correct for strict
     → "First >" (upper_bound) → gives non-decreasing (allows equal)
     → Getting this wrong → wrong LIS length for arrays with duplicates

→ DO NOT forget that dp[i] = 1 initially (not 0)
     → Every single element is a valid subsequence of length 1
     → Starting from 0 → every dp value is off by 1

→ DO NOT skip the O(n²) approach in interviews
     → Start with O(n²) → shows understanding
     → Then say "we can optimize using binary search"
     → Jumping straight to O(n log n) without explanation → risky
     → Interviewer might think you memorized the solution

→ DO NOT use this for "longest non-decreasing subsequence" without adjusting
     → Non-decreasing allows equal consecutive elements
     → Need upper_bound instead of lower_bound in binary search
     → Or: use nums[j] <= nums[i] in O(n²) DP

🔀 WHEN TO USE THIS PATTERN

✓ "Find the longest increasing/decreasing subsequence"
✓ "Minimum number of elements to remove for sorted order"
     → = n - LIS length
✓ "Maximum number of envelopes that can be nested"
     → Russian doll envelopes → 2D LIS variant
✓ "Longest chain of pairs"
     → Sort by first element → LIS on second element
✓ "Number of longest increasing subsequences" (count variant)
✓ Any problem reducible to LIS

REAL-WORLD APPLICATIONS:
  → Stock analysis: longest period of increasing prices (non-contiguous)
  → Task scheduling: longest chain of dependent tasks
  → DNA sequencing: longest common increasing subsequence
  → Patience sorting algorithm (actual sorting method)

🚫 WHEN NOT TO USE THIS PATTERN

✗ Problem asks for CONTIGUOUS increasing subarray
     → Simple linear scan or sliding window → O(n)
     → No DP or binary search needed

✗ Array is already sorted
     → LIS = entire array → answer is n
     → No computation needed

✗ Need longest COMMON subsequence of TWO arrays
     → Different problem → 2D DP (Project 50)
     → LIS is single-array; LCS is two-array

✗ Need the actual subsequence elements (not just length)
     → O(n log n) tails approach gives LENGTH only
     → Need additional tracking for actual elements
     → O(n²) DP with parent array is simpler for recovery

✗ Subsequence must satisfy complex constraints beyond "increasing"
     → LIS binary search trick relies on simple ordering
     → Complex constraints may need general backtracking or 2D DP

⚙️ HOW IT ONLY WORKS — Internal Mechanics

The tails Array Invariant

INVARIANT: At all times:
  1. tails is SORTED in strictly increasing order
  2. tails[k] = smallest possible ending value for an increasing 
     subsequence of length k+1
  3. tails.length = length of the longest increasing subsequence found so far

PROOF OF INVARIANT 1 (tails is always sorted):

  Initially: tails is empty → trivially sorted ✓

  When we APPEND (num > tails[last]):
    → num > tails[last] → sorted order maintained ✓

  When we REPLACE tails[pos] with num:
    → We found pos such that tails[pos] ≥ num (first such position)
    → Therefore: tails[pos-1] < num (previous element is smaller)
    → After replacement: tails[pos-1] < num ≤ old tails[pos]
    → And num < tails[pos+1] (since tails[pos] ≤ tails[pos+1] and 
      tails[pos] was the FIRST ≥ num → tails[pos+1] > num)
    → Sorted order maintained ✓

PROOF OF INVARIANT 2 (optimal tail values):

  When we replace tails[pos] with a SMALLER value num:
    → There exists an increasing subsequence of length pos+1 ending at num
    → num ≤ old tails[pos] → this is a BETTER (smaller) ending value
    → Smaller ending → easier to extend in the future
    → Optimal ✓

  When we append num:
    → There exists an increasing subsequence of length tails.length ending at num
    → This is the FIRST subsequence of this length → trivially optimal ✓

Binary Search Implementation Detail

We need: "first position in tails where tails[pos] >= num"
This is called "lower bound" search.

CUSTOM BINARY SEARCH:
  int lo = 0, hi = tailsLength;
  while (lo < hi) {
      int mid = lo + (hi - lo) / 2;
      if (tails[mid] < num) {
          lo = mid + 1;    // tails[mid] too small → look right
      } else {
          hi = mid;         // tails[mid] >= num → could be answer, look left
      }
  }
  // lo = first position where tails[pos] >= num

WHY NOT Arrays.binarySearch?
  → Arrays.binarySearch returns negative value when not found
  → Requires awkward transformation: -(result) - 1
  → Custom binary search is cleaner and more intuitive
  → Also: Arrays.binarySearch finds ANY match, not necessarily the FIRST

FOR NON-DECREASING (allowing equal):
  → Change condition to tails[mid] <= num → lo = mid + 1
  → This finds "first position where tails[pos] > num" (upper bound)
  → Allows equal values in the subsequence

Time Complexity — Precise Analysis

O(n²) DP:
  → Outer loop: n iterations
  → Inner loop: up to n iterations
  → Total: O(n²)
  → Space: O(n) for dp array

O(n log n) Binary Search:
  → For each of n elements:
       → Binary search on tails: O(log L) where L ≤ n
       → Append or replace: O(1)
  → Total: O(n log n)
  → Space: O(n) for tails array (worst case: entire array is sorted)

COMPARISON:
  n = 10,000:
    O(n²) = 10^8 → 1-2 seconds → borderline
    O(n log n) = 10^4 * 13 ≈ 130K → instant

  n = 100,000:
    O(n²) = 10^10 → minutes → TLE
    O(n log n) = 10^5 * 17 ≈ 1.7M → instant

  n = 1,000,000:
    O(n²) = impossible
    O(n log n) = 10^6 * 20 ≈ 20M → ~200ms → comfortable

Recovering the Actual LIS (Not Just Length)

THE PROBLEM: tails gives us LENGTH but not the actual subsequence.

SOLUTION FOR O(n²) APPROACH:
  → Maintain parent[] array alongside dp[]
  → parent[i] = index of previous element in LIS ending at i
  → After finding the max dp[i], trace back through parent pointers

SOLUTION FOR O(n log n) APPROACH:
  → Maintain a parallel array tracking which ORIGINAL index was placed at each position
  → Also maintain a predecessor array for each element
  → When replacing tails[pos] with nums[i]:
       → Record that nums[i]'s predecessor is whoever is at position pos-1
  → After processing all elements:
       → Start from the last element placed in the last tails position
       → Follow predecessor chain backward → reconstruct LIS

This recovery adds O(n) space but doesn't change time complexity.

⚠️ COMMON PITFALLS AND FAILURE MODES

Pitfall 1: Strictly Increasing vs Non-Decreasing

nums = [1, 3, 3, 5]

STRICTLY INCREASING (each > previous):
  LIS = [1, 3, 5] → length 3
  Binary search: "first ≥" → when processing second 3:
    tails = [1, 3] → first ≥ 3 is index 1 → replace tails[1] with 3 → [1, 3]
    (no change — 3 replaces 3)
    tails stays [1, 3] → next 5 appends → [1, 3, 5] → length 3 ✓

NON-DECREASING (each ≥ previous):
  LIS = [1, 3, 3, 5] → length 4
  Binary search: "first >" → when processing second 3:
    tails = [1, 3] → first > 3 is index 2 (beyond end) → APPEND 3
    tails = [1, 3, 3] → next 5 appends → [1, 3, 3, 5] → length 4 ✓

THE DIFFERENCE IS IN THE BINARY SEARCH CONDITION:
  Strictly increasing: tails[mid] < num → go right (find first ≥)
  Non-decreasing:      tails[mid] <= num → go right (find first >)
  
  ONE CHARACTER DIFFERENCE (< vs <=) → DIFFERENT ANSWER

Pitfall 2: Forgetting dp[i] = 1 Initialization

WRONG:
  int[] dp = new int[n]; // all zeros
  for (int i = 1; i < n; i++)
      for (int j = 0; j < i; j++)
          if (nums[j] < nums[i])
              dp[i] = Math.max(dp[i], dp[j] + 1);
  
  → dp[0] = 0 (never updated!) → wrong
  → dp values are all 1 less than correct

CORRECT:
  int[] dp = new int[n];
  Arrays.fill(dp, 1); // every element is LIS of length 1
  for (int i = 1; i < n; i++)
      for (int j = 0; j < i; j++)
          if (nums[j] < nums[i])
              dp[i] = Math.max(dp[i], dp[j] + 1);

Pitfall 3: All Equal Elements

nums = [7, 7, 7, 7, 7]

STRICTLY increasing: LIS = 1 (can't have equal neighbors)

With O(n²) DP:
  dp = [1, 1, 1, 1, 1] → nums[j] < nums[i] is NEVER true → correct ✓

With binary search:
  Process 7: tails = [7]
  Process 7: first ≥ 7 is index 0 → replace tails[0] with 7 → tails = [7]
  (repeats for all elements)
  tails = [7] → length 1 ✓

Both handle it correctly as long as we use STRICT comparison.

Pitfall 4: Single Element

nums = [42]
→ LIS = 1

Both approaches handle this:
  O(n²): dp = [1] → max = 1 ✓
  Binary search: tails = [42] → length 1 ✓

Edge case but trivially correct.

Pitfall 5: Assuming tails IS the LIS

nums = [3, 1, 4, 1, 5, 9, 2, 6]

Binary search process:
  3 → tails = [3]
  1 → replace 3 → tails = [1]
  4 → append → tails = [1, 4]
  1 → replace 1 → tails = [1, 4]   (no change)
  5 → append → tails = [1, 4, 5]
  9 → append → tails = [1, 4, 5, 9]
  2 → replace 4 → tails = [1, 2, 5, 9]
  6 → replace 9 → tails = [1, 2, 5, 6]

  tails = [1, 2, 5, 6] → length 4 ✓
  
  But is [1, 2, 5, 6] an actual LIS?
  → 1 is at index 1 or 3
  → 2 is at index 6
  → 5 is at index 4
  → 6 is at index 7
  → Order: 1(idx3), 5(idx4), 2(idx6), 6(idx7) → 2 < 5 → NOT increasing!
  
  tails = [1, 2, 5, 6] is NOT a valid subsequence!
  The actual LIS could be [1, 4, 5, 9] or [1, 4, 5, 6] or [3, 4, 5, 9]
  
  LENGTH is correct (4) → VALUES are misleading

Pitfall 6: Negative Numbers

nums = [-5, -3, -8, -1, -7]

Works identically — comparisons handle negatives naturally:
  -5: tails = [-5]
  -3: -3 > -5 → append → tails = [-5, -3]
  -8: first ≥ -8 is index 0 (-5) → replace → tails = [-8, -3]
  -1: -1 > -3 → append → tails = [-8, -3, -1]
  -7: first ≥ -7 is index 1 (-3) → replace → tails = [-8, -7, -1]

  LIS length = 3 ✓ (e.g., [-5, -3, -1] or [-8, -3, -1])

📈 SCALABILITY CONSIDERATIONS

SCENARIO 1: Small n (n ≤ 1000)
  → O(n²) = 10^6 → instant
  → Use O(n²) for simplicity + easy LIS recovery

SCENARIO 2: Medium n (n ≤ 10,000)
  → O(n²) = 10^8 → 1-2 seconds → borderline
  → O(n log n) recommended for safety

SCENARIO 3: Large n (n ≤ 100,000)
  → O(n²) = 10^10 → TLE
  → O(n log n) = 10^5 * 17 ≈ 2M → instant
  → MUST use binary search approach

SCENARIO 4: Very large n (n ≤ 10^6)
  → O(n log n) = 10^6 * 20 ≈ 20M → ~200ms
  → Comfortable but use int[] for tails (not ArrayList)
  → Avoid boxing/unboxing overhead

MEMORY:
  → O(n²) DP: dp array = O(n) (manageable)
  → O(n log n): tails array = O(n) worst case
  → Both use O(n) space — no advantage either way
  → But O(n²) DP with parent tracking also O(n)

🔍 INTERPRETABILITY CONSIDERATIONS

MENTAL MODEL — "Building the Tallest Tower of Blocks"

→ Blocks arrive one by one (array elements)
→ Each block has a height (its value)
→ You can only stack a block on top if it's TALLER than the current top
→ You want the TALLEST possible tower (most blocks)
→ You can SKIP blocks that don't help

O(n²) DP MODEL:
  → For each new block: "Which existing tower can I extend?"
  → Check ALL towers → find the tallest one I can extend → add my block
  → Track the height of each tower ending at each block

O(n log n) TAILS MODEL:
  → Maintain "showcase towers" — one per possible height
  → Each showcase shows the SHORTEST top block for that tower height
  → New block: if taller than all showcases → start new (taller) showcase
  → Otherwise: find the showcase where it can replace the top with a shorter block
  → The number of showcases = the tallest possible tower

DEBUGGING:
  → Print dp array for O(n²) → verify each value
  → Print tails array after each step for O(n log n)
  → Verify tails is always sorted
  → Compare both approaches' final lengths — must match

📊 THE CROSS-PATTERN FUSION — DP + Binary Search

┌───────────┬────────────────────────────────────────────────────────┐
│ Pattern   │ How It Contributes to LIS                             │
├───────────┼────────────────────────────────────────────────────────┤
│ DP        │ Defines the state: "LIS ending at index i"            │
│ (P43-44)  │ Recurrence: dp[i] = 1 + max(dp[j]) for valid j       │
│           │ Gives O(n²) solution → correct but slow               │
├───────────┼────────────────────────────────────────────────────────┤
│ Binary    │ Optimizes the "find best predecessor" step            │
│ Search    │ Instead of scanning all j < i → binary search on      │
│ (P9)      │ sorted tails array → O(log n) per element             │
│           │ Reduces O(n²) → O(n log n)                            │
├───────────┼────────────────────────────────────────────────────────┤
│ Greedy    │ The tails array maintenance is greedy:                │
│ (P19)     │ "Always keep the SMALLEST possible tail for each      │
│           │  subsequence length" → maximizes future extensibility  │
│           │ Local optimal (smallest tail) = global optimal (LIS)   │
└───────────┴────────────────────────────────────────────────────────┘

THIS IS THE FIRST TRUE CROSS-PATTERN FUSION:
  → DP provides the FRAMEWORK (what to compute)
  → Binary Search provides the SPEED (how to compute fast)
  → Greedy provides the STRATEGY (what to maintain)
  → Three patterns COMBINE into something none could achieve alone

WHY THIS MATTERS:
  → Real interview problems rarely use ONE pattern in isolation
  → Recognizing that "this DP bottleneck can be binary-searched"
     is the mark of an advanced problem solver
  → The ability to COMBINE patterns = competitive advantage

🧠 KEY TAKEAWAY

WHAT WE LEARNED:
→ LIS is solvable in O(n²) with classic DP → good starting point
→ The O(n²) bottleneck is "scan all predecessors" → O(n) per element
→ Maintaining a sorted tails array enables binary search → O(log n) per element
→ tails[k] = smallest ending value for increasing subsequence of length k+1
→ tails gives correct LENGTH but NOT the actual subsequence
→ To recover actual LIS → need additional predecessor tracking

THE BINARY SEARCH ON TAILS — WHY IT'S BRILLIANT:
→ The tails array is always sorted → binary search works
→ Each new element either EXTENDS (append) or OPTIMIZES (replace)
→ Extending increases LIS length → append to tails
→ Optimizing improves future extensibility → replace in tails
→ This greedy strategy provably gives the correct LIS length

THE PATTERN FUSION LESSON:
→ DP alone: O(n²) — good but not great
→ Binary search alone: can't solve LIS
→ DP + Binary Search + Greedy: O(n log n) — optimal
→ The COMBINATION is more powerful than any individual pattern
→ This cross-pollination thinking is what makes expert problem solvers

STRICTLY vs NON-DECREASING:
→ Strictly increasing: binary search for "first ≥" (lower_bound)
→ Non-decreasing: binary search for "first >" (upper_bound)
→ ONE character difference in code → completely different behavior
→ ALWAYS clarify which version the problem requires

PATTERN SIGNATURE:
→ "Longest increasing/decreasing subsequence" ✓
→ "Minimum deletions to make sorted" ✓
→ "Maximum chain length" ✓
→ Array elements with ordering constraint ✓
→ = LIS → O(n²) DP or O(n log n) DP + BINARY SEARCH
