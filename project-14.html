PROJECT 14: Merge Overlapping Intervals

🎯 Merge Intervals → Sort-then-Scan → Overlap Detection Logic

🔗 DEPENDENCY CHAIN

Project 9  → Binary Search → sorted data as prerequisite → half-elimination
Project 11 → Three Sum → WE sorted the data ourselves → sorting as a tool

             → detect overlaps → merge in-place
             → introduces INTERVAL PROCESSING — an entirely new pattern family
             → intervals appear in scheduling, calendars, resource allocation

🧩 THE HYPOTHETICAL PROBLEM

→ You are given a list of INTERVALS
→ Each interval is represented as [start, end]
→ Some intervals OVERLAP with each other
→ MERGE all overlapping intervals
→ Return the list of merged (non-overlapping) intervals

Example 1:
  Input:  [[1,3], [2,6], [8,10], [15,18]]
  Output: [[1,6], [8,10], [15,18]]
  
  → [1,3] and [2,6] overlap (3 ≥ 2) → merge into [1,6]
  → [8,10] stands alone
  → [15,18] stands alone

Example 2:
  Input:  [[1,4], [4,5]]
  Output: [[1,5]]
  
  → [1,4] and [4,5] share the point 4 → they TOUCH → considered overlapping
  → Merge into [1,5]

Example 3:
  Input:  [[1,4], [0,4]]
  Output: [[0,4]]
  
  → [0,4] completely contains [1,4]
  → Merge into [0,4]

Example 4:
  Input:  [[1,4], [2,3]]
  Output: [[1,4]]
  
  → [2,3] is completely INSIDE [1,4]
  → Merge into [1,4] (the larger one absorbs the smaller)

Example 5:
  Input:  [[6,8], [1,3], [2,4], [9,12]]
  Output: [[1,4], [6,8], [9,12]]
  
  → UNSORTED input → must sort first
  → After sorting by start: [[1,3], [2,4], [6,8], [9,12]]
  → [1,3] and [2,4] overlap → [1,4]
  → [6,8] and [9,12] don't overlap → stay separate

🔍 HOW TO APPROACH IT

Step 1 — Understanding What "Overlap" Means

Two intervals [a, b] and [c, d] OVERLAP when:

  CASE 1: Partial overlap
    [1,  5]
       [3,    8]
    → They share the range [3, 5]
    → Merged: [1, 8]

  CASE 2: Complete containment
    [1,      8]
      [3, 5]
    → [3,5] is entirely inside [1,8]
    → Merged: [1, 8]

  CASE 3: Touching at a point
    [1, 5]
         [5, 8]
    → They share the single point 5
    → Merged: [1, 8]

  CASE 4: No overlap
    [1, 3]
            [5, 8]
    → Gap between 3 and 5
    → NOT overlapping → stay separate

THE RULE (for sorted intervals where a.start ≤ b.start):
  → a and b OVERLAP if and only if: a.end ≥ b.start
  → If a ends at or after b starts → they overlap
  → If a ends before b starts → gap exists → no overlap

Step 2 — Why Sorting Is Essential

WITHOUT SORTING:
  Input: [[6,8], [1,3], [2,4], [9,12]]
  
  Compare [6,8] with [1,3]: do they overlap?
    → 6 > 3 → no overlap? But we need to check if [1,3] overlaps with [6,8]
    → Actually: 3 < 6 → no overlap ✓
    → But we had to check BOTH directions

  Compare [6,8] with [2,4]: overlap?
    → 8 ≥ 2 → yes? NO → [6,8] starts AFTER [2,4] ends → no overlap
    → Confusing because intervals are out of order

  Without sorting, we'd need to compare EVERY pair → O(n²)

WITH SORTING (by start time):
  Sorted: [[1,3], [2,4], [6,8], [9,12]]
  
  Now intervals are in ORDER → we only need to compare ADJACENT intervals
  → [1,3] vs [2,4]: 3 ≥ 2 → overlap → merge to [1,4]
  → [1,4] vs [6,8]: 4 < 6 → no overlap → keep separate
  → [6,8] vs [9,12]: 8 < 9 → no overlap → keep separate
  
  Result: [[1,4], [6,8], [9,12]] ✓

WHY ADJACENT-ONLY COMPARISON WORKS AFTER SORTING:
  → If interval A doesn't overlap with interval B (next one)
  → Then A can't overlap with C, D, E... (further ones)
  → Because C.start ≥ B.start > A.end
  → So once we find a gap, ALL subsequent intervals are separate from A
  → This is the power of sorted order → LINEAR scan suffices

Step 3 — The Merge Logic

After sorting by start time:

INITIALIZE:
  → Take the first interval as the "current" interval being built
  → merged = [first interval's start, first interval's end]

FOR EACH subsequent interval [start, end]:
  
  CASE A: Overlap (current.end ≥ start)
    → The new interval overlaps with the current one
    → EXTEND the current interval: current.end = max(current.end, end)
    → WHY max? Because the new interval might end BEFORE current
      → [1, 10] and [3, 5] → current.end = max(10, 5) = 10
      → [1, 3] and [2, 8] → current.end = max(3, 8) = 8

  CASE B: No overlap (current.end < start)
    → Gap exists between current interval and new interval
    → SAVE the current interval to result
    → START a new current interval = [start, end]

AFTER LOOP:
  → Save the last current interval (it was never saved inside the loop)

VISUAL:
  Sorted: [[1,3], [2,6], [8,10], [15,18]]
  
  current = [1, 3]
  
  Next [2, 6]: current.end(3) ≥ start(2) → overlap
    → current.end = max(3, 6) = 6 → current = [1, 6]
  
  Next [8, 10]: current.end(6) < start(8) → NO overlap
    → Save [1, 6] to result
    → current = [8, 10]
  
  Next [15, 18]: current.end(10) < start(15) → NO overlap
    → Save [8, 10] to result
    → current = [15, 18]
  
  End of loop → save [15, 18] to result
  
  Result: [[1, 6], [8, 10], [15, 18]] ✓

Step 4 — Why max(current.end, new.end) Not Just new.end

SCENARIO: Complete containment

  current = [1, 10]
  next = [3, 5]
  
  They overlap (10 ≥ 3) → merge
  
  IF we set current.end = new.end:
    → current = [1, 5] → WRONG → we LOST the range [5, 10]
  
  IF we set current.end = max(current.end, new.end):
    → current.end = max(10, 5) = 10 → current = [1, 10] → CORRECT

SCENARIO: Normal extension

  current = [1, 3]
  next = [2, 6]
  
  IF we set current.end = max(3, 6) = 6 → current = [1, 6] → CORRECT

→ max() handles BOTH cases correctly
→ Simple assignment handles NEITHER correctly for all cases
→ This is the MOST common bug in interval merging

Step 5 — Why Sort by Start Time (Not End Time)

SORT BY START:
  [[1,5], [2,3], [4,8]]
  
  → Process left-to-right
  → current = [1,5]
  → [2,3]: 5 ≥ 2 → overlap → current = [1, max(5,3)] = [1,5]
  → [4,8]: 5 ≥ 4 → overlap → current = [1, max(5,8)] = [1,8]
  → Result: [[1,8]] ✓

SORT BY END:
  [[2,3], [1,5], [4,8]]
  
  → current = [2,3]
  → [1,5]: 3 ≥ 1 → overlap → current = [2, max(3,5)] = [2,5]
  → BUT WAIT: current should start at 1 (min of starts), not 2!
  → We also need to update current.start = min(current.start, new.start)
  → More complex and error-prone

  → current = [min(2,1), max(3,5)] = [1,5]
  → [4,8]: 5 ≥ 4 → overlap → current = [1, max(5,8)] = [1,8]
  → Result: [[1,8]] ✓ → but required extra logic

SORT BY START is simpler because:
  → When intervals are sorted by start → current.start is ALWAYS correct
  → We only need to update current.end → ONE update per merge
  → Sort by end requires updating BOTH start and end → TWO updates per merge
  → Sort by start is the standard convention

🛠️ HOW CAN WE SOLVE IT

The Algorithm — Complete Logic

1. EDGE CASE: if intervals is empty or has one interval → return as-is

2. SORT intervals by start time ascending
     → Arrays.sort(intervals, (a, b) -> a[0] - b[0])
     → Or: Arrays.sort(intervals, (a, b) -> Integer.compare(a[0], b[0]))

3. INITIALIZE:
     → List<int[]> merged = new ArrayList<>()
     → int[] currentInterval = intervals[0]  (or add first to merged list)
     → Add currentInterval to merged list

4. FOR i = 1 to intervals.length - 1:
     → int[] nextInterval = intervals[i]
     
     a. IF currentInterval[1] >= nextInterval[0]:
          → OVERLAP → extend: currentInterval[1] = max(currentInterval[1], nextInterval[1])
          → (currentInterval is already in merged list → modifying it updates the list)
     
     b. ELSE:
          → NO OVERLAP → start new interval
          → currentInterval = nextInterval
          → Add currentInterval to merged list

5. CONVERT merged list to array and return

✅ WHAT SHOULD YOU DO

→ ALWAYS sort by start time first
     → This is the NON-NEGOTIABLE prerequisite for the linear scan
     → Without sorting → must compare all pairs → O(n²)
     → With sorting → adjacent comparison suffices → O(n)

→ ALWAYS use max() when extending the end of the current interval
     → Handles both partial overlap AND complete containment
     → Simple assignment fails for containment cases

→ ALWAYS add the current interval to the result list BEFORE the loop
     → Then modify it in-place when merging
     → The list holds a REFERENCE to the array → modifications propagate
     → This is a Java-specific trick: arrays are reference types

→ ALWAYS handle the edge cases upfront
     → Empty input → return empty
     → Single interval → return as-is
     → All overlapping → return single merged interval
     → No overlapping → return input sorted

→ ALWAYS use Integer.compare(a, b) over a - b for sorting
     → a - b can OVERFLOW for extreme values
     → Integer.compare is safe
     → Example: a = Integer.MIN_VALUE, b = 1 → a - b overflows → wrong sort order

🚫 WHAT SHOULD YOU NOT DO

→ DO NOT compare all pairs of intervals
     → O(n²) approach: for each interval, check overlap with every other
     → Unnecessary after sorting → adjacent comparison suffices
     → Sorting + scan = O(n log n) + O(n) = O(n log n) total

→ DO NOT modify the input array during processing
     → Sorting modifies the order (acceptable if problem allows)
     → But don't merge IN the original array → use a separate result list
     → If input must be preserved → clone it first

→ DO NOT forget to add the LAST interval
     → The loop saves intervals only when a gap is found
     → The very last interval is never followed by a gap → never saved in the loop
     → Must be saved AFTER the loop
     → OR use the trick: add first interval to result, modify in-place

→ DO NOT use current.end = next.end instead of max(current.end, next.end)
     → Fails for containment: [1,10] merging with [3,5] → would become [1,5] → WRONG
     → ALWAYS use max()

→ DO NOT sort by end time
     → Works but requires updating BOTH start and end during merge
     → Sort by start time → only update end → simpler and standard

→ DO NOT use a HashSet or HashMap for overlap detection
     → Intervals are continuous ranges, not discrete elements
     → Set-based approaches don't capture range overlap correctly
     → The sort-and-scan approach is purpose-built for intervals

🔀 WHEN TO USE THIS PATTERN

✓ Problem involves INTERVALS (ranges with start and end)
     → Meeting times, flight schedules, time ranges, coordinate ranges

✓ Problem asks to MERGE overlapping intervals
     → "Combine overlapping ranges"
     → "Simplify a list of intervals"
     → "Find the union of all intervals"

✓ Problem asks to COUNT overlapping intervals
     → "How many intervals overlap at the same point?"
     → "Maximum number of concurrent meetings" (Project 46)
     → Same sort-first approach, different processing logic

✓ Problem asks to INSERT a new interval and re-merge
     → "Add a meeting to a schedule" (Project 22)
     → Extension of the merge pattern

✓ Problem asks to find GAPS between intervals
     → "Free time between meetings"
     → Sort → scan → gaps are where current.end < next.start

✓ Data can be SORTED (or is already sorted by start)
     → The sort-then-scan paradigm applies

🚫 WHEN NOT TO USE THIS PATTERN

✗ Intervals are in higher dimensions (rectangles, cubes)
     → 2D interval overlap requires different algorithms
     → Sweep line algorithm or interval tree structures

✗ Problem asks for INTERSECTION (not union/merge)
     → "Find the range that ALL intervals share"
     → Different logic: track max(starts) and min(ends)

✗ Problem involves weighted intervals
     → "Choose non-overlapping intervals to maximize total weight"
     → Dynamic programming, not merge intervals

✗ Intervals are not numeric ranges
     → "Find overlapping strings" → not interval problem
     → Interval pattern requires numeric start/end with ordering

✗ You need to maintain original order
     → Sorting destroys original order
     → If problem requires preserving insertion order → different approach

⚙️ HOW IT ONLY WORKS — Internal Mechanics

Mechanic 1: Why Sorting Guarantees Adjacent-Only Comparison

CLAIM: After sorting by start time, if interval A doesn't overlap with
       interval B (the next one), then A doesn't overlap with ANY
       subsequent interval C, D, E, ...

PROOF:
  → Sorted: A.start ≤ B.start ≤ C.start ≤ D.start ≤ ...
  → A doesn't overlap with B: A.end < B.start
  → For any C after B: C.start ≥ B.start > A.end
  → Therefore: A.end < C.start → A doesn't overlap with C
  → By induction: A doesn't overlap with any interval after B
  
  QED.

CONSEQUENCE:
  → When we find A doesn't overlap with B:
    → We FINALIZE A (add to result) → it can't merge with anything else
    → We start fresh with B as the new "current"
  → When A overlaps with B:
    → We merge them → the merged interval might overlap with C
    → So we continue checking the merged interval against C
  → This CHAIN of merges is handled by the while/for loop naturally

Mechanic 2: The Reference Trick in Java

STANDARD APPROACH:
  List<int[]> merged = new ArrayList<>();
  int[] current = intervals[0];
  merged.add(current);
  
  for (int i = 1; i < intervals.length; i++) {
      if (current[1] >= intervals[i][0]) {
          // Overlap → extend current's end
          current[1] = Math.max(current[1], intervals[i][1]);
          // current is already IN merged list → list is auto-updated!
      } else {
          // No overlap → new current
          current = intervals[i];
          merged.add(current);
      }
  }

WHY THIS WORKS:
  → int[] is a reference type in Java
  → merged.add(current) adds a REFERENCE to the array, not a copy
  → When we modify current[1], the element in merged is ALSO modified
  → Because they point to the SAME array in memory

  → This is elegant: no need to track indices or update the list manually
  → One variable (current) controls what's in the list

ALTERNATIVE (explicit index tracking):
  merged.add(new int[]{intervals[0][0], intervals[0][1]});
  
  for (int i = 1; i < intervals.length; i++) {
      int[] lastMerged = merged.get(merged.size() - 1);
      if (lastMerged[1] >= intervals[i][0]) {
          lastMerged[1] = Math.max(lastMerged[1], intervals[i][1]);
      } else {
          merged.add(new int[]{intervals[i][0], intervals[i][1]});
      }
  }
  
  → Same logic, different style
  → merged.get(merged.size() - 1) accesses the last element

Mechanic 3: All Overlap Cases Handled by One Condition

After sorting (A.start ≤ B.start guaranteed):

CASE 1: Partial overlap
  A: [1───5]
  B:    [3──────8]
  Check: A.end(5) ≥ B.start(3) → YES → merge
  Result: [1, max(5,8)] = [1, 8]

CASE 2: B inside A (containment)
  A: [1──────────10]
  B:    [3──5]
  Check: A.end(10) ≥ B.start(3) → YES → merge
  Result: [1, max(10,5)] = [1, 10]

CASE 3: Touching at boundary
  A: [1───5]
  B:       [5───8]
  Check: A.end(5) ≥ B.start(5) → YES → merge
  Result: [1, max(5,8)] = [1, 8]

CASE 4: Same start
  A: [3───5]
  B: [3──────8]
  Check: A.end(5) ≥ B.start(3) → YES → merge
  Result: [3, max(5,8)] = [3, 8]

CASE 5: No overlap
  A: [1───3]
  B:         [5───8]
  Check: A.end(3) ≥ B.start(5) → NO → separate
  Save A, start new with B

→ ONE condition (current.end >= next.start) handles ALL five cases
→ ONE merge operation (max of ends) handles all merge cases
→ The simplicity comes from SORTING guaranteeing start order

Mechanic 4: Chain Merging

What if multiple consecutive intervals all overlap?

Input: [[1,4], [2,5], [3,7], [6,9]]

Sorted: [[1,4], [2,5], [3,7], [6,9]]

Step 1: current = [1,4]
Step 2: [2,5] → 4 ≥ 2 → merge → current = [1, max(4,5)] = [1,5]
Step 3: [3,7] → 5 ≥ 3 → merge → current = [1, max(5,7)] = [1,7]
Step 4: [6,9] → 7 ≥ 6 → merge → current = [1, max(7,9)] = [1,9]

Result: [[1,9]]

→ All four intervals merged into one
→ The "current" interval kept GROWING as each new overlap was found
→ No special handling needed for chain merges
→ The algorithm naturally chains because current keeps updating

KEY INSIGHT:
→ After merging [1,4] with [2,5] → current = [1,5]
→ Now current's END is 5, not 4
→ This EXTENDED END might overlap with intervals that the ORIGINAL [1,4] didn't reach
→ [3,7] overlaps with [1,5] but might NOT have overlapped with [1,4] directly
→ Wait, 4 ≥ 3 → it would have → bad example
→ Better: [[1,3], [2,5], [4,8]] 
→ [1,3] and [4,8]: 3 < 4 → NO overlap
→ But [1,3] merges with [2,5] → [1,5] → and [1,5] overlaps with [4,8] → [1,8]
→ The intermediate merge CREATED an overlap that didn't exist before
→ This is why we must use the MERGED interval for subsequent comparisons

Time and Space Complexity

TIME:
  → Sorting: O(n log n)
  → Single scan: O(n)
  → Total: O(n log n) — dominated by sorting

SPACE:
  → Sorting: O(log n) for Java's sort implementation
  → Result list: O(n) in worst case (no overlaps → all intervals preserved)
  → Total: O(n)

BEST CASE: all intervals overlap → result has 1 element → O(1) output space
WORST CASE: no intervals overlap → result has n elements → O(n) output space

⚠️ COMMON PITFALLS AND FAILURE MODES

Pitfall 1: Forgetting to Sort

Input: [[3,5], [1,4], [6,8]]

WITHOUT SORTING:
  current = [3,5]
  [1,4]: 5 ≥ 1 → overlap → merge → current = [3, max(5,4)] = [3,5]
  → BUT the merged interval should be [1,5] not [3,5]
  → We lost the start of [1,4] because it came AFTER [3,5]
  → current.start was 3, should have been 1
  
  → WRONG ANSWER

WITH SORTING:
  Sorted: [[1,4], [3,5], [6,8]]
  current = [1,4]
  [3,5]: 4 ≥ 3 → overlap → current = [1, max(4,5)] = [1,5]
  [6,8]: 5 < 6 → no overlap → save [1,5], current = [6,8]
  
  → CORRECT: [[1,5], [6,8]]

Pitfall 2: Using a - b Instead of Integer.compare for Sorting

WRONG:
  Arrays.sort(intervals, (a, b) -> a[0] - b[0]);
  
  If a[0] = Integer.MIN_VALUE and b[0] = 1:
    → a[0] - b[0] = -2147483648 - 1 = 2147483647 (overflow!)
    → Returns POSITIVE → sort thinks a > b → WRONG ORDER

CORRECT:
  Arrays.sort(intervals, (a, b) -> Integer.compare(a[0], b[0]));
  
  → Integer.compare handles all values safely
  → No overflow possible
  → Always use Integer.compare for production code

PRACTICALLY:
  → Most LeetCode test cases don't have extreme values
  → But in real-world code, a - b is a bug waiting to happen
  → Build the habit of using Integer.compare

Pitfall 3: Not Adding the Last Interval

WRONG APPROACH (without reference trick):
  for (int i = 1; i < intervals.length; i++) {
      if (overlap) {
          extend current
      } else {
          result.add(current)   // saves the OLD current
          current = intervals[i] // starts new current
      }
  }
  // MISSING: result.add(current) for the last interval!
  
  → The last interval (or last merged group) is NEVER added to result
  → Result is always missing one interval → WRONG

CORRECT:
  for (int i = 1; i < intervals.length; i++) {
      if (overlap) {
          extend current
      } else {
          result.add(current)
          current = intervals[i]
      }
  }
  result.add(current);  // ← CRITICAL: add the last one

OR USE THE REFERENCE TRICK:
  → Add current to result BEFORE the loop
  → Modify current in-place → result auto-updates
  → No need for post-loop addition

Pitfall 4: Treating Touching Intervals as Non-Overlapping

[1, 5] and [5, 8]
→ Do they overlap?

PROBLEM DEPENDENT:
  → Most problems: YES → 5 ≥ 5 → they touch → merge to [1,8]
  → Some problems: NO → touching is not overlapping
  → READ THE PROBLEM STATEMENT

Default assumption: touching = overlapping
→ current.end >= next.start (≥ includes equality → touching merges)

If touching should NOT merge:
→ current.end > next.start (strict > → touching stays separate)

→ One character difference (>= vs >) → completely different behavior
→ Be precise about the overlap condition

Pitfall 5: Modifying Input Array

→ Arrays.sort(intervals, ...) sorts the input IN-PLACE
→ The original order is LOST
→ If the caller needs the original order → they get a surprise

SAFE APPROACH:
  int[][] copy = intervals.clone(); // shallow clone
  // For deep clone: copy each sub-array too
  for (int i = 0; i < copy.length; i++) {
      copy[i] = intervals[i].clone();
  }
  Arrays.sort(copy, ...);
  // Process copy, return result

→ In LeetCode: input modification is usually fine
→ In production: always consider side effects

📈 SCALABILITY CONSIDERATIONS

ARRAY SIZE SCALING:
→ n = 100       → sort: ~700 ops, scan: ~100 ops → instant
→ n = 10,000    → sort: ~130K ops, scan: ~10K ops → instant
→ n = 1,000,000 → sort: ~20M ops, scan: ~1M ops → ~50ms
→ n = 10,000,000 → sort: ~230M ops → ~1-2 seconds

BOTTLENECK: Sorting (O(n log n))
→ The scan is O(n) → always fast
→ Sorting dominates for large n
→ If intervals are ALREADY SORTED → skip sort → O(n) total

COMPARISON WITH BRUTE FORCE:
  → Compare all pairs: O(n²)
  → n = 1,000,000 → 10^12 operations → HOURS
  → Sort + scan: 20M operations → milliseconds
  → Sort + scan is DRAMATICALLY better

MEMORY:
  → Result list: O(n) worst case
  → Sorting: O(log n) for stack
  → No heavy memory usage
  → Scales well for large inputs

🔍 INTERPRETABILITY CONSIDERATIONS

VISUAL INTUITION:
  → Draw intervals as horizontal bars on a number line
  → Overlapping bars visually merge into one longer bar
  → Gaps between bars are where separate intervals exist
  → Sorting = arranging bars from left to right
  → Scanning = sweeping left-to-right, merging as you go

DEBUGGING:
  → Print: each interval comparison (current vs next)
  → Print: whether overlap detected or not
  → Print: the merged interval or the new current
  → Easy to trace because it's a simple left-to-right scan

INTERVIEW COMMUNICATION:
  → "I sort intervals by start time"
  → "I scan left to right, comparing each interval with the current merged one"
  → "If they overlap, I extend the current by taking the max of ends"
  → "If they don't overlap, I save the current and start a new one"
  → "Time: O(n log n) for sorting, O(n) for scanning"

🧠 KEY TAKEAWAY

WHAT WE LEARNED:
→ Interval problems follow the SORT-THEN-SCAN paradigm
→ Sort by start time → scan left-to-right → compare adjacent pairs
→ ONE condition detects ALL overlap types: current.end >= next.start
→ ONE operation handles ALL merges: max(current.end, next.end)
→ Sorting transforms O(n²) all-pairs comparison into O(n) adjacent scan

THE SORT-THEN-SCAN PARADIGM:
  Step 1: SORT → impose order → enable directional scanning
  Step 2: SCAN → process left-to-right → make local decisions
  Step 3: MERGE → combine overlapping elements → build result

WHERE WE'VE SEEN SORT-FIRST BEFORE:
  Project 1:  Array was PRE-SORTED → two pointers worked directly
  Project 11: We SORTED the array → then used two pointers for 3Sum
  Project 14: We SORT intervals → then scan for overlaps
  
→ Sorting is a RECURRING PREREQUISITE for many patterns
→ The O(n log n) cost is almost always worth the simplification

THE INTERVAL PROCESSING FAMILY:

→ All built on the same sort-then-scan foundation
