PROJECT 40: Merge K Sorted Lists

🎯 K-way Merge → Min-Heap of List Heads → Multi-Source Merging

🧩 THE HYPOTHETICAL PROBLEM

→ You are given K sorted linked lists
→ Merge ALL of them into ONE sorted linked list
→ Return the head of the merged list

Example 1:
  Input:  lists = [
    1 → 4 → 5,
    1 → 3 → 4,
    2 → 6
  ]
  Output: 1 → 1 → 2 → 3 → 4 → 4 → 5 → 6

Example 2:
  Input:  lists = [
    -2 → 1 → 4,
    -3 → 2 → 5,
    -1 → 0 → 3
  ]
  Output: -3 → -2 → -1 → 0 → 1 → 2 → 3 → 4 → 5

Example 3:
  Input:  lists = []
  Output: null

Example 4:
  Input:  lists = [null, null, 1 → 2]
  Output: 1 → 2

🔍 HOW TO APPROACH IT

Step 1 — Understand the Shape of This Problem

→ We have K SEPARATE streams of sorted data
→ Each stream (linked list) is individually sorted
→ We need to INTERLEAVE them into one globally sorted stream
→ At any moment, the NEXT element in the merged result
     must be the SMALLEST among all current heads

Visualization (3 lists):
  List 0:  1 → 4 → 5
            ↑
  List 1:  1 → 3 → 4
            ↑
  List 2:  2 → 6
            ↑
  
  Current heads: [1, 1, 2]
  Smallest head = 1 (from List 0 or List 1)
  Pick it → advance that list's pointer → repeat

THIS IS A SELECTION PROBLEM:
  → At each step: "Which of K candidates is smallest?"
  → Pick it → replace it with its successor → repeat
  → Continue until ALL lists are exhausted

Step 2 — Think About the Brute Force Approaches

APPROACH A: Collect all values → sort → build new list
  → Traverse all K lists → collect N total values
  → Sort all N values → O(N log N)
  → Build new linked list from sorted values
  → Time: O(N log N)   Space: O(N)
  → WORKS but ignores that lists are ALREADY SORTED
  → We're throwing away sorted information

APPROACH B: Merge two lists at a time (sequential)
  → Merge list 0 and list 1 → result₁
  → Merge result₁ and list 2 → result₂
  → Merge result₂ and list 3 → result₃
  → ... repeat K-1 times
  → Each merge of two sorted lists is O(n₁ + n₂)
  → Total work: roughly O(N * K) in worst case
     → First merge: N/K + N/K = 2N/K elements
     → Second merge: 2N/K + N/K = 3N/K elements
     → ... grows each time
     → Sum ≈ O(N * K)
  → When K is large → very slow

APPROACH C: Merge two at a time (divide and conquer)
  → Pair up lists: merge (0,1), merge (2,3), merge (4,5)...
  → Now K/2 lists → pair again → K/4 lists → ...
  → log K rounds of merging
  → Each round processes all N elements once
  → Time: O(N log K)   Space: O(1) extra (or O(log K) recursion stack)
  → GOOD approach — but complex to implement

APPROACH D: Min-heap of K heads ← THE PATTERN
  → Always know the smallest of K current candidates in O(log K)
  → Pick it → advance → insert successor → repeat
  → Time: O(N log K)   Space: O(K)
  → Same time as divide-and-conquer but SIMPLER to implement
  → Uses the heap in a fundamentally NEW way vs Projects 38-39

Step 3 — The Fundamental Shift from Projects 38-39

PROJECTS 38-39: Heap as FILTER
  → Process n elements → keep only K in the heap
  → Heap size stays bounded at K
  → Elements ENTER and get REJECTED or REPLACE existing ones
  → At the end: heap contains the answer

PROJECT 40: Heap as SELECTOR
  → Heap always has at most K elements (one per list)
  → At each step: EXTRACT the minimum
  → Then INSERT the next element from that same list
  → Heap doesn't accumulate — it ROTATES through elements
  → The answer is built OUTSIDE the heap (in the merged list)

KEY DIFFERENCES:
  ┌──────────────────┬────────────────────┬────────────────────┐
  │ Aspect           │ Filter (38-39)     │ Selector (40)      │
  ├──────────────────┼────────────────────┼────────────────────┤
  │ Purpose          │ Keep top K         │ Pick next smallest │
  │ Heap operation   │ Offer + maybe poll │ Poll + maybe offer │
  │ Answer location  │ INSIDE the heap    │ OUTSIDE (built)    │
  │ Element flow     │ In → stay or eject │ In → out → replace │
  │ Heap size over   │ Grows to K, stays  │ Starts at K, may   │
  │ time             │                    │ shrink as lists end │
  │ When done        │ Read heap contents │ Heap is empty      │
  └──────────────────┴────────────────────┴────────────────────┘

Step 4 — Visualize the Full Process

Lists:
  L0: 1 → 4 → 5
  L1: 1 → 3 → 4
  L2: 2 → 6

INITIALIZE: Add head of each list to min-heap
  Heap: [1(L0), 1(L1), 2(L2)]    (min-heap → smallest at root)
  Merged: (empty)

STEP 1: Poll min = 1(L0)
  → Append 1 to merged list
  → L0's next = 4 → offer 4(L0)
  → Heap: [1(L1), 4(L0), 2(L2)]
  → Merged: 1

STEP 2: Poll min = 1(L1)
  → Append 1 to merged list
  → L1's next = 3 → offer 3(L1)
  → Heap: [2(L2), 4(L0), 3(L1)]
  → Merged: 1 → 1

STEP 3: Poll min = 2(L2)
  → Append 2 to merged list
  → L2's next = 6 → offer 6(L2)
  → Heap: [3(L1), 4(L0), 6(L2)]
  → Merged: 1 → 1 → 2

STEP 4: Poll min = 3(L1)
  → Append 3 to merged list
  → L1's next = 4 → offer 4(L1)
  → Heap: [4(L0), 4(L1), 6(L2)]
  → Merged: 1 → 1 → 2 → 3

STEP 5: Poll min = 4(L0)
  → Append 4 to merged list
  → L0's next = 5 → offer 5(L0)
  → Heap: [4(L1), 5(L0), 6(L2)]
  → Merged: 1 → 1 → 2 → 3 → 4

STEP 6: Poll min = 4(L1)
  → Append 4 to merged list
  → L1's next = null → DON'T offer (list exhausted)
  → Heap: [5(L0), 6(L2)]     ← heap SHRINKS
  → Merged: 1 → 1 → 2 → 3 → 4 → 4

STEP 7: Poll min = 5(L0)
  → Append 5 to merged list
  → L0's next = null → DON'T offer
  → Heap: [6(L2)]
  → Merged: 1 → 1 → 2 → 3 → 4 → 4 → 5

STEP 8: Poll min = 6(L2)
  → Append 6 to merged list
  → L2's next = null → DON'T offer
  → Heap: []     ← EMPTY → done!
  → Merged: 1 → 1 → 2 → 3 → 4 → 4 → 5 → 6  ✓

Step 5 — The Dummy Head Technique for Linked List Building

When building a linked list from scratch:
→ Problem: the "first node" needs special handling
→ Who does head point to? First extracted element?
→ Every append needs "if head == null" check

SOLUTION: Dummy head node
  → Create a fake node: dummy = new ListNode(-1)
  → tail = dummy
  → For each extracted element:
       tail.next = extractedNode
       tail = tail.next
  → At the end: return dummy.next (skip the fake node)

This is the standard technique from Project 16 (linked list reversal)
→ Avoids special-casing the first node
→ tail always points to the LAST node of the built list
→ Clean, uniform append logic throughout

🛠️ HOW CAN WE SOLVE IT

The Algorithm

1. Create a min-heap of ListNode, ordered by node.val
2. Add the HEAD of each non-null list to the heap
3. Create dummy head node → tail = dummy
4. WHILE heap is not empty:
     a. Poll the minimum node from heap
     b. Append it to tail: tail.next = polledNode; tail = tail.next
     c. If polledNode.next != null:
          → offer polledNode.next to heap (successor from same list)
5. tail.next = null (clean termination)
6. Return dummy.next

✅ WHAT SHOULD YOU DO

→ ALWAYS skip null lists during initialization
     → Some lists may be empty → don't add null to heap
     → Null in heap causes NullPointerException on comparison

→ ALWAYS use a dummy head for building the result list
     → Eliminates edge case for first node
     → Clean uniform code (from Project 16)

→ ALWAYS check polledNode.next before offering to heap
     → If next is null → list exhausted → don't add null
     → Heap naturally shrinks as lists are consumed

→ ALWAYS set tail.next = null at the end
     → Extracted nodes may still have old .next pointers
     → tail.next = null ensures clean termination
     → Without this → merged list may have dangling references

→ ALWAYS use node.val for heap comparison, NOT node reference
     → ListNode doesn't implement Comparable by default
     → Must provide explicit Comparator: (a, b) -> a.val - b.val
     → Or safer: (a, b) -> Integer.compare(a.val, b.val)

→ ALWAYS consider: what if K = 0 or all lists are null?
     → Heap starts empty → while loop never executes
     → Return dummy.next = null → correct!

🚫 WHAT SHOULD YOU NOT DO

→ DO NOT add null nodes to the heap
     → Causes NullPointerException when comparator accesses .val
     → Filter nulls during initialization AND after polling

→ DO NOT use subtraction in Comparator for ListNode
     → (a, b) -> a.val - b.val CAN overflow if vals are extreme
     → Use Integer.compare(a.val, b.val) instead

→ DO NOT create new ListNode objects for the merged list
     → We're REUSING the existing nodes from the input lists
     → Just re-wire the .next pointers
     → Creating new nodes wastes O(N) extra space unnecessarily

→ DO NOT forget to advance the tail pointer
     → tail.next = polledNode ← connects the node
     → tail = tail.next      ← advances tail
     → Missing the second line → tail stays in place → only last node attached

→ DO NOT try to merge all lists sequentially (one pair at a time)
     → O(N * K) time in worst case
     → Heap approach: O(N log K) → exponentially better for large K

→ DO NOT use a max-heap
     → We need the SMALLEST element at each step
     → Min-heap gives us that in O(1) peek, O(log K) poll
     → Max-heap would give us the LARGEST → wrong order

🔀 WHEN TO USE THIS PATTERN

✓ Multiple SORTED sources need to be merged into one sorted output
✓ Sources can be: linked lists, arrays, iterators, file streams
✓ The number of sources (K) is known and manageable
✓ You need elements in GLOBALLY sorted order
✓ Each source produces elements in sorted order independently

REAL-WORLD APPLICATIONS:
  → External merge sort (merge K sorted file chunks)
  → Database query: merge results from K sorted index scans
  → Log aggregation: merge K sorted log files by timestamp
  → Distributed systems: merge K sorted partitions
  → Calendar apps: merge K users' sorted events
  → Streaming: merge K sorted real-time data feeds

🚫 WHEN NOT TO USE THIS PATTERN

✗ Sources are NOT sorted
     → Heap-based merge REQUIRES each source to be sorted
     → Unsorted sources → collect all + sort → O(N log N)

✗ K = 2 (only two lists)
     → Simple two-pointer merge is sufficient → O(N)
     → Heap overhead unnecessary for just 2 sources
     → Standard merge from merge sort works perfectly

✗ K = 1 (single list)
     → Already sorted → return as-is

✗ You don't need sorted output
     → If order doesn't matter → just concatenate lists → O(N)
     → Heap adds unnecessary O(log K) per element

✗ N is very small regardless of K
     → Collect all → sort → O(N log N) is fast enough
     → Simpler code, fewer edge cases

✗ Sources are arrays and you can afford O(N) space
     → Merge all arrays into one → sort → simpler
     → Linked list merge is in-place (re-wire pointers)
     → Array merge needs extra space anyway

⚙️ HOW IT ONLY WORKS — Internal Mechanics

The Invariant — Why Heap Always Yields Global Minimum

CLAIM: At every step, the heap contains exactly one element from
       each NON-EXHAUSTED list — specifically, that list's current
       smallest unprocessed element.

WHY THIS GUARANTEES CORRECTNESS:
  → Each list is sorted → current head = smallest remaining in that list
  → Heap holds K heads → min of heap = smallest across ALL lists
  → Extracting this min = next element in global sorted order ✓
  → Replacing it with its successor maintains the invariant ✓

PROOF BY INDUCTION:
  Base: Initially, head of each list = smallest in that list ✓
        Heap min = global minimum ✓
  
  Step: After extracting min from list Lᵢ:
        → Lᵢ's new head = next smallest in Lᵢ (sorted property)
        → Offered to heap → heap now has new candidate from Lᵢ
        → All other lists' candidates unchanged
        → Heap min = new global minimum among remaining ✓
  
  Termination: Lists exhaust → heap shrinks → eventually empty → done ✓

Time Complexity — Precise Analysis

Let N = total number of elements across ALL K lists
Let K = number of lists

INITIALIZATION:
  → Add K heads to heap → K * O(log K) = O(K log K)

MAIN LOOP:
  → Runs exactly N times (once per element)
  → Each iteration:
       → poll: O(log K)     (heap has at most K elements)
       → offer: O(log K)    (if successor exists)
       → append: O(1)
  → Total: N * O(log K) = O(N log K)

OVERALL: O(K log K + N log K) = O(N log K)
  (since N ≥ K in any meaningful scenario)

COMPARISON:
  ┌─────────────────────────────┬──────────────┬──────────┐
  │ Approach                    │ Time         │ Space    │
  ├─────────────────────────────┼──────────────┼──────────┤
  │ Collect + Sort              │ O(N log N)   │ O(N)     │
  │ Sequential pairwise merge   │ O(N * K)     │ O(1)     │
  │ Divide-and-conquer merge    │ O(N log K)   │ O(log K) │
  │ Min-heap of K heads         │ O(N log K)   │ O(K)     │
  └─────────────────────────────┴──────────────┴──────────┘

When K = 1000, N = 10,000,000:
  → Collect+Sort: 10^7 * 23 ≈ 230M ops
  → Sequential:   10^7 * 1000 = 10^10 → WAY too slow
  → Heap:         10^7 * 10 ≈ 100M ops → best practical choice

Space Complexity

→ Heap: at most K elements → O(K)
→ Dummy node: O(1)
→ We REUSE existing ListNode objects → no new nodes created
→ Total: O(K)

NOTE: The merged list itself uses O(N) space — but these are
the SAME nodes from the input, just re-wired.
No ADDITIONAL space for the output.

Why Linked Lists Are Ideal for This Pattern

LINKED LIST ADVANTAGES:
  → Re-wiring .next pointers is O(1)
  → No shifting elements in memory
  → Natural "current position" via node pointer
  → Successor access via .next — perfect for "advance one step"

IF INPUTS WERE ARRAYS:
  → Would need index tracking per array (K indices)
  → Would need O(N) space for output array
  → Same algorithmic complexity but more bookkeeping
  → Linked lists make the code cleaner for this pattern

THE HEAP STORES NODES, NOT VALUES:
  → We offer ListNode objects to the heap
  → When polled, we get the node itself
  → We can access .next to find the successor
  → We can wire it into the result list directly
  → No separate index management needed

⚠️ COMMON PITFALLS AND FAILURE MODES

Pitfall 1: ListNode Comparable Issue

Java's PriorityQueue requires elements to be Comparable
OR a Comparator must be provided.

ListNode is NOT Comparable by default.

WRONG:
  PriorityQueue<ListNode> heap = new PriorityQueue<>();
  → Runtime error: ListNode cannot be cast to Comparable

CORRECT:
  PriorityQueue<ListNode> heap = new PriorityQueue<>(
      (a, b) -> Integer.compare(a.val, b.val)
  );
  → Explicit comparator provided → works ✓

Pitfall 2: Null Lists in Input Array

Input: lists = [1→2, null, 3→4, null]

WRONG (no null check):
  for (ListNode list : lists) {
      heap.offer(list);    // offers null → NPE on comparison
  }

CORRECT:
  for (ListNode list : lists) {
      if (list != null) {
          heap.offer(list);
      }
  }

Pitfall 3: Forgetting to Advance Successor

WRONG:
  while (!heap.isEmpty()) {
      ListNode min = heap.poll();
      tail.next = min;
      tail = tail.next;
      // FORGOT to offer min.next → only first elements are merged!
  }

CORRECT:
  while (!heap.isEmpty()) {
      ListNode min = heap.poll();
      tail.next = min;
      tail = tail.next;
      if (min.next != null) {
          heap.offer(min.next);   // ← CRITICAL: feed successor
      }
  }

Pitfall 4: Dangling .next on Last Node

After the loop, tail is the last node in the merged list.
But tail.next might still point to some old node from its original list.

MUST DO: tail.next = null;

Without this:
  → Merged list may have extra nodes hanging off the end
  → Or create cycles if original lists shared structure

Pitfall 5: Empty Input

Input: lists = [] (empty array) or lists = null

MUST HANDLE:
  if (lists == null || lists.length == 0) return null;

The heap-based approach handles this naturally:
  → No heads added to heap → heap empty → loop never runs
  → Return dummy.next = null ✓
  → But explicit check is cleaner and safer

Pitfall 6: Ties (Equal Values from Different Lists)

Lists: [1→3, 1→2, 1→4]
All three heads have value 1.

The heap breaks ties ARBITRARILY (based on internal ordering).
→ Any of the three 1's can be extracted first → all valid
→ The merged list will have three 1's at the start → correct
→ No special handling needed for ties

BUT: If you need STABLE merge (preserve original list order for ties):
  → Add list index as secondary sort key
  → Comparator: first by val, then by listIndex
  → Rarely required in LeetCode but important in real systems

📈 SCALABILITY CONSIDERATIONS

SCENARIO 1: Few long lists (K small, N large)
  → K = 3, N = 10,000,000
  → Heap: O(10^7 * log 3) ≈ O(10^7 * 1.6) ≈ 16M ops
  → Almost linear! Heap overhead is minimal when K is tiny
  → Practically equivalent to simple two-pointer merge

SCENARIO 2: Many short lists (K large, N moderate)
  → K = 100,000, each list has 10 elements, N = 1,000,000
  → Heap: O(10^6 * log 10^5) ≈ O(10^6 * 17) ≈ 17M ops
  → Initialization: adding 100K heads → O(100K * log 100K) ≈ 1.7M ops
  → Efficient even with massive K

SCENARIO 3: Streaming / External merge
  → K = 1000 sorted file chunks on disk
  → Each chunk = 1GB sorted data
  → Total = 1TB
  → Heap of 1000 elements fits in memory
  → Stream through each file → merge → write to output
  → THIS is the canonical real-world use case for K-way merge
  → Database external sort uses exactly this pattern

SCENARIO 4: K = 2 (degenerate case)
  → Heap of size 2 → each operation is O(log 2) = O(1)
  → Equivalent to simple two-pointer merge
  → Heap works but adds unnecessary constant overhead
  → Use direct two-pointer merge for K = 2

MEMORY PROFILE:
  → Heap: O(K) → stores one node per list
  → Merged list: O(N) → but reuses existing nodes
  → True extra memory: O(K) only
  → When K = 100, N = 10^8 → 100 pointers vs 10^8 nodes → negligible

🔍 INTERPRETABILITY CONSIDERATIONS

MENTAL MODEL — "Tournament Bracket"
  → K lists are K competitors
  → Each competitor presents their BEST (smallest) card
  → Heap = tournament ring → finds the OVERALL best card in O(log K)
  → Winner plays their card → draws their next card from their deck
  → Tournament continues until all decks are empty
  → Cards played in order = the merged sorted result

ALTERNATIVE MODEL — "K Checkout Lanes"
  → K lanes at a grocery store, each with customers sorted by ticket number
  → You (the merger) call the customer with the LOWEST ticket number
  → That lane's next customer steps forward
  → You repeat until all lanes are empty
  → The order you called them = merged sorted order

DEBUGGING:
  → Print heap contents after each poll+offer
  → Verify heap always has one entry per active list
  → Verify polled values are non-decreasing (sorted output)
  → If polled value decreases → comparator is wrong or null in heap

🧠 KEY TAKEAWAY

WHAT WE LEARNED:
→ K-way merge uses a min-heap as a SELECTOR, not a filter
→ Heap holds one candidate per source → always K entries max
→ Extract min → append to result → offer successor → repeat
→ O(N log K) time, O(K) space — optimal for K sorted sources
→ Works with linked lists, arrays, files, streams — any sorted source

→ Project 40: Heap PRODUCES the answer (elements flow THROUGH heap)
→ Filter vs Pipeline — same structure, different data flow

THE CORE MENTAL MODEL:
→ "I have K sorted sources and need one sorted output"
→ "Heap holds one candidate per source"
→ "Extract smallest → output it → refill from same source"
→ "Heap naturally shrinks as sources exhaust"
→ "When heap is empty → all sources consumed → done"

PATTERN SIGNATURE:
→ Multiple sorted sources ✓
→ Need globally sorted output ✓
→ Sources produce elements in order ✓
→ = K-WAY MERGE — MIN-HEAP OF CURRENT HEADS
