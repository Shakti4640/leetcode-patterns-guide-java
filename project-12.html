PROJECT 12: Longest Substring with K Distinct Characters

🎯 Sliding Window → HashMap Frequency Tracking → Window Validity Condition

🔗 DEPENDENCY CHAIN

Project 1  → Two Pointers → left/right convergence on sorted data
Project 5  → Sliding Window → fixed-size window → running sum maintenance
Project 6  → Sliding Window → variable-size window → shrink when condition violated
Project 11 → Three Sum → multi-pointer coordination → sorting + dedup
Project 12 → Sliding Window + HashMap → character frequency tracking
             → variable window with RICH STATE instead of simple sum
             → the window's validity is defined by a MAP condition
             → this unlocks the entire family of substring problems

🧩 THE HYPOTHETICAL PROBLEM

→ You are given a STRING of characters
→ You are given an integer K
→ Find the LENGTH of the LONGEST SUBSTRING that contains AT MOST K distinct characters
→ Return the length (not the substring itself)

Example 1:
  Input:  s = "araaci", k = 2
  Output: 4
  → "araa" has 2 distinct characters (a, r) → length 4
  → "raac" has 3 distinct characters (r, a, c) → INVALID (exceeds k=2)
  → "aaci" has 3 distinct characters (a, c, i) → INVALID
  → Longest valid: "araa" → 4

Example 2:
  Input:  s = "araaci", k = 1
  Output: 2
  → "aaa" doesn't exist contiguously
  → "aa" (positions 2,3) → 1 distinct character → length 2
  → Longest valid: "aa" → 2

Example 3:
  Input:  s = "cbbebi", k = 3
  Output: 5
  → "cbbeb" has 3 distinct characters (c, b, e) → length 5
  → "bbebi" has 3 distinct characters (b, e, i) → length 5
  → Longest valid: 5

Example 4:
  Input:  s = "aaaa", k = 2
  Output: 4
  → Entire string has 1 distinct character → 1 ≤ 2 → valid → length 4

🔍 HOW TO APPROACH IT

Step 1 — Why This Differs from Projects 5 and 6

PROJECT 5 (Fixed Window):
  → Window size was FIXED at K
  → Slide right by 1, slide left by 1
  → Track a simple SUM (one integer)
  → No decisions about when to shrink

PROJECT 6 (Variable Window — Sum ≥ Target):
  → Window size was VARIABLE
  → Expand right until condition met (sum ≥ target)
  → Shrink left to find minimum window
  → Track a simple SUM (one integer)
  → Shrink condition: sum ≥ target → simple numeric comparison

PROJECT 12 (Variable Window — K Distinct):
  → Window size is VARIABLE
  → Expand right continuously
  → Shrink left when condition VIOLATED (distinct > K)
  → Track CHARACTER FREQUENCIES (a HashMap, not a single number)
  → Shrink condition: map.size() > K → requires maintaining a MAP
  → This is the LEAP: from simple state to RICH state

Step 2 — Think About What "State" the Window Needs

The window needs to answer ONE question at all times:
  "How many DISTINCT characters are currently inside the window?"

To answer this, we need to know:
  → WHICH characters are in the window
  → HOW MANY of each character are in the window

WHY "how many"?
  → When we shrink the window (remove leftmost character):
    → If that character's count drops to 0 → it LEAVES the window
    → If that character's count is still > 0 → it STAYS in the window
    → We need the COUNT to know whether removing a character changes distinctness

DATA STRUCTURE:
  → HashMap<Character, Integer>
  → Key: the character
  → Value: its frequency (count) within the current window
  → map.size() = number of distinct characters in the window

Step 3 — The Sliding Window Mechanics

Two pointers: windowStart (left) and windowEnd (right)
Both start at 0 and move FORWARD (same direction)

EXPAND phase (always happens):
  → Move windowEnd to the right by 1
  → Add the new character to the frequency map
  → map.put(c, map.getOrDefault(c, 0) + 1)

CHECK phase (after every expansion):
  → Is map.size() > K?
  → If YES → window is INVALID → must SHRINK

SHRINK phase (only when invalid):
  → Remove the character at windowStart from the map
  → Decrease its count: map.put(leftChar, map.get(leftChar) - 1)
  → If count becomes 0 → remove the key entirely: map.remove(leftChar)
  → Move windowStart forward by 1
  → REPEAT shrinking until map.size() ≤ K

UPDATE phase (after window is valid):
  → maxLength = Math.max(maxLength, windowEnd - windowStart + 1)

VISUAL:
  s = "araaci", k = 2

  Step 1: windowEnd=0 → 'a' → map={a:1} → size=1 ≤ 2 ✓ → maxLen=1
           [a]raaci

  Step 2: windowEnd=1 → 'r' → map={a:1,r:1} → size=2 ≤ 2 ✓ → maxLen=2
           [ar]aaci

  Step 3: windowEnd=2 → 'a' → map={a:2,r:1} → size=2 ≤ 2 ✓ → maxLen=3
           [ara]aci

  Step 4: windowEnd=3 → 'a' → map={a:3,r:1} → size=2 ≤ 2 ✓ → maxLen=4
           [araa]ci

  Step 5: windowEnd=4 → 'c' → map={a:3,r:1,c:1} → size=3 > 2 ✗ → SHRINK!
           [araac]i
           
           Shrink: remove 'a' at windowStart=0 → map={a:2,r:1,c:1} → size=3 > 2 → continue
           [raac]i  windowStart=1
           
           Shrink: remove 'r' at windowStart=1 → map={a:2,c:1} → size=2 ≤ 2 ✓ → stop
           [aac]i  windowStart=2
           
           maxLen = max(4, 4-2+1) = max(4, 3) = 4

  Step 6: windowEnd=5 → 'i' → map={a:2,c:1,i:1} → size=3 > 2 ✗ → SHRINK!
           [aaci]
           
           Shrink: remove 'a' at windowStart=2 → map={a:1,c:1,i:1} → size=3 > 2 → continue
           [aci]  windowStart=3
           
           Shrink: remove 'a' at windowStart=3 → map={c:1,i:1} → size=2 ≤ 2 ✓ → stop
           [ci]  windowStart=4
           
           maxLen = max(4, 5-4+1) = max(4, 2) = 4

  FINAL: maxLen = 4 ✓

Step 4 — Why the Shrink Must Remove Until Valid

When map.size() > K:
  → The window has too many distinct characters
  → We MUST remove characters from the LEFT until distinct count drops to K
  → Removing one character from the left might not be enough
  → That character might appear multiple times → count decreases but doesn't reach 0
  → So distinct count doesn't change → must keep shrinking

Example:
  Window: "araac" → map={a:3, r:1, c:1} → 3 distinct > k=2
  
  Remove 'a' (windowStart=0):
    → map={a:2, r:1, c:1} → still 3 distinct → keep shrinking
  
  Remove 'r' (windowStart=1):
    → map={a:2, c:1} → count of 'r' became 0 → removed from map
    → now 2 distinct → VALID → stop shrinking

→ The shrink is a WHILE loop, not an IF statement
→ It runs until the window becomes valid again

Step 5 — Why windowEnd - windowStart + 1

Window covers indices: [windowStart, windowEnd] inclusive

Length = windowEnd - windowStart + 1

Example:
  windowStart = 2, windowEnd = 5
  Indices in window: 2, 3, 4, 5 → 4 elements
  Length: 5 - 2 + 1 = 4 ✓

The +1 accounts for both endpoints being INCLUSIVE
This is a universal formula for any contiguous range

🛠️ HOW CAN WE SOLVE IT

The Algorithm — Complete Logic

1. Initialize:
     → HashMap<Character, Integer> charFrequency = new HashMap<>()
     → int windowStart = 0
     → int maxLength = 0

2. FOR windowEnd = 0 to s.length() - 1:

     a. EXPAND: Add character at windowEnd to the map
        → char rightChar = s.charAt(windowEnd)
        → charFrequency.put(rightChar, charFrequency.getOrDefault(rightChar, 0) + 1)

     b. SHRINK: While map has more than K distinct characters
        → char leftChar = s.charAt(windowStart)
        → charFrequency.put(leftChar, charFrequency.get(leftChar) - 1)
        → if charFrequency.get(leftChar) == 0 → charFrequency.remove(leftChar)
        → windowStart++

     c. UPDATE: Record maximum valid window length
        → maxLength = Math.max(maxLength, windowEnd - windowStart + 1)

3. RETURN maxLength

✅ WHAT SHOULD YOU DO

→ ALWAYS remove map entries when their count reaches 0
     → map.size() represents DISTINCT characters in the window
     → If a character's count is 0 but key still exists → map.size() is WRONG
     → map.remove(key) when count hits 0 → keeps size accurate

→ ALWAYS use getOrDefault for safe frequency increments
     → charFrequency.getOrDefault(c, 0) + 1
     → Handles first occurrence (key doesn't exist yet) gracefully
     → Avoids NullPointerException

→ ALWAYS update maxLength AFTER the shrink loop
     → The window might have just been shrunk → need current valid size
     → Updating before shrink would record invalid window sizes

→ ALWAYS think of the window as [windowStart, windowEnd] inclusive
     → Both pointers point AT characters that are IN the window
     → Length = windowEnd - windowStart + 1

→ ALWAYS verify edge cases: k=0, empty string, k ≥ distinct chars in string
     → k=0: no characters allowed → return 0
     → Empty string: no characters → return 0
     → k ≥ total distinct: entire string is valid → return s.length()

🚫 WHAT SHOULD YOU NOT DO

→ DO NOT use a HashSet instead of HashMap
     → HashSet tracks WHICH characters exist → gives distinct count
     → But when you remove a character from the window:
       → You can't know if other copies still exist
       → "Remove 'a' from set" → but 'a' appears 3 more times in window
       → Set incorrectly removes 'a' → distinct count drops incorrectly
     → HashMap tracks COUNT per character → knows when count truly reaches 0
     → HashSet = existence only → HashMap = existence + quantity

→ DO NOT forget to remove zero-count entries from the map
     → map = {a:2, r:0, c:1} → map.size() = 3
     → But 'r' is NOT in the window (count=0)
     → True distinct count is 2, not 3
     → The shrink logic checks map.size() > K
     → Wrong size → wrong shrink decisions → wrong answer

→ DO NOT shrink with IF instead of WHILE
     WRONG:
       if (map.size() > k) { shrink once }
     
     → Removing one character might not reduce distinct count
     → Window stays invalid → maxLength records invalid size → WRONG

     CORRECT:
       while (map.size() > k) { shrink }
     
     → Keeps shrinking until valid → guarantees correctness

→ DO NOT reset windowStart to 0 or to windowEnd
     → windowStart only moves FORWARD → never backward
     → Resetting destroys the sliding window invariant
     → Each character is added at most once and removed at most once
     → This is what gives O(n) time complexity

→ DO NOT use substring extraction for comparison
     → s.substring(windowStart, windowEnd+1) creates a new String
     → Doing this every iteration → O(n) per call → O(n²) total
     → Use the HashMap to track state → O(1) per operation

→ DO NOT confuse this with fixed-size window (Project 5)
     → Project 5: window size was always K → slide by 1
     → Here: window size VARIES → expand freely, shrink as needed
     → The window can grow to cover the entire string if all valid
     → Or shrink to size 1 if every character is unique and K=1

🔀 WHEN TO USE THIS PATTERN

✓ Problem asks for "longest/shortest substring with condition on character diversity"
     → "at most K distinct" → this exact problem
     → "exactly K distinct" → variation (covered below)
     → "at most 2 types of fruits" → K=2 (Project 13 connection)

✓ Problem involves a CONTIGUOUS segment of a string or array
     → Sliding window = contiguous subarray/substring
     → If elements don't need to be contiguous → different technique (backtracking)

✓ The validity condition can be checked incrementally
     → Adding one element: update map → O(1)
     → Removing one element: update map → O(1)
     → Checking validity: map.size() → O(1)
     → All incremental → sliding window works

✓ You need the OPTIMAL (longest/shortest) valid window
     → For longest: expand greedily, shrink only when forced
     → For shortest: expand until valid, then shrink aggressively
     → Both follow the same expand/shrink skeleton

✓ O(n) time is required
     → Brute force checks all O(n²) substrings → too slow
     → Sliding window visits each character twice (once expand, once shrink) → O(n)

🚫 WHEN NOT TO USE THIS PATTERN

✗ Elements don't need to be contiguous
     → "Find K characters with maximum frequency" (no contiguity needed)
     → Use sorting or heap-based selection → not sliding window

✗ Validity condition is not incrementally maintainable
     → If adding/removing one element requires O(n) recomputation
     → Sliding window loses its O(n) advantage
     → Example: "substring where all characters appear in sorted order"
       → Adding a character might invalidate the ordering → need full re-check

✗ Problem asks for count of all valid substrings, not the optimal one
     → Sliding window finds the longest/shortest
     → Counting ALL valid substrings needs different math
     → Though sometimes sliding window CAN count (with clever tricks)

✗ The condition depends on elements OUTSIDE the window
     → Sliding window only sees what's INSIDE [windowStart, windowEnd]
     → If validity depends on global array properties → different approach

✗ Window needs to skip elements (non-contiguous selection)
     → Sliding window = contiguous range
     → Skipping breaks contiguity → need two pointers or backtracking instead

⚙️ HOW IT ONLY WORKS — Internal Mechanics

Mechanic 1: The Amortized O(n) Guarantee

CLAIM: The algorithm runs in O(n) time despite having a while loop inside a for loop.

PROOF:
  → windowEnd moves from 0 to n-1 → exactly n steps forward
  → windowStart moves from 0 to at most n-1 → at most n steps forward
  → windowStart NEVER moves backward
  
  TOTAL pointer movements: at most 2n
  Each movement does O(1) work (one map operation)
  TOTAL: O(2n) = O(n)

  This is AMORTIZED analysis:
  → Some iterations have no shrinking (windowStart stays)
  → Some iterations have heavy shrinking (windowStart jumps forward)
  → But across ALL iterations, windowStart moves at most n times total
  → So the AVERAGE cost per iteration is O(1)

ANALOGY:
  → Imagine paying rent monthly
  → Some months you pay $100, some months $500
  → But total yearly rent is capped at $2400
  → Average monthly cost is $200
  → The expensive months are "amortized" by the cheap months

Mechanic 2: Why HashMap Size Equals Distinct Count

INVARIANT: map.size() == number of distinct characters in window [windowStart, windowEnd]

MAINTAINED BY:
  → EXPAND: adding character c
    → If c is NEW: map didn't have c → put(c, 1) → size increases by 1 ✓
    → If c EXISTS: map had c → put(c, count+1) → size unchanged ✓

  → SHRINK: removing character c
    → If count > 1: put(c, count-1) → size unchanged ✓ (c still in window)
    → If count == 1: put(c, 0) then remove(c) → size decreases by 1 ✓
      (c no longer in window)

→ The remove-when-zero step is CRITICAL
→ Without it: size counts "ghost" characters that aren't in the window
→ The invariant breaks → algorithm produces wrong answers

Mechanic 3: Why We Maximize After Shrinking

EXPAND:
  → windowEnd++ → window grows → MIGHT become invalid
  
SHRINK:
  → while invalid: windowStart++ → window shrinks → becomes valid again
  
UPDATE:
  → maxLength = max(maxLength, currentWindowSize)
  → At this point, window is GUARANTEED valid
  → We record the size of the LARGEST valid window seen so far

IF WE UPDATED BEFORE SHRINKING:
  → Window might be invalid (too many distinct characters)
  → We'd record an invalid window size → WRONG answer

IF WE UPDATED DURING SHRINKING:
  → Window is getting SMALLER → never the maximum → wasted comparisons
  → Not wrong, but unnecessary

AFTER SHRINKING is the perfect time:
  → Window just became valid
  → Window is as LARGE as possible (we only shrunk the minimum needed)
  → This is potentially the new maximum

Time and Space Complexity

TIME: O(n)
  → Each character is added to the window exactly ONCE (windowEnd passes it)
  → Each character is removed from the window at most ONCE (windowStart passes it)
  → HashMap operations: O(1) average per put/get/remove
  → Total: O(n) + O(n) = O(n)

SPACE: O(K)
  → HashMap stores at most K+1 entries (K valid + 1 that triggered shrink)
  → After shrinking: at most K entries
  → If K is bounded by alphabet size (26 for lowercase) → O(1)
  → In general: O(min(K, alphabet_size))

⚠️ COMMON PITFALLS AND FAILURE MODES

Pitfall 1: Forgetting to Remove Zero-Count Entries

WRONG CODE:
  charFrequency.put(leftChar, charFrequency.get(leftChar) - 1);
  windowStart++;
  // MISSING: if count == 0, remove key

  Window: "ab" → map = {a:1, b:1} → size = 2
  Shrink: remove 'a' → map = {a:0, b:1} → size = 2
  
  → map.size() still says 2 → algorithm thinks 'a' is still in window
  → Won't stop shrinking when it should → OR won't shrink when it should
  → WRONG answers

CORRECT CODE:
  charFrequency.put(leftChar, charFrequency.get(leftChar) - 1);
  if (charFrequency.get(leftChar) == 0) {
      charFrequency.remove(leftChar);
  }
  windowStart++;
  
  → map = {b:1} → size = 1 → correct distinct count

Pitfall 2: Off-by-One in Window Length

WRONG: maxLength = Math.max(maxLength, windowEnd - windowStart)
  → Window [2, 5] → 5-2 = 3 → but actual length is 4 (indices 2,3,4,5)
  → Missing +1 → all lengths underreported by 1

CORRECT: maxLength = Math.max(maxLength, windowEnd - windowStart + 1)
  → Window [2, 5] → 5-2+1 = 4 ✓

Pitfall 3: K = 0 Edge Case

→ K = 0 means NO characters allowed in the window
→ Every single character makes the window invalid
→ Window shrinks to size 0 immediately
→ maxLength stays 0

BUT: some implementations crash:
  → map.size() > 0 → always true → infinite shrink loop if windowStart > windowEnd

SAFE GUARD:
  if (k == 0) return 0;
  → Handle at the start → avoid edge case bugs

Pitfall 4: Assuming Characters Are Only Lowercase Letters

→ Problem might have uppercase, digits, special characters, Unicode
→ HashMap handles ALL characters naturally (Character is the key)
→ But if you try to use int[26] array instead of HashMap:
  → Only works for lowercase a-z
  → Crashes or gives wrong answers for other characters

→ Use HashMap for generality
→ Use int[26] or int[128] ONLY when the character set is explicitly constrained

Pitfall 5: Confusing "At Most K" with "Exactly K"

"AT MOST K distinct": map.size() ≤ K → the problem we solved
"EXACTLY K distinct": map.size() == K → different problem

For "exactly K":
  → Cannot directly use the same algorithm
  → TRICK: longestWithExactlyK = longestWithAtMostK(K) - longestWithAtMostK(K-1)
  → This reuses our algorithm twice → elegant reduction

→ Don't mix these up → read the problem statement carefully

🧠 KEY TAKEAWAY

WHAT WE LEARNED:
→ Sliding Window + HashMap = the POWER COMBO for substring problems
→ The HashMap tracks RICH STATE (character frequencies) inside the window
→ map.size() gives distinct count → the window's validity metric
→ Remove zero-count entries → keeps size accurate → algorithm correct
→ Amortized O(n): each character added once, removed at most once

EVOLUTION OF SLIDING WINDOW:
  Project 5:  Fixed-size window → track SUM (single integer)
  Project 6:  Variable-size window → track SUM → shrink when sum ≥ target
  Project 12: Variable-size window → track FREQUENCIES (HashMap)
              → shrink when distinct > K → RICH state management

THE LEAP:
  → From tracking ONE number (sum) to tracking a MAP of frequencies
  → The expand/shrink skeleton is IDENTICAL to Project 6
  → Only the STATE and VALIDITY CONDITION changed
  → This is the power of patterns: same structure, different content

GENERAL SLIDING WINDOW TEMPLATE:
  1. Initialize state (sum, map, count, etc.)
  2. FOR each windowEnd:
     a. Update state with new element (expand)
     b. WHILE window is invalid: update state removing element (shrink)
     c. Record optimal window size (update answer)
