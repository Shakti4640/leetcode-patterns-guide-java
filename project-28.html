PROJECT 28: Generate All Subsets of a Set

🎯 Subsets/Backtracking → Inclusion/Exclusion Branching → Power Set Generation

🔗 DEPENDENCY CHAIN

Project 26 (All Root-to-Leaf Paths)
    → Introduced backtracking on a REAL tree
    → Add to path → explore → remove from path
    → Tree structure GUIDED which branches to explore

Project 27 (Serialize/Deserialize Binary Tree)
    → DFS traversal collecting data from an EXISTING structure
    → Reconstruction = building structure from flat data

PROJECT 28 (THIS):
    → No tree exists — WE create the branching decisions
    → For each element: INCLUDE it or EXCLUDE it
    → This creates an IMPLICIT binary decision tree
    → Backtracking explores ALL paths through this decision tree
    → FIRST TIME: Pure backtracking without a physical tree

🧩 THE HYPOTHETICAL PROBLEM

→ Given a set of DISTINCT integers
→ Generate ALL possible subsets (the power set)
→ A subset can include ANY combination of elements (including none and all)
→ Return all subsets as a list of lists

Example:
  Input:  [1, 2, 3]
  Output: [[], [1], [2], [3], [1,2], [1,3], [2,3], [1,2,3]]

  Total subsets = 2^n = 2^3 = 8

Why 2^n?
→ For each of the n elements, you make a binary choice: IN or OUT
→ n independent binary choices = 2 × 2 × 2 × ... × 2 = 2^n combinations

Another example:
  Input:  [5, 10]
  Output: [[], [5], [10], [5,10]]

  Element 5:  IN or OUT → 2 choices
  Element 10: IN or OUT → 2 choices
  Total: 2 × 2 = 4 subsets

🔍 HOW TO APPROACH IT

Step 1 — See the Invisible Decision Tree

This is the KEY conceptual leap from Projects 26/27 to Project 28.

In Project 26:
→ A binary tree EXISTED in memory
→ We traversed it, following left/right pointers
→ The tree told us where to go

In Project 28:
→ No tree exists in memory
→ But there IS a tree — a DECISION tree — in the LOGIC of the problem
→ Each level of this tree represents one element
→ At each level, we CHOOSE: include this element or skip it

For input [1, 2, 3], the decision tree looks like:

                          []
                   /              \
              [1]                   []
            /      \            /       \
        [1,2]      [1]      [2]         []
        /   \     /   \    /   \       /   \
   [1,2,3] [1,2] [1,3] [1] [2,3] [2] [3]  []

Level 0: decide about element 1 → include (left) or skip (right)
Level 1: decide about element 2 → include (left) or skip (right)
Level 2: decide about element 3 → include (left) or skip (right)
Level 3: LEAF → all decisions made → this IS a subset → collect it

Every LEAF of this decision tree is one valid subset.
There are 2^3 = 8 leaves → 8 subsets.

Step 2 — Map the Decision Tree to Backtracking Code

The decision tree above has this recursive structure:

At each level (index i):
  → CHOICE 1: Add nums[i] to current subset → recurse to level i+1
  → CHOICE 2: Don't add nums[i]             → recurse to level i+1
  → When i reaches nums.length → all decisions made → save the subset

This maps DIRECTLY to backtracking:

  backtrack(index, currentSubset):
      if index == nums.length:
          results.add(copy of currentSubset)   ← LEAF reached
          return
      
      // CHOICE 1: Include nums[index]
      currentSubset.add(nums[index])
      backtrack(index + 1, currentSubset)
      currentSubset.remove(last)               ← BACKTRACK
      
      // CHOICE 2: Exclude nums[index]
      backtrack(index + 1, currentSubset)

Notice the SAME backtracking pattern from Project 26:
  → Add before exploring
  → Remove after returning
  → This ensures clean state for the next choice

Step 3 — Alternative Approach: Cascading (Iterative)

There's a beautiful ITERATIVE way to build all subsets:

Start with: [[]]  (just the empty set)

For each new element, COPY all existing subsets and ADD the element to each copy:

Element 1:
  Existing: [[]]
  Add 1 to each: [[1]]
  Merge: [[], [1]]

Element 2:
  Existing: [[], [1]]
  Add 2 to each: [[2], [1,2]]
  Merge: [[], [1], [2], [1,2]]

Element 3:
  Existing: [[], [1], [2], [1,2]]
  Add 3 to each: [[3], [1,3], [2,3], [1,2,3]]
  Merge: [[], [1], [2], [1,2], [3], [1,3], [2,3], [1,2,3]]

Done! All 8 subsets generated.

This approach is elegant but doesn't teach BACKTRACKING.
We'll implement BOTH — backtracking is the primary technique to learn.

Step 4 — Yet Another Approach: Bit Manipulation

Since each element is either IN or OUT, we can use BITS:

For n=3 elements, we count from 0 to 2^3 - 1 = 7:

Binary  Decimal  Subset (bits map to elements [1,2,3])
000     0        []
001     1        [3]
010     2        [2]
011     3        [2,3]
100     4        [1]
101     5        [1,3]
110     6        [1,2]
111     7        [1,2,3]

Each bit position corresponds to an element:
→ Bit i is 1 → include nums[i]
→ Bit i is 0 → exclude nums[i]

This connects back to Project 10 (Bitwise XOR) — 
bits encode binary decisions naturally.
We'll implement this too for completeness.

🛠️ HOW CAN WE SOLVE IT

Approach A: Backtracking (Include/Exclude Decision Tree)

backtrack(nums, index, currentSubset, results):
    if index == nums.length:
        results.add(new ArrayList<>(currentSubset))   ← deep copy!
        return
    
    // Branch 1: INCLUDE nums[index]
    currentSubset.add(nums[index])
    backtrack(nums, index + 1, currentSubset, results)
    currentSubset.remove(currentSubset.size() - 1)    ← BACKTRACK
    
    // Branch 2: EXCLUDE nums[index]
    backtrack(nums, index + 1, currentSubset, results)

Approach B: Backtracking (Collect-At-Every-Node Variant)

There's a second backtracking style that collects subsets at EVERY node,
not just at leaves:

backtrack(nums, startIndex, currentSubset, results):
    results.add(new ArrayList<>(currentSubset))   ← collect HERE
    
    for i = startIndex to nums.length - 1:
        currentSubset.add(nums[i])
        backtrack(nums, i + 1, currentSubset, results)
        currentSubset.remove(currentSubset.size() - 1)   ← BACKTRACK

Why this works:
→ Each call represents a VALID subset (including partial ones)
→ The loop says "which element do I add NEXT?"
→ By starting from startIndex, we avoid creating duplicates
→ [1,2] and [2,1] are the same subset — starting from startIndex prevents this

Walkthrough for [1,2,3]:
  backtrack(start=0, current=[])
    SAVE []
    i=0: add 1 → backtrack(start=1, current=[1])
      SAVE [1]
      i=1: add 2 → backtrack(start=2, current=[1,2])
        SAVE [1,2]
        i=2: add 3 → backtrack(start=3, current=[1,2,3])
          SAVE [1,2,3]
          loop ends (start=3 == length)
        remove 3
      remove 2
      i=2: add 3 → backtrack(start=3, current=[1,3])
        SAVE [1,3]
      remove 3
    remove 1
    i=1: add 2 → backtrack(start=2, current=[2])
      SAVE [2]
      i=2: add 3 → backtrack(start=3, current=[2,3])
        SAVE [2,3]
      remove 3
    remove 2
    i=2: add 3 → backtrack(start=3, current=[3])
      SAVE [3]
    remove 3

Result: [[], [1], [1,2], [1,2,3], [1,3], [2], [2,3], [3]]
All 8 subsets ✓

Comparing the Two Backtracking Styles

STYLE A (Include/Exclude):
→ Binary decision at each level: yes or no for each element
→ Collects only at LEAVES (when all decisions are made)
→ Decision tree is a FULL binary tree of depth n
→ More intuitive for beginners
→ Exactly 2^n leaves = 2^n subsets

STYLE B (Collect-At-Every-Node):
→ For loop picks "which element to add next"
→ Collects at EVERY node (every partial subset is valid)
→ More common in interview solutions
→ More flexible — easier to adapt for combinations, permutations
→ This style is what Project 29 (Permutations) and Project 42 (Combination Sum) use

LEARN BOTH — but Style B is more versatile for interview patterns.

✅ WHAT SHOULD YOU DO

→ ALWAYS deep copy when saving to results
     → results.add(new ArrayList<>(currentSubset))
     → Same lesson from Project 26 — mutable list, snapshot needed

→ ALWAYS use startIndex to avoid duplicate subsets
     → [1,2] and [2,1] are the SAME subset
     → By only adding elements AFTER the current position, we avoid revisiting
     → startIndex ensures we only move FORWARD through the array

→ ALWAYS backtrack (remove last) after each recursive exploration
     → Same pattern as Project 26: add → recurse → remove

→ ALWAYS consider whether the input has duplicates
     → This problem says DISTINCT integers
     → If duplicates exist, need extra skip logic (covered separately)

→ ALWAYS count expected output size
     → 2^n subsets for n elements
     → n=20 → 2^20 = 1,048,576 subsets → still feasible
     → n=25 → 2^25 = 33,554,432 → borderline
     → n=30 → 2^30 = 1,073,741,824 → likely too many

→ ALWAYS understand which backtracking style the problem needs
     → Subsets: collect at every node (Style B) or leaves only (Style A)
     → Combinations: collect at specific depth (Style B with target)
     → Permutations: collect when all elements used (Style B with visited set)

🚫 WHAT SHOULD YOU NOT DO

→ DO NOT forget the deep copy
     BAD:  results.add(currentSubset);
     → All entries in results point to the SAME list
     → At the end, all entries are empty (after all backtracks)
     → Exact same bug from Project 26 — it's the #1 backtracking mistake

→ DO NOT allow backward element selection
     BAD (no startIndex):
       for (int i = 0; i < nums.length; i++) {
           currentSubset.add(nums[i]);
           backtrack(nums, i + 1, currentSubset, results);
           // Wait — this looks correct... but what if we used 'start' wrong?
       }
     
     ACTUALLY BAD:
       for (int i = 0; i < nums.length; i++) {  // always from 0
           // This generates [1,2] AND [2,1] → duplicates!
       }
     
     → Must use startIndex parameter and pass i+1 to recursive call
     → This ensures each element can only appear ONCE in each subset
     → And elements are always added in forward order → no duplicate subsets

→ DO NOT use a HashSet to deduplicate subsets
     → This works but is wasteful — O(2^n) memory for the set
     → Proper use of startIndex prevents duplicates structurally
     → Using a set is a band-aid for broken logic

→ DO NOT confuse subsets with permutations
     → Subset: ORDER doesn't matter → {1,2} == {2,1} → only generate once
     → Permutation: ORDER matters → [1,2] ≠ [2,1] → generate both
     → Subsets use startIndex to prevent reordering
     → Permutations use a visited/swap mechanism (Project 29)

→ DO NOT try to generate subsets of a very large set (n > 25)
     → 2^25 = 33 million subsets
     → Each subset up to 25 elements long
     → Total memory: hundreds of MB to GB
     → The problem is inherently exponential — no trick avoids this

🔀 WHEN TO USE THIS PATTERN

✓ Problem asks for "all subsets" or "power set"
✓ Problem asks for "all combinations" (subsets of specific size)
✓ Problem asks for "all ways to choose" from a set
✓ The output is exponential (2^n) and that's expected
✓ You need to explore ALL possible selections from a collection
✓ The selection order doesn't matter (if order matters → permutations)

RECOGNIZABLE KEYWORDS:
→ "generate all subsets"
→ "find all combinations"
→ "power set"
→ "all possible selections"
→ "choose k from n" (subsets of size k)

VARIANTS:
→ All subsets → collect at every node
→ Subsets of size k → collect only when currentSubset.size() == k
→ Subsets with target sum → collect when sum matches (Project 42)
→ Subsets with duplicates → sort + skip duplicate elements

🚫 WHEN NOT TO USE THIS PATTERN

✗ Problem asks for OPTIMAL subset (max/min value)
     → Don't enumerate all — use DP or greedy instead
     → Example: "subset with maximum sum" → just take all positive numbers
     → Example: "0/1 Knapsack" → DP (Project 44)

✗ Problem asks for COUNT of subsets (not the actual subsets)
     → Count is always 2^n → just compute pow(2, n)
     → Count of subsets with property X → often DP

✗ n is very large (> 20-25)
     → 2^n is too many to enumerate
     → Need mathematical formula or DP instead

✗ Order of elements matters
     → That's PERMUTATIONS, not subsets (Project 29)
     → Different backtracking structure

✗ Elements can be reused
     → That's "combinations with repetition" or "unbounded" variant
     → Different recursion: pass i instead of i+1

⚙️ HOW IT ONLY WORKS — Internal Mechanics

The Implicit Decision Tree

For nums = [1, 2, 3]:

STYLE A (Include/Exclude):

                           start: index=0, subset=[]
                          /                          \
                   include 1                      exclude 1
                  index=1, [1]                   index=1, []
                /           \                  /            \
          include 2      exclude 2       include 2       exclude 2
         index=2,[1,2]  index=2,[1]     index=2,[2]    index=2,[]
          /       \      /       \       /      \       /       \
       inc 3   exc 3  inc 3   exc 3  inc 3  exc 3  inc 3   exc 3
      [1,2,3]  [1,2]  [1,3]   [1]   [2,3]   [2]   [3]      []

8 leaves → 8 subsets ✓

STYLE B (For-loop / Collect-at-every-node):

                     backtrack(start=0, [])
                    /         |          \
              add 1          add 2        add 3
          bt(1,[1])       bt(2,[2])     bt(3,[3])
          /       \          |
       add 2    add 3      add 3
    bt(2,[1,2]) bt(3,[1,3]) bt(3,[2,3])
        |
     add 3
  bt(3,[1,2,3])

Collect at EVERY node:
  [] → [1] → [1,2] → [1,2,3] → [1,3] → [2] → [2,3] → [3]
  = 8 subsets ✓

Why startIndex Prevents Duplicates

Without startIndex:
  From [], we could add 1 → [1], then from [1], add 2 → [1,2]
  From [], we could add 2 → [2], then from [2], add 1 → [2,1]
  [1,2] and [2,1] are the SAME subset → duplicate!

With startIndex:
  From [] with start=0: can add elements at index 0,1,2
  After adding element at index 0 (value 1): start becomes 1
    From [1] with start=1: can add elements at index 1,2 (NOT 0)
    → Can never add 1 again → can never go backward
  After adding element at index 1 (value 2): start becomes 2
    From [2] with start=2: can add elements at index 2 only
    → Can never add 1 → [2,1] is IMPOSSIBLE

startIndex creates a ONE-WAY GATE:
→ Once you pass an element, you can never return to it
→ This ensures elements are always added in forward order
→ Forward order = canonical order = no duplicates

Backtracking State Invariant

INVARIANT: When backtrack(start, currentSubset) is called:
→ currentSubset contains exactly the elements chosen BEFORE this call
→ All elements in currentSubset have indices < start
→ The remaining choices are elements at indices >= start

PROOF:
→ Initial call: backtrack(0, []) → empty subset, all choices available ✓
→ When we add nums[i] and recurse with start=i+1:
     → currentSubset now has nums[i] added
     → start=i+1 means only indices > i are available
     → INVARIANT holds for recursive call ✓
→ When we backtrack (remove last):
     → currentSubset returns to state before adding nums[i]
     → Loop continues with i+1
     → INVARIANT holds for next iteration ✓

Time Complexity

→ Total subsets: 2^n
→ Each subset requires O(n) work to copy into results
→ Total: O(n × 2^n)

This is OPTIMAL — you cannot generate 2^n subsets faster than O(2^n)
And each subset has average length n/2, so copying costs O(n) per subset
Total: O(n × 2^n) time and O(n × 2^n) space for the output

Space Complexity

→ Output: O(n × 2^n) — unavoidable, this IS the answer
→ Recursion depth: O(n) — one level per element
→ currentSubset: O(n) — at most n elements
→ Total auxiliary (excluding output): O(n)
→ With output: O(n × 2^n)

⚠️ COMMON PITFALLS AND FAILURE MODES

Pitfall 1: The Deep Copy Bug (Again!)

// ❌ BROKEN
results.add(currentSubset);

// Run with [1,2]:
// Step 1: currentSubset = [] → results = [[]]  (reference to currentSubset)
// Step 2: currentSubset = [1] → results = [[1], [1]]  (BOTH point to same list!)
// Step 3: currentSubset = [1,2] → results = [[1,2], [1,2], [1,2]]
// Step 4: After all backtracks: currentSubset = [] → results = [[], [], [], []]
// ALL entries show [] because they all point to the SAME mutable list

// ✅ CORRECT
results.add(new ArrayList<>(currentSubset));
// Each add creates an INDEPENDENT snapshot

Pitfall 2: Off-by-One in Loop Bound

// ❌ WRONG — misses last element
for (int i = startIndex; i < nums.length - 1; i++) {
    // When i = nums.length - 2, we process second-to-last element
    // We NEVER process the last element
    // Missing subsets that include only the last element
}

// ✅ CORRECT
for (int i = startIndex; i < nums.length; i++) {
    // Processes ALL elements from startIndex to the end
}

Pitfall 3: Passing Wrong startIndex in Recursion

// ❌ WRONG — allows reuse of same element
for (int i = startIndex; i < nums.length; i++) {
    currentSubset.add(nums[i]);
    backtrack(nums, startIndex, currentSubset, results);  // should be i+1!
    currentSubset.remove(currentSubset.size() - 1);
}
// This passes startIndex (unchanged) → can add nums[i] again
// Results in subsets like [1,1], [1,1,1], etc.

// ❌ ALSO WRONG — skips elements
for (int i = startIndex; i < nums.length; i++) {
    currentSubset.add(nums[i]);
    backtrack(nums, startIndex + 1, currentSubset, results);  // should be i+1!
    currentSubset.remove(currentSubset.size() - 1);
}
// When i=2 and startIndex=0, this recurses with start=1
// But we already processed elements 0 and 1 → duplicates possible

// ✅ CORRECT — move past the element we just added
for (int i = startIndex; i < nums.length; i++) {
    currentSubset.add(nums[i]);
    backtrack(nums, i + 1, currentSubset, results);  // i+1, not startIndex+1
    currentSubset.remove(currentSubset.size() - 1);
}

Pitfall 4: Confusing Subsets with Combinations of Fixed Size

// SUBSETS: all sizes from 0 to n
// Collect at EVERY node

// COMBINATIONS of size k: only size k
// Collect only when currentSubset.size() == k

// If problem says "all subsets" → no size filter
// If problem says "all combinations of size 3" → add size check
// If problem says "combination sum = target" → add sum check (Project 42)

Pitfall 5: Handling Duplicate Elements (Preview)

If input has duplicates like [1, 2, 2]:
→ Naive approach generates [1,2] twice (using first 2, then second 2)
→ FIX: Sort the array, then skip consecutive duplicates in the loop

for (int i = startIndex; i < nums.length; i++) {
    if (i > startIndex && nums[i] == nums[i-1]) continue;  // skip duplicate
    currentSubset.add(nums[i]);
    backtrack(nums, i + 1, currentSubset, results);
    currentSubset.remove(currentSubset.size() - 1);
}

This problem says DISTINCT elements, so we don't need this yet.
But knowing the pattern prepares you for the duplicate variant.

🧠 KEY TAKEAWAY

WHAT PROJECT 26/27 GAVE US:
→ Backtracking on PHYSICAL trees (add → recurse → remove)
→ The tree structure dictated our recursion (left child, right child)
→ We FOLLOWED the tree

WHAT PROJECT 28 ADDS:
→ Backtracking on IMPLICIT decision trees
→ No physical tree exists — WE create the branching
→ Each element = one level of decisions
→ At each level: "include this element?" → two branches
→ The for-loop with startIndex creates the branching structure
→ Backtracking (add → recurse → remove) is IDENTICAL mechanically
→ But now WE control what choices exist

THE BACKTRACKING TEMPLATE:
  backtrack(choices, startIndex, currentState, results):
      results.add(copy of currentState)          ← or check condition
      for i = startIndex to end:
          currentState.add(choices[i])            ← CHOOSE
          backtrack(choices, i + 1, currentState) ← EXPLORE
          currentState.remove(last)               ← UN-CHOOSE

This template works for:
→ Subsets (this project) — collect at every node
→ Combinations of size K — collect when size == K
→ Combination Sum (Project 42) — collect when sum == target
→ Permutations (Project 29) — different startIndex logic

THE KEY MENTAL MODEL:
→ Every backtracking problem is a TREE of decisions
→ The nodes are states (current partial solution)
→ The edges are choices (add an element, pick a number, etc.)
→ The leaves are complete solutions (or dead ends)
→ Backtracking = DFS through this decision tree
→ "Add-Recurse-Remove" ensures clean state at every node

🌉 BRIDGE TO NEXT PROJECT

PROJECT 28 taught us:
→ Subsets: for each element, include or exclude
→ startIndex ensures we only move FORWARD → no duplicate subsets
→ Order doesn't matter: {1,2} and {2,1} are the SAME subset

PROJECT 29 (Generate All Permutations) will ask:
→ What if ORDER MATTERS?
→ [1,2,3] and [3,2,1] are DIFFERENT permutations
→ We can no longer use startIndex to prevent revisiting
→ Instead: we need a SWAP mechanism or VISITED set
→ Every element must appear exactly ONCE in each permutation
→ But in ANY position

The key difference:
  Subsets:      Choose WHICH elements to include (order irrelevant)
  Permutations: Choose WHICH POSITION each element occupies (order matters)

  Subsets:      for i = startIndex → prevents going backward
  Permutations: for i = 0 → allows any position → uses visited[] to prevent reuse

→ Same backtracking skeleton
→ Different branching logic
→ This is why learning the TEMPLATE matters more than memorizing solutions
