PROJECT 37: Find Two Single Numbers in Array

🎯 Bitwise XOR → XOR Grouping by Set Bit → Partition-Then-Cancel

🧩 THE HYPOTHETICAL PROBLEM

→ You are given an array of integers
→ EVERY number appears EXACTLY TWICE, except for TWO numbers
→ Those two numbers appear EXACTLY ONCE each
→ Find and return both single numbers
→ You MUST solve it in O(n) time and O(1) space
→ The two single numbers are DIFFERENT from each other

Example 1:
  Input:  [1, 2, 1, 3, 2, 5]
  Output: [3, 5]  (or [5, 3] — order doesn't matter)
  → 1 appears twice, 2 appears twice
  → 3 appears once, 5 appears once

Example 2:
  Input:  [4, 1, 4, 6]
  Output: [1, 6]
  → 4 appears twice
  → 1 appears once, 6 appears once

Example 3:
  Input:  [2, 3]
  Output: [2, 3]
  → Both appear once (minimum case)

Example 4:
  Input:  [10, 20, 10, 30, 20, 40, 50, 50]
  Output: [30, 40]

🔍 HOW TO APPROACH IT

Step 1 — Recall Project 10 (Single Number — ONE Unique)

PROJECT 10 RECAP:
  → Array where every number appears twice EXCEPT ONE
  → XOR all numbers together
  → Pairs cancel: a ^ a = 0
  → Single number remains: 0 ^ single = single

Example: [4, 1, 2, 1, 2]
  → 4 ^ 1 ^ 2 ^ 1 ^ 2
  → = 4 ^ (1^1) ^ (2^2)
  → = 4 ^ 0 ^ 0
  → = 4 ✓

KEY XOR PROPERTIES (from Project 10):
  → a ^ a = 0        (self-cancellation)
  → a ^ 0 = a        (identity)
  → a ^ b = b ^ a    (commutative)
  → (a^b) ^ c = a ^ (b^c)  (associative)
  → Order doesn't matter — pairs cancel regardless of position

Step 2 — What Happens When We XOR Everything with TWO Singles?

Array: [1, 2, 1, 3, 2, 5]
Single numbers: 3 and 5

XOR everything:
  1 ^ 2 ^ 1 ^ 3 ^ 2 ^ 5
  = (1^1) ^ (2^2) ^ 3 ^ 5
  = 0 ^ 0 ^ 3 ^ 5
  = 3 ^ 5
  = 6 (in binary: 011 ^ 101 = 110)

RESULT: We get singleA ^ singleB
  → NOT singleA or singleB individually
  → We know their XOR but not their individual values
  → This is the DIFFERENCE from Project 10
  → Project 10: XOR gives the answer directly
  → Project 37: XOR gives a CLUE — not the answer

THE CLUE: What does singleA ^ singleB tell us?
  → XOR produces 1 in bit positions where A and B DIFFER
  → XOR produces 0 in bit positions where A and B are SAME
  → Since A ≠ B → at least one bit position has a 1
  → That bit position tells us WHERE A and B differ

Step 3 — The Partition Insight

singleA = 3 = 011 (binary)
singleB = 5 = 101 (binary)
xorResult = 3 ^ 5 = 110 (binary)

The rightmost set bit of xorResult is at position 1 (value = 2)
  → This means: singleA has a 0 at bit 1, singleB has a 1 at bit 1
  → (Or vice versa — we don't know which is which yet)

NOW THE KEY IDEA:
  → Use this differentiating bit to SPLIT ALL numbers into two groups
  → Group 0: numbers with bit 1 = 0
  → Group 1: numbers with bit 1 = 1

  Array: [1, 2, 1, 3, 2, 5]
  Bit 1 (value 2):
    1 = 01 → bit 1 = 0 → Group 0
    2 = 10 → bit 1 = 1 → Group 1
    1 = 01 → bit 1 = 0 → Group 0
    3 = 11 → bit 1 = 1 → Group 1
    2 = 10 → bit 1 = 1 → Group 1
    5 = 101 → bit 1 = 0 → Group 0

  Group 0: [1, 1, 5]     → contains exactly ONE single (5)
  Group 1: [2, 3, 2]     → contains exactly ONE single (3)

WHY DOES THIS WORK?
  → singleA and singleB DIFFER at this bit → they go to DIFFERENT groups
  → Every PAIR of identical numbers has the SAME bit value
  → So both copies of a pair go to the SAME group
  → Each group has: some complete pairs + exactly ONE single number
  → Each group is now exactly the problem from Project 10!

XOR Group 0: 1 ^ 1 ^ 5 = 5 ✓
XOR Group 1: 2 ^ 3 ^ 2 = 3 ✓

ANSWER: [5, 3] ✓

Step 4 — How to Find the Differentiating Bit

Given xorResult = singleA ^ singleB (a non-zero number):
→ We need ANY bit that is set to 1
→ The simplest choice: the RIGHTMOST set bit

HOW TO ISOLATE THE RIGHTMOST SET BIT:
  → diffBit = xorResult & (-xorResult)

WHY THIS WORKS — Two's Complement Magic:
  → -xorResult in two's complement = ~xorResult + 1
  → ~xorResult flips ALL bits
  → Adding 1 flips bits from the right until the first 0→1

  Example: xorResult = 6 = 110
  → ~6 = ...11111001 (all bits flipped)
  → ~6 + 1 = ...11111010 = -6
  → 6 & (-6) = 110 & 010 = 010 = 2

  The rightmost set bit of 6 is at position 1 (value 2) ✓

  Example: xorResult = 12 = 1100
  → -12 = ...11110100
  → 12 & (-12) = 1100 & 0100 = 0100 = 4

  The rightmost set bit of 12 is at position 2 (value 4) ✓

ALTERNATIVE: Use any set bit, not necessarily the rightmost
  → The rightmost is just the easiest to isolate
  → Any set bit would work for partitioning

Step 5 — Full Walkthrough

Array: [10, 20, 10, 30, 20, 40, 50, 50]
Singles: 30 and 40

STEP 1 — XOR everything:
  10 ^ 20 ^ 10 ^ 30 ^ 20 ^ 40 ^ 50 ^ 50
  = (10^10) ^ (20^20) ^ (50^50) ^ 30 ^ 40
  = 0 ^ 0 ^ 0 ^ 30 ^ 40
  = 30 ^ 40

  30 = 011110
  40 = 101000
  XOR = 110110 = 54

STEP 2 — Find rightmost set bit:
  diffBit = 54 & (-54)
  54  = 110110
  -54 = 001010
  54 & (-54) = 000010 = 2

  Differentiating bit: position 1 (value 2)

STEP 3 — Partition by bit 1:
  10 = 001010 → bit 1 = 1 → Group 1
  20 = 010100 → bit 1 = 0 → Group 0
  10 = 001010 → bit 1 = 1 → Group 1
  30 = 011110 → bit 1 = 1 → Group 1
  20 = 010100 → bit 1 = 0 → Group 0
  40 = 101000 → bit 1 = 0 → Group 0
  50 = 110010 → bit 1 = 1 → Group 1
  50 = 110010 → bit 1 = 1 → Group 1

  Group 0: [20, 20, 40]     → single = 40
  Group 1: [10, 10, 30, 50, 50] → single = 30

STEP 4 — XOR each group:
  Group 0: 20 ^ 20 ^ 40 = 40 ✓
  Group 1: 10 ^ 10 ^ 30 ^ 50 ^ 50 = 30 ✓

ANSWER: [40, 30] ✓

Step 6 — Why We Don't Actually Create Two Groups

→ Creating two separate arrays would use O(n) space
→ We need O(1) space
→ Instead: we XOR in a SINGLE PASS with two accumulators

int xorA = 0;  // Will accumulate XOR of Group 0
int xorB = 0;  // Will accumulate XOR of Group 1

for (int num : nums) {
    if ((num & diffBit) == 0) {
        xorA ^= num;  // Group 0
    } else {
        xorB ^= num;  // Group 1
    }
}

→ xorA = single number from Group 0
→ xorB = single number from Group 1
→ O(1) space — no arrays created
→ O(n) time — single pass through the array

🛠️ HOW CAN WE SOLVE IT

The Algorithm — Three Steps

STEP 1: XOR all numbers → get singleA ^ singleB
  int xorAll = 0;
  for (int num : nums) {
      xorAll ^= num;
  }
  // xorAll = singleA ^ singleB (non-zero since A ≠ B)

STEP 2: Find any differentiating bit
  int diffBit = xorAll & (-xorAll);  // Rightmost set bit

STEP 3: Partition-and-XOR in one pass
  int num1 = 0, num2 = 0;
  for (int num : nums) {
      if ((num & diffBit) == 0) {
          num1 ^= num;  // Group where this bit is 0
      } else {
          num2 ^= num;  // Group where this bit is 1
      }
  }
  // num1 and num2 are the two single numbers

Why Each Step is Necessary

STEP 1 — WHY XOR all?
  → Eliminates all pairs (a ^ a = 0)
  → Leaves only singleA ^ singleB
  → This is the INFORMATION we extract from the array

STEP 2 — WHY find a set bit?
  → A set bit means singleA and singleB DIFFER at this position
  → This difference lets us SEPARATE them into different groups
  → Without this: we can't split them apart

STEP 3 — WHY partition by this bit?
  → Pairs always go to the SAME group (same number = same bits)
  → Singles go to DIFFERENT groups (they differ at this bit)
  → Each group becomes a standard "one single number" problem (Project 10)
  → XOR each group → each yields one single number

THE REDUCTION:
  → Project 37 (two singles) REDUCES TO two instances of Project 10 (one single)
  → The partition step is what enables this reduction
  → This is a beautiful example of PROBLEM REDUCTION

✅ WHAT SHOULD YOU DO

→ ALWAYS compute XOR of entire array first
     → This gives you singleA ^ singleB
     → Without this: you have no information to work with

→ ALWAYS use xorAll & (-xorAll) for rightmost set bit
     → Clean, one-line, O(1) operation
     → Works for all non-zero integers
     → The result is a power of 2 (exactly one bit set)

→ ALWAYS check (num & diffBit) == 0 vs != 0 for partitioning
     → NOT (num & diffBit) == 1 → diffBit might not be 1
     → diffBit could be 2, 4, 8, etc.
     → (num & diffBit) gives either 0 or diffBit → check == 0

→ ALWAYS initialize both accumulators to 0
     → XOR with 0 is identity: 0 ^ x = x
     → Starting at 0 means the first XOR just stores the value

→ ALWAYS verify the two results are different
     → If both return the same value → something went wrong
     → The problem guarantees two DIFFERENT singles

→ ALWAYS handle the edge case of two-element array
     → [a, b] where a ≠ b → both are singles
     → XOR = a ^ b → diffBit exists → partition works
     → Each group has exactly one element → works correctly

🚫 WHAT SHOULD YOU NOT DO

→ DO NOT use a HashMap to count frequencies
     → HashMap: O(n) time, O(n) space
     → Problem requires O(1) space
     → XOR approach is O(n) time, O(1) space → superior

→ DO NOT try to extract individual numbers from xorAll directly
     → xorAll = singleA ^ singleB
     → You CANNOT recover singleA or singleB from this alone
     → You need the partition step to separate them

→ DO NOT use (num & diffBit) == 1 for the check
     → If diffBit = 4 (binary 100):
          → num = 5 (binary 101) → num & diffBit = 4 (not 1!)
          → Checking == 1 would FAIL → num goes to wrong group
     → Use == 0 or != 0 instead

→ DO NOT use any bit other than a SET bit of xorAll for partitioning
     → If you pick a bit where xorAll has 0:
          → singleA and singleB have the SAME value at that bit
          → They go to the SAME group → not separated → WRONG

→ DO NOT confuse this with finding THREE single numbers
     → Two singles → partition by one bit → two groups
     → Three singles → much harder → different technique entirely
     → This specific partition trick only works for EXACTLY two singles

→ DO NOT worry about which single goes to which group
     → The partition is deterministic but you don't control it
     → num1 might be the larger or smaller single
     → Order doesn't matter → return both

🔀 WHEN TO USE THIS PATTERN

✓ "Find two numbers that appear once, all others appear twice"
     → This exact problem

✓ "Find two missing numbers from a sequence"
     → XOR expected sequence with actual → reduces to two-singles problem

✓ Any problem reducible to "separate two unknown values using XOR"
     → The partition-by-bit technique applies whenever:
          → You have XOR of two unknowns
          → You can partition the data by a bit position
          → Each partition yields one unknown via XOR cancellation

✓ Problems where O(1) space is mandatory
     → HashMap/Set solutions use O(n) space
     → XOR solutions use O(1) space
     → When space constraint is strict → think bitwise

✓ Interview context: "Can you solve it without extra space?"
     → This is the classic follow-up to the HashMap solution
     → Demonstrates deep understanding of bit manipulation

🚫 WHEN NOT TO USE THIS PATTERN

✗ Numbers appear more than twice (e.g., three times)
     → XOR cancellation: a ^ a = 0 → works for PAIRS
     → a ^ a ^ a = a → triples don't cancel
     → Need different approach (bit counting modulo 3)

✗ More than two single numbers
     → Two singles: partition by 1 bit → 2 groups → solvable
     → Three singles: partition by 1 bit → can't guarantee separation
     → K singles with K > 2: much more complex

✗ Numbers can be negative AND you're on a system without two's complement
     → Two's complement is standard for modern systems
     → But some edge cases with Integer.MIN_VALUE need care
     → -Integer.MIN_VALUE overflows in Java

✗ You need the POSITIONS (indices) of the single numbers
     → XOR gives you the VALUES but not WHERE they are
     → After finding values: linear scan to find indices → O(n) extra pass
     → But this is fine — still O(n) time, O(1) space

✗ The "duplicates" appear different numbers of times
     → "Some appear 3 times, some 2 times, find the one appearing once"
     → XOR doesn't help cleanly → need bit counting approach

⚙️ HOW IT ONLY WORKS — Internal Mechanics

XOR as Information Extraction

WHAT XOR TELLS US:
  → a ^ b produces a bitmask of DIFFERENCES between a and b
  → Bit = 1 → a and b differ at this position
  → Bit = 0 → a and b agree at this position

EXAMPLE:
  30 = 011110
  40 = 101000
  XOR= 110110

  Position 0: 0 vs 0 → same → XOR bit = 0
  Position 1: 1 vs 0 → differ → XOR bit = 1
  Position 2: 1 vs 0 → differ → XOR bit = 1
  Position 3: 1 vs 1 → same → XOR bit = 0
  Position 4: 1 vs 0 → differ → XOR bit = 1
  Position 5: 0 vs 1 → differ → XOR bit = 1

→ XOR is a "difference detector"
→ For our problem: it tells us WHERE the two singles differ
→ We use ANY one of these difference positions to separate them

Why Rightmost Set Bit? (And Why Any Bit Works)

xorAll & (-xorAll) isolates the RIGHTMOST set bit

WHY RIGHTMOST?
  → It's the easiest to isolate with a one-line formula
  → No loops, no counting → pure bit manipulation → O(1)

COULD WE USE A DIFFERENT BIT?
  → YES — any set bit of xorAll works
  → The rightmost is just the most convenient to extract

ALTERNATIVE METHODS to find a set bit:
  → Integer.lowestOneBit(xorAll) → same as xorAll & (-xorAll)
  → Integer.highestOneBit(xorAll) → uses the leftmost set bit
  → Loop: while ((xorAll & 1) == 0) xorAll >>= 1; → finds rightmost manually
  → All work, first is cleanest

The Partition Correctness Proof

CLAIM: After partitioning by diffBit:
  → singleA is in one group, singleB is in the other
  → Every pair {x, x} has both copies in the SAME group

PROOF — Singles are separated:
  → diffBit is a set bit of singleA ^ singleB
  → This means singleA and singleB DIFFER at this bit position
  → One has 0, the other has 1
  → (singleA & diffBit) ≠ (singleB & diffBit)
  → They go to DIFFERENT groups ✓

PROOF — Pairs stay together:
  → For any number x that appears twice:
       → Both copies have identical bit patterns
       → (x & diffBit) is the SAME for both copies
       → Both copies go to the SAME group ✓

PROOF — Each group is a "single number" problem:
  → Group has: some complete pairs + exactly one single
  → XOR of the group: pairs cancel (a ^ a = 0), single remains
  → Result: the single number ✓

Two's Complement: Why -x = ~x + 1

In two's complement (used by Java, C, Python, etc.):
  → Negative numbers are stored as bitwise NOT plus 1
  → -x = ~x + 1

EXAMPLE: x = 6
  6    = 00000110
  ~6   = 11111001
  ~6+1 = 11111010 = -6

  6 & (-6):
  00000110
  11111010
  --------
  00000010 = 2 (rightmost set bit of 6)

WHY IT ISOLATES THE RIGHTMOST SET BIT:
  → ~x flips all bits → the rightmost 1 becomes 0, all 0s to its right become 1s
  → Adding 1 to ~x: the trailing 1s carry over, turning back to 0
  → Until the first 0 (which was the rightmost 1 of x) becomes 1
  → Now x and -x share ONLY that one bit position as 1
  → AND isolates it

VISUAL:
  x     = ...1010  1  0  0  0
  ~x    = ...0101  0  1  1  1
  ~x+1  = ...0101  1  0  0  0
  x & -x= ...0000  1  0  0  0  ← only the rightmost 1 survives

⚠️ COMMON PITFALLS AND FAILURE MODES

Pitfall 1: Integer.MIN_VALUE Edge Case

ISSUE:
  → xorAll could be Integer.MIN_VALUE (= -2^31)
  → -Integer.MIN_VALUE OVERFLOWS in Java
  → -(-2^31) = 2^31 → exceeds Integer.MAX_VALUE (2^31 - 1)
  → Result wraps around to Integer.MIN_VALUE again

DOES THIS CAUSE A BUG?
  → Let's check: Integer.MIN_VALUE & (-Integer.MIN_VALUE)
  → Integer.MIN_VALUE = 10000...0 (1 followed by 31 zeros)
  → -Integer.MIN_VALUE overflows to Integer.MIN_VALUE
  → Integer.MIN_VALUE & Integer.MIN_VALUE = Integer.MIN_VALUE
  → Result: 10000...0 → which IS the rightmost (and only) set bit
  → IT WORKS CORRECTLY despite the overflow!

  → This is a happy accident of two's complement arithmetic
  → But it's worth knowing about in case an interviewer asks

SAFE ALTERNATIVE:
  → int diffBit = Integer.lowestOneBit(xorAll);
  → Java library method → handles all edge cases correctly

Pitfall 2: Using == 1 Instead of != 0 for Bit Check

WRONG:
  if ((num & diffBit) == 1) {  // ← BUG if diffBit ≠ 1

EXAMPLE:
  diffBit = 4 (binary 100)
  num = 7 (binary 111)
  num & diffBit = 111 & 100 = 100 = 4
  4 == 1? NO → goes to wrong group!

CORRECT:
  if ((num & diffBit) != 0) {  // ← Works for any diffBit value

  OR:
  if ((num & diffBit) == 0) {  // Group 0
  } else {                      // Group 1
  }

Pitfall 3: Forgetting That XOR of Empty Set is 0

→ If a group happens to be empty (impossible in this problem but good to know):
     → XOR of no elements = 0 (the identity)
     → This would incorrectly report 0 as a "single number"

→ In THIS problem: each group has at least one single + possibly pairs
→ Each group is guaranteed non-empty
→ So this pitfall doesn't apply here
→ But in variants: be careful about empty groups

Pitfall 4: Assuming the Two Singles Are Adjacent

→ The two single numbers can be ANYWHERE in the array
→ They might be adjacent or far apart
→ The algorithm doesn't care about positions
→ XOR is commutative and associative → order irrelevant
→ Don't try to optimize based on adjacency assumptions

Pitfall 5: Returning Wrong Order

→ The problem says "return both numbers" — usually order doesn't matter
→ But some problems say "return in ascending order"
→ If so: return new int[]{ Math.min(num1, num2), Math.max(num1, num2) }
→ Don't forget this final step if the problem requires it

Pitfall 6: Trying to Apply to Three Singles

→ Two singles: partition by 1 bit → 2 groups → each has 1 single → solved
→ Three singles: partition by 1 bit → 2 groups → one has 2 singles, other has 1
     → The group with 2 singles is STILL a two-singles problem
     → But we don't know which group has 2 and which has 1
     → More complex logic needed → different algorithm entirely
→ Don't assume the two-singles technique generalizes to three

📈 SCALABILITY CONSIDERATIONS

Time Complexity: O(n)
  → Pass 1: XOR all elements → O(n)
  → Rightmost set bit: O(1)
  → Pass 2: Partition and XOR → O(n)
  → Total: O(n)

Space Complexity: O(1)
  → Three integer variables: xorAll, diffBit, num1/num2
  → No arrays, no hash maps, no data structures
  → True constant space

Practical Performance:
  → n = 10^7: ~2 passes × 10^7 → ~20M operations → ~50ms
  → n = 10^8: ~200M operations → ~500ms
  → XOR is one of the fastest CPU operations → excellent constant factor

Comparison:
  → HashMap approach: O(n) time, O(n) space
  → Sorting approach: O(n log n) time, O(1) space (if in-place)
  → XOR approach: O(n) time, O(1) space → OPTIMAL on both axes
  → For n = 10^8: HashMap needs ~800MB for the map → might OOM
  → XOR: needs ~12 bytes extra → trivial

🔍 INTERPRETABILITY CONSIDERATIONS

→ The XOR approach is elegant but NOT intuitive:
     → "XOR everything" → "find a set bit" → "partition" → "XOR each group"
     → Each step requires understanding of binary operations
     → Interviewers are impressed when you explain it clearly

→ Step-by-step explanation strategy:
     → "First, I XOR everything. Pairs cancel, leaving singleA XOR singleB."
     → "This XOR tells me where the two singles DIFFER in their bits."
     → "I pick any differing bit and use it to split all numbers into two groups."
     → "Each group has exactly one single number plus complete pairs."
     → "XOR each group — pairs cancel — single remains."
     → "This reduces the two-single problem to two one-single problems."

→ Visual aid:
     → Draw the binary representations
     → Circle the differing bit
     → Show the partition visually
     → This makes the abstract bit manipulation concrete

→ Common interviewer reaction:
     → "That's clever. Can you prove the partition always separates the singles?"
     → Answer: "The partition bit is set in singleA XOR singleB, meaning 
       singleA and singleB have DIFFERENT values at that bit. So they go to 
       different groups. Pairs have identical bits, so both copies go to the 
       same group. QED."

→ A set bit in singleA ^ singleB reveals WHERE the two singles DIFFER
→ That bit lets us PARTITION all numbers into two groups
→ Each group contains exactly ONE single + complete pairs
→ XOR each group → pairs cancel → single remains → SOLVED

THE PROBLEM REDUCTION:
  → Project 37 (two singles) REDUCES TO two instances of Project 10 (one single)
  → The reduction tool is: partition by a differentiating bit
  → This is a CLASSIC technique in algorithm design:
       → Can't solve the hard problem directly?
       → Split it into easier subproblems you already know how to solve

THE RIGHTMOST SET BIT TRICK:
  → diffBit = xorAll & (-xorAll)
  → Isolates exactly one bit → perfect for binary partitioning
  → Works via two's complement arithmetic
  → O(1) time, pure bit manipulation

PROJECT 38 (Kth Largest Element in an Array) opens the TOP K ELEMENTS pattern:
→ Building on Project 35 (Two Heaps — PriorityQueue fundamentals)
→ Project 35: TWO heaps to find the MEDIAN (middle element)
→ Project 38: ONE heap to find the Kth LARGEST element

→ The key insight:
     → Maintain a MIN-HEAP of size K
     → The heap always contains the K LARGEST elements seen so far
     → The heap's TOP (minimum of the K largest) = the Kth largest overall
     → Adding a new element:
          → If smaller than heap top → ignore (not in top K)
          → If larger than heap top → remove top, add new → heap still has K largest

→ Why MIN-heap for "largest"?
     → Counterintuitive but brilliant:
     → We want to DISCARD the SMALLEST of the K candidates
     → Min-heap gives us the smallest → we discard it → keep the K largest
     → This is the "bouncer at the door" model:
          → The bouncer checks: "Are you bigger than the smallest person inside?"
          → If yes: smallest person leaves, you enter
          → If no: you're rejected
