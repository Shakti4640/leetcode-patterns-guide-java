🧩 THE HYPOTHETICAL PROBLEM

→ Design a data structure that supports these operations efficiently:
     → insert(word): Add a word to the structure
     → search(word): Return true if the EXACT word exists
     → startsWith(prefix): Return true if ANY word starts with this prefix

→ All inputs consist of lowercase English letters (a-z)
→ All operations should run in O(L) time where L = word/prefix length

Example:
  Trie trie = new Trie();
  trie.insert("apple");
  trie.search("apple");    → true  (exact match)
  trie.search("app");      → false (exists as prefix, not complete word)
  trie.startsWith("app");  → true  (prefix of "apple")
  trie.insert("app");
  trie.search("app");      → true  (now "app" is also a complete word)

Why can't we just use a HashSet?
  → HashSet does search() in O(1) average → faster for exact search
  → But startsWith() requires checking ALL words → O(N × L) worst case
  → Trie does startsWith() in O(L) → much faster for prefix queries
  → Trie also enables autocomplete, spell-checking, pattern matching

🔍 HOW TO APPROACH IT

Step 1 — Understand What a Trie Looks Like

After inserting: "apple", "app", "ape", "bat", "ball"

The Trie looks like:

              ROOT
             /    \
            a      b
            |      |
            p      a
           / \     |  \
          p   e    t   l
          |        ●   |
          l             l
          |             |
          e             ●
          ●

● = "end of word" marker

Key observations:
→ "apple" and "app" share the path: ROOT → a → p → p
→ "apple" and "ape" share the path: ROOT → a → p
→ "bat" and "ball" share the path: ROOT → b → a
→ Shared prefixes = shared nodes = memory efficiency
→ Each NODE has up to 26 children (one per letter a-z)
→ Each EDGE represents one character

Step 2 — The TrieNode Structure

Each node in the Trie contains:
  1. children: an array/map of child nodes (one per possible character)
  2. isEndOfWord: boolean flag marking if a complete word ends here

Why children array of size 26?
→ Only lowercase a-z → 26 possible characters
→ children[0] = child for 'a', children[1] = child for 'b', etc.
→ Index = character - 'a'
→ null means no child for that character

Why isEndOfWord flag?
→ "app" and "apple" share the path a → p → p
→ The node after the second 'p' must know:
     → "app" ends here → isEndOfWord = true
     → But there's also a child 'l' leading to "apple"
→ Without the flag, we can't distinguish:
     → "app" was inserted (word exists)
     → "app" is just a prefix of "apple" (word doesn't exist)

Step 3 — How Insert Works

insert("apple"):

Start at ROOT
  Character 'a': children['a'-'a'] = children[0]
    → null? Create new TrieNode → move to it
  Character 'p': children['p'-'a'] = children[15]
    → null? Create new TrieNode → move to it
  Character 'p': children[15]
    → null? Create new TrieNode → move to it
  Character 'l': children['l'-'a'] = children[11]
    → null? Create new TrieNode → move to it
  Character 'e': children['e'-'a'] = children[4]
    → null? Create new TrieNode → move to it
  End of word → mark current node: isEndOfWord = true

After insert("apple"), insert("app"):

insert("app"):
  'a' → exists → move to it
  'p' → exists → move to it
  'p' → exists → move to it
  End of word → mark isEndOfWord = true

The second insert REUSES existing nodes
→ No new nodes created
→ Only the flag changes
→ This is the POWER of Trie: shared prefix = shared storage

Step 4 — How Search Works

search("apple"):
  Start at ROOT
  'a' → children[0] exists? YES → move to it
  'p' → children[15] exists? YES → move to it
  'p' → children[15] exists? YES → move to it
  'l' → children[11] exists? YES → move to it
  'e' → children[4] exists? YES → move to it
  End of word → check isEndOfWord → TRUE → return true ✓

search("app"):
  'a' → exists → move
  'p' → exists → move
  'p' → exists → move
  End of word → check isEndOfWord → depends:
    → If "app" was inserted → TRUE
    → If only "apple" was inserted → FALSE

search("apx"):
  'a' → exists → move
  'p' → exists → move
  'x' → children['x'-'a'] = null → DOESN'T EXIST → return false
  → We "fell off" the Trie → word not found

Step 5 — How StartsWith Works

startsWith("app"):
  Start at ROOT
  'a' → exists → move
  'p' → exists → move
  'p' → exists → move
  Reached end of prefix → ALL characters found → return true ✓
  → We DON'T check isEndOfWord
  → Just reaching the end of the prefix is enough
  → Because SOME word must continue from here

startsWith("apx"):
  'a' → exists → move
  'p' → exists → move
  'x' → null → DOESN'T EXIST → return false
  → No word in the Trie starts with "apx"

KEY DIFFERENCE between search and startsWith:
  search("app"):      traverse "app" → check isEndOfWord
  startsWith("app"):  traverse "app" → return true (reached the end)
  
  Both traverse the same path
  → search needs isEndOfWord = true (exact word must exist)
  → startsWith just needs the path to exist (any word with this prefix)

🛠️ HOW CAN WE SOLVE IT

The Node Design

class TrieNode:
  children: TrieNode[26]    → one slot per lowercase letter
  isEndOfWord: boolean       → true if a complete word ends here

Why array[26] instead of HashMap?
  → Fixed alphabet size (26 lowercase letters)
  → Array access is O(1) with constant factor
  → HashMap has overhead (hashing, boxing, etc.)
  → For lowercase-only: array is faster and simpler
  → For Unicode/large alphabets: HashMap is better

Index mapping:
  'a' → index 0
  'b' → index 1
  ...
  'z' → index 25
  Formula: index = character - 'a'

The Three Operations — Pseudocode

INSERT(word):
  node = root
  for each character c in word:
    index = c - 'a'
    if node.children[index] is null:
      node.children[index] = new TrieNode()
    node = node.children[index]
  node.isEndOfWord = true

SEARCH(word):
  node = root
  for each character c in word:
    index = c - 'a'
    if node.children[index] is null:
      return false    → word doesn't exist
    node = node.children[index]
  return node.isEndOfWord    → true only if complete word ends here

STARTS_WITH(prefix):
  node = root
  for each character c in prefix:
    index = c - 'a'
    if node.children[index] is null:
      return false    → no word starts with this prefix
    node = node.children[index]
  return true    → reached end of prefix → some word has this prefix

✅ WHAT SHOULD YOU DO

→ ALWAYS separate the TrieNode class from the Trie class
     → TrieNode: data container (children array + isEndOfWord flag)
     → Trie: operations (insert, search, startsWith) starting from root
     → Clean separation of concerns

→ ALWAYS initialize root as an empty TrieNode
     → Root represents the "empty prefix"
     → Root has no character of its own
     → All words start their traversal from root
     → Root's isEndOfWord is typically false (unless "" is inserted)

→ ALWAYS use character - 'a' for index mapping
     → Clean, constant-time conversion
     → Works for lowercase a-z
     → For other character sets → adjust the formula

→ ALWAYS create new nodes ONLY when needed (during insert)
     → Don't preallocate the entire tree
     → Most children slots remain null
     → This is LAZY initialization → memory efficient

→ ALWAYS extract a common helper method for traversal
     → search() and startsWith() share the same traversal logic
     → Extract a "findNode(prefix)" helper that returns the node
     → search() = findNode(word) != null AND findNode(word).isEndOfWord
     → startsWith() = findNode(prefix) != null
     → DRY principle — Don't Repeat Yourself

→ ALWAYS think about what ADDITIONAL operations you might need
     → delete(word): mark isEndOfWord as false (or prune unused nodes)
     → countWordsWithPrefix(prefix): count words under a prefix
     → getAllWordsWithPrefix(prefix): autocomplete functionality
     → The basic Trie supports all of these with minor extensions

🚫 WHAT SHOULD YOU NOT DO

→ DO NOT use HashMap<Character, TrieNode> for lowercase-only problems
     → Array[26] is faster and uses less memory for fixed alphabet
     → HashMap adds boxing overhead (char → Character)
     → HashMap adds hashing overhead
     → Use HashMap ONLY for large/dynamic alphabets (Unicode, etc.)

→ DO NOT forget the isEndOfWord flag
     → Without it: search("app") returns true even if only "apple" was inserted
     → The flag distinguishes "this is a complete word" from "this is just a prefix"
     → This is the SINGLE MOST COMMON mistake with Tries

→ DO NOT confuse search() with startsWith()
     → search: exact word must exist → check isEndOfWord
     → startsWith: any word with this prefix → just check path exists
     → They share traversal logic but differ in the final check

→ DO NOT create a node for the character BEFORE checking if it exists
     → During search/startsWith: if children[index] is null → return false
     → During insert: if children[index] is null → create new node
     → Mixing these up causes either crashes or phantom nodes

→ DO NOT store the character in the node itself
     → The character is encoded in the EDGE (parent → child relationship)
     → children[3] means the edge carries character 'd' (index 3 + 'a')
     → Storing it in the node is redundant (wastes memory)
     → Some implementations do store it for debugging — that's OK

→ DO NOT assume the root node represents a character
     → Root is the "empty string" node
     → Root's children represent the FIRST character of words
     → Root itself has no character

TRIE vs HASHSET:
┌────────────────────┬───────────┬───────────┐
│ Operation          │ HashSet   │ Trie      │
├────────────────────┼───────────┼───────────┤
│ insert(word)       │ O(L)      │ O(L)      │
│ search(word)       │ O(L) avg  │ O(L)      │
│ startsWith(prefix) │ O(N × L)  │ O(L)      │
│ delete(word)       │ O(L)      │ O(L)      │
│ autocomplete       │ O(N × L)  │ O(L + K)  │
│ Space              │ O(N × L)  │ O(N × L)  │
└────────────────────┴───────────┴───────────┘
  N = number of words, L = avg word length, K = results count

→ Trie wins decisively on PREFIX operations
→ HashSet is simpler for exact-match-only problems

🚫 WHEN NOT TO USE THIS PATTERN

✗ You only need exact word lookup (no prefix operations)
     → HashSet is simpler and often faster
     → Trie is overkill for pure existence checks

✗ The alphabet is very large (e.g., full Unicode)
     → Array[26] doesn't work → need HashMap per node
     → Each node's HashMap has overhead
     → Memory consumption explodes
     → Consider other structures (ternary search tree, etc.)

✗ Words are very long and share few prefixes
     → Trie's advantage is shared prefixes
     → If words share no prefixes → Trie degrades to array of strings
     → No memory savings, just added complexity

✗ You need sorted iteration over all words
     → Trie CAN do this (DFS traversal gives sorted order)
     → But a TreeSet/sorted array is simpler for this specific need

✗ Dynamic deletion is frequent
     → Trie deletion is tricky (need to prune empty branches)
     → If you're constantly adding and removing → HashSet is simpler
     → Trie excels at BUILD ONCE, QUERY MANY times

⚙️ HOW IT ONLY WORKS — Internal Mechanics

The Character-Per-Edge Model

Traditional tree: each NODE stores a value
  → Binary tree node: { value: 5, left: ..., right: ... }

Trie: each EDGE stores a character, each NODE stores metadata
  → TrieNode: { children[26], isEndOfWord }
  → The "value" of a node is the PATH from root to that node

Example: inserting "cat"
  
  ROOT ──(c)──→ Node1 ──(a)──→ Node2 ──(t)──→ Node3 [isEnd=true]
  
  The word "cat" is NOT stored in any single node
  It's stored as the PATH: root → c → a → t
  Each node represents a PREFIX:
    ROOT = ""
    Node1 = "c"
    Node2 = "ca"
    Node3 = "cat" (marked as end of word)

This is why it's called a PREFIX tree:
  → Every node represents a prefix
  → The path from root to any node spells out that prefix
  → Words that share prefixes share paths

Why Array[26] Works for Lowercase

Each TrieNode has: TrieNode[] children = new TrieNode[26]

Memory per node: 26 pointers + 1 boolean ≈ 26 × 8 + 1 = 209 bytes (64-bit)

For a Trie with 1000 nodes:
  → 209 × 1000 ≈ 209 KB
  → Very manageable

But MOST children slots are null:
  → Average English word has ~5 characters
  → Average node has 2-3 non-null children
  → So 23-24 out of 26 slots are wasted per node
  → This is the SPACE TRADE-OFF for O(1) child access

Alternative: HashMap<Character, TrieNode>
  → No wasted slots
  → But HashMap overhead per entry: ~50-100 bytes
  → For 3 children: HashMap ≈ 150-300 bytes vs Array ≈ 209 bytes
  → HashMap actually uses MORE memory for small child counts!
  → Array wins for alphabets ≤ ~50 characters

Why Insert is O(L)

insert("apple") — L = 5 characters

Step 1: index = 'a' - 'a' = 0  → check/create children[0]  → O(1)
Step 2: index = 'p' - 'a' = 15 → check/create children[15] → O(1)
Step 3: index = 'p' - 'a' = 15 → check/create children[15] → O(1)
Step 4: index = 'l' - 'a' = 11 → check/create children[11] → O(1)
Step 5: index = 'e' - 'a' = 4  → check/create children[4]  → O(1)
Step 6: set isEndOfWord = true → O(1)

Total: L steps × O(1) per step = O(L)

→ No comparison with other words needed
→ No sorting, no hashing, no balancing
→ Pure character-by-character traversal
→ Independent of how many words are already in the Trie

Why Trie Enables Efficient Prefix Queries

HashSet approach to startsWith("app"):
  → Iterate over ALL N words in the set
  → Check if each word starts with "app"
  → O(N × L) worst case

Trie approach to startsWith("app"):
  → Follow the path: root → a → p → p
  → If we reach the end → return true
  → 3 steps regardless of how many words exist
  → O(L) where L = prefix length

WHY?
→ Trie ORGANIZES words by prefix
→ All words starting with "app" are in the subtree under the "app" node
→ We don't need to check other words
→ The Trie structure PRUNES the search space automatically
→ This is the same principle as binary search (Project 9):
     → Organized data enables elimination of irrelevant elements
     → Binary search eliminates half → Trie eliminates all non-matching prefixes

The Relationship Between Trie and Other Tree Structures

Binary Tree (Projects 17-18):
  → Each node has ≤ 2 children
  → Stores values in nodes
  → Used for searching, sorting, hierarchical data

Binary Search Tree:
  → Binary tree with ordering invariant
  → Left < parent < right
  → O(log n) search

Trie (This project):
  → Each node has ≤ 26 children (or alphabet size)
  → Stores characters on EDGES, metadata on NODES
  → Used for STRING operations
  → O(L) search where L = string length
  → No comparison between words needed

The progression:
  Binary Tree → general hierarchical storage
  BST → ordered storage with efficient search
  Trie → string storage with efficient prefix operations
  
  Each specializes the tree structure for a different use case

⚠️ COMMON PITFALLS AND FAILURE MODES

Pitfall 1: Forgetting isEndOfWord Check in search()

Words inserted: "apple"

search("app"):
  Traverse: root → a → p → p → arrives at a valid node
  
WRONG: return true (node exists!)
RIGHT: return node.isEndOfWord (false — "app" was never inserted)

"app" is a PREFIX of "apple" but not a WORD itself
→ The node exists (because "apple" passes through it)
→ But isEndOfWord is false (no word ends here)
→ search() must check this flag
→ startsWith() does NOT check this flag

Pitfall 2: Off-by-One in Index Calculation

WRONG: int index = c;           // ASCII value, not 0-25 range
WRONG: int index = c - 'A';     // uppercase offset, not lowercase
RIGHT: int index = c - 'a';     // correct for lowercase a-z

If input can include uppercase:
  → Convert to lowercase first: c = Character.toLowerCase(c)
  → Or use separate ranges for upper and lower

Pitfall 3: Not Creating Root Node

WRONG:
  class Trie {
      TrieNode root;  // null!
      void insert(String word) {
          TrieNode node = root;  // NullPointerException!
          ...
      }
  }

RIGHT:
  class Trie {
      TrieNode root;
      Trie() {
          root = new TrieNode();  // Initialize empty root!
      }
      ...
  }

Pitfall 4: Modifying Trie During Search

search() and startsWith() should be READ-ONLY operations
→ They should NOT create new nodes
→ They should NOT modify isEndOfWord

WRONG (in search):
  if (node.children[index] == null) {
      node.children[index] = new TrieNode();  // DON'T create during search!
  }

RIGHT (in search):
  if (node.children[index] == null) {
      return false;  // Path doesn't exist → word not found
  }

Pitfall 5: Memory Leak with Unused Nodes

If you implement delete():
  → Setting isEndOfWord = false is NOT enough
  → The nodes along the path may now be useless
  → Example: insert("apple"), then delete("apple")
     → All 5 nodes are now useless (no word passes through them)
     → They should be removed to free memory
  
  → But if "app" was also inserted:
     → Only nodes for 'l' and 'e' should be removed
     → Nodes for 'a', 'p', 'p' are still needed
  
  → Proper deletion requires checking if a node has any children
     or isEndOfWord after unmarking
  → This is complex → many implementations skip it
  → For interview problems → lazy deletion (just unmark) is usually fine

📈 SCALABILITY CONSIDERATIONS

TIME COMPLEXITY:
→ insert(word):       O(L) where L = word length
→ search(word):       O(L) where L = word length
→ startsWith(prefix): O(L) where L = prefix length
→ ALL operations are independent of N (number of words stored)
→ This is the key advantage over HashSet for prefix operations

SPACE COMPLEXITY:
→ Worst case: O(N × L × ALPHABET_SIZE) where N = words, L = avg length
     → Every word shares no prefix → maximum nodes
     → Each node has array of 26 pointers
→ Best case: O(L × ALPHABET_SIZE) where all words share one prefix
     → Example: "a", "aa", "aaa", "aaaa" → only L nodes
→ Practical case: somewhere in between
     → English words share many prefixes (un-, re-, pre-, etc.)
     → Significant memory savings over storing full strings

MEMORY OPTIMIZATION OPTIONS:
→ Array[26] per node: fast but wastes space for sparse nodes
→ HashMap per node: space-efficient but slower
→ Compressed Trie (Radix Tree): merge single-child chains
     → "apple" with no branching → single node "apple"
     → Saves nodes but complicates implementation
→ Ternary Search Tree: binary search within each node's children
     → Middle ground between Trie and BST

PRACTICAL LIMITS:
→ 10,000 words × avg 10 chars → ~100,000 nodes → ~20 MB → fine
→ 1,000,000 words × avg 10 chars → ~10M nodes → ~2 GB → tight
→ For very large dictionaries → consider compressed tries or other structures

🧠 KEY TAKEAWAY

WHAT WE LEARNED:
→ Trie = tree where each edge represents one character
→ Shared prefixes share nodes → memory efficient for related strings
→ Three core operations:
     insert(word):       O(L) → create nodes along the path
     search(word):       O(L) → traverse + check isEndOfWord
     startsWith(prefix): O(L) → traverse only (no end check)
→ The isEndOfWord flag is CRITICAL — distinguishes words from prefixes
→ Trie dominates HashSet for PREFIX operations

THE TRIE MENTAL MODEL:
→ Think of it as a HIGHWAY SYSTEM for words
→ Words that share a beginning share the same road
→ At each fork, you choose which character to follow
→ End-of-word markers are like DESTINATION signs
→ You can check "does this road exist?" without traveling all roads

TRIE vs OTHER DATA STRUCTURES:

┌──────────────────┬──────────┬─────────────┬─────────────┐
│ Operation        │ Array    │ HashSet     │ Trie        │
├──────────────────┼──────────┼─────────────┼─────────────┤
│ Insert           │ O(1)*    │ O(L) avg    │ O(L)        │
│ Exact Search     │ O(N×L)   │ O(L) avg    │ O(L)        │
│ Prefix Search    │ O(N×L)   │ O(N×L)      │ O(L)        │
│ Autocomplete     │ O(N×L)   │ O(N×L)      │ O(L+K)      │
│ Sorted Iteration │ O(NlogN) │ O(NlogN)    │ O(N) free!  │
│ Wildcard Search  │ O(N×L)   │ O(N×L)      │ O(26^d × L) │
└──────────────────┴──────────┴─────────────┴─────────────┘
  N=words, L=length, K=result size, d=wildcards

→ Trie wins for PREFIX and PATTERN operations
→ HashSet wins for pure EXACT match (simpler, less memory)
→ Choose based on what operations you need

→ The Trie PRUNES paths that can't lead to any word

The combination:
  Project 58: Trie as standalone data structure

  → Build Trie from word list
  → DFS on grid, following Trie edges
  → At each grid cell, check: does Trie have this character?
  → If yes → continue DFS along Trie path
  → If no → prune this branch (backtrack)
  → If isEndOfWord → found a word! Add to results
