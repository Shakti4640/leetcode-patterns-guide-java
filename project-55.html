🔍 HOW TO APPROACH IT

Step 1 — Understand What Traps Water

→ Water can sit on TOP of a bar only if TALLER bars exist on BOTH sides
→ The water level at position i is determined by:
     → The tallest bar to the LEFT of i (including i's own height)
     → The tallest bar to the RIGHT of i (including i's own height)
     → Water level = min(maxLeft, maxRight)
     → Water trapped at i = waterLevel - height[i]
     → If waterLevel ≤ height[i] → no water trapped (bar sticks out)

This is the FUNDAMENTAL INSIGHT:
     water[i] = max(0, min(maxLeft[i], maxRight[i]) - height[i])

Step 2 — Contrast with Project 54

Project 54 (Largest Rectangle):
→ Each bar expands until it hits a SHORTER bar
→ Bounded by NEAREST SMALLER elements
→ INCREASING stack → pop on SMALLER
→ Computes AREA of solid rectangle

Project 55 (Trapping Rain Water):
→ Water fills VALLEYS between TALLER bars
→ Bounded by NEAREST GREATER elements (walls on both sides)
→ DECREASING stack → pop on GREATER
→ Computes VOLUME of water in valley

KEY FLIP:
  Project 54: "How far does this tall bar extend?" → INCREASING stack
  Project 55: "How deep is this valley?" → DECREASING stack

Step 3 — Four Ways to Solve This

Approach A: Brute Force → O(n²) time, O(1) space
→ For each position, scan left and right to find max heights
→ Simple but slow

Approach B: Prefix Max Arrays → O(n) time, O(n) space
→ Precompute maxLeft[] and maxRight[] arrays
→ Then compute water at each position in one pass

Approach C: Monotonic Stack → O(n) time, O(n) space
→ Use decreasing stack to detect valleys
→ Compute water layer by layer when popping

Approach D: Two Pointers → O(n) time, O(1) space ← OPTIMAL
→ Converge from both ends (callback to Project 1!)
→ Track running maxLeft and maxRight
→ Process the side with the smaller max

Each approach reveals a DIFFERENT way of thinking
→ All produce the same answer
→ I will implement ALL FOUR

🛠️ HOW CAN WE SOLVE IT

Approach A — Brute Force Logic

For each index i:
  1. Scan LEFT from i to 0 → find max height
  2. Scan RIGHT from i to n-1 → find max height
  3. Water at i = min(maxLeft, maxRight) - height[i]
  4. If negative → clamp to 0

Time: O(n) per position × O(n) positions = O(n²)
Space: O(1)

Approach B — Prefix Max Arrays Logic

1. Build maxLeft[] scanning left to right
     maxLeft[0] = height[0]
     maxLeft[i] = max(maxLeft[i-1], height[i])

2. Build maxRight[] scanning right to left
     maxRight[n-1] = height[n-1]
     maxRight[i] = max(maxRight[i+1], height[i])

3. For each i:
     water += min(maxLeft[i], maxRight[i]) - height[i]

Time: O(n)
Space: O(n) for the two arrays

Approach C — Monotonic Stack Logic (Layer-by-Layer)

KEY DIFFERENCE from Project 54:
→ Project 54 used INCREASING stack (pop on smaller) for rectangles
→ Here we use DECREASING stack (pop on taller) for water valleys

The DECREASING stack stores indices of bars in decreasing height order
→ When a TALLER bar arrives → it forms a RIGHT WALL
→ The popped element is the VALLEY BOTTOM
→ The new stack top (after pop) is the LEFT WALL
→ Water trapped = width × (min(leftWall, rightWall) - valleyBottom)

This computes water LAYER BY LAYER horizontally
→ Not column by column like prefix approach
→ Same total — different decomposition

Approach D — Two Pointers Logic (Optimal)

KEY INSIGHT:
→ Water at position i = min(maxLeft, maxRight) - height[i]
→ We don't need to know BOTH maxLeft and maxRight simultaneously
→ We only need to know the SMALLER of the two

→ If maxLeft < maxRight:
     → min(maxLeft, maxRight) = maxLeft
     → Water at left position is determined ENTIRELY by maxLeft
     → We don't care what maxRight actually is — it's bigger
     → Process the LEFT pointer

→ If maxRight ≤ maxLeft:
     → min(maxLeft, maxRight) = maxRight
     → Water at right position is determined ENTIRELY by maxRight
     → Process the RIGHT pointer

ALGORITHM:
  left = 0, right = n-1
  maxLeft = 0, maxRight = 0
  
  while left < right:
    if height[left] < height[right]:
      → The right side has a bar ≥ height[right]
      → So maxRight ≥ height[right] > height[left]
      → Water at left is bounded by maxLeft
      maxLeft = max(maxLeft, height[left])
      water += maxLeft - height[left]
      left++
    else:
      → The left side has a bar ≥ height[left]
      → So maxLeft ≥ height[left] ≥ height[right]
      → Water at right is bounded by maxRight
      maxRight = max(maxRight, height[right])
      water += maxRight - height[right]
      right--

Time: O(n)
Space: O(1) ← OPTIMAL

✅ WHAT SHOULD YOU DO

→ ALWAYS start by understanding the water formula:
     water[i] = min(maxLeft[i], maxRight[i]) - height[i]
     This is the FOUNDATION — every approach is a different way to compute this

→ ALWAYS pick the approach based on constraints:
     → Need O(1) space? → Two Pointers (Approach D)
     → Need clearest code? → Prefix Arrays (Approach B)
     → Need to demonstrate stack mastery? → Monotonic Stack (Approach C)
     → Interview setting? → Explain prefix first, then optimize to two pointers

→ ALWAYS handle edge cases:
     → Empty array → return 0
     → Array of length 1 or 2 → cannot trap water → return 0
     → All bars same height → no valleys → return 0
     → Strictly ascending or descending → no valleys → return 0

→ ALWAYS verify with the column-by-column mental model:
     → For each position, ask "how much water sits HERE?"
     → This catches off-by-one errors in stack or pointer logic

🚫 WHAT SHOULD YOU NOT DO

→ DO NOT confuse with Project 54 (Largest Rectangle)
     → Rectangle: INCREASING stack → pop on SMALLER → area of solid bar
     → Water: DECREASING stack → pop on TALLER → volume of water valley
     → Using the wrong stack direction = wrong answer

→ DO NOT forget the valley bottom subtraction in stack approach
     → Water height = min(leftWall, rightWall) - bottomHeight
     → NOT just min(leftWall, rightWall)
     → The valley bottom is already occupied by a bar — no water there
     → Forgetting this DOUBLE-COUNTS the bar height as water

→ DO NOT confuse "height at position" with "water at position"
     → height[i] is the bar itself
     → Water sits ON TOP of the bar
     → Total at position i = height[i] + water[i] = min(maxLeft, maxRight)
     → We want JUST the water part

→ DO NOT process both pointers simultaneously in two-pointer approach
     → Process ONLY the side with the smaller height
     → The other side guarantees a wall exists
     → Moving both loses the guarantee

→ DO NOT use an INCREASING stack for this problem
     → Increasing stack finds "Next Smaller" (Project 54)
     → We need "Next Greater" to find valley WALLS
     → DECREASING stack finds taller bars that form walls

🔀 WHEN TO USE THIS PATTERN

✓ Problem involves TRAPPING or CONTAINING between boundaries
✓ Each position's value depends on the MAXIMUM on both sides
✓ The answer decomposes into per-position contributions
✓ Data is a 1D elevation map or similar bounded structure

SIGNATURE PROBLEMS:
→ Trapping Rain Water (1D) ← THIS
→ Trapping Rain Water II (2D) → BFS + heap extension
→ Container With Most Water → two pointers variant
→ Pour Water → simulation variant
→ Any problem asking "how much fits BETWEEN elements"

🚫 WHEN NOT TO USE THIS PATTERN

✗ Boundaries are not defined by MAX on both sides
     → If water level depends on something other than max
     → The min(maxLeft, maxRight) formula doesn't apply

✗ The trapped quantity can "leak" through gaps
     → This problem assumes bars are solid walls
     → If bars have holes → different problem entirely

✗ You need the shape of the water, not just the volume
     → This approach computes total volume efficiently
     → Recovering which cells have water needs additional work

✗ 2D grid version
     → 1D techniques don't directly generalize to 2D
     → 2D Trapping Rain Water uses BFS + min-heap (different approach)

⚙️ HOW IT ONLY WORKS — Internal Mechanics

The Water Formula — Why min(maxLeft, maxRight)?

Think physically:
→ Water at position i rises until it spills over the SHORTER wall
→ The left wall is the tallest bar to the left → maxLeft
→ The right wall is the tallest bar to the right → maxRight
→ Water rises to the height of the SHORTER of these two walls
→ Because it would spill over the shorter one

     maxLeft = 5        maxRight = 3
     ┌───┐                ┌───┐
     │   │                │   │
     │   │   water level  │   │
     │   │ ═══════════════│   │ ← min(5, 3) = 3
     │   │ ░░░░░░░░░░░░░░│   │
     │   │ ░░░░░░░░░░░░░░│   │
     │   │ ░░░░░bar░░░░░░│   │
     └───┴───────┴────────┴───┘

Water level = 3 (limited by shorter wall)
Water at bar position = 3 - bar_height

The Stack Approach — Layer-by-Layer Decomposition

The stack approach computes water HORIZONTALLY, layer by layer
→ Each pop event computes one horizontal slice of water

Example valley:
     ┌───┐               ┌───┐
     │   │   ┌───┐       │   │
     │   │   │   │       │   │
     │   │   │   │       │   │
     │   ├───┤   ├───┐   │   │
     │   │ ░ │   │░░░│   │   │
     │   │░░░│   │░░░│░░░│   │
     └───┴───┴───┴───┴───┴───┘
      h=5 h=2 h=4 h=1 h=2 h=6

Pop 1: bottom=h=1, left=h=4, right=h=2 → wait, let me re-examine...

Actually the stack processes this as:
  Push 5, 2 → decreasing ✓
  At 4: pop 2 (bottom=2, left=5, right=4) → water = (min(5,4)-2) × 1 = 2
  Push 4 → stack: [5, 4]
  Push 1 → stack: [5, 4, 1]
  At 2: pop 1 (bottom=1, left=4, right=2) → water = (min(4,2)-1) × 1 = 1
  2 ≤ 4 → stop, push 2 → stack: [5, 4, 2]
  At 6: pop 2 (bottom=2, left=4, right=6) → water = (min(4,6)-2) × 2 = 4
        pop 4 (bottom=4, left=5, right=6) → water = (min(5,6)-4) × 4 = 4
        pop 5 (bottom=5, stack empty) → no water
  
Total = 2 + 1 + 4 + 4 = 11

This is the HORIZONTAL SLICING:
→ First pop: fills the bottom layer of a narrow valley
→ Later pops: fill wider layers above
→ All layers sum to the correct total

The Two Pointers — Why Processing the Smaller Side Works

INVARIANT:
  At any point, maxLeft tracks the max from the left side up to left pointer
  maxRight tracks the max from the right side up to right pointer

When height[left] < height[right]:
  → We KNOW there exists a bar of height ≥ height[right] on the right
  → So maxRight will eventually be ≥ height[right] > height[left]
  → Therefore: min(maxLeft, future_maxRight) = maxLeft
  → We can safely compute water at left using ONLY maxLeft
  → We don't need to know the exact maxRight

When height[right] ≤ height[left]:
  → Symmetric argument
  → min(future_maxLeft, maxRight) = maxRight
  → Compute water at right using ONLY maxRight

THIS IS WHY O(1) SPACE IS POSSIBLE:
  → We don't need to precompute both arrays
  → At each step, we KNOW which side is the bottleneck
  → Process that side immediately with the information we have

Why All Four Approaches Give the Same Answer

ALL approaches compute the same thing:
  Total water = Σ max(0, min(maxLeft[i], maxRight[i]) - height[i])

Brute Force: Computes maxLeft[i] and maxRight[i] per position by scanning
Prefix Arrays: Precomputes maxLeft[] and maxRight[] in advance
Monotonic Stack: Decomposes the same total into horizontal layers
Two Pointers: Computes the per-position contribution on the fly

Different decompositions of the SAME sum
→ Column-by-column (prefix, brute, two-pointer)
→ Layer-by-layer (stack)
→ Both sum to the same total

⚠️ COMMON PITFALLS AND FAILURE MODES

Pitfall 1: Negative Water Values

→ If height[i] > min(maxLeft, maxRight) → bar sticks ABOVE water level
→ Water at this position = 0, NOT negative
→ ALWAYS clamp to 0

WRONG: water += min(maxLeft, maxRight) - height[i];
RIGHT: water += Math.max(0, min(maxLeft, maxRight) - height[i]);

Note: With proper maxLeft/maxRight computation (including position i),
this situation means maxLeft or maxRight IS height[i],
so the formula gives 0 naturally.
But be careful with exclusive vs inclusive boundary definitions.

Pitfall 2: Stack — Forgetting Bottom Height Subtraction

When popping from stack:
→ Water layer height = min(leftWall, rightWall) - bottomHeight
→ NOT just min(leftWall, rightWall)

WRONG:
  int waterHeight = Math.min(leftHeight, rightHeight);
  water += waterHeight * width;

RIGHT:
  int waterHeight = Math.min(leftHeight, rightHeight) - bottomHeight;
  water += waterHeight * width;

Without subtracting bottomHeight:
→ You count the bar itself as water
→ You also double-count lower layers already computed

Pitfall 3: Stack — Empty Stack After Pop

After popping the bottom element:
→ If stack is empty → no left wall exists → no water can be trapped
→ MUST check stack.isEmpty() before accessing left wall

WRONG:
  int bottom = stack.pop();
  int leftWall = heights[stack.peek()];  // CRASH if stack empty

RIGHT:
  int bottom = stack.pop();
  if (stack.isEmpty()) continue;  // no left wall → skip
  int leftWall = heights[stack.peek()];

Pitfall 4: Two Pointers — Wrong Update Order

Must update maxLeft/maxRight BEFORE computing water
→ Because the current bar itself could be the new max
→ If current bar IS the max → water = max - height = 0 (correct)

WRONG ORDER:
  water += maxLeft - height[left];  // maxLeft might be stale
  maxLeft = Math.max(maxLeft, height[left]);

CORRECT ORDER:
  maxLeft = Math.max(maxLeft, height[left]);
  water += maxLeft - height[left];

Pitfall 5: Confusing This with "Container With Most Water"

Trapping Rain Water:
→ Bars are SOLID walls of given height and width 1
→ Water fills ALL valleys between bars
→ Compute TOTAL water across entire histogram

Container With Most Water:
→ Bars are vertical LINES (no width)
→ Pick TWO lines that form a container
→ Compute MAX water for a single container
→ Different problem, different formula, different approach

Both use two pointers but with DIFFERENT logic

🔍 INTERPRETABILITY CONSIDERATIONS

→ Prefix Array approach is the MOST intuitive:
     → "Precompute the tallest bar to the left of each position"
     → "Precompute the tallest bar to the right of each position"
     → "Water at each position is limited by the shorter of these two"
     → Easy to verify by hand, easy to debug

→ Two Pointer approach is the MOST elegant:
     → "Process the side with the shorter maximum"
     → "That side's water level is fully determined"
     → Harder to prove correct but beautiful once understood

→ Stack approach is the MOST complex:
     → Layer-by-layer horizontal slicing is non-obvious
     → Requires understanding of what each pop represents
     → Best for demonstrating stack pattern mastery

→ For interviews: START with prefix explanation, CODE two pointers
→ For competitive programming: CODE two pointers (fastest to write, optimal)
→ For learning: IMPLEMENT all four, verify they give same answer

KEY DISTINCTION BETWEEN PROJECT 54 AND 55:

Project 54 (Rectangle):
→ Bar EXPANDS outward until it meets something SHORTER
→ "How far can this tall thing reach?"
→ INCREASING stack → pop when SHORTER bar arrives
→ Popped bar is the HEIGHT of the rectangle
→ Boundaries are SHORTER bars on both sides

Project 55 (Water):
→ Valley FILLS upward until it meets something TALLER
→ "How deep is this low thing?"
→ DECREASING stack → pop when TALLER bar arrives
→ Popped bar is the BOTTOM of the valley
→ Walls are TALLER bars on both sides

THEY ARE MIRROR IMAGES:
→ Rectangle: tall bar bounded by short bars → area under
→ Water: short bar bounded by tall bars → volume above
→ Same stack machinery → opposite direction → opposite geometry

THE TWO-POINTER CALLBACK:

Project 1:  Two pointers → pair sum → move based on sum comparison
Project 55: Two pointers → water → move based on height comparison

The SAME convergence framework from Project 1 reappears here:
→ Two pointers at opposite ends
→ Move inward based on a comparison
→ Each step makes irreversible progress
→ O(n) time, O(1) space

But the DECISION LOGIC is different:
→ Project 1: "sum too big → move right, sum too small → move left"
→ Project 55: "left height smaller → process left, right height smaller → process right"

This demonstrates that Two Pointers is a UNIVERSAL FRAMEWORK
→ The skeleton (converge from both ends) stays constant
→ The decision logic (what to compare, how to act) changes per problem
→ Mastering the framework means you only need to figure out the decision
