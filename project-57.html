🧩 THE HYPOTHETICAL PROBLEM

→ You are given a list of ACCOUNTS
→ Each account is a list of strings:
     → First string = person's NAME
     → Remaining strings = person's EMAIL addresses
→ Two accounts belong to the SAME person if they share at least one email
→ MERGE all accounts belonging to the same person
→ Return merged accounts with:
     → Name (first element)
     → All unique emails SORTED alphabetically

Example:
  Input:
    accounts = [
      ["John", "john@mail.com", "john_work@mail.com"],
      ["John", "john@mail.com", "john00@mail.com"],
      ["Mary", "mary@mail.com"],
      ["John", "johnny@mail.com"]
    ]

  Analysis:
  → Account 0 and Account 1 share "john@mail.com"
  → So they belong to the SAME person
  → Account 2 has no shared email with anyone → separate person
  → Account 3 has name "John" but NO shared email → separate person
  → (Same name does NOT mean same person!)

  Output:
    [
      ["John", "john00@mail.com", "john@mail.com", "john_work@mail.com"],
      ["Mary", "mary@mail.com"],
      ["John", "johnny@mail.com"]
    ]

  → Account 0 and 1 merged → 3 unique emails sorted
  → Account 2 unchanged
  → Account 3 unchanged (different John!)

🔍 HOW TO APPROACH IT

Step 1 — Understand the Grouping Relationship

→ Two accounts are CONNECTED if they share an email
→ Connection is TRANSITIVE:
     → Account A shares email with Account B
     → Account B shares email with Account C
     → Then A, B, C are ALL the same person
     → Even if A and C share NO direct email

Example:
  Account 0: [email1, email2]
  Account 1: [email2, email3]
  Account 2: [email3, email4]

  → 0 and 1 share email2 → same person
  → 1 and 2 share email3 → same person
  → Therefore 0, 1, 2 are ALL the same person
  → Merged: [email1, email2, email3, email4]

This is TRANSITIVE CLOSURE → classic UNION FIND territory

Step 2 — Why Union Find?

Recall Project 48:
→ Union Find groups elements into connected components
→ union(a, b) → merge the groups containing a and b
→ find(a) → return the representative (root) of a's group
→ Two elements are in the same group iff find(a) == find(b)

For Accounts Merge:
→ Each EMAIL is an element
→ If two emails appear in the same account → union them
→ After all unions → emails with the same root = same person
→ Collect all emails per root → that's one merged account

WHY NOT use BFS/DFS instead?
→ BFS/DFS also finds connected components
→ But Union Find is more NATURAL here:
     → We process accounts one at a time
     → Each account gives us "these emails are connected"
     → Union Find handles incremental merging perfectly
     → BFS/DFS would require building full adjacency list first
→ Both work, but Union Find is cleaner for this problem

Step 3 — The Key Mapping Challenge

Project 48 used Union Find on INTEGERS (node indices 0 to n-1)
→ Parent array: int[] parent = new int[n]
→ find(3) → returns root index

This problem has STRINGS (email addresses) as elements
→ Cannot use int[] parent
→ Need: Map<String, String> parent
→ find("john@mail.com") → returns root email string

Additionally:
→ We need to map each email to the account NAME
→ Because the output needs the person's name
→ Use: Map<String, String> emailToName

This is the progression from Project 48:
  Project 48: Union Find on integers → array-based → simple
  Project 57: Union Find on strings → map-based → flexible

Step 4 — The Three Phases

Phase 1: BUILD Union Find relationships
→ For each account:
     → The first email is the "anchor"
     → Union every other email in this account with the anchor
     → Map every email to the account name

Phase 2: GROUP emails by their root
→ For each email encountered:
     → Find its root via find()
     → Add it to the root's group

Phase 3: FORMAT output
→ For each group:
     → Sort the emails
     → Prepend the name
     → Add to result list

🛠️ HOW CAN WE SOLVE IT

Full Algorithm — Detailed Steps

STEP 1: Initialize Union Find (Map-based)
→ parent: Map<String, String> → each email points to its parent
→ rank: Map<String, Integer> → for union by rank optimization
→ emailToName: Map<String, String> → email → person name

STEP 2: Process each account
→ For account = [name, email1, email2, email3, ...]:
     → emailToName[email1] = name
     → emailToName[email2] = name
     → emailToName[email3] = name
     → Initialize parent[email1] = email1 (if new)
     → Initialize parent[email2] = email2 (if new)
     → Initialize parent[email3] = email3 (if new)
     → union(email1, email2)
     → union(email1, email3)
     → (union all with the first email as anchor)

STEP 3: Group emails by root
→ components: Map<String, List<String>>
→ For each email in parent.keySet():
     → root = find(email)
     → components[root].add(email)

STEP 4: Build result
→ For each root in components:
     → emails = components[root]
     → Sort emails alphabetically
     → name = emailToName[root]
     → result.add([name] + emails)

✅ WHAT SHOULD YOU DO

→ ALWAYS use the FIRST email in each account as the union anchor
     → Union every subsequent email WITH the first one
     → This creates a star topology per account
     → Minimizes the number of union operations
     → Any email in the account works as anchor — first is convention

→ ALWAYS initialize parent[email] = email for NEW emails
     → Union Find requires every element to start as its own root
     → Check if email already exists before initializing
     → If it exists → it's already in the structure from a previous account

→ ALWAYS use path compression in find()
     → Without it → find() degrades to O(n)
     → With it → amortized nearly O(1)
     → Same optimization as Project 48

→ ALWAYS map email → name BEFORE doing unions
     → After unions, root might change
     → We need to know the name for any email
     → Store the mapping during account processing

→ ALWAYS sort emails within each merged group
     → Problem requires alphabetical order
     → Sort AFTER grouping, BEFORE building output
     → Collections.sort() on the email list

→ ALWAYS use find() when grouping — NOT direct parent lookup
     → parent[email] might not be the ROOT
     → It might be an intermediate node
     → find(email) with path compression gives the actual root

🚫 WHAT SHOULD YOU NOT DO

→ DO NOT merge accounts by NAME
     → Same name ≠ same person
     → "John" can be multiple different people
     → Only SHARED EMAILS indicate same person
     → The name is just metadata attached to accounts

→ DO NOT use integer-based Union Find directly
     → Emails are strings, not integers
     → You COULD map emails to integers and use array-based UF
     → But Map-based UF is cleaner and more readable
     → Both work — choose based on clarity preference

→ DO NOT union emails ACROSS accounts without shared emails
     → Account 0: [email1, email2]
     → Account 1: [email3, email4]
     → These are separate unless email1==email3, email1==email4, etc.
     → Union only happens WITHIN an account's email list
     → Cross-account merging happens AUTOMATICALLY through Union Find
        when the same email appears in multiple accounts

→ DO NOT forget to handle single-email accounts
     → Account with one email → no union operations
     → But the email still needs to be in the output
     → It forms its own group of size 1

→ DO NOT sort the final result list by name
     → Problem typically says "return in any order"
     → Sort only the EMAILS within each group
     → The groups themselves can be in any order

→ DO NOT skip path compression thinking "it's just a small input"
     → Without path compression → worst case O(n) per find
     → With it → amortized O(α(n)) ≈ O(1)
     → Always implement it — it's one extra line

🔀 WHEN TO USE THIS PATTERN

✓ You need to GROUP elements by shared properties
✓ The grouping is TRANSITIVE (A-B connected, B-C connected → A-C connected)
✓ Elements arrive INCREMENTALLY (process one account at a time)
✓ The key question is "which elements belong together?"

SIGNATURE PROBLEMS:
→ Accounts Merge ← THIS
→ Number of Connected Components (Project 48)
→ Redundant Connection (find the extra edge in a tree)
→ Number of Islands II (dynamic grid connectivity)
→ Sentence Similarity II (transitive word similarity)
→ Smallest String with Swaps
→ Any problem with "merge groups that share an element"

THE PATTERN:
→ "Same X means same group" → Union Find
→ Where X can be: email, word, cell, node, etc.

🚫 WHEN NOT TO USE THIS PATTERN

✗ You need ORDERING, not grouping
     → Ordering = directed relationships → Topological Sort (Project 56)
     → Grouping = undirected relationships → Union Find

✗ Groups have HIERARCHY (parent-child structure)
     → Union Find creates FLAT groups (all equal)
     → If you need tree structure → use actual tree/graph

✗ You need to SPLIT groups (un-merge)
     → Union Find is designed for MERGING only
     → Splitting/disconnecting is extremely expensive
     → Use different data structure (link-cut trees, etc.)

✗ The relationships are NOT transitive
     → "A is friends with B" and "B is friends with C"
     → Does NOT mean "A is friends with C" in some problems
     → Union Find assumes transitivity — will over-merge

✗ You need the actual PATH between elements
     → Union Find tells you IF two elements are connected
     → It does NOT tell you HOW they are connected
     → For path finding → use BFS/DFS

⚙️ HOW IT ONLY WORKS — Internal Mechanics

Map-Based Union Find vs Array-Based Union Find

PROJECT 48 (Array-Based):
  int[] parent = new int[n];
  int[] rank = new int[n];
  
  // Initialize
  for (int i = 0; i < n; i++) parent[i] = i;
  
  // Find
  int find(int x) {
      if (parent[x] != x) parent[x] = find(parent[x]);
      return parent[x];
  }
  
  // Union
  void union(int x, int y) {
      int rx = find(x), ry = find(y);
      // merge by rank
  }

PROJECT 57 (Map-Based):
  Map<String, String> parent = new HashMap<>();
  Map<String, Integer> rank = new HashMap<>();
  
  // Initialize (on demand)
  void init(String x) {
      if (!parent.containsKey(x)) {
          parent.put(x, x);
          rank.put(x, 0);
      }
  }
  
  // Find (identical logic, different data structure)
  String find(String x) {
      if (!parent.get(x).equals(x)) {
          parent.put(x, find(parent.get(x)));
      }
      return parent.get(x);
  }
  
  // Union (identical logic)
  void union(String x, String y) {
      String rx = find(x), ry = find(y);
      // merge by rank
  }

THE LOGIC IS IDENTICAL
→ Only the data structure changes (array → map)
→ This is the power of Union Find as an ABSTRACT pattern
→ The algorithm doesn't care what the elements ARE
→ It only cares about find() and union() operations

How Cross-Account Merging Happens Automatically

Account 0: ["John", "A@", "B@"]
Account 1: ["John", "B@", "C@"]
Account 2: ["John", "C@", "D@"]

Processing Account 0:
  → init(A@), init(B@)
  → union(A@, B@) → parent: {A@: A@, B@: A@}
  → Group: {A@, B@}

Processing Account 1:
  → B@ already exists!
  → init(C@)
  → union(B@, C@) → find(B@)=A@, find(C@)=C@ → merge: C@ → A@
  → parent: {A@: A@, B@: A@, C@: A@}
  → Group: {A@, B@, C@}

Processing Account 2:
  → C@ already exists!
  → init(D@)
  → union(C@, D@) → find(C@)=A@, find(D@)=D@ → merge: D@ → A@
  → parent: {A@: A@, B@: A@, C@: A@, D@: A@}
  → Group: {A@, B@, C@, D@}

ALL FOUR EMAILS END UP IN THE SAME GROUP
→ Even though A@ and D@ were never in the same account
→ The chain A@-B@ (account 0) + B@-C@ (account 1) + C@-D@ (account 2)
→ Transitively connects them all
→ Union Find handles this AUTOMATICALLY through find() root tracking

Why "Anchor to First Email" Strategy Works

Account: ["Name", "E1", "E2", "E3", "E4"]

Strategy: union(E1, E2), union(E1, E3), union(E1, E4)
→ Creates a STAR: E1 is center, all others point to E1
→ 3 union operations

Alternative: union(E1, E2), union(E2, E3), union(E3, E4)
→ Creates a CHAIN: E1 → E2 → E3 → E4
→ 3 union operations (same count)
→ But find() needs to traverse the chain (longer paths)
→ Path compression eventually flattens it, but more work initially

Alternative: union ALL pairs
→ union(E1,E2), union(E1,E3), union(E1,E4), union(E2,E3), ...
→ C(n,2) = n*(n-1)/2 operations → wasteful
→ Many are redundant (already in same group)

STAR STRATEGY IS OPTIMAL:
→ n-1 union operations (minimum to connect n elements)
→ Short paths from the start (star topology)
→ Clean and simple code

Post-Union Grouping — The Collection Phase

After all unions are done, we need to COLLECT groups.

APPROACH: Iterate over all emails, find root, group by root

for (String email : parent.keySet()) {
    String root = find(email);  // WITH path compression
    components.computeIfAbsent(root, k -> new ArrayList<>()).add(email);
}

WHY find() and not parent.get()?
→ parent.get(email) might return an INTERMEDIATE node
→ Only find(email) with path compression returns the TRUE ROOT
→ Two emails in the same group might have different parent values
→ But find() on both returns the SAME root

Example:
  parent: {A: A, B: A, C: B, D: C}
  
  parent.get(D) = C (NOT the root!)
  parent.get(C) = B (NOT the root!)
  find(D) = A ✓ (with path compression, also updates D→A)
  find(C) = A ✓ (with path compression, also updates C→A)

⚠️ COMMON PITFALLS AND FAILURE MODES

Pitfall 1: Merging by Name Instead of Email

WRONG THINKING:
  "Both accounts have name 'John' → merge them"
  
CORRECT THINKING:
  "Both accounts share email 'john@mail.com' → merge them"

Two different people can have the same name
→ Only shared EMAILS prove they're the same person
→ Name is metadata, not identity

Pitfall 2: Forgetting to Initialize New Emails

WRONG:
  void union(String x, String y) {
      String rx = find(x);  // CRASH — x not in parent map!
      ...
  }

CORRECT:
  void union(String x, String y) {
      init(x);  // Ensure x exists in parent map
      init(y);  // Ensure y exists in parent map
      String rx = find(x);
      ...
  }

OR initialize all emails BEFORE any union operations
→ Process each account: first add all emails, then union them

Pitfall 3: Using parent.get() Instead of find() for Grouping

After all unions:
  parent = {A: A, B: A, C: B, D: B}
  
Using parent.get():
  A → group A: [A]
  B → group A: [A, B]
  C → group B: [C]     ← WRONG! C should be in group A
  D → group B: [C, D]  ← WRONG! D should be in group A

Using find():
  find(A) = A → group A: [A]
  find(B) = A → group A: [A, B]
  find(C) = A → group A: [A, B, C]     ✓
  find(D) = A → group A: [A, B, C, D]  ✓

ALWAYS use find() for grouping — NEVER raw parent lookup

Pitfall 4: Not Sorting Emails in Output

Problem requires emails sorted alphabetically within each group.

WRONG: Return emails in whatever order they were discovered
CORRECT: Collections.sort(emailList) before adding to result

This is easy to forget but will fail test cases

Pitfall 5: Losing the Name Mapping

After all unions, we need the NAME for each group.
→ The root email might not be the "first" email of any account
→ We need a reliable email → name mapping

SOLUTION: Store emailToName for EVERY email encountered
→ Every email maps to its account's name
→ When building output: name = emailToName.get(rootEmail)
→ Since all emails in the same group belong to the same person
→ Any email's name will give the correct name

Pitfall 6: Handling Accounts with One Email

Account: ["Mary", "mary@mail.com"]

→ Only one email → no union operation needed
→ But email must still be initialized in parent map
→ And must appear in output as its own group

If you only process emails during union operations:
→ Single-email accounts might be skipped entirely
→ ALWAYS initialize emails in a separate loop before unions

📈 SCALABILITY CONSIDERATIONS

Let:
  N = number of accounts
  K = maximum emails per account
  E = total number of emails across all accounts
  U = number of unique emails

TIME COMPLEXITY:
→ Step 1 (initialize): O(E) → process all emails
→ Step 2 (unions): O(E × α(U)) → one union per email pair
     → α(U) is inverse Ackermann → practically O(1)
     → So effectively O(E)
→ Step 3 (grouping): O(U × α(U)) → one find per unique email
     → Effectively O(U)
→ Step 4 (sorting): O(U log U) → sort emails within groups
     → This is actually the bottleneck!
→ TOTAL: O(E + U log U)

SPACE COMPLEXITY:
→ Parent map: O(U) unique emails
→ Rank map: O(U)
→ Email-to-name map: O(U)
→ Components map: O(U)
→ TOTAL: O(U)

PRACTICAL PERFORMANCE:
→ E = 1,000 → instant
→ E = 100,000 → ~10ms (dominated by sorting)
→ E = 10,000,000 → ~1 second (sorting bottleneck)

COMPARISON WITH BFS/DFS APPROACH:
→ BFS/DFS: Build adjacency list → O(E), BFS → O(E) → Total O(E)
→ Union Find: Process unions → O(E × α), Group → O(U) → Total O(E)
→ Both are O(E) in practice
→ Union Find is often cleaner for incremental merging
→ BFS/DFS might be simpler if you're more comfortable with graph traversal
