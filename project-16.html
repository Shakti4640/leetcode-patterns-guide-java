Project 16 → Reverse a Linked List → WRITE operation on linked list
             → the Prev/Curr/Next pointer dance
             → each node's .next pointer is REDIRECTED
             → this is the most fundamental linked list MODIFICATION operation
             → building block for: palindrome, merge sort, reorder, sub-reversal

🧩 THE HYPOTHETICAL PROBLEM

→ You are given the HEAD of a singly linked list
→ REVERSE the list
→ Return the new head (which was the old tail)
→ Must be done IN-PLACE (no new list, no array conversion)

Example 1:
  Input:  1 → 2 → 3 → 4 → 5
  Output: 5 → 4 → 3 → 2 → 1

Example 2:
  Input:  1 → 2
  Output: 2 → 1

Example 3:
  Input:  1
  Output: 1

Example 4:
  Input:  null (empty list)
  Output: null

WHAT "REVERSE" MEANS AT THE POINTER LEVEL:

  BEFORE:
    1 → 2 → 3 → 4 → 5 → null
    Each node points FORWARD to the next node

  AFTER:
    null ← 1 ← 2 ← 3 ← 4 ← 5
    Each node points BACKWARD to the previous node
    The old tail (5) is now the head
    The old head (1) now points to null (it's the new tail)

→ We don't move nodes in memory
→ We only REDIRECT their .next pointers
→ The values stay in the same node objects
→ Only the LINKS between them change direction

🔍 HOW TO APPROACH IT

Step 1 — Understand What Needs to Change

Node 1: currently points to Node 2 → must point to null (new tail)
Node 2: currently points to Node 3 → must point to Node 1
Node 3: currently points to Node 4 → must point to Node 2
Node 4: currently points to Node 5 → must point to Node 3
Node 5: currently points to null   → must point to Node 4 (new head)

PATTERN:
  → Every node's .next must point to its PREVIOUS node
  → The first node's .next becomes null
  → The last node becomes the new head

TO REDIRECT node.next TO point backward:
  → We need to KNOW the previous node
  → But singly linked list has NO "previous" pointer
  → We must TRACK the previous node ourselves
  → This is where the "prev" variable comes in

Step 2 — The Fundamental Challenge

PROBLEM:
  Consider node 2: currently 2.next → 3
  We want to change: 2.next → 1 (point backward)

  But THE MOMENT we do 2.next = 1:
    → We LOSE the link to node 3
    → Node 3 is now unreachable
    → The rest of the list is GONE

VISUALIZATION:
  Before: 1 ← 2    3 → 4 → 5 → null
                ↑    ↑
                |    LOST! Can't reach 3 anymore
                |
          We set 2.next = 1

SOLUTION:
  → SAVE the reference to node 3 BEFORE redirecting 2.next
  → Store it in a temporary variable: next = 2.next (saves reference to 3)
  → THEN redirect: 2.next = 1 (safe now, we have 3 saved)
  → THEN move forward using the saved reference: current = next (move to 3)

THIS IS THE "POINTER DANCE":
  1. SAVE the next node       → next = current.next
  2. REDIRECT the pointer     → current.next = prev
  3. ADVANCE prev             → prev = current
  4. ADVANCE current          → current = next

  These four operations, in THIS ORDER, are the entire algorithm

Step 4 — Why prev Starts at null

The FIRST node of the original list becomes the LAST node of the reversed list
The last node's .next must be null (end of list)

By starting prev = null:
  → First iteration: curr.next = prev → 1.next = null
  → Node 1 correctly becomes the new tail

If prev started at something else:
  → Node 1 would point to that something → corrupted list
  → The reversed list would never terminate → potential infinite loop

prev = null is the "anchor" of the reversal
  → It represents "there is nothing before the first node"
  → After reversal: the first node points to nothing → correct tail

Step 5 — Why the Order of Operations Matters

THE CORRECT ORDER:
  1. next = curr.next      (SAVE)
  2. curr.next = prev      (REDIRECT)
  3. prev = curr           (ADVANCE prev)
  4. curr = next           (ADVANCE curr)

WHAT HAPPENS IF WE CHANGE THE ORDER?

WRONG ORDER 1: Redirect before save
  1. curr.next = prev      → we just LOST the reference to the next node
  2. next = curr.next      → this now gives us prev, NOT the original next!
  → BROKEN: we go backward instead of forward → infinite loop

WRONG ORDER 2: Advance curr before redirect
  1. next = curr.next
  2. curr = next            → we moved past the node we wanted to redirect
  3. curr.next = prev       → we're redirecting the WRONG node now
  → BROKEN: wrong node gets modified

WRONG ORDER 3: Advance prev before redirect
  1. next = curr.next
  2. prev = curr            → prev now points to curr
  3. curr.next = prev       → curr.next = curr → SELF-LOOP!
  → BROKEN: node points to itself → infinite loop

→ The order is NON-NEGOTIABLE
→ SAVE → REDIRECT → ADVANCE PREV → ADVANCE CURR
→ Each step depends on the previous step's state
→ Think of it as a careful HANDOFF sequence

🛠️ HOW CAN WE SOLVE IT

The Iterative Algorithm

1. Initialize:
     → prev = null
     → curr = head

2. WHILE curr is not null:
     a. next = curr.next        (save next before we break the link)
     b. curr.next = prev        (reverse the pointer)
     c. prev = curr             (advance prev one step)
     d. curr = next             (advance curr one step)

3. RETURN prev (new head of reversed list)

The Recursive Algorithm

BASE CASE:
  → If head is null OR head.next is null → return head
  → Empty list or single node → already reversed

RECURSIVE STEP:
  → Recursively reverse everything after the first node
  → Then make the second node point back to the first
  → Then make the first node point to null

VISUALIZATION:
  reverseList(1 → 2 → 3 → 4 → 5)
  
  → Recurse: reverseList(2 → 3 → 4 → 5) returns 5 → 4 → 3 → 2 → null
  → Now: head = 1, head.next = 2, reversed rest = 5 → 4 → 3 → 2 → null
  → Make 2 point back to 1: head.next.next = head → 2.next = 1
  → Make 1 point to null: head.next = null → 1.next = null
  → Return the new head (5) from the recursive call

  Result: 5 → 4 → 3 → 2 → 1 → null ✓

THE KEY LINE:
  head.next.next = head
  
  → head.next is node 2 (the node after head)
  → head.next.next is where node 2 points
  → We set node 2 to point back to node 1 (head)
  → This creates the backward link

✅ WHAT SHOULD YOU DO

→ ALWAYS save curr.next BEFORE redirecting curr.next
     → Once you change curr.next, the original next is GONE
     → The save must happen FIRST — this is the most critical step

→ ALWAYS start prev at null
     → The old head becomes the new tail → must point to null
     → prev = null ensures this happens in the first iteration

→ ALWAYS return prev (not curr) at the end
     → When loop ends, curr is null (went past the last node)
     → prev is the last node we processed → the new head

→ ALWAYS handle edge cases before the loop
     → null head → return null
     → Single node → return head (already reversed)
     → Both handled naturally by the loop (0 or 1 iterations)
     → But explicit checks improve clarity

→ ALWAYS draw the pointer diagram when debugging
     → Linked list bugs are INVISIBLE without visualization
     → Draw boxes and arrows → trace each step → find the break

→ ALWAYS prefer iterative over recursive for interviews (unless asked)
     → Iterative: O(1) space → truly in-place
     → Recursive: O(n) space for call stack → not truly in-place
     → But know BOTH approaches → shows depth

🚫 WHAT SHOULD YOU NOT DO

→ DO NOT create a new list and copy values in reverse
     WRONG:
       Create new nodes → traverse original → add each value to front of new list
     → This uses O(n) extra space → violates in-place requirement
     → Also creates garbage (old nodes) → memory waste

→ DO NOT use a stack to reverse
     WRONG:
       Push all nodes onto stack → pop them out → relink
     → Uses O(n) extra space for the stack
     → Correct but not in-place

→ DO NOT use an array to reverse
     WRONG:
       Collect all values into array → reverse array → build new list
     → O(n) extra space → not in-place
     → Also loses the original node objects

→ DO NOT change the ORDER of the four operations
     → As proven above, any reordering breaks the algorithm
     → SAVE → REDIRECT → ADVANCE PREV → ADVANCE CURR → memorize this

→ DO NOT forget to update BOTH prev AND curr
     → Missing prev = curr → prev stays at null forever → all nodes point to null
     → Missing curr = next → curr stays at same node → infinite loop

→ DO NOT return curr at the end
     → curr is null when the loop ends → returning null means "empty list"
     → prev is the new head → return prev

→ DO NOT try to reverse by swapping values
     WRONG (for linked lists):
       Swap first and last values, then second and second-to-last, etc.
     → Singly linked list → can't efficiently access the end
     → Would need O(n) per swap to find the partner → O(n²) total
     → Works for arrays → NOT for linked lists

🔀 WHEN TO USE THIS PATTERN

✓ Problem explicitly asks to REVERSE a linked list
     → The direct application

✓ Problem asks to check if a linked list is a PALINDROME
     → Find middle (Project 15) → reverse second half → compare
     → Reversal is the key building block

✓ Problem asks to REORDER a linked list (L0→Ln→L1→Ln-1→...)
     → Find middle → reverse second half → interleave

✓ Problem asks to reverse a SUB-PORTION of a linked list
     → "Reverse nodes from position m to n"
     → Same pointer dance, with boundary stitching (Project 23)

✓ Problem asks to reverse in GROUPS of K
     → "Reverse every K nodes" → apply reversal K times, reconnect groups
     → Extension of basic reversal

✓ Problem involves linked list MERGE SORT
     → Split at middle (Project 15) → recursively sort halves → merge
     → Reversal not directly used but pointer manipulation skills transfer

✓ Any problem requiring STRUCTURAL MODIFICATION of a linked list
     → Reversal teaches the core skill: safely redirecting .next pointers
     → All linked list modifications use the same careful save-then-modify approach

🚫 WHEN NOT TO USE THIS PATTERN

✗ Data is in an ARRAY (not linked list)
     → Array reversal: swap elements from both ends → two pointers → O(n)
     → No pointer dance needed → simple swaps
     → Collections.reverse() or manual two-pointer swap

✗ List is DOUBLY linked
     → Each node has .prev and .next
     → Reversal: swap .prev and .next for each node → simpler
     → OR just swap head and tail pointers
     → The prev/curr/next dance is overkill

✗ You only need the reversed VALUES (not the reversed structure)
     → Traverse the list → collect values in array → reverse array
     → Simpler if you don't need the linked structure afterward

✗ The list is IMMUTABLE (functional programming context)
     → Can't modify .next pointers
     → Must create a new reversed list → O(n) space
     → Different approach entirely

✗ You need STABLE references to nodes
     → If other parts of the code hold references to specific nodes
     → Reversing changes what .next points to → may break external references
     → Consider creating a copy instead

⚙️ HOW IT ONLY WORKS — Internal Mechanics

Mechanic 1: The Three-Variable Invariant

AT EVERY POINT IN THE LOOP, three regions exist:

  [REVERSED PORTION]  ←  prev  ←  ...  ←  null
  [CURRENT NODE]      =  curr
  [REMAINING PORTION] →  curr.next  →  ...  →  null

BEFORE FIRST ITERATION:
  Reversed: empty (prev = null)
  Current:  node 1 (curr = head)
  Remaining: node 2 → ... → null

AFTER EACH ITERATION:
  → One node moves from "remaining" to "reversed"
  → curr advances to the next remaining node
  → prev advances to the newly reversed node

AFTER LAST ITERATION:
  Reversed: all nodes (prev = last node = new head)
  Current:  null (past the end)
  Remaining: empty

→ The algorithm PEELS OFF one node at a time from the front of "remaining"
→ And PREPENDS it to the front of "reversed"
→ Like moving cards from one pile to another, one at a time, reversing order

Mechanic 2: Why Iterative is O(1) Space

VARIABLES USED:
  → prev: one pointer → 8 bytes (64-bit JVM)
  → curr: one pointer → 8 bytes
  → next: one pointer → 8 bytes
  → Total: 24 bytes → CONSTANT regardless of list size

NO ADDITIONAL ALLOCATIONS:
  → No new nodes created
  → No arrays or lists
  → No recursion stack
  → Existing nodes are reused — only their .next pointers change

THIS IS TRUE IN-PLACE:
  → The same memory footprint before and after
  → Only the LINKS between existing objects change
  → The objects themselves don't move in memory

Mechanic 3: Why Recursive Uses O(n) Space

RECURSIVE VERSION:
  reverseList(node 1)
    → reverseList(node 2)
      → reverseList(node 3)
        → reverseList(node 4)
          → reverseList(node 5)
            → return node 5 (base case)
          → redirect 5→4, return node 5
        → redirect 4→3, return node 5
      → redirect 3→2, return node 5
    → redirect 2→1, return node 5
  → return node 5

CALL STACK DEPTH: n frames (one per node)
  → Each frame stores: head, the return value, local variables
  → n frames × ~20-40 bytes each ≈ O(n) space

FOR n = 10,000:
  → 10,000 stack frames → might cause StackOverflowError in Java
  → Java default stack size: ~512KB-1MB
  → Each frame: ~40 bytes → 10,000 × 40 = 400KB → borderline
  → For n = 100,000 → likely stack overflow

→ Iterative is ALWAYS safer for large lists
→ Recursive is elegant but has practical limitations

Mechanic 4: The Recursive Trick Explained

The key line: head.next.next = head

WHAT DOES THIS MEAN?

  Consider: head = node 2, head.next = node 3
  
  After recursion, everything after node 2 is already reversed:
    5 → 4 → 3 → null
  
  But node 2 still points to node 3:
    2 → 3 (and 3 now points to null from the reversal)
  
  We need: 3 → 2 (make 3 point back to 2)
  
  head.next is node 3
  head.next.next = head means: node3.next = node2
  
  Now: 3 → 2
  
  But node 2 still points to node 3: 2 → 3 → 2 → 3 → ... (CYCLE!)
  
  Fix: head.next = null → node2.next = null
  
  Now: 5 → 4 → 3 → 2 → null ✓

→ head.next.next = head creates the backward link
→ head.next = null breaks the forward link
→ Both are necessary to avoid cycles

Time and Space Complexity

ITERATIVE:
  TIME:  O(n) → visit each node exactly once
  SPACE: O(1) → three pointer variables

RECURSIVE:
  TIME:  O(n) → one recursive call per node
  SPACE: O(n) → call stack depth equals list length

⚠️ COMMON PITFALLS AND FAILURE MODES

Pitfall 1: Forgetting to Save next Before Redirecting

WRONG:
  while (curr != null) {
      curr.next = prev;        // ← OOPS: lost reference to original next
      prev = curr;
      curr = curr.next;        // ← curr.next is now prev, not the original next!
  }

  → After curr.next = prev: curr.next now points BACKWARD
  → curr = curr.next moves us BACKWARD → we revisit prev → infinite loop

CORRECT:
  while (curr != null) {
      ListNode next = curr.next;   // ← SAVE first
      curr.next = prev;            // ← now safe to redirect
      prev = curr;
      curr = next;                 // ← use the saved reference
  }

Pitfall 2: Returning curr Instead of prev

WRONG:
  return curr;   // curr is null when loop ends → returns null → "empty list"

CORRECT:
  return prev;   // prev is the last processed node → the new head

TRACE:
  List: 1 → 2 → 3

  After iteration 3:
    null ← 1 ← 2 ← 3
                     ↑     ↑
                    prev   curr = null

  return prev → returns node 3 → correct new head ✓
  return curr → returns null → WRONG

Pitfall 3: Creating a Cycle in Recursive Version

WRONG:
  public ListNode reverseList(ListNode head) {
      if (head == null || head.next == null) return head;
      ListNode newHead = reverseList(head.next);
      head.next.next = head;
      // MISSING: head.next = null
      return newHead;
  }

  → After head.next.next = head: node2 → node1 AND node1 → node2
  → CYCLE: 1 ↔ 2 → traversal never ends
  → Must add: head.next = null to break the forward link

CORRECT:
  head.next.next = head;
  head.next = null;        // ← CRITICAL: break the old forward link

Pitfall 4: Not Handling Single-Node or Empty Lists

Empty list: head = null
  → Loop condition: curr != null → false → loop skipped
  → Return prev = null ✓ (empty reversed list is still empty)

Single node: head = node1, node1.next = null
  → Iteration 1: next = null, 1.next = null (already null), prev = 1, curr = null
  → Loop ends → return prev = node1 ✓

→ Both handled correctly by the algorithm naturally
→ But explicit guards improve readability:
  if (head == null || head.next == null) return head;

Pitfall 5: Assuming the Original Head is Still Usable as Head

AFTER REVERSAL:
  Original head (node 1) is now the TAIL → its .next is null
  
  If caller still uses the old head reference:
    ListNode head = buildList(1,2,3,4,5);
    ListNode reversed = reverseList(head);
    
    // head still points to node 1
    // but node 1.next is now null
    // traversing from head gives: 1 → null (only one node visible)
    
    // Must use reversed as the new head:
    // reversed points to node 5 → 5 → 4 → 3 → 2 → 1 → null ✓

→ Always use the RETURNED value as the new head
→ The old head variable is now a reference to the TAIL
→ This is a common source of confusion

📈 SCALABILITY CONSIDERATIONS

ITERATIVE:
→ n = 100         → 100 pointer redirections → instant
→ n = 1,000,000   → 1M redirections → ~5ms
→ n = 100,000,000 → 100M redirections → ~500ms
→ Memory: O(1) always → never a bottleneck

RECURSIVE:
→ n = 1,000   → 1,000 stack frames → fine
→ n = 10,000  → 10,000 stack frames → might overflow (Java default stack)
→ n = 100,000 → 100,000 stack frames → WILL overflow
→ Memory: O(n) → scales linearly → problematic for large lists

JAVA STACK SIZE:
  → Default: ~512KB-1MB
  → Each frame: ~40-80 bytes (references, return address)
  → Max safe depth: ~10,000-25,000 frames
  → Can increase with -Xss flag: java -Xss4m MyProgram
  → But iterative is always better for production code

RECOMMENDATION:
  → ALWAYS use iterative for production/interviews
  → Show recursive as "I also know this approach" bonus
  → Recursive is more elegant but less practical

🔍 INTERPRETABILITY CONSIDERATIONS

MENTAL MODEL:
  → Imagine a train of carriages
  → Each carriage is coupled to the next one
  → To reverse: disconnect each coupling and reconnect it backward
  → Work from front to back, one coupling at a time
  → The caboose becomes the engine

DEBUGGING:
  → Print: prev, curr, next at each step
  → Draw the pointer diagram after each iteration
  → Verify: reversed portion grows by one node per iteration
  → Verify: curr advances forward through the original list

INTERVIEW COMMUNICATION:
  → "I use three pointers: prev, curr, and next"
  → "At each step, I save the next node, reverse the current link, then advance"
  → "prev starts at null because the old head becomes the new tail"
  → "When curr reaches null, prev is the new head"
  → "Time: O(n), Space: O(1)"

THE FOUR-STEP MANTRA:
  → "Save, Redirect, Advance Prev, Advance Curr"
  → Say this out loud while coding → never miss a step
  → This is the linked list equivalent of "left, right, mid" for binary search

🧠 KEY TAKEAWAY

WHAT WE LEARNED:
→ Linked list reversal uses the Prev/Curr/Next pointer dance
→ Four steps per node: SAVE → REDIRECT → ADVANCE PREV → ADVANCE CURR
→ Order is NON-NEGOTIABLE → any reordering breaks the algorithm
→ prev starts at null → ensures old head becomes new tail
→ Return prev (not curr) → curr is null when loop ends
→ Iterative: O(n) time, O(1) space → preferred for production
→ Recursive: O(n) time, O(n) space → elegant but stack-limited
