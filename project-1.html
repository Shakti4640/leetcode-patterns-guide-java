PROJECT 1: Pair Sum in Sorted Array

🎯 Two Pointers → Left/Right Convergence on Sorted Data

🧩 THE HYPOTHETICAL PROBLEM

→ You are given a SORTED array of integers (ascending order)
→ You are given a TARGET sum
→ Find TWO numbers in the array that add up to the target
→ Return their indices (1-based indexing)
→ Exactly one solution is guaranteed to exist
→ You cannot use the same element twice

Example:
  Input:  numbers = [1, 3, 5, 7, 11, 15], target = 16
  Output: [2, 5]  →  because numbers[1]=3 and numbers[4]=11 → 3+11=14... 
  Actually: [1, 6] → numbers[0]=1 and numbers[5]=15 → 1+15=16 ✓

Another:
  Input:  numbers = [2, 4, 6, 10, 14], target = 16
  Output: [1, 4] → 2+14=16 ✓  OR  [2, 3] → 4+6=10 ✗  → 6+10=16 ✓
  So output: [3, 4] → numbers[2]=6 and numbers[3]=10 → 6+10=16 ✓

🔍 HOW TO APPROACH IT

Step 1 — Observe the Data Shape

→ The array is SORTED in ascending order
→ This is the single most important clue
→ Sorted = directional → small values on LEFT, large values on RIGHT
→ This means we can make DECISIONS based on direction

Step 2 — Think About the Brute Force First

→ Naive approach: Check every pair
→ Pick element i, pick element j, check if arr[i] + arr[j] == target
→ Two nested loops → O(n²) time

for i = 0 to n-1:
    for j = i+1 to n-1:
        if arr[i] + arr[j] == target:
            return [i, j]

→ This WORKS but IGNORES the sorted property entirely
→ We are wasting information
→ The sorted order gives us a COMPASS — we should use it

Step 3 — What Does "Sorted" Give Us?

→ If we pick the SMALLEST element (leftmost) and LARGEST element (rightmost)
→ We get the EXTREME sum
→ If that sum is TOO BIG → the large number is too large → move it DOWN
→ If that sum is TOO SMALL → the small number is too small → move it UP
→ If that sum is EXACT → we found our answer

This is the CORE INSIGHT:
  → Two pointers starting at opposite ends
  → Moving INWARD based on comparison with target
  → Each move ELIMINATES one candidate permanently

Step 4 — Visualize the Convergence

Array:  [1, 3, 5, 7, 11, 15]    Target = 16

Step 1:  left=0, right=5  → 1+15 = 16 → MATCH → return [0, 5]
Done in ONE step.

Another example: target = 12
Step 1:  left=0, right=5  → 1+15 = 16 > 12 → too big → right--
Step 2:  left=0, right=4  → 1+11 = 12 → MATCH → return [0, 4]

Another example: target = 18
Step 1:  left=0, right=5  → 1+15 = 16 < 18 → too small → left++
Step 2:  left=1, right=5  → 3+15 = 18 → MATCH → return [1, 5]

Another example: target = 8
Step 1:  left=0, right=5  → 1+15 = 16 > 8  → right--
Step 2:  left=0, right=4  → 1+11 = 12 > 8  → right--
Step 3:  left=0, right=3  → 1+7  = 8  → MATCH → return [0, 3]

🛠️ HOW CAN WE SOLVE IT

The Algorithm — Line by Line Logic

1. Place LEFT pointer at index 0 (smallest element)
2. Place RIGHT pointer at index n-1 (largest element)
3. WHILE left < right:
     a. Compute currentSum = arr[left] + arr[right]
     b. IF currentSum == target → FOUND → return [left, right]
     c. IF currentSum < target → need BIGGER sum → left++
     d. IF currentSum > target → need SMALLER sum → right--
4. If loop ends without return → no pair exists

Why Does This Work? — The Elimination Logic

Consider: arr = [1, 3, 5, 7, 11, 15], target = 8

When left=0, right=5: sum = 1+15 = 16 > 8
→ We move right to 4
→ This ELIMINATES 15 from ALL future considerations
→ WHY? Because if 15 + 1 (the SMALLEST) is already too big
→ Then 15 + anything else is ALSO too big
→ So 15 can NEVER be part of the answer
→ We safely discard it

When left=0, right=4: sum = 1+11 = 12 > 8
→ Same logic → 11 + smallest is still too big → discard 11

When left=0, right=3: sum = 1+7 = 8 == target → FOUND

The REVERSE works too:
If sum < target → arr[left] + arr[right] < target
→ arr[left] + arr[right] is already the BIGGEST sum possible with arr[left]
→ Because arr[right] is the largest remaining
→ So arr[left] can NEVER reach target → discard it → left++

✅ WHAT SHOULD YOU DO

→ ALWAYS verify the array is sorted before applying this pattern
→ ALWAYS use left < right as the loop condition (not left <= right)
     → left == right means same element → problem says two DIFFERENT elements
→ ALWAYS move only ONE pointer per step
     → Either left++ OR right-- → never both simultaneously
→ ALWAYS think about what each pointer movement ELIMINATES
→ ALWAYS return immediately upon finding the match (exactly one solution)
→ ALWAYS consider 1-based vs 0-based indexing based on problem statement

🚫 WHAT SHOULD YOU NOT DO

→ DO NOT use a HashMap for this problem
     → HashMap works but ignores the sorted property
     → HashMap uses O(n) extra space
     → Two pointers uses O(1) space
     → When data is sorted → two pointers is the OPTIMAL choice

→ DO NOT sort the array yourself then apply two pointers
     → The problem GIVES you sorted data
     → If data were unsorted → sorting costs O(n log n)
     → HashMap on unsorted data would be O(n) which is better

→ DO NOT move both pointers in the same direction
     → Left and right must move TOWARD each other
     → Moving both right or both left loses the convergence guarantee

→ DO NOT forget the loop termination
     → left < right guarantees the loop ends
     → Without it → infinite loop or index out of bounds

→ DO NOT use left <= right
     → When left == right → you're pointing at ONE element
     → You'd be checking arr[i] + arr[i] = target → same element used twice
     → Problem forbids this

🔀 WHEN TO USE THIS PATTERN

✓ Array/list is SORTED (or can be sorted without losing information)
✓ You need to find a PAIR that satisfies a condition (sum, difference, etc.)
✓ The condition has a MONOTONIC relationship with pointer movement
     → Moving left increases the sum
     → Moving right decreases the sum
     → This monotonicity is what makes the technique work
✓ You want O(n) time and O(1) space
✓ The problem involves CONVERGENCE from two ends toward a middle
✓ Variants: pair sum, pair difference, container with most water, etc.

🚫 WHEN NOT TO USE THIS PATTERN

✗ Array is NOT sorted AND sorting would lose required information
     → Example: "return original indices" on unsorted array
     → Sorting destroys original positions
     → Use HashMap instead

✗ You need MORE than two elements
     → Two pointers finds PAIRS
     → For triplets → need outer loop + two pointers (covered in Project 11)

✗ No monotonic relationship exists
     → If moving left pointer could either increase OR decrease the metric
     → Two pointers cannot make a reliable decision
     → Technique breaks down

✗ Data structure is not random-access
     → Singly linked list → can't move backward efficiently
     → Two pointers on linked lists needs different handling (Project 7)

✗ Duplicates need special handling for unique results
     → Basic two pointers finds ONE pair
     → Finding ALL unique pairs needs duplicate-skipping logic (Project 11)

⚙️ HOW IT ONLY WORKS — Internal Mechanics

The Invariant

At every step of the algorithm, the answer pair (if it exists)
is ALWAYS within the range [left, right] inclusive.

PROOF:
→ Initial state: left=0, right=n-1 → entire array → answer is inside
→ When we move left++: 
     → We proved arr[left] + arr[right] < target
     → arr[right] is the MAXIMUM remaining value
     → So arr[left] + anything_remaining ≤ arr[left] + arr[right] < target
     → arr[left] CANNOT be part of any valid pair
     → Eliminating it does NOT eliminate the answer
→ When we move right--:
     → We proved arr[left] + arr[right] > target
     → arr[left] is the MINIMUM remaining value
     → So anything_remaining + arr[right] ≥ arr[left] + arr[right] > target
     → arr[right] CANNOT be part of any valid pair
     → Eliminating it does NOT eliminate the answer
→ Therefore: the answer is NEVER eliminated
→ Eventually left and right converge on the answer

Time Complexity Breakdown

→ Each step moves exactly ONE pointer inward
→ left starts at 0, can move up to n-1 times
→ right starts at n-1, can move down to 0 times
→ TOTAL pointer movements ≤ n
→ Each movement is O(1) work
→ TOTAL: O(n) time

Space Complexity Breakdown

→ Only two integer variables: left and right
→ No additional data structures
→ TOTAL: O(1) space

Why Sorted Order is NON-NEGOTIABLE

→ The decision "move left or move right" depends on:
     → left++ will INCREASE the sum (because sorted → next element is bigger)
     → right-- will DECREASE the sum (because sorted → previous element is smaller)
→ If array is NOT sorted:
     → left++ might increase OR decrease the sum → unpredictable
     → right-- might increase OR decrease the sum → unpredictable
     → No reliable decision can be made
     → Technique FAILS

⚠️ COMMON PITFALLS AND FAILURE MODES

Pitfall 1: Off-by-One on Index Return

→ Problem says "1-based indexing" → return [left+1, right+1]
→ Problem says "0-based indexing" → return [left, right]
→ READ THE PROBLEM → most LeetCode problems specify this
→ Getting the logic right but indices wrong = wrong answer

Pitfall 2: Integer Overflow on Sum

→ arr[left] + arr[right] can overflow if values are large
→ In Java: int range is -2^31 to 2^31-1
→ If arr[left] = 2,000,000,000 and arr[right] = 1,000,000,000
→ Sum = 3,000,000,000 → OVERFLOWS int
→ FIX: use long for the sum computation

long currentSum = (long) arr[left] + (long) arr[right];

Pitfall 3: Assuming Exactly One Solution Exists

→ If the problem guarantees one solution → simple return inside loop
→ If the problem says "return empty if no solution" → handle the case after loop
→ If the problem says "return all pairs" → don't return early, collect results
→ Different problem variants need different handling

Pitfall 4: Negative Numbers

→ Sorted array with negatives: [-5, -3, 0, 2, 8]
→ Target = -1 → pair is (-3, 2) → indices [1, 3]
→ The algorithm works IDENTICALLY with negatives
→ Sorted is sorted → direction logic still holds
→ No special handling needed

📈 SCALABILITY CONSIDERATIONS

→ n = 100 → brute force O(n²) = 10,000 ops → fast enough
→ n = 10,000 → brute force = 100,000,000 ops → borderline slow
→ n = 1,000,000 → brute force = 10^12 ops → WAY too slow
→ n = 1,000,000 → two pointers = 10^6 ops → perfectly fast

→ Two pointers scales LINEARLY
→ It handles arrays of size 10^7+ without issue
→ Memory footprint stays constant regardless of input size
→ This is why recognizing "sorted + pair" = two pointers is critical

🔍 INTERPRETABILITY CONSIDERATIONS

→ The logic is visually intuitive:
     → Imagine a number line with two fingers
     → Slide them inward from both ends
     → Sum too big? Pull the right finger left
     → Sum too small? Push the left finger right
     → They meet at the answer

→ Easy to trace and debug:
     → Print left, right, sum at each step
     → You can see the convergence happening
     → Each step makes measurable progress toward the goal

→ Easy to explain in interviews:
     → "Since the array is sorted, I use two pointers from both ends"
     → "I exploit the monotonic property: left++ increases sum, right-- decreases sum"
     → "This guarantees O(n) time and O(1) space"

🧠 KEY TAKEAWAY

WHAT WE LEARNED:
→ Two pointers is a SPACE-EFFICIENT alternative to brute force on SORTED data
→ The technique works because sorted order creates MONOTONIC behavior
→ Moving left pointer ALWAYS increases the sum
→ Moving right pointer ALWAYS decreases the sum
→ This predictability allows GUARANTEED elimination of one element per step
→ Result: O(n) time, O(1) space — optimal for this problem class

THE CORE MENTAL MODEL:
→ "I have sorted data and need to find a pair"
→ "Place two fingers at opposite ends"
→ "Move the appropriate finger based on comparison with target"
→ "Each move eliminates one candidate forever"

PATTERN SIGNATURE:
→ Sorted input ✓
→ Pair-based condition ✓  
→ Monotonic pointer movement effect ✓
→ = TWO POINTERS — LEFT/RIGHT CONVERGENCE
