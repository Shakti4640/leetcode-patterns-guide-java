PROJECT 26: All Root-to-Leaf Paths for a Target Sum

🎯 Tree DFS → Path Accumulation → Backtracking on DFS Return

🔗 DEPENDENCY CHAIN

Project 17 (Tree BFS — Level Order Traversal)
    → Introduced tree structure, TreeNode, queue-based traversal
Project 18 (Root-to-Leaf Path Sum)
    → Introduced DFS recursion, subtraction technique, leaf base case
    → Asked: "Does ANY path sum to target?" → returned boolean

PROJECT 26 (THIS):
    → Asks: "Find ALL paths that sum to target" → return list of lists
    → NEW SKILL: Path accumulation + Backtracking
    → This is the FIRST time we explicitly manage state on DFS return

🧩 THE HYPOTHETICAL PROBLEM

→ You are given a BINARY TREE and a TARGET SUM
→ Find ALL root-to-leaf paths where the sum of node values equals the target
→ Return each path as a list of node values from root to leaf
→ Return ALL such paths (not just one, not just true/false)

Example:
            5
           / \
          4    8
         /    / \
        11   13   4
       / \       / \
      7   2     5   1

  Target = 22

  Path 1: 5 → 4 → 11 → 2  = 22 ✓
  Path 2: 5 → 8 → 4 → 5   = 22 ✓

  Output: [[5,4,11,2], [5,8,4,5]]

Other paths:
  5 → 4 → 11 → 7  = 27 ✗
  5 → 8 → 13       = 26 ✗
  5 → 8 → 4 → 1    = 18 ✗

🔍 HOW TO APPROACH IT

Step 1 — Recall What Project 18 Taught Us

In Project 18 we solved: "Does ANY path sum to target?"
→ We subtracted current node value from remaining target
→ At leaf: checked if remaining == 0
→ Returned true/false
→ The moment ONE path matched, we returned true

KEY LIMITATION OF PROJECT 18:
→ We only needed a YES/NO answer
→ We never needed to REMEMBER which nodes formed the path
→ We never needed to find ALL such paths

PROJECT 26 DEMANDS:
→ Find ALL matching paths (not just one)
→ RECORD the actual node values along each path
→ Return them as complete lists
→ This requires a fundamentally new technique: BACKTRACKING

Step 2 — The Core New Concept: Path Accumulation

Imagine walking through the tree with a NOTEBOOK:
→ When you ENTER a node → WRITE its value in the notebook
→ When you reach a LEAF → CHECK if notebook values sum to target
→ If YES → COPY the notebook page and save it as a result
→ When you BACKTRACK (return from a node) → ERASE the last entry

This "write on enter, erase on exit" is BACKTRACKING

Visual walkthrough:
            5
           / \
          4    8

Step 1: Enter node 5 → notebook = [5]
Step 2: Go left → Enter node 4 → notebook = [5, 4]
Step 3: Node 4 is leaf (in simple tree) → check sum → backtrack
Step 4: Erase 4 → notebook = [5]
Step 5: Go right → Enter node 8 → notebook = [5, 8]
Step 6: Node 8 is leaf → check sum → backtrack
Step 7: Erase 8 → notebook = [5]
Step 8: Erase 5 → notebook = []

Step 3 — Why Backtracking is ESSENTIAL

Without backtracking — what goes WRONG:

            5
           / \
          4    8

Enter 5 → path = [5]
Enter 4 → path = [5, 4]
Return from 4 → path is STILL [5, 4]   ← PROBLEM!
Enter 8 → path = [5, 4, 8]             ← WRONG! Should be [5, 8]

The left subtree's nodes CONTAMINATE the right subtree's path
because we never REMOVED them.

With backtracking:
Enter 5 → path = [5]
Enter 4 → path = [5, 4]
Return from 4 → REMOVE last → path = [5]    ← CLEAN!
Enter 8 → path = [5, 8]                      ← CORRECT!

Step 4 — Deep Copy vs Reference Trap

CRITICAL PITFALL that catches almost everyone:

When we find a valid path, we must COPY it into results.
We CANNOT just add the path reference.

WHY?
→ The path list is being MODIFIED throughout the DFS
→ If you add a reference: results.add(currentPath)
→ That reference points to the SAME list object
→ As backtracking erases entries, your "saved" result gets erased too
→ At the end, ALL entries in results point to the SAME empty list

FIX:
→ results.add(new ArrayList<>(currentPath))   ← DEEP COPY
→ This creates a SNAPSHOT of the path at that moment
→ Future modifications to currentPath don't affect the snapshot

🛠️ HOW CAN WE SOLVE IT

The Algorithm — Step by Step

1. Create a results list (list of lists) to store all valid paths
2. Create a currentPath list to track the current DFS path
3. Start DFS from root

DFS(node, remainingSum, currentPath, results):
    a. If node is null → return (base case — dead end)
    b. ADD node.val to currentPath
    c. Subtract node.val from remainingSum
    d. If node is a LEAF (no left, no right):
         → If remainingSum == 0:
              → COPY currentPath into results
    e. Else (not a leaf):
         → Recurse LEFT:  DFS(node.left, remainingSum, currentPath, results)
         → Recurse RIGHT: DFS(node.right, remainingSum, currentPath, results)
    f. BACKTRACK: REMOVE last element from currentPath
       → This is the KEY step — restores state for sibling exploration

Walkthrough on Full Example

         /    / \
        11   13   4
       / \       / \
      7   2     5   1

Target = 22

        Leaf! remaining==0 → ✅ SAVE COPY [5,4,11,2]

    BACKTRACK → remove 4 → path=[5]
  Enter 8 → path=[5,8], remaining=9
    Enter 13 → path=[5,8,13], remaining=-4
      Leaf! remaining≠0 → no match
      BACKTRACK → remove 13 → path=[5,8]
    Enter 4 → path=[5,8,4], remaining=5
      Enter 5 → path=[5,8,4,5], remaining=0
        Leaf! remaining==0 → ✅ SAVE COPY [5,8,4,5]
        BACKTRACK → remove 5 → path=[5,8,4]
      Enter 1 → path=[5,8,4,1], remaining=4
        Leaf! remaining≠0 → no match
        BACKTRACK → remove 1 → path=[5,8,4]
      BACKTRACK → remove 4 → path=[5,8]

[[5,4,11,2], [5,8,4,5]]

✅ WHAT SHOULD YOU DO

→ ALWAYS backtrack (remove last element) AFTER processing both children
     → This is non-negotiable — forgetting it corrupts ALL subsequent paths
→ ALWAYS deep-copy the path when saving to results
     → new ArrayList<>(currentPath) — not just currentPath
→ ALWAYS subtract BEFORE checking leaf condition
     → Add to path → subtract → check → recurse → backtrack
     → This order is the canonical DFS-with-path pattern
→ ALWAYS check leaf condition as: node.left == null && node.right == null
     → Not just "no children in one direction"
→ ALWAYS pass the results list and currentPath by REFERENCE
     → Java passes object references naturally
     → Both the path and results accumulate across recursive calls
→ ALWAYS handle null node as the FIRST check in recursion
     → Prevents NullPointerException on node.val access

🚫 WHAT SHOULD YOU NOT DO

→ DO NOT create a new path list at every recursive call
     → This wastes O(n) memory per call → total O(n × 2^n) memory
     → Backtracking reuses ONE list → O(n) memory for the path

     BAD (creates new list every call):
       List<Integer> newPath = new ArrayList<>(currentPath);
       newPath.add(node.val);
       dfs(node.left, remaining, newPath, results);
     
     → This "works" but is extremely wasteful
     → It avoids the need for backtracking but at massive memory cost
     → Real interviews expect the backtracking approach

→ DO NOT forget the deep copy when saving results

     BAD:  results.add(currentPath);         ← reference to mutable list
     GOOD: results.add(new ArrayList<>(currentPath)); ← snapshot copy

→ DO NOT try to use BFS for this problem
     → BFS processes level by level → paths cross levels
     → Tracking individual root-to-leaf paths in BFS is awkward
     → DFS naturally follows ONE path at a time → perfect fit

→ DO NOT check remaining < 0 as an early exit IF negative values exist
     → If tree has negative values, a path can go below zero then come back
     → 5 → (-3) → 20 with target 22: remaining goes 17 → 20 → 0 ✓
     → Pruning at remaining < 0 would MISS this path
     → Only prune on remaining < 0 if ALL values are guaranteed POSITIVE

→ DO NOT confuse "root-to-leaf" with "root-to-any-node"
     → Leaf = node with NO children (both left and right are null)
     → A node with one child is NOT a leaf
     → Stopping at non-leaf nodes gives wrong answers

🔀 WHEN TO USE THIS PATTERN

✓ Problem says "find ALL paths" in a tree
✓ Problem says "root-to-leaf" → DFS is the natural traversal
✓ You need to COLLECT results, not just check existence
✓ The path itself (sequence of nodes) matters, not just a boolean
✓ The tree has potential for MULTIPLE valid answers
✓ Backtracking is needed whenever:
     → You build up state going DOWN the tree
     → You must UNDO that state coming BACK UP
     → The state is shared across sibling subtrees

RECOGNIZABLE KEYWORDS:
→ "all paths", "find every path", "list all", "enumerate"
→ "root to leaf", "path sum", "path with property X"
→ "return list of lists"

🚫 WHEN NOT TO USE THIS PATTERN

✗ Problem only asks "does ANY path exist?" → Project 18 approach (boolean DFS)
✗ Problem asks for "any path in the tree" (not root-to-leaf)
     → Different problem: path can start/end anywhere
     → Needs different technique (path between any two nodes)
✗ Problem asks for COUNT of paths, not the actual paths
     → Counting doesn't need path accumulation
     → Simpler DFS with integer return suffices
✗ Tree is extremely deep (>10,000 levels)
     → Recursive DFS may cause StackOverflowError
     → Need iterative DFS with explicit stack (rare in interviews)
✗ Problem involves graph cycles
     → Trees have no cycles → no visited tracking needed
     → Graphs need visited sets → different approach (Project 47)

⚙️ HOW IT ONLY WORKS — Internal Mechanics

The Call Stack as an Implicit Path Tracker

The Java call stack itself mirrors the tree path:

Call stack when at node 2 (path [5,4,11,2]):

  main()
    └─ dfs(node=5, rem=22)      ← entered 5
        └─ dfs(node=4, rem=17)   ← entered 4
            └─ dfs(node=11, rem=13) ← entered 11
                └─ dfs(node=2, rem=2)  ← entered 2 (LEAF, rem=0 ✓)

Each stack frame holds its own local variables.
But the currentPath list is SHARED across all frames.
That's why we must manually add/remove from it.

When dfs(node=2) returns:
→ We remove 2 from currentPath
→ Control returns to dfs(node=11)
→ Which then calls dfs(node=7) with clean path [5,4,11]

Memory Layout During Execution

At the moment we find path [5,4,11,2]:

  currentPath (ONE shared list): [5, 4, 11, 2]
  results: []

  Action: results.add(new ArrayList<>(currentPath))
  
  After action:
  currentPath: [5, 4, 11, 2]  ← still mutable, will be modified
  results: [[5, 4, 11, 2]]     ← independent copy, safe from changes

  After backtrack (remove 2):
  currentPath: [5, 4, 11]      ← modified
  results: [[5, 4, 11, 2]]     ← copy unchanged ✓

If we had done results.add(currentPath) WITHOUT copy:
  After backtrack:
  currentPath: [5, 4, 11]
  results: [[5, 4, 11]]        ← ALSO modified because same reference!
  
  At end of all DFS:
  currentPath: []
  results: [[], []]             ← ALL entries are empty! DISASTER!

Backtracking Invariant

INVARIANT: Before and after calling dfs(node, ...),
           currentPath contains exactly the path from root
           to node's PARENT.

PROOF:
→ Before entering dfs(node): path = [root, ..., parent]
→ First line of dfs: path.add(node.val) → path = [root, ..., parent, node]
→ Process children → path grows and shrinks through their backtracking
→ Last line of dfs: path.remove(last) → path = [root, ..., parent]
→ After returning from dfs(node): path = [root, ..., parent]
→ INVARIANT RESTORED ✓

This means the CALLER always sees the path in the same state
before and after the recursive call. This is the ESSENCE of backtracking.

Time Complexity Analysis

→ We visit every node exactly ONCE → O(n) visits
→ At each leaf, we MIGHT copy the path → path length is O(h) where h = tree height
→ In worst case, every leaf is a valid path
→ Number of leaves ≤ n/2
→ Each path copy costs O(h)
→ Total copy cost: O(n/2 × h) = O(n × h)
→ For balanced tree: h = log(n) → O(n log n)
→ For skewed tree: h = n → O(n²)
→ Overall: O(n × h) where h is tree height

Space Complexity Analysis

→ currentPath list: O(h) — at most tree height elements
→ Call stack depth: O(h) — recursive depth equals tree height
→ Results storage: O(k × h) where k = number of valid paths
→ Total auxiliary: O(h) + O(k × h)
→ For balanced tree: O(log n) auxiliary (excluding output)
→ For skewed tree: O(n) auxiliary

⚠️ COMMON PITFALLS AND FAILURE MODES

Pitfall 1: Forgetting to Backtrack

// ❌ BROKEN — no backtracking
void dfs(TreeNode node, int rem, List<Integer> path, List<List<Integer>> res) {
    if (node == null) return;
    path.add(node.val);
    if (node.left == null && node.right == null && rem - node.val == 0) {
        res.add(new ArrayList<>(path));
    }
    dfs(node.left, rem - node.val, path, res);
    dfs(node.right, rem - node.val, path, res);
    // MISSING: path.remove(path.size() - 1);
}

// Result: paths accumulate ALL visited nodes
// Path for rightmost leaf would contain EVERY node in the tree

Pitfall 2: Reference vs Copy Bug

// ❌ BROKEN — saves reference, not copy
if (isLeaf && rem == 0) {
    res.add(path);  // ALL entries in res point to SAME list
}

// ✅ CORRECT — saves independent snapshot
if (isLeaf && rem == 0) {
    res.add(new ArrayList<>(path));  // each entry is independent
}

Pitfall 3: Wrong Leaf Check

// ❌ WRONG — this is NOT a leaf check
if (node.left == null || node.right == null) {  // OR instead of AND
    // A node with ONE child satisfies this
    // But it's NOT a leaf — it has one child
}

// ✅ CORRECT — both children must be null
if (node.left == null && node.right == null) {
    // TRUE leaf — no children at all
}

Pitfall 4: Subtract Before or After Adding?

The order matters for consistency:

APPROACH A (subtract after add — used in this project):
  path.add(node.val)
  newRemaining = remaining - node.val
  if leaf && newRemaining == 0 → match

APPROACH B (subtract before recursive call):
  path.add(node.val)
  if leaf && remaining == node.val → match
  dfs(node.left, remaining - node.val, ...)

Both work — but be CONSISTENT
Mixing approaches causes off-by-one in the sum check

Pitfall 5: Negative Values and Early Pruning

// ❌ DANGEROUS — only safe if ALL values are positive
if (remaining < 0) return;  // prunes negative remaining

// Tree:    5
//         / \
//       -3   8
//       /
//      20
// Target: 22
// Path 5 → -3 → 20: remaining goes 22 → 17 → 20 → 0 ✓
// But remaining after -3 is 20 (positive), so this specific case works
// However: 5 → -3: remaining = 20, not negative, so no pruning issue HERE
// But consider target = 2, path 5 → -3: remaining = -3+... 
// Point: only prune if problem GUARANTEES positive values

// ✅ SAFE — always works regardless of value signs
// Simply don't prune based on remaining sign
// Let the leaf check handle validity

📈 SCALABILITY CONSIDERATIONS

TREE SIZE vs PERFORMANCE:

n = 100 (tiny tree):
→ Visit 100 nodes, ~50 leaves, paths of length ~7
→ Instant — no concerns

n = 10,000 (medium tree):
→ Visit 10,000 nodes
→ Balanced: paths of length ~14, up to 5,000 leaves
→ Still fast — under 1ms

n = 100,000 (large tree):
→ Balanced: height ~17, manageable stack depth
→ Skewed: height ~100,000 → STACK OVERFLOW RISK
→ Java default stack size: ~512KB to 1MB
→ Each stack frame: ~50-100 bytes
→ 100,000 frames: ~5-10MB → likely overflow
→ FIX: increase stack with -Xss flag or use iterative approach

n = 1,000,000:
→ Balanced: height ~20 → fine for recursion
→ Skewed: definitely stack overflow
→ Iterative DFS with explicit stack needed

OUTPUT SIZE:
→ Worst case: every leaf is valid → up to n/2 paths
→ Each path up to O(h) long
→ Total output: O(n × h) → can be large
→ Cannot optimize this — the output itself is that big

🔍 INTERPRETABILITY CONSIDERATIONS

WHY THIS PATTERN IS INTUITIVE:
→ Think of exploring a maze with breadcrumbs
→ Drop a breadcrumb at each turn (add to path)
→ Dead end? Pick up the last breadcrumb (backtrack)
→ Found the exit? Photograph your breadcrumb trail (copy path)
→ Then pick up breadcrumbs anyway (backtrack to find OTHER exits)

DEBUGGING STRATEGY:
→ Print the path at every node entry and exit
→ Watch the path grow and shrink
→ Verify it matches the tree structure
→ Check that backtracking restores exactly one element

INTERVIEW COMMUNICATION:
→ "I'll DFS from root to leaves, maintaining a running path"
→ "At each leaf, I check if the path sum matches the target"
→ "If yes, I save a COPY of the current path"
→ "After exploring both children, I BACKTRACK by removing the current node"
→ "This ensures the path is clean for sibling subtree exploration"
