PROJECT 49: Jump Game — Can You Reach the End?

🎯 Greedy → Farthest-Reachable Tracking → Forward Greedy Scan

🧩 THE HYPOTHETICAL PROBLEM

→ You are given an array of NON-NEGATIVE integers
→ Each element represents the MAXIMUM jump length from that position
→ You start at index 0
→ Determine if you can REACH the last index

→ nums[i] = maximum number of steps you can jump FORWARD from index i
→ You can jump anywhere from 1 to nums[i] steps (not just exactly nums[i])
→ If nums[i] = 0, you are STUCK at position i (cannot move forward)

Example 1:
  Input:  nums = [2, 3, 1, 1, 4]
  Output: true
  
  → From index 0: can jump 1 or 2 steps
  → Jump 1 → index 1: can jump 1, 2, or 3 steps
  → Jump 2 → index 3: can jump 1 step
  → Jump 1 → index 4: REACHED THE END ✓
  
  OR: index 0 → jump 2 → index 2 → jump 1 → index 3 → jump 1 → index 4 ✓

Example 2:
  Input:  nums = [3, 2, 1, 0, 4]
  Output: false
  
  → From index 0: can reach indices 1, 2, 3
  → From index 1: can reach indices 2, 3
  → From index 2: can reach index 3
  → From index 3: nums[3] = 0 → STUCK
  → Index 4 is unreachable from any path
  → Every path leads to index 3 which is a dead end

Example 3:
  Input:  nums = [0]
  Output: true
  
  → Already at the last index → trivially reachable

Example 4:
  Input:  nums = [2, 0, 0]
  Output: true
  
  → From index 0: can jump 2 → directly to index 2 (last index) ✓

Example 5:
  Input:  nums = [1, 1, 1, 1, 1]
  Output: true
  
  → Hop one step at a time → reach the end

Example 6:
  Input:  nums = [1, 0, 1, 0]
  Output: false
  
  → From index 0: jump 1 → index 1
  → From index 1: nums[1] = 0 → STUCK at index 1
  → Cannot reach index 2 or 3

🔍 HOW TO APPROACH IT

Step 1 — Recognize the Problem Structure

→ This is an ARRAY problem, not a graph problem
→ No explicit edges like Project 48 (Union Find)
→ No explicit dependencies like Project 47 (Topological Sort)
→ Connectivity is IMPLIED by the array values

→ From each index i, you can reach indices i+1, i+2, ..., i+nums[i]
→ This is like an IMPLICIT directed graph:
     → Nodes = array indices
     → Edges = possible jumps from each index
→ But building this graph explicitly would be O(n²) edges → wasteful

Step 2 — Consider Multiple Approaches

APPROACH A — BFS/DFS on Implicit Graph:
  → Treat each index as a node
  → From index i, explore all reachable indices
  → Check if last index is visited
  → Time: O(n²) worst case (each node can reach up to n others)
  → Space: O(n) for visited array
  → Works but slow

APPROACH B — Dynamic Programming (Bottom-Up):
  → dp[i] = can we reach index i from index 0?
  → dp[0] = true (starting point)
  → dp[i] = true if ANY dp[j] is true AND j + nums[j] >= i (for j < i)
  → Answer = dp[n-1]
  → Time: O(n²) → for each i, scan all previous j
  → Space: O(n)
  → Correct but slow — builds on Project 43 (Climbing Stairs) foundation

APPROACH C — Dynamic Programming (Top-Down / Backward):
  → Start from the last index, work backward
  → Mark each index as "good" (can reach end) or "bad" (cannot)
  → dp[n-1] = GOOD
  → dp[i] = GOOD if any index reachable from i is GOOD
  → Time: O(n²) worst case
  → Space: O(n)

APPROACH D — Greedy (Farthest Reachable):
  → Track the FARTHEST index reachable so far
  → Scan left to right
  → At each index i: if i <= farthest → we CAN stand here → update farthest
  → If farthest >= last index → return true
  → If i > farthest → stranded → return false
  → Time: O(n) → single pass
  → Space: O(1)
  → OPTIMAL — this is the interview answer

WHY is Greedy sufficient?
  → We don't need to know the EXACT path
  → We only need to know IF we can reach the end
  → The farthest reachable index is the ONLY thing that matters
  → If we can reach index i, and i + nums[i] extends our reach, that's all we need
  → No need to track which specific jumps we make

Step 3 — Understand the Greedy Insight

KEY OBSERVATION:
  → If you can reach index i, then you can reach ALL indices 0 to i
  → Why? Because from index 0 you can reach some index j ≤ i
  → From j you can reach some index k ≤ i
  → The reachable set is always a CONTIGUOUS PREFIX [0, farthest]
  → It NEVER has gaps

PROOF:
  → Suppose we can reach index 5 but NOT index 3
  → To reach index 5, we must pass through some path
  → That path goes through indices < 5
  → At some point, a jump goes OVER index 3 (from j to k where j < 3 < k)
  → But that same jump position j can also jump to 3 (since k > 3, nums[j] ≥ k-j > 3-j)
  → Contradiction: index 3 IS reachable
  → Therefore: reachable set is always [0, farthest] — no gaps

THIS MEANS:
  → We only need to track ONE number: farthest reachable index
  → No need for arrays, sets, or queues
  → Pure O(1) space greedy

Step 4 — Visualize the Greedy Scan

nums = [2, 3, 1, 1, 4]
Goal: reach index 4

Index:    0   1   2   3   4
Value:    2   3   1   1   4

Step 0: i=0, farthest=0
  → i(0) <= farthest(0) → can stand here
  → farthest = max(0, 0+2) = 2
  → Can reach up to index 2

Step 1: i=1, farthest=2
  → i(1) <= farthest(2) → can stand here
  → farthest = max(2, 1+3) = 4
  → Can reach up to index 4 → that's the last index!
  → return TRUE

We didn't even need to check indices 2, 3, 4.
nums = [3, 2, 1, 0, 4]
Goal: reach index 4

Index:    0   1   2   3   4
Value:    3   2   1   0   4

Step 0: i=0, farthest=0
  → i(0) <= farthest(0) → can stand here
  → farthest = max(0, 0+3) = 3

Step 1: i=1, farthest=3
  → i(1) <= farthest(3) → can stand here
  → farthest = max(3, 1+2) = 3 (no improvement)

Step 2: i=2, farthest=3
  → i(2) <= farthest(3) → can stand here
  → farthest = max(3, 2+1) = 3 (no improvement)

Step 3: i=3, farthest=3
  → i(3) <= farthest(3) → can stand here
  → farthest = max(3, 3+0) = 3 (STUCK — 0 jump)

Step 4: i=4, farthest=3
  → i(4) > farthest(3) → CANNOT stand here → STRANDED
  → return FALSE

COMMON GREEDY TRAITS:
  → Make one simple local decision at each step
  → Never backtrack or reconsider
  → Single pass through the data
  → O(n) time, O(1) space
  → Correctness requires a PROOF (not obvious)

🛠️ HOW CAN WE SOLVE IT

The Greedy Algorithm — Step by Step

1. Initialize farthest = 0
2. FOR i = 0 to n-1:
     a. IF i > farthest → cannot reach index i → return FALSE
     b. farthest = max(farthest, i + nums[i])
     c. IF farthest >= n-1 → can reach last index → return TRUE
3. Return TRUE (loop completed → last index reachable)

Why the Early Exit in Step 2c?

→ Once farthest >= n-1, we KNOW the end is reachable
→ No need to continue scanning
→ This is an OPTIMIZATION, not a correctness requirement
→ Without it, the algorithm still works — just does unnecessary iterations
→ With it: best case O(1) (if nums[0] >= n-1)

✅ WHAT SHOULD YOU DO

→ ALWAYS check i > farthest BEFORE updating farthest
     → If you can't reach index i, you can't use nums[i]
     → Order matters: check accessibility first, then extend reach

→ ALWAYS use max(farthest, i + nums[i]) for the update
     → i + nums[i] = farthest index reachable FROM index i
     → We keep the MAXIMUM across all reachable indices
     → A previous index might have given a farther reach than current

→ ALWAYS handle the single-element case
     → nums = [0] → already at last index → return true
     → nums = [X] for any X → already at last index → return true

→ CONSIDER early termination
     → If farthest >= n-1 at any point → return true immediately
     → Saves unnecessary iterations in best case

→ CONSIDER the DP approach for understanding
     → Implement DP first to verify correctness
     → Then optimize to greedy
     → This builds the intuition bridge from Project 43

🚫 WHAT SHOULD YOU NOT DO

→ DO NOT try to find the actual PATH
     → This problem asks BOOLEAN: can you reach the end?
     → Finding the actual path is a different (harder) problem
     → Greedy gives yes/no, not the sequence of jumps

→ DO NOT build an explicit graph
     → Creating adjacency list from jump possibilities = O(n²) edges
     → Then BFS/DFS on that graph = O(n²) time
     → Greedy does it in O(n) without any graph

→ DO NOT use recursion without memoization
     → Naive recursion: "can I reach end from index 0?"
       → Try each possible jump, recurse
     → Exponential time without memoization
     → Even with memoization → O(n²) → still worse than greedy

→ DO NOT confuse "maximum jump" with "exact jump"
     → nums[i] = 3 means you can jump 1, 2, OR 3 steps
     → NOT that you MUST jump exactly 3 steps
     → This is why the farthest reachable approach works
     → Every index between current and current+nums[i] is reachable

→ DO NOT update farthest for unreachable indices
     → If i > farthest, index i is unreachable
     → nums[i] is irrelevant — you can't stand there to jump
     → Updating farthest with unreachable index = WRONG answer

→ DO NOT use Union Find or Topological Sort
     → This is an array problem, not a graph connectivity problem
     → No undirected edges → Union Find doesn't apply (Project 48)
     → No dependencies → Topological Sort doesn't apply (Project 47)
     → Greedy is the right tool here

🔀 WHEN TO USE THIS PATTERN

✓ You need to determine REACHABILITY in an array
✓ Each position defines a RANGE of forward movement
✓ The reachable set is CONTIGUOUS (no gaps)
✓ You only need a YES/NO answer (not the actual path)
✓ A single forward scan can capture all necessary information
✓ The problem has OPTIMAL SUBSTRUCTURE and GREEDY CHOICE property

VARIANTS OF THIS PATTERN:
  → Jump Game I: Can you reach the end? (this project)
  → Jump Game II: Minimum jumps to reach the end (BFS-like greedy)
  → Jump Game III: Can you reach index with value 0? (BFS — bidirectional)
  → Jump Game IV: Min jumps with equal-value teleportation (BFS)

REAL-WORLD ANALOGS:
  → Can a car with fuel at each station reach the destination?
  → Can a signal propagate through a network of repeaters?
  → Can a project be completed given resource ranges at each stage?
  → Can a character cross a platformer level given jump abilities?

🚫 WHEN NOT TO USE THIS PATTERN

✗ You need the ACTUAL PATH (not just reachability)
     → Greedy tracks farthest reach, not the jump sequence
     → For the path → use DP with backtracking or BFS with parent tracking

✗ You need MINIMUM number of jumps
     → That's Jump Game II → BFS-like greedy with level counting
     → Different algorithm (though related)

✗ Jumps can go BACKWARD
     → Farthest-reachable greedy assumes FORWARD-only movement
     → Backward jumps break the contiguous reachable set property
     → Need BFS/DFS for bidirectional jump problems

✗ Reachable set is NOT contiguous
     → If you can reach index 5 but not index 3 → gaps exist
     → Greedy farthest-reach assumes no gaps
     → Need full BFS/DFS for gap-prone reachability

✗ The cost of jumping matters (weighted jumps)
     → Greedy doesn't account for jump costs
     → Need DP or Dijkstra for weighted optimization

⚙️ HOW IT ONLY WORKS — Internal Mechanics

The Farthest-Reachable Invariant

INVARIANT:
  After processing index i, 'farthest' equals the maximum index
  reachable from any index in [0, i] that is itself reachable from index 0.

FORMALLY:
  farthest = max(j + nums[j]) for all j in [0, i] where j is reachable

INITIALIZATION:
  Before processing: farthest = 0 (only index 0 is reachable)

MAINTENANCE:
  At index i:
    → If i > farthest: index i is unreachable → stop (invariant preserved trivially)
    → If i <= farthest: index i IS reachable → update farthest = max(farthest, i + nums[i])
    → This correctly extends the reach using index i's jump ability

TERMINATION:
  → If farthest >= n-1 at any point → last index reachable → return true
  → If we reach an index i where i > farthest → stranded → return false
  → If loop completes → farthest >= n-1 (otherwise would have returned false)

Why Contiguity is the Key Insight

CLAIM: The set of reachable indices from index 0 is always [0, farthest]

PROOF BY INDUCTION:

Base: Only index 0 is reachable → reachable set = [0, 0] ✓

Inductive step: 
  Assume reachable set at some point is [0, k]
  Consider any index j in [0, k]
  From j, we can reach j+1, j+2, ..., j+nums[j]
  The new reachable set includes all these indices
  Since j is in [0, k], and we can reach j+1 to j+nums[j]:
    → All indices from 0 to max(k, j+nums[j]) are reachable
    → This is still a contiguous range [0, new_farthest]

Therefore: reachable set is ALWAYS a contiguous prefix [0, farthest]

CONSEQUENCE:
  → We ONLY need to track the right boundary (farthest)
  → Everything to the left of farthest is automatically reachable
  → This reduces the problem to tracking a single integer

Why Greedy Beats DP Here

DP APPROACH (from Project 43 foundation):
  dp[i] = can index i be reached?
  dp[0] = true
  dp[i] = true if EXISTS j < i such that dp[j] == true AND j + nums[j] >= i

  → For each i, we scan ALL previous j → O(n) per index → O(n²) total
  → We're doing REDUNDANT work: checking many j values

GREEDY APPROACH:
  → Instead of asking "can I reach index i?"
  → Ask "what's the FARTHEST I can reach overall?"
  → If farthest >= i → yes, I can reach index i
  → This collapses the inner loop to a single comparison

  → DP: O(n²) time, O(n) space
  → Greedy: O(n) time, O(1) space
  → Greedy wins by exploiting the contiguity property

Greedy Correctness Proof

THEOREM: The greedy algorithm returns TRUE if and only if
         the last index is reachable from index 0.

PROOF (⟹ direction — if greedy returns TRUE, then end is reachable):
  → Greedy returns true when farthest >= n-1
  → farthest = max(j + nums[j]) for reachable j
  → If farthest >= n-1, then some reachable index j has j + nums[j] >= n-1
  → From j, we can jump to index n-1 → end is reachable ✓

PROOF (⟸ direction — if end is reachable, greedy returns TRUE):
  → Suppose there exists a path: 0 = p₀, p₁, p₂, ..., pₖ = n-1
  → At index p₀=0: farthest >= 0+nums[0] >= p₁ (since p₁ is reachable from 0)
  → At index p₁: p₁ <= farthest → we process it → farthest >= p₁+nums[p₁] >= p₂
  → By induction: at each pᵢ, farthest >= pᵢ₊₁
  → At pₖ₋₁: farthest >= pₖ = n-1 → greedy returns true ✓

Time Complexity

→ Single pass through the array: O(n)
→ Each index: O(1) work (comparison, max, comparison)
→ TOTAL: O(n)

→ Best case: O(1) if nums[0] >= n-1 (jump directly to end)
→ Worst case: O(n) full scan
→ Average case: O(n)

Space Complexity

→ One integer variable: farthest
→ No arrays, no data structures
→ TOTAL: O(1)

⚠️ COMMON PITFALLS AND FAILURE MODES

Pitfall 1: Not Checking Reachability Before Updating

WRONG:
  for (int i = 0; i < n; i++) {
      farthest = Math.max(farthest, i + nums[i]);  // ← always update
  }
  return farthest >= n - 1;

→ This updates farthest even for UNREACHABLE indices
→ nums = [1, 0, 5] → index 2 is unreachable (stuck at 1)
→ But wrong code computes farthest = max(1, 2+5) = 7 → returns true
→ WRONG! Correct answer is false

RIGHT:
  for (int i = 0; i < n; i++) {
      if (i > farthest) return false;               // ← check FIRST
      farthest = Math.max(farthest, i + nums[i]);
  }
  return true;

Pitfall 2: Off-by-One in Loop Bound

OPTION A — Loop to n-1 (inclusive):
  for (int i = 0; i < n; i++) { ... }
  → Checks all indices including the last
  → Works correctly (if we reach last index, farthest >= n-1)

OPTION B — Loop to n-2 (exclusive of last):
  for (int i = 0; i < n - 1; i++) { ... }
  → Doesn't process last index
  → If farthest >= n-1 after this loop → return true
  → This is also correct because:
       → We only need to REACH the last index, not jump FROM it
       → nums[n-1] doesn't matter — we're already there
  → Slightly cleaner — avoids checking i > farthest for the last index

BOTH are correct — choose one and be consistent

Pitfall 3: Single Element Array

nums = [0]
→ n = 1 → last index = 0 → already there
→ farthest = 0 → 0 >= 0 → return true ✓

→ The loop doesn't even execute if we use i < n-1
→ Or executes once and returns true if we use i < n
→ Either way: correct

→ EDGE CASE: always verify single-element arrays

Pitfall 4: Array of All Zeros

nums = [0, 0, 0, 0]
→ i=0: farthest = max(0, 0+0) = 0
→ i=1: i(1) > farthest(0) → return FALSE ✓

nums = [0]
→ Already at end → return TRUE ✓

→ All zeros except possibly the first → can never move → false (unless n=1)

Pitfall 5: Confusing with Jump Game II (Minimum Jumps)

Jump Game I (this problem):
  → CAN you reach the end? → YES/NO → Greedy farthest tracking
  → O(n) time, O(1) space

Jump Game II (different problem):
  → MINIMUM number of jumps to reach the end
  → Need to count "levels" of BFS-like expansion
  → Still greedy but with additional tracking (currentEnd, jumps)
  → O(n) time, O(1) space but more complex logic

→ Don't mix up the two — different questions, different algorithms

📈 SCALABILITY CONSIDERATIONS

n = array length

→ n = 100 → O(100) → instant
→ n = 100,000 → O(100,000) → instant
→ n = 10,000,000 → O(10,000,000) → fast (~100ms)
→ n = 1,000,000,000 → O(10^9) → borderline (memory for array is the issue)

→ Greedy is the MOST SCALABLE approach for this problem
→ Single pass, no extra memory, no recursion
→ Limited only by the array size itself

COMPARISON:
  → BFS/DFS on implicit graph: O(n²) → fails at n > 10,000
  → DP: O(n²) → fails at n > 10,000
  → Greedy: O(n) → handles n up to 10^8+ easily

🔍 INTERPRETABILITY CONSIDERATIONS

Greedy Approach — Mental Model:
  → "Imagine walking along a number line"
  → "At each position, you can see how far ahead you could jump"
  → "You keep track of the FARTHEST point you could possibly reach"
  → "As you walk forward, if you're still within reachable range, keep going"
  → "If you ever step beyond the farthest reachable point — you're stranded"
  → "If the farthest point covers the end — you can make it"

Alternative Mental Model — Fuel Tank:
  → "Imagine a car that starts with fuel = nums[0]"
  → "At each step forward, fuel decreases by 1"
  → "At each gas station (index), you can REFUEL up to nums[i]"
  → "You take max(current_fuel, nums[i]) — the better option"
  → "If fuel ever hits 0 before the end — stranded"
  → "This is equivalent to the farthest-reachable approach"

Interview Explanation:
  → "I scan left to right, tracking the farthest reachable index"
  → "At each index, if it's within reach, I update the farthest"
  → "If the farthest ever covers the last index, I return true"
  → "If I encounter an index beyond reach, I return false"
  → "Time: O(n), Space: O(1)"

COMMON TRAITS OF GREEDY PROBLEMS:
  → One simple metric captures everything (end time, farthest reach, frequency)
  → Local decision = global optimal (provable by exchange argument)
  → Single pass through data
  → O(n) or O(n log n) time, O(1) space
  → Correctness is NOT obvious → requires proof/intuition

🧠 KEY TAKEAWAY

WHAT WE LEARNED:
→ Array reachability can be solved with a SINGLE INTEGER (farthest)
→ The contiguity property (no gaps in reachable set) enables greedy
→ Greedy reduces O(n²) DP to O(n) single pass
→ Two equivalent mental models: farthest-reachable and fuel-tank
→ Early termination in both directions: success AND failure
→ Greedy correctness requires PROOF — not intuition alone

THE CORE MENTAL MODEL:
→ "Scan left to right"
→ "At each step, can I stand here? (i <= farthest)"
→ "If yes, extend my reach (farthest = max(farthest, i + nums[i]))"
→ "If no, I'm stranded (return false)"
→ "If reach covers the end (farthest >= n-1), I'm done (return true)"

PATTERN SIGNATURE:
→ Array of forward-jump capacities ✓
→ Reachability question (yes/no) ✓
→ Contiguous reachable set ✓
→ No backward movement ✓
→ = GREEDY FARTHEST-REACHABLE TRACKING
