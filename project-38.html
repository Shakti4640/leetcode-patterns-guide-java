PROJECT 38: Kth Largest Element in an Array

🎯 Top K Elements → Min-Heap of Size K → Heap-Based Selection

🧩 THE HYPOTHETICAL PROBLEM

→ You are given an UNSORTED array of integers
→ You are given a value K
→ Find the Kth LARGEST element in the array
→ Not the Kth distinct — the Kth in sorted order (duplicates count)

Example 1:
  Input:  nums = [3, 2, 1, 5, 6, 4], k = 2
  Sorted descending: [6, 5, 4, 3, 2, 1]
                       1st 2nd
  Output: 5

Example 2:
  Input:  nums = [3, 2, 3, 1, 2, 4, 5, 5, 6], k = 4
  Sorted descending: [6, 5, 5, 4, 3, 3, 2, 2, 1]
                       1st 2nd 3rd 4th
  Output: 4

Example 3:
  Input:  nums = [7, 7, 7, 7], k = 2
  Sorted descending: [7, 7, 7, 7]
                       1st 2nd
  Output: 7  (duplicates count as separate positions)

🔍 HOW TO APPROACH IT

Step 1 — Understand What "Kth Largest" Really Means

→ If array were sorted DESCENDING → answer is at index k-1
→ If array were sorted ASCENDING → answer is at index n-k
→ We don't need the FULL sorted order
→ We only need to identify ONE specific element
→ Sorting the entire array does MORE work than necessary
→ Key insight: we need to PARTIALLY order — just enough to find position K

Step 2 — Think About the Brute Force

Approach A: Sort the array → pick index n-k
→ Time: O(n log n) for sort + O(1) for access
→ Space: O(1) if in-place sort, O(n) if not
→ WORKS but does unnecessary work
→ We sorted ALL n elements when we only need K of them

Approach B: Find max K times, removing each time
→ Scan array → find max → remove → repeat K times
→ Time: O(n * K)
→ If K is close to n → O(n²) → worse than sorting
→ If K is small → O(n) → better than sorting
→ INCONSISTENT performance

Step 3 — The Heap Insight

→ From Project 35 we learned: heaps give us efficient access to extremes
→ A max-heap gives us the LARGEST element at the root
→ A min-heap gives us the SMALLEST element at the root

NAIVE HEAP APPROACH:
→ Put ALL elements into a max-heap
→ Extract max K times
→ The Kth extraction is the answer
→ Time: O(n) to build heap + O(K log n) for K extractions
→ Space: O(n) for the heap
→ Better than sorting when K << n
→ But still stores ALL n elements

THE BRILLIANT INSIGHT:
→ What if we only keep K elements in the heap?
→ Use a MIN-HEAP of size K
→ The root of this min-heap = the SMALLEST among the K largest
→ The SMALLEST of the K largest = the Kth LARGEST
→ That's our answer!

Step 4 — Visualize Why Min-Heap of Size K Works

nums = [3, 2, 1, 5, 6, 4], k = 3
Goal: 3rd largest = 4

Process each element, maintain min-heap of size K=3:

Step 1: Process 3 → heap = [3]          size=1 ≤ 3 → just add
Step 2: Process 2 → heap = [2, 3]       size=2 ≤ 3 → just add
Step 3: Process 1 → heap = [1, 3, 2]    size=3 = 3 → just add
Step 4: Process 5 → heap has size 3, 5 > root(1)
        → remove root(1), add 5
        → heap = [2, 3, 5]
Step 5: Process 6 → 6 > root(2)
        → remove root(2), add 6
        → heap = [3, 5, 6]              ← these ARE the top 3!
Step 6: Process 4 → 4 > root(3)
        → remove root(3), add 4
        → heap = [4, 5, 6]              ← these ARE the top 3!

Answer = root = 4 ✓ (3rd largest)

THE MAGIC:
→ At every point, the heap holds the K largest elements SEEN SO FAR
→ The min-heap root = smallest of those K = Kth largest overall
→ Any element smaller than root CANNOT be in the top K → reject it
→ Any element larger than root BELONGS in the top K → swap it in

Step 5 — Why MIN-Heap and Not MAX-Heap?

This is the most COUNTERINTUITIVE part of the pattern:
→ We want K LARGEST elements
→ Instinct says: use a MAX-heap!
→ But MAX-heap would give us the 1st largest at root — not the Kth

WITH MIN-HEAP OF SIZE K:
→ Root = smallest of the K largest = Kth largest ← EXACTLY what we need
→ We can COMPARE incoming elements against root in O(1)
→ If new element > root → it deserves to be in top K → swap in
→ If new element ≤ root → it's NOT in top K → skip it
→ The root acts as a GATEKEEPER

WITH MAX-HEAP OF SIZE K:
→ Root = largest of the K largest = 1st largest
→ To find Kth largest → we'd need to dig DEEP into the heap
→ No efficient way to access the Kth position
→ Useless for this problem

RULE: To find K LARGEST → use MIN-heap of size K
      To find K SMALLEST → use MAX-heap of size K
      The heap type is OPPOSITE to what you're looking for
      Because the ROOT acts as the BOUNDARY/GATEKEEPER

🛠️ HOW CAN WE SOLVE IT

The Algorithm — Line by Line

1. Create a min-heap (PriorityQueue in Java)
2. For each element in the array:
     a. If heap size < K → add element directly
     b. Else if element > heap.peek() (root):
          → heap.poll()    (remove the smallest of top K)
          → heap.offer(element)  (add the new larger element)
     c. Else → skip (element is not in top K)
3. After processing all elements:
     → heap.peek() = root of min-heap = Kth largest
     → Return heap.peek()

Simplified Version (Even Cleaner)

1. Create a min-heap
2. For each element:
     → Add to heap
     → If heap size > K → poll (remove smallest)
3. Return heap.peek()

WHY THIS WORKS:
→ We always add first, then trim
→ Trimming removes the SMALLEST (min-heap root)
→ After trim, heap has exactly K elements
→ The K elements remaining are always the K largest seen so far
→ Slightly more operations than the gatekeeper version
→ But simpler code, same asymptotic complexity

✅ WHAT SHOULD YOU DO

→ ALWAYS use min-heap for "K largest" problems
→ ALWAYS use max-heap for "K smallest" problems
→ ALWAYS keep heap size bounded at K
     → This is what gives O(n log K) instead of O(n log n)
→ ALWAYS peek (don't poll) for the final answer
     → The answer IS the root — no need to remove it
→ ALWAYS consider: does K change over time or is it fixed?
     → Fixed K → this approach is perfect
     → Changing K → might need different strategy
→ ALWAYS think about whether you need the Kth element or ALL top K
     → Kth element → peek at root
     → All top K → drain the entire heap into result list

🚫 WHAT SHOULD YOU NOT DO

→ DO NOT sort the entire array
     → O(n log n) vs O(n log K)
     → When K << n, the difference is massive
     → n = 10,000,000 and K = 10:
       Sort: 10^7 * 23 ≈ 230,000,000 operations
       Heap: 10^7 * 3.3 ≈ 33,000,000 operations → 7x faster

→ DO NOT use a max-heap of size n
     → Stores ALL elements → O(n) space
     → Then extract K times → O(K log n) for extractions
     → Total: O(n + K log n) time, O(n) space
     → Min-heap of size K: O(n log K) time, O(K) space → better

→ DO NOT use a max-heap of size K for "K largest"
     → Root = largest of top K = 1st largest
     → Cannot efficiently access Kth position
     → Wrong tool for the job

→ DO NOT forget that PriorityQueue in Java is a MIN-heap by default
     → new PriorityQueue<>() → min-heap ✓ (what we want here)
     → For max-heap: new PriorityQueue<>(Collections.reverseOrder())

→ DO NOT poll the heap at the end
     → peek() returns the answer without removing it
     → poll() also works but destroys the heap state
     → If you might need the heap again → peek is safer

→ DO NOT skip the "element > root" check
     → Adding element then polling works (simplified version)
     → But gatekeeper version avoids unnecessary add+remove operations
     → Gatekeeper = faster in practice (fewer heap operations)

🔀 WHEN TO USE THIS PATTERN

✓ "Find the Kth largest / smallest element"
✓ "Find the top K / bottom K elements"
✓ "Find the K most frequent elements"
✓ "Find the K closest points"
✓ K is significantly smaller than n (K << n)
✓ You need a SINGLE pass through the data
✓ Data might be streaming (elements arrive one by one)
✓ You need O(K) space, not O(n) space

SIGNATURE PHRASES IN PROBLEMS:
→ "Kth largest" → min-heap of size K
→ "Kth smallest" → max-heap of size K
→ "Top K" → min-heap of size K (collect all K at end)
→ "K closest" → max-heap of size K (with distance comparator)
→ "K most frequent" → min-heap of size K (with frequency comparator)

🚫 WHEN NOT TO USE THIS PATTERN

✗ K is close to n (K ≈ n)
     → O(n log K) ≈ O(n log n) → no advantage over sorting
     → Sorting might be simpler to implement

✗ You need the EXACT position (median = n/2th element)
     → For median specifically → Two Heaps pattern (Project 35)
     → Two Heaps maintains BOTH halves efficiently

✗ You need the result in SORTED order
     → Heap of size K gives top K but NOT sorted
     → Would need additional O(K log K) sort of the K elements
     → Sometimes sorting the full array is cleaner

✗ Array is already sorted
     → Just access index n-K directly → O(1)
     → Heap would be O(n log K) → wasteful

✗ You need to find Kth largest for MULTIPLE different K values
     → Building a new heap for each K → O(n log K) per query
     → Better: sort once O(n log n) → answer any K in O(1)

✗ QuickSelect is acceptable
     → QuickSelect: O(n) average, O(n²) worst case
     → Heap: O(n log K) guaranteed
     → If average case is fine → QuickSelect wins
     → If worst case matters → Heap is safer

⚙️ HOW IT ONLY WORKS — Internal Mechanics

The Invariant — Why the Heap Always Holds the Top K

CLAIM: After processing the first i elements, the heap contains
       the K largest among those i elements (or all i if i < K).

BASE CASE (i ≤ K):
→ We add every element → heap has all i elements
→ All i elements = the i largest of i elements ✓

INDUCTIVE STEP (i > K, heap has K elements):
→ New element arrives: nums[i]

CASE A: nums[i] ≤ heap.peek() (root = current Kth largest)
→ nums[i] is ≤ ALL K elements in the heap? NO — only ≤ the smallest
→ But: nums[i] ≤ smallest of current top K
→ Therefore: nums[i] is NOT in the top K
→ We skip it → heap still holds top K ✓

CASE B: nums[i] > heap.peek()
→ nums[i] is LARGER than the current Kth largest
→ nums[i] DESERVES to be in the top K
→ The current root (smallest of top K) gets KICKED OUT
→ nums[i] takes its place
→ Heap now holds the new top K ✓

THEREFORE: The invariant holds at every step.
After all n elements → heap holds top K of entire array.
Root = smallest of top K = Kth largest. ∎

Min-Heap Operations Refresher (from Project 35)

→ Project 35 covered heap internals in detail
→ Quick recap of what matters here:

PriorityQueue<Integer> minHeap = new PriorityQueue<>();
  → offer(x): Add element → bubbles UP → O(log size)
  → poll():   Remove root (minimum) → bubbles DOWN → O(log size)
  → peek():   View root without removing → O(1)
  → size():   Number of elements → O(1)

Since we keep heap size ≤ K:
  → Each offer/poll is O(log K), NOT O(log n)
  → This is the KEY performance advantage

Time Complexity Breakdown

For each of n elements:
  → Compare with root: O(1)
  → If added: offer O(log K) + possibly poll O(log K)
  → Worst case per element: O(log K)

Total: O(n log K)

COMPARE:
  → Sorting: O(n log n)
  → Max-heap full: O(n + K log n)
  → QuickSelect: O(n) average, O(n²) worst
  → Min-heap K: O(n log K) guaranteed

When K = 10 and n = 10,000,000:
  → log K = 3.3
  → log n = 23.3
  → Heap approach is ~7x fewer operations than sorting

Space Complexity Breakdown

→ Heap stores exactly K elements → O(K) space
→ No additional data structures needed
→ COMPARE with sorting: O(1) extra but modifies input
→ COMPARE with max-heap: O(n) space

When K = 10 and n = 10,000,000:
  → Heap: 10 integers in memory
  → Max-heap: 10,000,000 integers in memory
  → 1,000,000x difference in memory usage

Why the Gatekeeper Check Matters in Practice

SIMPLIFIED VERSION (always add, then trim):
  → Every element triggers: offer + conditional poll
  → Even elements smaller than root get added then immediately removed
  → Wasted work: 2 * O(log K) for elements that don't belong

GATEKEEPER VERSION (check before adding):
  → Compare with root: O(1)
  → If element ≤ root → skip entirely → O(1) total
  → If element > root → poll + offer → O(log K)
  → Only elements that ACTUALLY enter the heap pay O(log K)

In practice:
  → As we process more elements, the root (Kth largest) rises
  → More and more incoming elements fail the gatekeeper check
  → Vast majority of elements are handled in O(1)
  → Theoretical worst case is same: O(n log K)
  → But average constant factor is much smaller

⚠️ COMMON PITFALLS AND FAILURE MODES

Pitfall 1: Confusing Min-Heap vs Max-Heap for the Task

THE #1 MISTAKE IN INTERVIEWS:

"I need K largest... so I'll use a max-heap!"

WRONG → Max-heap root = LARGEST = 1st largest
     → To get Kth largest from max-heap → extract K times
     → But then you need ALL elements in the heap → O(n) space

CORRECT → Min-heap of size K
       → Root = SMALLEST of top K = Kth largest
       → O(K) space

MEMORY AID:
  → "K largest" → MIN-heap (opposite!)
  → "K smallest" → MAX-heap (opposite!)
  → The heap type is the OPPOSITE of what you seek
  → Because the root is the GATEKEEPER/BOUNDARY

Pitfall 2: Off-by-One in Heap Size

→ Heap should have EXACTLY K elements at the end
→ If heap.size() > K after adding → poll to trim
→ If heap.size() < K (array has fewer than K elements) → edge case!

int[] nums = {5, 3};  k = 5;
→ Only 2 elements but K=5 → impossible to find 5th largest
→ Problem constraints usually guarantee K ≤ n
→ But ALWAYS verify in the problem statement

Pitfall 3: Duplicate Elements

nums = [3, 3, 3, 3, 3], k = 3
→ Sorted descending: [3, 3, 3, 3, 3]
→ 3rd largest = 3

The heap handles this AUTOMATICALLY:
→ heap = [3, 3, 3] after processing 3 elements
→ Next 3: 3 is NOT > peek(3) → skip
→ Next 3: same → skip
→ Answer = peek() = 3 ✓

No special duplicate handling needed.

Pitfall 4: All Negative Numbers

nums = [-5, -3, -8, -1, -7], k = 2
→ Sorted descending: [-1, -3, -5, -7, -8]
→ 2nd largest = -3

Heap process:
→ Add -5: heap = [-5]
→ Add -3: heap = [-5, -3]           size=2=K → full
→ Process -8: -8 < peek(-5) → skip
→ Process -1: -1 > peek(-5) → remove -5, add -1 → heap = [-3, -1]
→ Process -7: -7 < peek(-3) → skip
→ Answer = peek() = -3 ✓

Works perfectly — comparison operators handle negatives naturally.

Pitfall 5: K = 1 (Find Maximum)

→ Min-heap of size 1
→ Root is always the single largest element seen
→ Equivalent to tracking a running maximum
→ Works but OVERKILL — a simple variable would suffice:
     int max = Integer.MIN_VALUE;
     for (int num : nums) max = Math.max(max, num);
→ Heap adds unnecessary overhead for K=1
→ Recognize K=1 as a special case in interviews

Pitfall 6: K = n (Find Minimum)

→ Min-heap of size n = the entire array in a heap
→ Root = smallest element = nth largest = minimum
→ Works but no advantage over simply finding the minimum
→ O(n log n) vs O(n) for a simple scan
→ Recognize K=n as a special case too

📈 SCALABILITY CONSIDERATIONS

SCENARIO 1: Static array, single query
  → n = 100, K = 10 → any approach works
  → n = 10^6, K = 100 → heap shines: O(10^6 * 7) ≈ 7M ops
  → n = 10^6, K = 10^6 → heap = sorting → use sort instead

SCENARIO 2: Streaming data (elements arrive over time)
  → Heap approach is NATURALLY streaming
  → Process each element as it arrives → O(log K) per element
  → Don't need to store all elements — only K at a time
  → Sorting CANNOT handle streaming — needs all data upfront

SCENARIO 3: Multiple queries with same K
  → Build heap once → O(n log K)
  → Answer: peek() → O(1)
  → If new elements arrive → update heap → O(log K) each
  → Sorting would need full re-sort for each new element

SCENARIO 4: Multiple queries with different K values
  → Heap approach: rebuild for each K → O(n log K) per query
  → Better: sort once → O(n log n) → answer any K in O(1)
  → Choose based on number of queries vs array size

MEMORY SCALING:
  → n = 10^9 (1 billion elements), K = 1000
  → Cannot fit all elements in memory
  → Heap of size 1000 fits easily → ONLY viable approach
  → This is the real-world use case for Top K pattern
  → Log files, sensor data, network packets — all streaming

🔍 INTERPRETABILITY CONSIDERATIONS

MENTAL MODEL — "The Bouncer at a VIP Club"
  → Club has exactly K spots (heap size = K)
  → Bouncer checks each person arriving (new element)
  → If club isn't full → let them in (heap.size() < K → add)
  → If club is full → compare with the LEAST important person inside
     → New person more important? → kick out least important, let new one in
     → New person less important? → reject them
  → At any time, the club holds the K most important people
  → The bouncer (root of min-heap) = least important person in the club
  → That person IS the Kth most important overall

WHY THIS IS INTUITIVE:
  → The root/bouncer is the THRESHOLD for entry
  → Only elements that EXCEED the threshold get in
  → The threshold naturally RISES as better elements arrive
  → At the end, the threshold = the Kth largest element

DEBUGGING:
  → Print the heap after each insertion
  → Watch the root value rise over time
  → The final root = your answer
  → If root seems wrong → check if you used min vs max heap
