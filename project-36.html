PROJECT 36: Next Greater Element

🎯 Monotonic Stack → Decreasing Stack Invariant → Right-Side Lookup

🧩 THE HYPOTHETICAL PROBLEM

→ You are given an array of integers
→ For EACH element, find the NEXT element to its RIGHT that is GREATER than it
→ If no such element exists, return -1 for that position
→ Return an array of the same size with the answers

Example 1:
  Input:  [4, 5, 2, 10, 8]
  Output: [5, 10, 10, -1, -1]
  
  → 4's next greater: 5  (immediately to the right)
  → 5's next greater: 10 (skip 2, find 10)
  → 2's next greater: 10 (immediately to the right)
  → 10's next greater: -1 (nothing to the right is bigger)
  → 8's next greater: -1  (nothing to the right at all)

Example 2:
  Input:  [3, 1, 2, 4]
  Output: [4, 2, 4, -1]

  → 3's next greater: 4  (skip 1, skip 2, find 4)
  → 1's next greater: 2  (immediately right)
  → 2's next greater: 4  (immediately right)
  → 4's next greater: -1 (end of array)

Example 3:
  Input:  [5, 4, 3, 2, 1]
  Output: [-1, -1, -1, -1, -1]

  → Strictly decreasing → no element has a next greater

Example 4:
  Input:  [1, 2, 3, 4, 5]
  Output: [2, 3, 4, 5, -1]

  → Strictly increasing → each element's next greater is its immediate neighbor

🔍 HOW TO APPROACH IT

Step 1 — The Brute Force Is Obvious but Slow

→ For each element at index i:
     → Scan right from i+1 to n-1
     → Find the first element > nums[i]
     → Record it (or -1 if not found)

for (int i = 0; i < n; i++) {
    result[i] = -1;
    for (int j = i + 1; j < n; j++) {
        if (nums[j] > nums[i]) {
            result[i] = nums[j];
            break;
        }
    }
}

→ Time: O(n²) worst case (descending array → each element scans to the end)
→ For n = 10^5: ~10^10 operations → TIMEOUT
→ We need O(n)

Step 2 — Why Brute Force Wastes Work

Consider: [4, 5, 2, 10, 8]

Brute force processing:
  i=0: 4 → scan 5 → FOUND (stop)
  i=1: 5 → scan 2 → no → scan 10 → FOUND
  i=2: 2 → scan 10 → FOUND
  i=3: 10 → scan 8 → no → END → -1
  i=4: 8 → END → -1

Notice:
  → When we process 5 (i=1), we scan past 2 to reach 10
  → When we process 2 (i=2), we ALSO scan to reach 10
  → The element 10 "answers" BOTH 5 and 2
  → But brute force discovers this independently for each

INSIGHT:
  → Multiple elements can be WAITING for the same answer
  → When a large element arrives, it answers ALL smaller waiting elements
  → If we could track "who is waiting" → we avoid redundant scanning
  → This is what the STACK does

Step 3 — The Stack as a "Waiting Room"

METAPHOR:
  → Imagine a queue of people standing in line
  → Each person is looking for someone TALLER to their right
  → When a tall person arrives, everyone shorter in front says "Found you!"
  → They leave the line (pop from stack)
  → The tall person then joins the line to wait for THEIR taller person

THE STACK HOLDS: elements that haven't found their "next greater" yet
THE INVARIANT: stack is in DECREASING order from bottom to top

WHY DECREASING?
  → If element A is below element B in the stack (A entered first)
  → And A < B → then B would have been A's next greater → A should have left
  → Contradiction → A must be ≥ B → stack is decreasing (or non-increasing)

WHEN A NEW ELEMENT ARRIVES:
  → Compare with stack top
  → If new > stack top → stack top FOUND its next greater → POP
  → Keep popping while new > stack top
  → Then PUSH new onto stack (it's now waiting for ITS next greater)

AFTER PROCESSING ALL ELEMENTS:
  → Anything remaining on the stack → never found a next greater → answer is -1

Step 4 — Full Walkthrough

Array: [4, 5, 2, 10, 8]
Stack stores INDICES (so we can fill the result array)
Result: [_, _, _, _, _]

Process right to left? Or left to right?
→ We'll do LEFT TO RIGHT (more intuitive for "next greater")

i=0, nums[0]=4:
  Stack is empty → nothing to compare
  Push 0 onto stack
  Stack: [0]  (values: [4])

i=1, nums[1]=5:
  Stack top = index 0, nums[0]=4
  5 > 4 → POP index 0 → result[0] = 5
  Stack empty → stop popping
  Push 1 onto stack
  Stack: [1]  (values: [5])
  Result: [5, _, _, _, _]

i=2, nums[2]=2:
  Stack top = index 1, nums[1]=5
  2 > 5? NO → stop popping
  Push 2 onto stack
  Stack: [1, 2]  (values: [5, 2])
  Result: [5, _, _, _, _]

i=3, nums[3]=10:
  Stack top = index 2, nums[2]=2
  10 > 2 → POP index 2 → result[2] = 10
  Stack top = index 1, nums[1]=5
  10 > 5 → POP index 1 → result[1] = 10
  Stack empty → stop popping
  Push 3 onto stack
  Stack: [3]  (values: [10])
  Result: [5, 10, 10, _, _]

i=4, nums[4]=8:
  Stack top = index 3, nums[3]=10
  8 > 10? NO → stop popping
  Push 4 onto stack
  Stack: [3, 4]  (values: [10, 8])
  Result: [5, 10, 10, _, _]

Done processing. Remaining stack: [3, 4]
  → result[3] = -1
  → result[4] = -1

Final Result: [5, 10, 10, -1, -1] ✓

OBSERVATIONS:
  → Element 10 resolved BOTH 2 and 5 in a single pass
  → No redundant scanning
  → Each element was pushed once and popped at most once
  → Total operations: exactly 2n (n pushes + at most n pops)

Step 5 — Why This is a NEW Pattern (Not Related to Previous Structures)

Connection to Project 9 (Binary Search):
  → Binary search: eliminate half based on comparison with target
  → Monotonic stack: eliminate elements based on comparison with NEW arrival
  → Both reduce work through elimination
  → But the elimination MECHANISM is completely different

Connection to Project 35 (Two Heaps):
  → Heaps: maintain partial order for efficient top-element access
  → Monotonic stack: maintain total order for efficient "next greater" resolution
  → Both maintain ORDER INVARIANTS on a data structure
  → But the ACCESS PATTERN differs:
       → Heaps: query the current best (peek)
       → Stacks: resolve waiting elements when their answer arrives (pop)

WHAT MAKES MONOTONIC STACK UNIQUE:
  → Elements WAIT on the stack until their condition is met
  → A single arrival can resolve MULTIPLE waiting elements
  → This "batch resolution" is what gives O(n) amortized time
  → No other pattern we've seen does this

THE KEY MENTAL MODEL:
  → Stack = "waiting room"
  → Push = "start waiting"
  → Pop = "found my answer"
  → New element = "potential answer for everyone waiting"

Step 6 — Left-to-Right vs Right-to-Left Processing

LEFT TO RIGHT (what we showed above):
  → Push index onto stack
  → When new element pops old elements → new element IS the answer
  → result[popped_index] = nums[current_index]
  → Elements remaining on stack at the end → answer is -1

RIGHT TO LEFT (alternative):
  → Process from right end
  → Stack holds POTENTIAL answers (elements to the right)
  → For current element: pop stack until we find something greater
  → Stack top (if exists) IS the next greater
  → Then push current element

BOTH produce identical results
LEFT TO RIGHT is more intuitive for "next greater"
RIGHT TO LEFT is more intuitive for some variants

We'll implement BOTH in the code

🛠️ HOW CAN WE SOLVE IT

Approach A — Left to Right (Push Index, Pop When Resolved)

1. Initialize stack (stores INDICES)
2. Initialize result array with -1 (default: no next greater)
3. For i = 0 to n-1:
     a. While stack is not empty AND nums[i] > nums[stack.peek()]:
          → Pop index from stack
          → result[popped_index] = nums[i]  (current element is the answer)
     b. Push i onto stack
4. Return result (remaining stack indices already have -1)

Approach B — Right to Left (Stack Holds Potential Answers)

1. Initialize stack (stores VALUES or INDICES)
2. Initialize result array
3. For i = n-1 down to 0:
     a. While stack is not empty AND stack.peek() ≤ nums[i]:
          → Pop (this element is too small to be the answer)
     b. If stack is empty:
          → result[i] = -1 (no greater element to the right)
     c. Else:
          → result[i] = stack.peek() (top of stack is the next greater)
     d. Push nums[i] onto stack
4. Return result

Which to Use When?

→ LEFT TO RIGHT:
     → Stack stores INDICES of elements waiting for their answer
     → New element resolves waiting elements by POPPING them
     → More natural for: "next greater to the RIGHT"
     → Elements find their answer at the moment it arrives

→ RIGHT TO LEFT:
     → Stack stores CANDIDATES (potential answers)
     → For each element, we search the stack for its answer
     → More natural for: "find answer among elements already processed"
     → Simpler to understand for some people

→ Both are O(n) with identical results
→ Interview: use whichever you can explain more clearly
→ We'll use LEFT TO RIGHT as the primary approach

✅ WHAT SHOULD YOU DO

→ ALWAYS store INDICES on the stack (not values)
     → You need to know WHERE to write the result
     → result[popped_index] = nums[current_index]
     → If you store values → you can't fill the result array correctly
     → Exception: right-to-left approach can store values if just looking up answers

→ ALWAYS initialize result array with -1
     → Default answer for elements that never find a next greater
     → Elements remaining on the stack at the end automatically get -1
     → No need to process the remaining stack separately

→ ALWAYS use STRICT inequality for "next greater"
     → nums[i] > nums[stack.peek()] → strictly greater
     → Using >= would find "next greater or EQUAL" → different problem
     → Read the problem carefully: "greater" vs "greater or equal"

→ ALWAYS verify the stack maintains decreasing order
     → After each push: stack top should be ≤ element below it
     → If not → you're popping in the wrong direction

→ ALWAYS think about what the stack REPRESENTS
     → In left-to-right: "elements still waiting for their next greater"
     → In right-to-left: "candidates that could be someone's next greater"
     → Clear mental model prevents bugs

→ ALWAYS consider the amortized complexity argument
     → Inner while loop looks like it could make this O(n²)
     → But each element is pushed ONCE and popped AT MOST ONCE
     → Total pushes: n, total pops: ≤ n → total operations: ≤ 2n → O(n)

🚫 WHAT SHOULD YOU NOT DO

→ DO NOT use a regular stack and scan it from bottom
     → Stacks only give efficient access to the TOP
     → Scanning from bottom is O(n) per element → back to O(n²)
     → The whole point is: we only interact with the TOP

→ DO NOT forget to push the current element AFTER popping
     → After resolving all smaller elements → current element joins the wait
     → Forgetting to push → current element is never considered
     → Its next greater is never found → wrong result

→ DO NOT pop elements that are GREATER than the current
     → We pop only elements SMALLER than the current (they found their answer)
     → Greater elements are still waiting → leave them on the stack
     → Popping greater elements → they lose their chance to find their answer

→ DO NOT use this for "next SMALLER element" without flipping the comparison
     → Next greater: pop when current > stack top (decreasing stack)
     → Next smaller: pop when current < stack top (increasing stack)
     → The stack order FLIPS for the opposite query

→ DO NOT confuse "next greater" with "maximum to the right"
     → Next greater: FIRST element to the right that is larger
     → Maximum to right: LARGEST element to the right
     → Different problems → different solutions
     → Next greater uses monotonic stack
     → Maximum to right uses suffix maximum array

→ DO NOT try to use a heap for this problem
     → Heap gives the overall max/min → not the NEXT positional greater
     → Positional information matters → stack preserves it, heap doesn't

🔀 WHEN TO USE THIS PATTERN

✓ "Find the NEXT greater/smaller element" for each position
     → Next greater to the right (this problem)
     → Next smaller to the right
     → Previous greater to the left
     → Previous smaller to the left

✓ "For each element, find the nearest element satisfying a comparison"
     → The comparison is always > or < (monotonic relationship)
     → The search direction is always left or right

✓ Problems involving "bounded regions"
     → Largest rectangle in histogram (Project 44)
     → Trapping rain water (Project 45)
     → These need "next smaller on both sides" → two passes

✓ "How many days until a warmer temperature"
     → Classic variant: instead of the VALUE, return the DISTANCE
     → result[i] = index_of_next_greater - i

✓ Stock span problems
     → "How many consecutive days was the price ≤ today's?"
     → Previous greater element variant

✓ Any problem where brute force does "scan right/left until condition"
     → If you see nested loops where inner loop scans for first match
     → Monotonic stack likely applies

✓ Circular array variants
     → "Next greater in a circular array"
     → Process the array TWICE (2n iterations)
     → Use modulo for index wrapping

🚫 WHEN NOT TO USE THIS PATTERN

✗ You need the Kth greater element (not the NEXT/first)
     → Monotonic stack finds the FIRST greater
     → Kth greater needs different approach (heap or sorting)

✗ The comparison is not monotonic
     → "Find next element whose value is within range [a, b]"
     → Non-monotonic condition → stack invariant doesn't help

✗ You need ALL greater elements to the right (not just the next)
     → Monotonic stack gives ONE answer per element
     → All greater elements needs different structure

✗ The array is modified between queries
     → Monotonic stack processes the array ONCE
     → If elements change → need segment tree or other dynamic structure

✗ Elements have complex multi-dimensional comparison
     → "Next element where both height AND weight are greater"
     → Two-dimensional comparison → monotonic stack for 1D only
     → Need sorting + stack combination or different approach

⚙️ HOW IT ONLY WORKS — Internal Mechanics

The Decreasing Stack Invariant

INVARIANT: At all times, elements on the stack are in DECREASING order
           from bottom to top.

Stack (bottom → top): [10, 7, 5, 3]
                        ↓   ↓  ↓  ↓
                      biggest    smallest (top)

WHY DECREASING?
  → Suppose stack has [A, B] where A is below B and A < B
  → A entered the stack BEFORE B (A is older)
  → When B was pushed, we should have checked: B > A?
  → If B > A → A should have been POPPED (A found its next greater = B)
  → But A is still here → contradiction
  → Therefore A ≥ B → stack is decreasing ✓

MAINTAINING THE INVARIANT:
  → New element arrives with value X
  → While stack top < X:
       → Pop stack top (it found its answer: X)
       → Stack top was the smallest waiting element
       → X is the first thing bigger than it → correct answer
  → Push X
  → Stack is still decreasing (X ≤ new stack top, or stack was empty)

Amortized O(n) Proof

CLAIM: Total time is O(n) despite the inner while loop

PROOF using the ACCOUNTING METHOD:
  → Each element is PUSHED onto the stack exactly ONCE (when we reach it)
  → Each element is POPPED from the stack AT MOST ONCE (when its answer arrives)
  → Push cost: O(1) per push → n pushes → O(n) total
  → Pop cost: O(1) per pop → at most n pops → O(n) total
  → Total: O(n) + O(n) = O(n)

WHY THE INNER WHILE LOOP DOESN'T MAKE IT O(n²):
  → Yes, for a single element, the while loop might pop many elements
  → Example: [1, 2, 3, 4, 5, 100] → when 100 arrives, it pops 5, 4, 3, 2, 1
  → That's 5 pops in one iteration!
  → But those 5 elements can NEVER be popped again
  → Total pops across ALL iterations ≤ n
  → The "burst" of pops is paid for by the previous "quiet" pushes

ANALOGY:
  → Imagine n coins on a table
  → Each push puts a coin in a jar (1 coin per element)
  → Each pop removes a coin from the jar
  → You can never remove more coins than you put in
  → Total removals ≤ n → total work ≤ 2n → O(n)

The Four Variants of Monotonic Stack

┌─────────────────────────┬──────────────┬────────────────┬──────────────────┐
│ Problem                 │ Stack Order  │ Process Dir    │ Pop Condition    │
├─────────────────────────┼──────────────┼────────────────┼──────────────────┤
│ Next Greater (right)    │ Decreasing   │ Left → Right   │ curr > top       │
│ Next Smaller (right)    │ Increasing   │ Left → Right   │ curr < top       │
│ Previous Greater (left) │ Decreasing   │ Left → Right   │ curr > top       │
│ Previous Smaller (left) │ Increasing   │ Left → Right   │ curr < top       │
└─────────────────────────┴──────────────┴────────────────┴──────────────────┘

NEXT vs PREVIOUS:
  → NEXT greater: "the answer IS the current element" → result[popped] = curr
  → PREVIOUS greater: "the answer is WHAT'S LEFT on stack" → result[curr] = stack.peek()

GREATER vs SMALLER:
  → GREATER: use DECREASING stack → pop when current is BIGGER
  → SMALLER: use INCREASING stack → pop when current is SMALLER

MNEMONIC:
  → "Next Greater" → Decreasing stack → pop the SMALLER ones
  → "Next Smaller" → Increasing stack → pop the BIGGER ones
  → The stack holds elements that are "safe" from popping
  → "Safe from greater" = you're already big = decreasing stack
  → "Safe from smaller" = you're already small = increasing stack

⚠️ COMMON PITFALLS AND FAILURE MODES

Pitfall 1: Storing Values Instead of Indices

WRONG (for left-to-right approach):
  Stack<Integer> stack = new Stack<>();
  // ...
  stack.push(nums[i]);  // ← Pushing VALUE
  // ...
  int popped = stack.pop();
  result[???] = nums[i];  // ← WHERE to write? We lost the index!

CORRECT:
  Stack<Integer> stack = new Stack<>();
  // ...
  stack.push(i);  // ← Pushing INDEX
  // ...
  int poppedIdx = stack.pop();
  result[poppedIdx] = nums[i];  // ← Write to the correct position ✓

Pitfall 2: Using >= Instead of > for "Strictly Greater"

WRONG:
  while (!stack.isEmpty() && nums[i] >= nums[stack.peek()]) {
      // This pops elements that are EQUAL too
  }

→ Array: [3, 3, 3, 4]
→ With >=: when second 3 arrives, it pops first 3 → result[0] = 3
→ But 3 is NOT greater than 3 → WRONG

CORRECT:
  while (!stack.isEmpty() && nums[i] > nums[stack.peek()]) {
      // Only pops elements that are STRICTLY smaller
  }

→ With >: second 3 arrives, 3 > 3? NO → don't pop → correct
→ When 4 arrives: 4 > 3? YES → pop all three 3's → result = [4, 4, 4, -1] ✓

Pitfall 3: Forgetting to Push After Popping

WRONG:
  for (int i = 0; i < n; i++) {
      while (!stack.isEmpty() && nums[i] > nums[stack.peek()]) {
          result[stack.pop()] = nums[i];
      }
      // FORGOT: stack.push(i);
  }

→ Stack stays empty after first element
→ No elements ever wait → all results stay -1
→ Only result[0] might be correct (if second element is greater)

CORRECT:
  for (int i = 0; i < n; i++) {
      while (!stack.isEmpty() && nums[i] > nums[stack.peek()]) {
          result[stack.pop()] = nums[i];
      }
      stack.push(i);  // ← ALWAYS push current element
  }

Pitfall 4: Processing in Wrong Direction for the Variant

→ "Next greater to the RIGHT" → process LEFT to RIGHT
→ "Previous greater to the LEFT" → ALSO process LEFT to RIGHT
     → But the answer comes from the stack, not the current element
→ Mixing up which direction → wrong results

RULE OF THUMB:
  → For "NEXT" (looking right): answer is the CURRENT element → result[popped] = curr
  → For "PREVIOUS" (looking left): answer is the STACK TOP → result[curr] = stack.peek()

Pitfall 5: Circular Array Handling

→ Problem: "Next Greater Element in a CIRCULAR array"
→ Array [2, 1, 3] circularly → after 3, we wrap to 2, 1, 3 again
→ 3's next greater (circularly) might be found by wrapping

SOLUTION: Process the array TWICE (2n iterations)
  for (int i = 2 * n - 1; i >= 0; i--) {
      int idx = i % n;
      // ... same logic but use idx instead of i
  }

→ This simulates the circular wrap
→ Use modulo for index
→ Only write results for indices 0 to n-1

Pitfall 6: Using Java Stack vs ArrayDeque

→ java.util.Stack is legacy → synchronized → slower
→ Preferred: java.util.ArrayDeque used as a stack

CORRECT:
  Deque<Integer> stack = new ArrayDeque<>();
  stack.push(i);        // Push to top
  stack.peek();         // Look at top
  stack.pop();          // Remove from top
  stack.isEmpty();      // Check if empty

→ ArrayDeque is faster (no synchronization overhead)
→ Both work correctly → but ArrayDeque is best practice in Java

📈 SCALABILITY CONSIDERATIONS

Time Complexity: O(n)
  → Each element pushed once, popped at most once
  → Total operations ≤ 2n
  → Strictly linear

Space Complexity: O(n)
  → Stack can hold at most n elements (worst case: decreasing array)
  → Result array: O(n)
  → Total: O(n)

Practical Performance:
  → n = 10^5: ~200K operations → instant
  → n = 10^6: ~2M operations → few milliseconds
  → n = 10^7: ~20M operations → tens of milliseconds
  → n = 10^8: ~200M operations → ~1 second

Comparison with Brute Force:
  → Brute force: O(n²) → n=10^5 → 10^10 → TIMEOUT
  → Monotonic stack: O(n) → n=10^5 → 2×10^5 → instant
  → Speedup: ~50,000x for n=10^5

Memory Considerations:
  → Stack stores ints (indices) → 4 bytes each
  → n = 10^7 → 40MB → acceptable
  → n = 10^8 → 400MB → might be tight
  → For extreme cases: can use int array instead of Deque

🔍 INTERPRETABILITY CONSIDERATIONS

→ The "waiting room" metaphor is very powerful:
     → Elements enter the waiting room (push onto stack)
     → They're waiting for someone taller (greater) to arrive
     → When a tall person enters, shorter people leave (pop)
     → They leave saying "THAT person is my next greater"
     → The tall person then sits down to wait for THEIR taller person

→ Visual debugging is easy:
     → Print stack contents at each step
     → Watch elements enter and leave
     → Verify the decreasing invariant holds

→ Interview communication:
     → "I use a monotonic decreasing stack"
     → "Elements wait on the stack until their next greater arrives"
     → "When a new element pops smaller ones, IT is their answer"
     → "Each element is pushed and popped at most once → O(n)"
     → "The stack maintains decreasing order from bottom to top"

→ Common interviewer follow-ups:
     → "What if we want next SMALLER?" → flip the comparison, use increasing stack
     → "What about circular array?" → process 2n elements with modulo
     → "What about PREVIOUS greater?" → answer comes from stack, not current
     → "Can you do it with O(1) extra space?" → generally no, stack is needed

🧠 KEY TAKEAWAY

WHAT WE LEARNED:
→ Monotonic stack is a stack that maintains elements in sorted order
→ For "next greater": use a DECREASING stack (bottom=big, top=small)
→ When a new element is bigger than stack top → stack top found its answer
→ Multiple elements can be resolved by a single new arrival
→ Each element is pushed once and popped at most once → O(n) amortized

THE FOUR VARIANTS:
  → Next Greater (right):     decreasing stack, answer = current element
  → Next Smaller (right):     increasing stack, answer = current element
  → Previous Greater (left):  decreasing stack, answer = stack top
  → Previous Smaller (left):  increasing stack, answer = stack top

THE "WAITING ROOM" MENTAL MODEL:
  → Elements sit on the stack WAITING for their condition to be met
  → A new arrival checks: "Am I the answer for anyone waiting?"
  → Resolved elements POP off → their result is recorded
  → The new element then sits down to wait for ITS answer
  → This "wait + batch resolve" pattern is the essence of monotonic stacks

WHY IT'S O(n):
  → Each of n elements: 1 push + at most 1 pop = 2 operations
  → Total: ≤ 2n operations → O(n)
  → The inner while loop is amortized over all iterations

PATTERN SIGNATURE:
  → "For each element, find the nearest greater/smaller" ✓
  → "First element to the right/left satisfying a comparison" ✓
  → Brute force has nested loop scanning for first match ✓
  → = MONOTONIC STACK

→ That information lets us SEPARATE the problem into two simpler subproblems
