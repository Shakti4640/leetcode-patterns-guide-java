🧩 THE HYPOTHETICAL PROBLEM

→ You are given a 2D grid (board) of characters
→ You are given a list of WORDS
→ Find ALL words from the list that can be formed on the board
→ A word is formed by ADJACENT cells (horizontally or vertically)
→ The SAME cell cannot be used twice in a single word
→ Return the list of found words (no duplicates)

Example:
  Board:                    Words: ["oath", "pea", "eat", "rain"]
  ┌───┬───┬───┬───┐
  │ o │ a │ a │ n │
  ├───┼───┼───┼───┤
  │ e │ t │ a │ e │
  ├───┼───┼───┼───┤
  │ i │ h │ k │ r │
  ├───┼───┼───┼───┤
  │ i │ f │ l │ v │
  └───┴───┴───┴───┘

  "oath":
    o(0,0) → a(0,1) → t(1,1) → h(2,1) ✓ FOUND

  "pea":
    No 'p' on board → NOT FOUND

  "eat":
    e(1,0) → a(0,0)? No, a(0,1)? 
    e(1,3) → a(1,2) → t(1,1) ✓ FOUND

  "rain":
    r(2,3) → a(1,2) → i(2,2)? No 'i' at (2,2), k is there
    → NOT FOUND on this board

  Output: ["oath", "eat"]

🔍 HOW TO APPROACH IT

Step 1 — Understand the Naive Approach and Why It Fails

NAIVE APPROACH:
→ For EACH word in the word list:
     → For EACH cell on the board:
          → Run DFS trying to match the word character by character
          → If all characters match → word found

Time complexity:
→ W words × M×N starting cells × 4^L DFS branches per word
→ Where L = max word length
→ If W = 30,000 and board is 12×12 and L = 10:
     → 30,000 × 144 × 4^10 ≈ 30,000 × 144 × 1,048,576
     → ≈ 4.5 × 10^12 operations → WAY TOO SLOW

THE PROBLEM:
→ Each word triggers a SEPARATE DFS
→ Words sharing prefixes ("eat", "ear", "east") trigger redundant work
→ DFS for "eat" explores the same path as DFS for "ear" up to "ea"
→ Massive duplication of effort

Step 2 — The Trie Insight

KEY REALIZATION:
→ Instead of searching for ONE word at a time
→ Search for ALL words SIMULTANEOUSLY using a Trie

HOW?
→ Build a Trie from the word list
→ Start DFS from each cell on the board
→ At each cell, check: does the Trie have a child for this character?
     → YES → continue DFS (this path could lead to a word)
     → NO → PRUNE (no word in our list has this prefix → stop exploring)
→ If we reach a node where isEndOfWord = true → FOUND a word

WHY IS THIS FASTER?
→ One DFS explores paths for ALL words simultaneously
→ Shared prefixes are explored only ONCE
→ Dead-end prefixes are pruned IMMEDIATELY
→ Instead of W separate DFS runs → ONE DFS guided by Trie

Time improvement:
→ M×N starting cells × 4^L DFS per cell (but heavily pruned by Trie)
→ In practice: Trie prunes >99% of branches
→ From ~10^12 operations to ~10^6 operations

Step 3 — The Three Components Working Together

COMPONENT 1: TRIE (from Project 58)
→ Built from the word list
→ Provides prefix validation: "does any word start with this?"
→ Provides word detection: "is this a complete word?"

COMPONENT 2: GRID DFS (new in this project)
→ From each cell, explore all 4 directions
→ Track visited cells (no reuse in same path)
→ Build the current path character by character

COMPONENT 3: BACKTRACKING (from Projects 26, 28, 42)
→ Mark cell as visited BEFORE exploring neighbors
→ UNMARK cell AFTER exploring (so other paths can use it)
→ This is the "choose → explore → unchoose" pattern

THE FUSION:
→ At each DFS step:
     1. Check if current character has a Trie child → if not, PRUNE
     2. Move to that Trie child node
     3. If isEndOfWord → record the word
     4. Mark grid cell as visited
     5. Explore all 4 neighbors recursively
     6. Unmark grid cell (backtrack)

🛠️ HOW CAN WE SOLVE IT

Full Algorithm

STEP 1: Build Trie from word list
→ For each word: trie.insert(word)
→ Store the COMPLETE WORD at the end node (for easy retrieval)

STEP 2: For each cell (i, j) on the board
→ If Trie root has a child matching board[i][j]:
     → Start DFS from (i, j) with Trie root's child

STEP 3: DFS function — dfs(board, i, j, trieNode, result)
→ Base cases:
     → Out of bounds → return
     → Cell already visited → return
     → Trie node has no child for board[i][j] → PRUNE → return
→ Get the Trie child node for current character
→ If this Trie node marks end of word:
     → Add the word to result
     → Set isEndOfWord = false (prevent duplicates)
→ Mark current cell as visited (board[i][j] = '#')
→ Explore all 4 neighbors: up, down, left, right
→ Unmark current cell (restore original character)
→ OPTIMIZATION: if Trie node has no remaining children → prune node

STEP 4: Return result list

Critical Optimization — Trie Node Pruning

PROBLEM:
→ After finding "oath", the Trie still has the path o→a→t→h
→ Future DFS from other cells will still explore this dead path
→ If "oath" is the ONLY word with prefix "oath"
→ We're wasting time exploring a path we've already found

SOLUTION: Remove leaf nodes after finding a word
→ When we find a word at a Trie node:
     → Mark it found (remove word / set isEndOfWord = false)
     → If the node has NO children → it's a dead leaf
     → Remove it from its parent's children array
     → This "bubbles up": if parent also becomes childless → remove it too

IMPACT:
→ Each found word REMOVES its unique suffix from the Trie
→ Shared prefixes remain until ALL words using them are found
→ The Trie SHRINKS as words are found → DFS gets faster over time
→ This turns worst-case scenarios from TLE to AC

Visited Marking — In-Place Trick

Instead of using a boolean[][] visited array:
→ Temporarily replace board[i][j] with a special character '#'
→ '#' won't match any Trie child → naturally prevents revisiting
→ After DFS returns → restore the original character

WHY this works:
→ board[i][j] = '#' → when any neighbor tries to visit this cell:
     → It checks Trie child for '#' → no Trie node for '#' → PRUNE
     → Effectively "visited" without extra data structure
→ O(1) mark and unmark
→ No extra O(M×N) space for visited array

✅ WHAT SHOULD YOU DO

→ ALWAYS build the Trie FIRST, then DFS
     → Trie construction: O(sum of word lengths)
     → DFS guided by Trie: much faster than word-by-word DFS
     → This order is non-negotiable

→ ALWAYS store the complete word at the Trie leaf node
     → When you find a word during DFS, you need to know WHAT word it is
     → Building the word character by character during DFS is error-prone
     → Store word as a field in TrieNode → instant retrieval
     → Example: node.word = "oath" at the 'h' node

→ ALWAYS handle duplicate word findings
     → The same word might be findable from multiple starting cells
     → Example: "eat" might start from (0,0) and (2,3)
     → Use a Set<String> for results, or null-out node.word after finding
     → Setting node.word = null after first find prevents duplicates

→ ALWAYS use in-place visited marking (board[i][j] = '#')
     → Saves O(M×N) space
     → Cleaner code than maintaining separate visited array
     → Remember to RESTORE the character after backtracking

→ ALWAYS implement Trie node pruning (remove dead leaves)
     → Without it: TLE on large inputs
     → With it: significant speedup as words are found
     → This is the difference between AC and TLE on LeetCode

→ ALWAYS check bounds BEFORE accessing board or Trie
     → DFS explores 4 directions → some go out of bounds
     → Check i >= 0, i < rows, j >= 0, j < cols FIRST
     → Then check if character has Trie child

🚫 WHAT SHOULD YOU NOT DO

→ DO NOT search for each word separately
     → Word-by-word DFS: O(W × M × N × 4^L) → too slow
     → Trie-guided DFS: O(M × N × 4^L) → W is eliminated
     → The Trie AMORTIZES the search across all words

→ DO NOT forget to backtrack (unmark visited cells)
     → If you mark cell as visited and never unmark:
          → Other paths from other starting cells can't use that cell
          → You'll miss valid words
     → ALWAYS restore board[i][j] after recursive calls return

→ DO NOT skip Trie pruning for "simplicity"
     → Without pruning: found words leave dead branches in Trie
     → DFS keeps exploring these dead branches
     → On large boards with many words → TLE guaranteed
     → The pruning optimization is ESSENTIAL, not optional

→ DO NOT use a StringBuilder to build words during DFS
     → Append on enter, delete on backtrack → error-prone
     → Off-by-one errors in string building are common
     → Instead: store complete word at Trie leaf → instant retrieval
     → Simpler, faster, fewer bugs

→ DO NOT return after finding one word at a cell
     → A cell might be part of MULTIPLE words
     → "oat" and "oath" share the prefix "oat"
     → Finding "oat" should NOT stop the DFS
     → Continue exploring deeper for "oath"

→ DO NOT forget that diagonal movement is NOT allowed
     → Only up, down, left, right (4 directions)
     → NOT 8 directions like some other grid problems
     → Using 8 directions would find words that don't exist
     → Read the problem carefully for movement rules

→ DO NOT add duplicates to result
     → "eat" might be findable starting from (1,0) AND (1,3)
     → Adding it twice → wrong answer
     → Null out node.word after first find → prevents re-adding

🔀 WHEN TO USE THIS PATTERN

✓ You need to search for MULTIPLE strings in a structure simultaneously
✓ The search space has EXPONENTIAL branching (grid DFS, tree paths)
✓ The strings share PREFIXES that can be exploited for pruning
✓ Brute force per-string search would be too slow

SIGNATURE PROBLEMS:
→ Word Search II ← THIS
→ Boggle Game (find all valid words on a board)
→ Stream of Characters (match suffixes using reversed Trie)
→ Word Squares (Trie-guided backtracking)
→ Concatenated Words (Trie + DP)
→ Any problem combining "word list" + "search space exploration"

THE GENERAL PATTERN:
  1. Build Trie from the TARGET strings (what you're looking for)
  2. Explore the SOURCE space (grid, tree, string) guided by Trie
  3. At each step: Trie child exists? → continue : prune
  4. At end-of-word nodes → record the find
  5. Prune found words from Trie to accelerate remaining search

🚫 WHEN NOT TO USE THIS PATTERN

✗ Only ONE word to search for
     → Simple DFS/BFS on grid is sufficient
     → Trie adds unnecessary complexity for single word search
     → Word Search I (single word) → plain backtracking DFS

✗ Words don't share prefixes
     → If all words start with different characters
     → Trie provides minimal pruning benefit
     → Still works but doesn't gain much over per-word DFS

✗ The grid is very small but word list is huge
     → Grid exploration is cheap anyway
     → Trie construction becomes the bottleneck
     → Consider HashSet-based approach instead

✗ Movement rules are complex (not simple adjacency)
     → Trie-guided DFS assumes simple branching
     → If movement depends on previous moves or game state
     → The DFS structure becomes more complex than standard grid DFS

✗ You need the PATH, not just the word
     → Trie-guided DFS finds THAT a word exists
     → To find WHERE (exact cells used) → need path tracking
     → Adds complexity to the backtracking

⚙️ HOW IT ONLY WORKS — Internal Mechanics

The Trie as a DFS Compass

Without Trie — Blind DFS:
  → At each cell, try all 4 directions
  → Each direction tries all 4 directions
  → 4^L total paths for word of length L
  → MOST paths are useless (don't match any word)
  → Like searching a maze without a map

With Trie — Guided DFS:
  → At each cell, check: does Trie have child for this character?
  → Only follow directions where Trie says "possible word ahead"
  → MOST branches are pruned immediately
  → Like searching a maze WITH a map that highlights valid routes

Example pruning power:
  Board has 26 possible characters per cell
  Trie for ["cat", "car", "card"] has:
    → Root: only child 'c' → 25/26 starting cells pruned immediately
    → After 'c': only child 'a' → 25/26 neighbors pruned
    → After 'ca': children 't', 'r' → 24/26 neighbors pruned
    
  Without Trie: 26 × 26 × 26 = 17,576 paths explored
  With Trie: 1 × 1 × 2 = 2 paths explored
  Pruning factor: 8,788x faster

Why Backtracking is Essential for Grid DFS

Finding "oath" on the board:

  o → a → t → h  ✓
  
  But what about finding "oat" starting from the same 'o'?
  
  o → a → t  ✓ (found "oat")
  
  The cells o, a, t were used for "oath"
  They MUST be available again for other paths
  
  Backtracking ensures:
  → When we FINISH exploring from cell (i,j)
  → We UNMARK it as visited
  → Other DFS paths can reuse it
  
  Without backtracking:
  → Finding "oath" marks o, a, t, h as permanently visited
  → Can't find "oat" or "eat" or any word using those cells
  → WRONG ANSWERS

  The backtracking pattern:
  
  MARK cell as visited
    EXPLORE all 4 neighbors (recursive calls)
  UNMARK cell as visited
  
  This is the same "choose → explore → unchoose" from Project 42

Why Trie Pruning Matters — Quantified

Scenario: Board 12×12, Word list has 30,000 words

Without pruning:
  → Find "cat" → Trie still has path c→a→t
  → Every future DFS that reaches 'c' still explores c→a→t
  → "cat" path explored thousands of times after already found
  → Total wasted explorations: ~10^6

With pruning:
  → Find "cat" → remove 't' leaf → if 'a' has no other children → remove 'a'
  → Future DFS reaching 'c' doesn't see c→a→t anymore
  → Zero wasted explorations for "cat"
  → Total saved: ~10^6 operations

The pruning cascading effect:
  Words: ["cat", "car", "card"]
  
  After finding "cat":
    → Remove 't' from 'a's children
    → 'a' still has child 'r' (for "car", "card") → keep 'a'
  
  After finding "car":
    → Set isEndOfWord=false at 'r' node
    → 'r' still has child 'd' (for "card") → keep 'r'
  
  After finding "card":
    → Remove 'd' from 'r's children → 'r' is now childless
    → Remove 'r' from 'a's children → 'a' is now childless
    → Remove 'a' from 'c's children → 'c' is now childless
    → Remove 'c' from root's children
    → Trie is now EMPTY → all future DFS from any cell terminates instantly!

The Four-Direction Grid DFS Pattern

Standard grid DFS directions:

  int[][] dirs = {{-1, 0}, {1, 0}, {0, -1}, {0, 1}};
  //               up       down     left     right

For each direction:
  newRow = row + dirs[d][0]
  newCol = col + dirs[d][1]

Bounds checking:
  if (newRow >= 0 && newRow < rows && newCol >= 0 && newCol < cols)
  
This pattern appears in MANY grid problems:
  → Number of Islands
  → Flood Fill
  → Shortest Path in Grid
  → Word Search I and II
  → The 4-direction array is a REUSABLE tool

Why Storing the Word in TrieNode is Better Than Building It

APPROACH A — Build word during DFS:
  void dfs(board, i, j, node, StringBuilder path, result) {
      path.append(board[i][j]);    // add character
      if (node.isEndOfWord) {
          result.add(path.toString());  // copy string
      }
      // ... recurse ...
      path.deleteCharAt(path.length() - 1);  // remove character
  }

  PROBLEMS:
  → StringBuilder operations at every recursive call
  → toString() creates a copy at every word find
  → Off-by-one errors in append/delete
  → More code, more bugs

APPROACH B — Store word at TrieNode:
  class TrieNode {
      String word;  // null except at end-of-word nodes
  }
  
  void dfs(board, i, j, node, result) {
      if (node.word != null) {
          result.add(node.word);  // instant retrieval
          node.word = null;       // prevent duplicates
      }
      // ... recurse ...
  }

  ADVANTAGES:
  → No StringBuilder management
  → No character-by-character building
  → No copy at find time (word already exists as String)
  → Duplicate prevention built-in (null-out after find)
  → Cleaner, faster, fewer bugs

⚠️ COMMON PITFALLS AND FAILURE MODES

Pitfall 1: Not Restoring Board After Backtracking

WRONG:
  void dfs(char[][] board, int i, int j, TrieNode node, List<String> result) {
      board[i][j] = '#';  // mark visited
      // explore neighbors
      // FORGOT TO RESTORE!
  }

RIGHT:
  void dfs(char[][] board, int i, int j, TrieNode node, List<String> result) {
      char original = board[i][j];
      board[i][j] = '#';  // mark visited
      // explore neighbors
      board[i][j] = original;  // RESTORE
  }

Without restoring:
→ After first DFS from (0,0), many cells are permanently '#'
→ Subsequent DFS from other cells can't use those cells
→ Miss valid words → wrong answer

Pitfall 2: Duplicate Words in Result

Word "eat" can be found starting from (1,0) and from (1,3)

WRONG:
  if (node.isEndOfWord) {
      result.add(node.word);  // adds "eat" TWICE
  }

RIGHT (Option A — null out the word):
  if (node.word != null) {
      result.add(node.word);
      node.word = null;  // won't be added again
  }

RIGHT (Option B — use a Set):
  Set<String> resultSet = new HashSet<>();
  // ... in DFS:
  if (node.isEndOfWord) {
      resultSet.add(node.word);  // Set prevents duplicates
  }
  return new ArrayList<>(resultSet);

Pitfall 3: Not Continuing DFS After Finding a Word

Words: ["oat", "oath"]
Path so far: o → a → t (found "oat")

WRONG:
  if (node.word != null) {
      result.add(node.word);
      return;  // STOPS HERE — never finds "oath"!
  }

RIGHT:
  if (node.word != null) {
      result.add(node.word);
      node.word = null;
      // DON'T return — continue exploring for "oath"
  }
  // ... explore neighbors ...

"oat" is a PREFIX of "oath"
→ Finding "oat" should NOT stop exploration
→ Must continue to find "oath"

Pitfall 4: Trie Child Check at Wrong Level

The DFS should check Trie child for the CURRENT cell's character
BEFORE entering the recursive call, not after.

WRONG (check inside recursive call):
  void dfs(board, i, j, node, result) {
      // Already moved to this cell — but does Trie even have this char?
      char c = board[i][j];
      if (node.children[c - 'a'] == null) return;  // TOO LATE
      node = node.children[c - 'a'];
      // ...
  }

This works but the check placement matters for clarity.

CLEANER approach:
  // In the calling code:
  for (int[] dir : dirs) {
      int ni = i + dir[0], nj = j + dir[1];
      if (inBounds(ni, nj) && board[ni][nj] != '#') {
          int idx = board[ni][nj] - 'a';
          if (node.children[idx] != null) {  // CHECK BEFORE ENTERING
              dfs(board, ni, nj, node.children[idx], result);
          }
      }
  }

Checking BEFORE entering the recursive call:
→ Avoids unnecessary function call overhead
→ Makes the pruning explicit and clear
→ Fewer stack frames

Pitfall 5: Forgetting Edge Cases

Edge case 1: Empty word list → return empty list
Edge case 2: Empty board → return empty list
Edge case 3: Single cell board, single character word
  → Board: [["a"]], Words: ["a"] → return ["a"]
Edge case 4: Word longer than total board cells → impossible → skip
Edge case 5: Duplicate words in word list
  → ["eat", "eat"] → Trie stores once → found once → return ["eat"]

📈 SCALABILITY CONSIDERATIONS

VARIABLES:
  M × N = board dimensions
  W = number of words
  L = maximum word length
  S = sum of all word lengths

TIME COMPLEXITY:
→ Trie construction: O(S)
→ DFS: O(M × N × 4^L) worst case
     → But Trie pruning makes this MUCH better in practice
     → With pruning: closer to O(M × N × L) for typical inputs
→ Total: O(S + M × N × 4^L) worst case
→ Practical: O(S + M × N × L) with good pruning

SPACE COMPLEXITY:
→ Trie: O(S) for all word characters
→ Recursion stack: O(L) depth (max word length)
→ Result list: O(W × L) for all found words
→ Total: O(S + L)

LEETCODE CONSTRAINTS (typical):
→ Board: up to 12 × 12 = 144 cells
→ Words: up to 30,000 words
→ Word length: up to 10 characters
→ Without Trie: 30,000 × 144 × 4^10 ≈ impossible
→ With Trie + pruning: ~1 second → AC

OPTIMIZATION PRIORITY:
  1. Trie pruning (most impactful)
  2. In-place visited marking (saves space)
  3. Store word at node (cleaner code)
  4. Check Trie child before recursive call (fewer stack frames)

THE OPTIMIZATION HIERARCHY:

  Level 0 — Naive:
    → Per-word DFS on grid
    → O(W × M × N × 4^L)
    → Fails on large inputs

  Level 1 — Trie-guided DFS:
    → All words searched simultaneously
    → O(M × N × 4^L) — W is eliminated
    → Passes most inputs

  Level 2 — Trie-guided DFS + word dedup:
    → Null out found words to prevent duplicates
    → Same complexity but correct output

  Level 3 — Trie-guided DFS + leaf pruning:
    → Remove dead Trie branches after finding words
    → Trie shrinks over time → DFS gets faster
    → Passes ALL inputs including worst-case edge cases
    → This is the ACCEPTED solution

Each level builds on the previous:
  → You can't do Level 3 without understanding Level 1
  → The optimizations are LAYERED, not independent
  → Interview tip: explain Level 1 first, then add optimizations

THE GRID DFS TEMPLATE (Reusable):

  int[][] DIRS = {{-1,0}, {1,0}, {0,-1}, {0,1}};
  
  void dfs(char[][] board, int i, int j, ...) {
      // 1. Bounds check
      if (outOfBounds(i, j)) return;
      
      // 2. Visit check
      if (board[i][j] == '#') return;
      
      // 3. Validity check (problem-specific)
      if (!isValid(...)) return;
      
      // 4. Process current cell
      process(board[i][j]);
      
      // 5. Mark visited
      char saved = board[i][j];
      board[i][j] = '#';
      
      // 6. Explore neighbors
      for (int[] dir : DIRS) {
          dfs(board, i + dir[0], j + dir[1], ...);
      }
      
      // 7. Unmark (backtrack)
      board[i][j] = saved;
  }
