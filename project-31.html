PROJECT 31: Find Peak Element in an Array

🎯 Modified Binary Search → Gradient-Based Elimination → No Strict Sorting Needed

🧩 THE HYPOTHETICAL PROBLEM

→ You are given an UNSORTED array of integers
→ A "peak element" is an element STRICTLY GREATER than its neighbors
→ Find ANY peak element and return its INDEX
→ The array may contain multiple peaks — returning ANY one is valid
→ Assume: nums[-1] = nums[n] = -∞ (imaginary boundaries are negative infinity)
→ Adjacent elements are NEVER equal: nums[i] ≠ nums[i+1]
→ You MUST solve it in O(log n) time

Example 1:
  Input:  [1, 3, 20, 4, 1, 0]
  Output: 2  →  because nums[2]=20 > nums[1]=3 AND nums[2]=20 > nums[3]=4

Example 2:
  Input:  [1, 2, 1, 3, 5, 6, 4]
  Output: 1 OR 5
  →  nums[1]=2 is a peak (2>1 and 2>1)
  →  nums[5]=6 is a peak (6>5 and 6>4)
  →  Either answer is accepted

Example 3:
  Input:  [5, 4, 3, 2, 1]
  Output: 0  →  because nums[-1] = -∞ < nums[0]=5 > nums[1]=4

Example 4:
  Input:  [1, 2, 3, 4, 5]
  Output: 4  →  because nums[4]=5 > nums[3]=4 AND nums[5] = -∞

🔍 HOW TO APPROACH IT

Step 1 — Understand Why This Problem is Confusing

→ In Project 9 (Classic Binary Search): array was FULLY SORTED
→ In Project 30 (Rotated Sorted Array): array was PARTIALLY SORTED
→ Here: array is NOT SORTED AT ALL
→ Your first instinct: "Binary search can't work on unsorted data"
→ That instinct is WRONG — and understanding WHY is this project's lesson

KEY REALIZATION:
→ Binary search does NOT require sorting
→ Binary search requires a DECISION CRITERION that eliminates half the search space
→ Sorting is ONE WAY to get that criterion
→ But it is NOT THE ONLY WAY

Step 2 — What Decision Criterion Exists Here?

→ Consider any middle element nums[mid]
→ Compare it with its RIGHT neighbor nums[mid + 1]

CASE A: nums[mid] < nums[mid + 1]
  → We are on an ASCENDING slope
  → The terrain is going UP to the right
  → A peak MUST exist somewhere to the RIGHT
  → WHY? Because either:
       → The array keeps going up → hits the right boundary → nums[n] = -∞
       → So the last element before -∞ is a peak
       OR
       → The array eventually goes down → the turning point is a peak
  → GUARANTEE: right side has at least one peak

CASE B: nums[mid] > nums[mid + 1]
  → We are on a DESCENDING slope
  → The terrain is going DOWN to the right
  → A peak MUST exist somewhere to the LEFT (including mid itself)
  → WHY? Because either:
       → The array keeps going up leftward → hits the left boundary → nums[-1] = -∞
       → So the first element after -∞ is a peak
       OR
       → The array eventually goes down leftward → the turning point is a peak
  → GUARANTEE: left side (including mid) has at least one peak

This is the GRADIENT — the slope direction tells us which half has a peak

  nums[1]=3, nums[2]=20
  3 < 20 → ascending → peak is to the RIGHT → left = mid + 1 = 2

Step 4 — Why This is Different from Projects 9 and 30

Project 9 (Classic Binary Search):
  → Array is fully sorted
  → Decision: compare nums[mid] with TARGET
  → Eliminate half based on value comparison with external target

Project 30 (Rotated Sorted Array):
  → Array is partially sorted (two sorted halves)
  → Decision: identify which HALF is sorted, then check if target is in that half
  → Eliminate half based on sorted-half identification

Project 31 (This — Peak Element):
  → Array is NOT sorted at all
  → Decision: compare nums[mid] with nums[mid + 1]
  → Eliminate half based on SLOPE DIRECTION (gradient)
  → No external target — we're searching for a STRUCTURAL PROPERTY

EVOLUTION OF BINARY SEARCH:
  Project 9:  "Which half contains the TARGET VALUE?"
  Project 30: "Which half is SORTED and does it contain the target?"
  Project 31: "Which half GUARANTEES the structural property I need?"

🛠️ HOW CAN WE SOLVE IT

The Algorithm — Line by Line

1. Initialize left = 0, right = nums.length - 1
2. WHILE left < right:
     a. Compute mid = left + (right - left) / 2
          → Overflow-safe mid calculation (covered in Project 9)
     b. IF nums[mid] < nums[mid + 1]:
          → Ascending slope → peak is to the RIGHT
          → left = mid + 1
     c. ELSE (nums[mid] > nums[mid + 1]):
          → Descending slope → peak is at mid or to the LEFT
          → right = mid
3. When left == right → that index IS a peak → return left

Why left < right (NOT left <= right)?

→ In Project 9: we used left <= right because we were searching for a TARGET
→ If target not found → loop must exhaust all possibilities → need the = case

→ Here: we are GUARANTEED a peak exists
→ The loop NARROWS the window until left == right
→ At that point, the single remaining element IS the peak
→ Using left <= right would cause:
     → left == right → mid = left = right
     → We'd check nums[mid] vs nums[mid + 1]
     → But mid + 1 could be OUT OF BOUNDS
     → CRASH
→ left < right ensures mid + 1 is always valid
     → Because if left < right → mid < right → mid + 1 ≤ right → safe

Why right = mid (NOT right = mid - 1)?

→ When nums[mid] > nums[mid + 1]:
     → mid itself COULD be the peak
     → We cannot exclude it
     → So right = mid (keep mid in the search space)

→ When nums[mid] < nums[mid + 1]:
     → mid is DEFINITELY NOT a peak (its right neighbor is bigger)
     → We can safely exclude it
     → So left = mid + 1 (exclude mid)

Compare with Project 9:
  → In classic binary search: right = mid - 1 and left = mid + 1
  → Because we check nums[mid] == target explicitly
  → Here: no explicit "found" check inside the loop
  → The found condition IS when left == right

✅ WHAT SHOULD YOU DO

→ ALWAYS compare nums[mid] with nums[mid + 1] — NOT with nums[mid - 1]
     → Comparing with mid + 1 is safe when left < right
     → Comparing with mid - 1 risks index -1 when mid = 0

→ ALWAYS use right = mid (NOT right = mid - 1) for descending case
     → mid could be the peak itself
     → Excluding it loses a valid candidate

→ ALWAYS use left = mid + 1 for ascending case
     → mid is definitely NOT the peak
     → Including it wastes iterations and risks infinite loop

→ ALWAYS use left < right as loop condition
     → Ensures mid + 1 is always within bounds
     → Convergence point (left == right) is the answer

→ ALWAYS think "which half GUARANTEES a peak" — not "which half MIGHT have one"
     → The guarantee comes from the boundary conditions (nums[-1] = nums[n] = -∞)

→ ALWAYS handle the edge case of single-element array
     → left = 0, right = 0 → loop doesn't execute → return 0
     → Single element is always a peak (both neighbors are -∞)

🚫 WHAT SHOULD YOU NOT DO

→ DO NOT use left <= right
     → Causes out-of-bounds access on nums[mid + 1]
     → Also creates potential infinite loop scenarios

→ DO NOT use right = mid - 1 in the descending case
     → You'd skip mid which could be the peak
     → Could miss the answer entirely

→ DO NOT compare nums[mid] with both neighbors simultaneously
     → Tempting to check: nums[mid] > nums[mid-1] AND nums[mid] > nums[mid+1]
     → This works but adds boundary checks and complexity
     → The gradient approach is cleaner and achieves the same result

→ DO NOT assume only one peak exists
     → Multiple peaks can exist
     → The algorithm finds ONE — whichever the binary search converges to
     → Different implementations might find different peaks — all are valid

→ DO NOT use linear scan
     → Walking left to right checking each element: O(n)
     → Problem explicitly requires O(log n)
     → Linear scan works but fails the time constraint

→ DO NOT confuse this with "find the maximum element"
     → Maximum element IS a peak, but peaks are NOT necessarily the maximum
     → [1, 3, 2, 5, 1] → 3 is a peak, 5 is a peak (and maximum)
     → We only need ANY peak

🔀 WHEN TO USE THIS PATTERN

✓ You need to find a LOCAL property (not a specific value)
     → Peak, valley, transition point, inflection point

✓ You can make a BINARY DECISION at any point
     → "Which half guarantees the property exists?"
     → Not "which half contains value X"

✓ Boundary conditions create guarantees
     → nums[-1] = -∞ and nums[n] = -∞ → peak MUST exist
     → Without these boundaries → no guarantee → technique fails

✓ Adjacent elements are distinct
     → nums[i] ≠ nums[i + 1] ensures clear slope direction
     → Equal adjacent elements create "plateaus" → ambiguous gradient

✓ Problem requires O(log n) on unsorted data
     → This is the signature clue in interviews
     → "O(log n) on unsorted?" → think gradient-based binary search

✓ Similar problems:
     → Find a valley (minimum between two higher points)
     → Find transition from false to true in boolean array
     → Find rotation point in mountain array
     → Bitonic array search

🚫 WHEN NOT TO USE THIS PATTERN

✗ Adjacent elements CAN be equal
     → nums = [1, 2, 2, 2, 1]
     → nums[mid] == nums[mid + 1] → no clear gradient direction
     → Cannot reliably eliminate half → degrades to O(n) worst case

✗ No boundary guarantees exist
     → If the problem doesn't say nums[-1] = nums[n] = -∞
     → A monotonically increasing array would have no peak
     → The guarantee of peak existence breaks down

✗ You need to find ALL peaks
     → Binary search finds ONE peak
     → Finding all peaks requires O(n) scan — no shortcut

✗ You need the GLOBAL maximum (not just any peak)
     → Binary search converges to A peak — not necessarily the highest
     → Global max requires examining all elements → O(n)

✗ Array is sorted
     → If fully sorted ascending → peak is last element (trivially)
     → If fully sorted descending → peak is first element (trivially)
     → No need for binary search at all

⚙️ HOW IT ONLY WORKS — Internal Mechanics

The Mountain Guarantee Theorem

CLAIM: In any subarray [left, right] where:
  → nums[left - 1] < nums[left]  (or left is at boundary with -∞)
  → nums[right] > nums[right + 1]  (or right is at boundary with -∞)
  → At least one peak exists in [left, right]

PROOF BY CONTRADICTION:
→ Suppose no peak exists in [left, right]
→ Then every element has at least one neighbor that is ≥ to it
→ Start at any element → it's not a peak → move to the larger neighbor
→ Repeat → you must eventually go out of bounds
→ But the boundaries are -∞ → you'd have to be larger than -∞ to exit
→ The last element before exiting would be > -∞ and > its inner neighbor
→ That element IS a peak → contradiction

This guarantee is why the algorithm works:
→ At each step, the half we choose MAINTAINS the boundary conditions
→ So the guarantee carries forward into every smaller subproblem

Invariant Preservation — Step by Step

INVARIANT: A peak exists in [left, right]

Initial state:
  → left = 0, right = n-1
  → nums[-1] = -∞ < nums[0] (if we walk left, we fall off)
  → nums[n] = -∞ < nums[n-1] (if we walk right, we fall off)
  → Peak guaranteed in [0, n-1] ✓

When nums[mid] < nums[mid + 1] → left = mid + 1:
  → New search space: [mid + 1, right]
  → Left boundary: nums[mid] < nums[mid + 1] → entering from left is ASCENDING
  → Right boundary: unchanged → still has -∞ guarantee
  → Peak guaranteed in [mid + 1, right] ✓

  Visual:
       ... nums[mid] < nums[mid+1] ... ? ... ? ... -∞
                        ↑ entering ascending
       Peak must exist somewhere between mid+1 and right

When nums[mid] > nums[mid + 1] → right = mid:
  → New search space: [left, mid]
  → Left boundary: unchanged → still has -∞ guarantee
  → Right boundary: nums[mid] > nums[mid + 1] → exiting from right is DESCENDING
  → Peak guaranteed in [left, mid] ✓

  Visual:
       -∞ ... ? ... ? ... nums[mid] > nums[mid+1] ...
                              ↑ exiting descending
       Peak must exist somewhere between left and mid

Termination: left == right
  → Single element → it IS the peak
  → Left boundary says: coming from left, values were ascending toward it
  → Right boundary says: going right, values descend from it
  → Or boundary is -∞
  → Either way: this element ≥ both neighbors → peak ✓

Convergence Proof

→ Each iteration: either left increases or right decreases
→ left = mid + 1: left increases by at least 1
→ right = mid: right decreases (since mid < right when left < right)
→ Search space shrinks by at least 1 element per step
→ After at most log₂(n) steps: left == right → terminate

WHY log₂(n)?
→ mid = (left + right) / 2 → roughly the midpoint
→ Each step eliminates approximately HALF the search space
→ n → n/2 → n/4 → ... → 1
→ Takes log₂(n) halvings

⚠️ COMMON PITFALLS AND FAILURE MODES

Pitfall 1: Infinite Loop with right = mid

DANGER: If you use left = mid instead of left = mid + 1

→ When left + 1 == right:
     → mid = (left + right) / 2 = left  (integer division floors)
     → If nums[mid] < nums[mid + 1]:
          → left = mid = left → NO CHANGE → INFINITE LOOP

→ This is why the ascending case MUST use left = mid + 1
→ It guarantees progress by moving left PAST mid

SAFE COMBINATION:
  → Ascending:  left = mid + 1  ← excludes mid (safe, mid isn't peak)
  → Descending: right = mid     ← includes mid (necessary, mid might be peak)
  → Loop: left < right          ← terminates when they meet

UNSAFE COMBINATIONS:
  → left = mid + right = mid         → infinite loop possible
  → left = mid + 1 + right = mid - 1 → might skip the peak

Pitfall 2: Accessing nums[mid + 1] Out of Bounds

→ If mid = n - 1 → mid + 1 = n → OUT OF BOUNDS
→ Does this happen? Let's check:
     → Loop condition: left < right
     → mid = (left + right) / 2
     → Since left < right → mid < right → mid ≤ right - 1 → mid + 1 ≤ right ≤ n-1
     → SAFE — mid + 1 is always within bounds ✓

→ But if you used left <= right:
     → left = right → mid = left = right
     → mid + 1 = right + 1 → could be n → OUT OF BOUNDS
     → This is why left < right is essential

Pitfall 3: Equal Adjacent Elements

→ Problem states: nums[i] ≠ nums[i + 1]
→ If this constraint is REMOVED:
     → nums[mid] == nums[mid + 1] → which direction?
     → No gradient → can't decide → must check both sides
     → Worst case degrades to O(n)

→ In interviews: ALWAYS ask "Can adjacent elements be equal?"
→ If yes → this O(log n) approach doesn't work cleanly
→ May need to handle the equal case separately (move either direction)

Pitfall 4: Confusing with Find Minimum in Rotated Array

→ Find Peak: ascending slope → go RIGHT (toward higher ground)
→ Find Min in Rotated: we'd go toward LOWER ground
→ The gradient direction is OPPOSITE
→ Don't mix up the pointer movements

Pitfall 5: Thinking "What if the Array is Monotonic?"

Monotonically Increasing: [1, 2, 3, 4, 5]
  → Every mid: nums[mid] < nums[mid + 1] → left keeps moving right
  → Converges to index 4 (last element)
  → nums[4]=5 > nums[3]=4 and nums[5]=-∞ → VALID PEAK ✓

Monotonically Decreasing: [5, 4, 3, 2, 1]
  → Every mid: nums[mid] > nums[mid + 1] → right keeps moving left
  → Converges to index 0 (first element)
  → nums[-1]=-∞ < nums[0]=5 > nums[1]=4 → VALID PEAK ✓

→ The algorithm handles monotonic arrays correctly
→ The -∞ boundaries ensure a peak always exists at the extremes

📈 SCALABILITY CONSIDERATIONS

Time Complexity: O(log n)
  → Each step halves the search space
  → n = 1,000,000 → ~20 steps
  → n = 1,000,000,000 → ~30 steps
  → Virtually instant for any practical input size

Space Complexity: O(1) iterative / O(log n) recursive
  → Iterative version: only left, right, mid variables → O(1)
  → Recursive version: call stack depth = log n → O(log n)
  → Always prefer iterative in interviews unless asked for recursive

Comparison with Linear Scan:
  → Linear scan: walk through array, check each element → O(n)
  → For n = 10^8: linear scan = 10^8 ops ≈ 0.1 seconds
  → For n = 10^8: binary search = 27 ops ≈ instant
  → The difference is massive at scale

Real-world applications:
  → Signal processing: finding local maxima in waveform data
  → Stock prices: finding local price peaks
  → Terrain analysis: finding hilltops in elevation data
  → All benefit from O(log n) peak finding

     → If the ground rises to your right → walk right (you'll hit a peak)

🧠 KEY TAKEAWAY

WHAT WE LEARNED:
→ Binary search does NOT require a sorted array
→ It requires a DECISION CRITERION that reliably eliminates half the space
→ For peak finding: the GRADIENT (slope direction) is that criterion
→ Ascending slope → peak guaranteed to the right
→ Descending slope → peak guaranteed to the left (including current)
→ Boundary conditions (nums[-1] = nums[n] = -∞) provide the existence guarantee

THE EVOLUTION OF BINARY SEARCH ACROSS THREE PROJECTS:
→ Project 9:  Sorted data + target value → compare mid with target
→ Project 30: Partially sorted + target → identify sorted half, then compare
→ Project 31: Unsorted + structural property → compare mid with NEIGHBOR

THE CORE MENTAL MODEL:
→ "Can I make a binary decision at any midpoint?"
→ "Does that decision GUARANTEE the answer is in one half?"
→ "Does the guarantee PRESERVE itself in the smaller subproblem?"
→ If yes to all three → binary search works — regardless of sorting

CRITICAL IMPLEMENTATION DETAILS:
→ Loop: left < right (not <=) → prevents out-of-bounds
→ Ascending: left = mid + 1 → exclude mid (not a peak)
→ Descending: right = mid → include mid (could be a peak)
→ Answer: return left when left == right

PATTERN SIGNATURE:
→ O(log n) required ✓
→ Array not sorted but has directional property ✓
→ Local comparison reveals global structure ✓
→ Boundary conditions guarantee existence ✓
→ = GRADIENT-BASED BINARY SEARCH
