PROJECT 20: Find the Start Node of a Linked List Cycle

🎯 Fast & Slow Pointers → Mathematical Cycle Entry Detection → Two-Phase Technique

🧩 THE HYPOTHETICAL PROBLEM

→ You are given the HEAD of a linked list
→ The list MAY contain a cycle
→ If a cycle exists → return the NODE where the cycle BEGINS
→ If no cycle exists → return null

Example 1:
  3 → 2 → 0 → -4
      ↑           |
      └───────────┘
  
  Cycle starts at node with value 2
  Output: node(2)

Example 2:
  1 → 2
  ↑   |
  └───┘

  Cycle starts at node with value 1
  Output: node(1)

Example 3:
  1 → 2 → 3 → 4 → null

  No cycle
  Output: null

IMPORTANT:
→ We return the NODE reference, not the value
→ Multiple nodes could have the same value
→ The POSITION in the list matters, not the value
→ We need O(1) space — no HashSet allowed for optimal solution

🔍 HOW TO APPROACH IT

Step 1 — Recall Project 7: Cycle Detection

→ Project 7 taught us the Tortoise and Hare algorithm
→ Slow pointer moves 1 step at a time
→ Fast pointer moves 2 steps at a time
→ If cycle exists → they WILL meet inside the cycle
→ If no cycle → fast reaches null

→ Project 7 answered: "IS there a cycle?" → true/false
→ Project 20 asks: "WHERE does the cycle begin?"

→ The meeting point from Project 7 is NOT necessarily the cycle start
→ They meet SOMEWHERE inside the cycle
→ We need a SECOND phase to find the exact entry point

Step 2 — Understand the Geometry

Let's label the distances:

  HEAD ───── F ─────→ CYCLE START ───── a ─────→ MEETING POINT
                           ↑                           |
                           |                           |
                           └──────── (C - a) ──────────┘

  F = distance from HEAD to CYCLE START (the "tail" length)
  C = total cycle length
  a = distance from CYCLE START to MEETING POINT (within cycle)
  C - a = distance from MEETING POINT back to CYCLE START

  When slow and fast meet:
  → Slow has traveled: F + a steps
  → Fast has traveled: F + a + n*C steps (for some integer n ≥ 1)
       → Fast may have gone around the cycle multiple times
       
  → Fast travels TWICE as far as slow:
       F + a + n*C = 2(F + a)
       
  → Simplify:
       F + a + n*C = 2F + 2a
       n*C = F + a
       F = n*C - a
       F = (n-1)*C + (C - a)

Step 3 — The Key Mathematical Insight

  F = (n-1)*C + (C - a)

  What does this mean?

  → F is the distance from HEAD to CYCLE START
  → (C - a) is the distance from MEETING POINT to CYCLE START (going forward in cycle)
  → (n-1)*C means some number of FULL LOOPS around the cycle

  THEREFORE:
  → If a pointer starts at HEAD and walks F steps → reaches CYCLE START
  → If a pointer starts at MEETING POINT and walks (C-a) + (n-1)*C steps → reaches CYCLE START
  → Both distances are EQUAL: F = (n-1)*C + (C-a)

  THE TRICK:
  → Place pointer1 at HEAD
  → Place pointer2 at MEETING POINT
  → Move BOTH one step at a time
  → They will meet EXACTLY at the CYCLE START

  WHY?
  → pointer1 walks F steps from HEAD → arrives at CYCLE START
  → pointer2 walks F = (n-1)*C + (C-a) steps from MEETING POINT
       → First (C-a) steps: reaches CYCLE START
       → Then (n-1)*C steps: goes around cycle (n-1) times → back at CYCLE START
  → Both arrive at CYCLE START at the same time → they MEET there

Step 4 — Visualize With Concrete Numbers

Example:
  HEAD → A → B → C → D → E
                  ↑           |
                  └─── F ←────┘

  Nodes: HEAD→A→B→C→D→E→F→C (cycle back to C)

  F (tail length) = 3 (HEAD→A→B→C, but we count edges: A,B,C = 3 steps? 
  Let me re-index:
  
  Index:  0    1    2    3    4    5
  Node:   A →  B →  C →  D →  E →  F
                    ↑                |
                    └────────────────┘

  F = 2 (steps from head to cycle start C: A→B→C)
  C = 4 (cycle length: C→D→E→F→C)

  Phase 1: Detect cycle (find meeting point)
  
  Step 0: slow=A(0), fast=A(0)
  Step 1: slow=B(1), fast=C(2)
  Step 2: slow=C(2), fast=E(4)
  Step 3: slow=D(3), fast=C(2)  ← fast went F→C
  Step 4: slow=E(4), fast=E(4)  ← MEET at E!

  Meeting point = E
  a = distance from C to E within cycle = 2 (C→D→E)
  
  Verify: F = n*C - a → 2 = 1*4 - 2 = 2 ✓

  Phase 2: Find cycle start
  → pointer1 at HEAD (A)
  → pointer2 at MEETING POINT (E)
  → Both move 1 step at a time

  Step 1: p1=B, p2=F
  Step 2: p1=C, p2=C  ← MEET at C!

  C is the cycle start ✓
  Distance walked by p1: 2 steps (A→B→C) = F ✓
  Distance walked by p2: 2 steps (E→F→C) = C-a = 4-2 = 2 ✓

Step 5 — The Two-Phase Algorithm

PHASE 1: Find the meeting point (same as Project 7)
  → slow moves 1 step
  → fast moves 2 steps
  → If they meet → cycle exists → save meeting point
  → If fast reaches null → no cycle → return null

PHASE 2: Find the cycle entry
  → pointer1 starts at HEAD
  → pointer2 starts at MEETING POINT
  → Both move 1 step at a time
  → When they meet → that node IS the cycle start
  → Return that node

🛠️ HOW CAN WE SOLVE IT

The Algorithm — Precise Steps

detectCycleStart(head):

  1. IF head is null OR head.next is null → return null (no cycle possible)

  2. PHASE 1 — Find meeting point:
     → slow = head
     → fast = head
     → WHILE fast != null AND fast.next != null:
          → slow = slow.next
          → fast = fast.next.next
          → IF slow == fast → BREAK (meeting point found)
     
     → IF fast == null OR fast.next == null → return null (no cycle)

  3. PHASE 2 — Find cycle start:
     → pointer1 = head
     → pointer2 = slow (meeting point)
     → WHILE pointer1 != pointer2:
          → pointer1 = pointer1.next
          → pointer2 = pointer2.next
     
     → RETURN pointer1 (or pointer2 — they're the same node)

Dry Run — Complete Trace

List: 3 → 2 → 0 → -4 → (back to 2)

Nodes by position:
  pos 0: val=3
  pos 1: val=2  ← cycle start
  pos 2: val=0
  pos 3: val=-4 → points back to pos 1

F = 1 (head to cycle start: 3→2)
C = 3 (cycle length: 2→0→-4→2)

═══ PHASE 1: Find Meeting Point ═══

Step 0: slow=3(pos0), fast=3(pos0)
Step 1: slow=2(pos1), fast=0(pos2)
Step 2: slow=0(pos2), fast=2(pos1)  ← fast went -4→2
Step 3: slow=-4(pos3), fast=-4(pos3) ← MEET at -4!

Meeting point = node(-4) at pos 3
a = distance from cycle start(pos1) to meeting(pos3) = 2
C - a = 3 - 2 = 1

Verify: F = n*C - a → 1 = 1*3 - 2 = 1 ✓

═══ PHASE 2: Find Cycle Start ═══

pointer1 = HEAD = node(3) at pos 0
pointer2 = MEETING = node(-4) at pos 3

Step 1: p1 = node(2) at pos 1
        p2 = node(2) at pos 1   ← -4.next points to 2

p1 == p2 → MEET at node(2)!

Return node(2) → THIS IS THE CYCLE START ✓

✅ WHAT SHOULD YOU DO

→ ALWAYS run Phase 1 completely before Phase 2
     → You MUST find the meeting point first
     → Without it, Phase 2 has no starting position for pointer2

→ ALWAYS check for "no cycle" after Phase 1
     → If fast reaches null → no cycle → return null immediately
     → Don't proceed to Phase 2

→ ALWAYS start both pointers at HEAD in Phase 1
     → Some implementations start slow=head, fast=head
     → Others start slow=head.next, fast=head.next.next
     → The math works for both but the code differs slightly
     → Starting both at HEAD is cleaner and more intuitive

→ ALWAYS move both Phase 2 pointers at the SAME SPEED (1 step each)
     → Phase 1: different speeds (1 and 2) to detect cycle
     → Phase 2: same speed (1 and 1) to find entry
     → Mixing up the speeds is a fatal error

→ ALWAYS compare NODE REFERENCES, not values
     → Two different nodes can have the same value
     → pointer1 == pointer2 checks if they're the SAME OBJECT
     → pointer1.val == pointer2.val checks if values match (WRONG)

🚫 WHAT SHOULD YOU NOT DO

→ DO NOT use a HashSet
     → HashSet approach: traverse list, add each node to set
     → First node you see AGAIN is the cycle start
     → This works BUT uses O(n) space
     → The two-pointer approach uses O(1) space
     → Interviewers specifically want the O(1) space solution
     
→ DO NOT confuse meeting point with cycle start
     → Phase 1 finds where slow and fast COLLIDE
     → This is INSIDE the cycle but NOT necessarily the START
     → Phase 2 is needed to find the actual start

→ DO NOT try to count cycle length first
     → Some solutions: find meeting point → count cycle → use that count
     → This works but adds unnecessary complexity
     → The two-phase approach is simpler and equally efficient

→ DO NOT move fast pointer in Phase 2
     → Phase 2 uses TWO SLOW pointers
     → One from head, one from meeting point
     → Both at speed 1
     → Using fast pointer (speed 2) in Phase 2 breaks the math

→ DO NOT forget edge cases
     → Empty list (head == null) → return null
     → Single node, no cycle → return null
     → Single node, self-cycle (node points to itself) → return that node
     → Cycle at the very beginning (head is cycle start) → must handle
     → Cycle at the very end (last node points back) → must handle

→ DO NOT modify the linked list
     → Some approaches mark visited nodes by changing values
     → This destroys the data structure
     → Problem typically requires non-destructive solution

🔀 WHEN TO USE THIS PATTERN

✓ Finding the ENTRY POINT of a cycle in a linked list
✓ Finding the START of a repeated pattern in a sequence
✓ Floyd's algorithm applications:
     → Duplicate number detection in array (each value is a "pointer")
     → Functional graph cycle detection
     → Any system where you follow "next" pointers and need cycle info
✓ When O(1) space is required for cycle-related problems
✓ When you need EXACT position, not just existence of cycle

🚫 WHEN NOT TO USE THIS PATTERN

✗ No cycle possible in the data structure
     → Trees (by definition acyclic) → use DFS/BFS instead
     → Sorted arrays → use binary search
     
✗ You need ALL nodes in the cycle, not just the start
     → After finding start, you'd need additional traversal
     → Two-phase finds the entry → then traverse cycle to collect all nodes
     
✗ Graph with multiple cycles
     → Floyd's works for single-path structures (linked lists)
     → For general graphs → use DFS with visited set
     
✗ O(n) space is acceptable
     → HashSet approach is simpler to code and understand
     → If space isn't a constraint → HashSet may be preferred for clarity
     
✗ The structure doesn't support "next pointer" navigation
     → Two pointers need sequential traversal capability
     → Random-access structures (arrays) need different cycle detection

⚙️ HOW IT ONLY WORKS — Internal Mechanics

Why Do Slow and Fast Always Meet? (Recap from Project 7)

→ Once slow enters the cycle, both pointers are in the cycle
→ Fast moves 2 steps, slow moves 1 step
→ Each step, fast CLOSES THE GAP by 1 position
→ Gap decreases: ..., 5, 4, 3, 2, 1, 0 → MEET
→ They MUST meet within one full cycle of the slow pointer

Mathematical proof:
→ Let gap = distance between fast and slow (measured forward in cycle)
→ Each step: gap decreases by 1 (fast gains 1 on slow)
→ Gap starts at some value ≤ C-1
→ Gap reaches 0 in at most C-1 steps → guaranteed meeting

The Full Mathematical Proof for Phase 2

SETUP:
→ Let F = number of nodes from head to cycle start (not counting cycle start)
→ Let C = number of nodes in the cycle
→ Let a = number of nodes from cycle start to meeting point (within cycle)

WHEN THEY MEET (Phase 1):
→ Slow traveled: F + a steps
→ Fast traveled: F + a + k*C steps (k ≥ 1, fast made k extra loops)
→ Fast speed = 2 × slow speed:
     2(F + a) = F + a + k*C
     F + a = k*C
     F = k*C - a

PHASE 2 ANALYSIS:
→ pointer1 starts at HEAD, needs to walk F steps to reach cycle start
→ pointer2 starts at meeting point, which is 'a' steps into the cycle

→ pointer2 needs to walk (C - a) steps to reach cycle start
→ But F = k*C - a = (k-1)*C + (C - a)
→ So pointer2 walks (C - a) + (k-1)*C steps
→ The (k-1)*C part means going around the cycle (k-1) times → back to same spot
→ Net displacement: (C - a) steps forward in cycle → reaches cycle start

→ pointer1 walks F steps → reaches cycle start
→ pointer2 walks F steps → reaches cycle start (with possible extra loops)
→ BOTH arrive at cycle start after F steps → they MEET there ∎

SPECIAL CASE: F = 0 (cycle starts at head)
→ F = k*C - a → 0 = k*C - a → a = k*C
→ Meeting point is at a = k*C steps from cycle start
→ k full loops → meeting point IS the cycle start
→ Phase 2: both start at head/meeting point
→ If meeting point IS the cycle start → they're already there
→ pointer1 starts at head (= cycle start) → immediate match ✓

SPECIAL CASE: a = 0 (they meet at cycle start)
→ F = k*C → F is a multiple of cycle length
→ Phase 2: pointer1 walks F steps → cycle start
→ pointer2 walks F = k*C steps → k full loops → back at cycle start ✓

Why Same Speed in Phase 2?

→ Phase 1 uses different speeds to CREATE a meeting event inside cycle
→ Phase 2 uses same speed to SYNCHRONIZE arrival at cycle start

→ If we used different speeds in Phase 2:
     → They might overshoot the cycle start
     → The mathematical relationship F = (k-1)*C + (C-a) would break
     → They could meet at a DIFFERENT point, not the cycle start

→ Same speed = same distance traveled = synchronized arrival
→ The MATH guarantees they cover exactly F steps to meet at cycle start

Time and Space Complexity

TIME: O(n)
→ Phase 1: Slow pointer enters cycle after F steps
     → Then meets fast within C steps
     → Total Phase 1: F + C steps ≤ n + n = O(n)
→ Phase 2: Both pointers walk F steps
     → F < n → O(n)
→ Total: O(n) + O(n) = O(n)

SPACE: O(1)
→ Only pointer variables: slow, fast, pointer1, pointer2
→ No additional data structures
→ This is the key advantage over HashSet approach

COMPARISON:
  Two-pointer: O(n) time, O(1) space ← OPTIMAL
  HashSet:     O(n) time, O(n) space ← simpler but more memory

The Pointer Reference Comparison Detail

→ In Java, == on objects compares REFERENCES (memory addresses)
→ slow == fast checks: do both point to the SAME node object?
→ This is EXACTLY what we want → same physical node in memory

→ .equals() or .val comparison would be WRONG:
     → Two different nodes could have value 5
     → They are NOT the same node
     → We need reference equality, not value equality

→ Java's == for objects = reference equality = correct for this problem

⚠️ COMMON PITFALLS AND FAILURE MODES

Pitfall 1: Phase 1 Initialization Variants

VARIANT A: Both start at head
  slow = head;
  fast = head;
  do {
      slow = slow.next;
      fast = fast.next.next;
  } while (slow != fast);

→ Uses do-while because initially slow == fast (both at head)
→ Must check fast != null AND fast.next != null inside loop

VARIANT B: Start one step ahead
  slow = head.next;
  fast = head.next.next;
  while (slow != fast) {
      slow = slow.next;
      fast = fast.next.next;
  }

→ Uses while because initially slow != fast
→ Must pre-check that head.next and head.next.next exist

VARIANT A is cleaner — use it consistently.
But you MUST use do-while or check after first move, not before.
The code below uses while loop with a moved-flag approach for clarity.

Pitfall 2: Forgetting Null Checks in Phase 1

CRASH scenario:
  List: 1 → 2 → 3 → null (no cycle)
  
  Step 1: slow=2, fast=3
  Step 2: slow=3, fast=null.next → NullPointerException!

→ ALWAYS check: fast != null AND fast.next != null
→ Before EVERY fast movement
→ If either is null → no cycle → return null

Pitfall 3: Self-Cycle (Node Points to Itself)

  1 → (back to 1)

  F = 0 (cycle starts at head)
  C = 1 (cycle has one node)
  
  Phase 1: slow=1, fast=1
  Step 1: slow = 1.next = 1, fast = 1.next.next = 1
  slow == fast → meet at node 1
  
  Phase 2: p1 = head = 1, p2 = meeting = 1
  p1 == p2 immediately → return node 1 ✓

→ Algorithm handles self-cycles correctly without special casing

Pitfall 4: Cycle at Head (F = 0)

  1 → 2 → 3 → (back to 1)

  F = 0, C = 3

  Phase 1:
  Step 0: slow=1, fast=1
  Step 1: slow=2, fast=3
  Step 2: slow=3, fast=2
  Step 3: slow=1, fast=1 → MEET at 1

  Phase 2: p1=head=1, p2=meeting=1
  p1 == p2 immediately → return 1 ✓

→ Works because F=0 → both start at the same point → instant match

Pitfall 5: Very Long Tail, Short Cycle

  1 → 2 → 3 → 4 → 5 → 6 → 7 → (back to 6)

  F = 5, C = 2

  → Fast enters cycle, loops many times while slow is still in tail
  → Eventually slow enters cycle
  → They meet within at most C steps after slow enters
  → Phase 2 works: both walk 5 steps → meet at node 6

→ No matter the ratio of F to C, the algorithm works
→ The math doesn't depend on F vs C size relationship

Pitfall 6: Returning Wrong Type

→ Problem asks for NODE, not VALUE, not INDEX
→ return pointer1 (the node reference)
→ NOT return pointer1.val
→ NOT return some index number

→ In interview: clarify return type
→ LeetCode typically expects ListNode return

📈 SCALABILITY CONSIDERATIONS

→ n = 100 → O(100) → instant
→ n = 1,000,000 → O(1,000,000) → fast
→ n = 100,000,000 → O(100,000,000) → few seconds
→ Space is ALWAYS O(1) regardless of n

→ Compare with HashSet approach:
     → n = 1,000,000 nodes → HashSet stores 1M entries
     → Each entry ≈ 32-64 bytes → 32-64 MB
     → Two-pointer: 0 extra bytes

→ In memory-constrained environments:
     → Embedded systems, real-time systems
     → O(1) space is critical
     → Two-pointer is the only viable option

→ The algorithm makes exactly:
     → Phase 1: ≤ F + 2C pointer moves
     → Phase 2: exactly F pointer moves
     → Total: ≤ 2F + 2C ≤ 2n moves
     → Constant factor is very small → cache-friendly

🔍 INTERPRETABILITY CONSIDERATIONS

→ Phase 1 is intuitive: "fast runner laps slow runner on a track"
     → Everyone understands this from running on circular tracks
     → Fast catches up by 1 step per iteration → must eventually meet

→ Phase 2 is LESS intuitive: "walk from head and meeting point at same speed"
     → This requires the mathematical proof to believe
     → In interviews: state the property, sketch the proof
     → "F = kC - a, so both pointers travel F steps to reach cycle start"

→ The two-phase structure is easy to remember:
     → Phase 1: Find IF cycle exists + WHERE they meet
     → Phase 2: Find WHERE cycle starts

→ Debugging tip:
     → If Phase 2 seems to loop forever → Phase 1 had a bug
     → If Phase 2 returns wrong node → check the pointer initialization
     → Print positions at each step to verify

     → This demonstrates mathematical thinking → strong signal
