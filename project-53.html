PROJECT 53: Smallest Range Covering Elements from K Lists

🎯 K-way Merge → Sliding Min-Max Across K Pointers → Heap + Global Tracking

🧩 THE HYPOTHETICAL PROBLEM

→ You are given K sorted lists of integers
→ Find the SMALLEST range [a, b] such that the range
  includes AT LEAST ONE number from EACH of the K lists
→ "Smallest range" = minimize (b - a)
→ If there are ties, return the range with the smaller starting point

Example 1:
  lists = [[4, 10, 15, 24, 26],
           [0, 9, 12, 20],
           [5, 18, 22, 30]]
  Output: [20, 24]
  
  → 20 is from list 2 (index 3)
  → 24 is from list 1 (index 3)
  → 22 is from list 3 (index 2) → 22 is in [20, 24] ✓
  → Range size = 24 - 20 = 4
  → No smaller range covers all three lists

Example 2:
  lists = [[1, 2, 3],
           [1, 2, 3],
           [1, 2, 3]]
  Output: [1, 1]
  
  → All lists contain 1 → range [1, 1] covers all → size 0

Example 3:
  lists = [[1, 5, 10],
           [3, 7, 12],
           [2, 6, 11]]
  Output: [1, 3] or [5, 7] or [10, 12]
  
  → [1, 3]: 1 from list 1, 3 from list 2, 2 from list 3 ✓ → size 2
  → [5, 7]: 5 from list 1, 7 from list 2, 6 from list 3 ✓ → size 2
  → [10, 12]: 10 from list 1, 12 from list 2, 11 from list 3 ✓ → size 2
  → All have size 2 → return [1, 3] (smallest start)

Example 4:
  lists = [[1],
           [2],
           [3]]
  Output: [1, 3]
  
  → Must include at least one from each → range must span 1 to 3

🔍 HOW TO APPROACH IT

Step 1 — Understand What "Covering All Lists" Means

→ At any moment, we're looking at ONE element from each list
→ These K elements define a "window" of values
→ The RANGE of this window = max(elements) - min(elements)
→ We want to MINIMIZE this range

→ This is like having K pointers, one in each list
→ Each pointer points to the "current element" from that list
→ The range = max(current elements) - min(current elements)
→ We want to advance pointers strategically to minimize the range

Step 2 — Connect to Previous Knowledge

Project 40 (Merge K Sorted Lists):
  → Used a min-heap to track the smallest element across K lists
  → Each heap entry: (value, listIndex, elementIndex)
  → Poll minimum → advance that list's pointer → push next element
  → This gave us a MERGED sorted stream

THIS Problem:
  → Same K-list structure → same min-heap approach
  → But instead of just tracking the MIN, we also track the MAX
  → At every step: range = currentMax - heap.peek() (heap gives min)
  → We want to MINIMIZE this range

KEY INSIGHT:
  → The min-heap always gives us the MINIMUM across K current elements
  → We separately track the MAXIMUM across K current elements
  → Range = max - min = currentMax - heap.peek()
  → To SHRINK the range: we must INCREASE the minimum
  → Increasing the minimum = advancing the pointer in the list that has the minimum
  → This is EXACTLY what polling from the min-heap does!

Project 34 (Sliding Window Median) / Project 41 (Sliding Window Median):
  → Maintained heaps while sliding a window
  → Rebalanced on each slide

THIS Problem:
  → Similar "sliding" behavior across K lists
  → But instead of a fixed window on ONE array
  → It's K pointers sliding across K arrays
  → The "window" is defined by the K current elements

Step 3 — The Algorithm Intuition

INITIAL STATE:
  → Start with the FIRST element from each list
  → These K elements define our initial range
  → Put all K elements in a min-heap
  → Track the maximum separately

IMPROVEMENT STEP:
  → The range = currentMax - minHeap.peek()
  → To shrink the range, we have two options:
       a. DECREASE the maximum → but we can only move FORWARD in sorted lists
          → Moving forward means the next element is ≥ current → max won't decrease
          → This option is IMPOSSIBLE (lists are sorted ascending)
       b. INCREASE the minimum → advance the list that currently has the minimum
          → Poll from min-heap → get the list with smallest current element
          → Push the NEXT element from that list into the heap
          → The new minimum is ≥ old minimum (might be larger)
          → This CAN shrink the range!

  → So we ALWAYS advance the minimum → this is the ONLY productive move

TERMINATION:
  → When ANY list is exhausted (no more elements to advance to)
  → At that point, we can't cover all K lists anymore
  → The best range seen so far is the answer

THIS IS GREEDY:
  → At each step, advance the minimum element's list
  → Track the best range seen
  → Stop when any list runs out

Step 5 — Why This Works

CLAIM: The algorithm finds the optimal (smallest) range.

INTUITION:
  → At each step, we consider a "configuration" of K elements (one per list)
  → The range of this configuration = max - min
  → We MUST advance the minimum to potentially shrink the range
  → Advancing any other pointer would only increase the max or leave it same
  → Advancing the min is the ONLY way to potentially reduce range

EXHAUSTIVE COVERAGE:
  → Every possible "optimal range" corresponds to some configuration
  → The algorithm visits ALL configurations reachable by advancing minimums
  → Since advancing the minimum is the only productive move,
    ALL potentially optimal configurations are visited

FORMAL PROOF SKETCH:
  → Let [a*, b*] be the optimal range
  → b* is the maximum in some configuration C*
  → In C*, the minimum is a* (or the range would be smaller)
  → The algorithm will reach a configuration where:
       → All elements ≤ a* have been advanced past
       → The current minimum is a* (or close to it)
       → The current maximum includes b*
  → At that moment, the algorithm records [a*, b*] as the best range ✓

🛠️ HOW CAN WE SOLVE IT

Algorithm — Min-Heap + Global Max Tracking

1. Create a min-heap storing (value, listIndex, elementIndex)
2. Initialize:
     → For each list i, push (lists[i][0], i, 0) into heap
     → Track currentMax = max of all lists[i][0]
3. Initialize bestRange = [heap.peek().value, currentMax]
4. WHILE true:
     a. Poll minimum element from heap → (minVal, listIdx, elemIdx)
     b. Check if this list has a next element:
          → If elemIdx + 1 >= lists[listIdx].length → STOP (list exhausted)
     c. Push next element: (lists[listIdx][elemIdx+1], listIdx, elemIdx+1)
     d. Update currentMax = max(currentMax, lists[listIdx][elemIdx+1])
     e. New range = [heap.peek().value, currentMax]
     f. If new range is smaller than bestRange → update bestRange
5. RETURN bestRange

Why We Track Max Separately (Not in Heap)

→ The min-heap gives us the MINIMUM in O(1) via peek()
→ But getting the MAXIMUM from a min-heap is O(K) — need to scan all entries
→ Instead, we maintain currentMax as a separate variable

→ currentMax ONLY INCREASES (or stays same) as we advance:
     → When we push a new element, it's the next in a sorted list
     → Next element ≥ current element (sorted ascending)
     → So the new element might be a new maximum
     → currentMax = max(currentMax, newElement)
     → It NEVER decreases → simple tracking works

→ This is the KEY TRICK: min from heap (O(1)) + max from variable (O(1))
→ Range computation at each step: O(1)

✅ WHAT SHOULD YOU DO

→ ALWAYS push ONE element from EACH list initially
     → The heap must have exactly K elements at all times
     → Each element represents one list's "current pointer"
     → Missing a list = range doesn't cover all lists

→ ALWAYS update currentMax when pushing new elements
     → New element might be larger than current max
     → Forgetting this = wrong range calculation

→ ALWAYS compare range using (max - min) not just min or max
     → We minimize (max - min), not min or max individually
     → A range [100, 101] (size 1) is better than [0, 50] (size 50)

→ ALWAYS stop when ANY list is exhausted
     → If one list has no more elements, we can't cover all K lists
     → Continuing would violate the "at least one from each list" constraint

→ ALWAYS store (value, listIndex, elementIndex) in heap entries
     → value: for heap ordering (min-heap sorts by value)
     → listIndex: to know WHICH list this element came from
     → elementIndex: to know WHERE in the list to get the NEXT element

→ CONSIDER using an int[] array as heap entry (cleaner than custom class)
     → int[] entry = {value, listIndex, elementIndex}
     → Comparator: (a, b) -> a[0] - b[0]

🚫 WHAT SHOULD YOU NOT DO

→ DO NOT advance ALL pointers simultaneously
     → Only advance the MINIMUM pointer
     → Advancing others would increase the range unnecessarily
     → Each step: poll one (min), push one (next from same list)

→ DO NOT use a max-heap
     → We need the MINIMUM to decide which pointer to advance
     → Min-heap gives minimum in O(1)
     → Max is tracked separately (only increases)

→ DO NOT try to find max from the heap
     → Scanning the heap for max is O(K)
     → Doing this at every step makes total O(n×K) instead of O(n×log K)
     → Track max separately in O(1)

→ DO NOT merge all lists and then search
     → Merging: O(N log K) where N = total elements
     → Then searching for smallest range: complex and no better
     → The heap approach does both simultaneously

→ DO NOT confuse with "merge K sorted lists" (Project 40)
     → Project 40: produce a single merged sorted list
     → This project: find smallest range covering all lists
     → Same heap setup, DIFFERENT objective
     → Project 40 doesn't need max tracking or range comparison

→ DO NOT continue after a list is exhausted
     → Exhausted list = can't be covered = range is invalid
     → Any range found after this point is incomplete

⚙️ HOW IT ONLY WORKS — Internal Mechanics

The Min-Heap Invariant

AT EVERY STEP:
  → Heap contains exactly K elements
  → Each element represents one list's "current" value
  → No two elements are from the same list
  → heap.peek() = minimum across all K current values
  → currentMax = maximum across all K current values

WHEN WE POLL AND PUSH:
  → Poll: remove the minimum → one list loses its representative
  → Push: add the NEXT element from that list → list regains a representative
  → Heap still has exactly K elements → invariant maintained
  → currentMax: new element might be larger → update → invariant maintained

WHEN WE STOP:
  → Poll minimum → its list has no next element
  → Cannot push → heap would have K-1 elements → invariant broken
  → One list is uncovered → no valid range → STOP

Why Advancing Minimum is Optimal

CURRENT STATE:
  → K current elements: e₁, e₂, ..., eₖ
  → min = eₘᵢₙ, max = eₘₐₓ
  → range = eₘₐₓ - eₘᵢₙ

TO SHRINK THE RANGE:
  Option A: Decrease eₘₐₓ
    → Lists are sorted ascending → next element ≥ current
    → Advancing eₘₐₓ's list gives element ≥ eₘₐₓ → max stays or increases
    → CANNOT decrease max by advancing

  Option B: Increase eₘᵢₙ
    → Advance eₘᵢₙ's list → next element ≥ eₘᵢₙ
    → New minimum might be larger → range might shrink
    → THIS IS THE ONLY PRODUCTIVE MOVE

  Option C: Advance some middle element
    → Middle element ≥ eₘᵢₙ and ≤ eₘₐₓ
    → Advancing it: next ≥ current → might increase max
    → But doesn't help decrease range (min stays same)
    → WASTEFUL — no benefit

THEREFORE:
  → Advancing the minimum is the ONLY move that can improve the range
  → This is why the algorithm is correct and greedy

Why All Optimal Configurations Are Visited

THEOREM: Every potentially optimal configuration is visited.

DEFINITION: A "configuration" is a choice of one element per list.

KEY OBSERVATION:
  → The algorithm generates configurations in order of INCREASING minimum
  → Each step increases the minimum (or keeps it same)
  → Once we pass a minimum value, we never go back

  → For any optimal configuration with minimum value m:
       → The algorithm will reach a state where its minimum ≥ m
       → At the moment the minimum first reaches m (or passes through m),
         the maximum is the smallest it can be given that minimum
       → Because we haven't unnecessarily advanced any other pointer
       → So the range at that moment is optimal for that minimum

  → Since we check ALL possible minimum values (in order),
    we will find the global optimum ✓

Time Complexity

→ Total elements across all K lists: N = n₁ + n₂ + ... + nₖ
→ Each element is pushed into the heap at most once
→ Each element is polled from the heap at most once
→ Each push/poll: O(log K) (heap of size K)
→ TOTAL: O(N × log K)

→ Initialization: O(K × log K) to build initial heap

→ OVERALL: O(N × log K)
→ This is OPTIMAL — we must examine each element at least once: O(N)
→ The log K factor is for heap maintenance

Space Complexity

→ Heap stores exactly K elements: O(K)
→ No additional arrays needed
→ TOTAL: O(K)
→ Very space-efficient

⚠️ COMMON PITFALLS AND FAILURE MODES

Pitfall 1: Not Tracking Max Separately

WRONG:
  → Using only min-heap
  → Trying to compute max by scanning heap entries: O(K) per step
  → Total: O(N × K) instead of O(N × log K)
  → Or worse: forgetting max entirely → wrong range

RIGHT:
  → Track currentMax as a separate variable
  → Update it when pushing new elements: currentMax = max(currentMax, newVal)
  → currentMax only increases → simple O(1) update

Pitfall 2: Forgetting to Update Max When Pushing

WRONG:
  // Push new element without updating max
  heap.offer(new int[]{nextVal, listIdx, elemIdx + 1});
  // currentMax is stale → range calculation is wrong

RIGHT:
  int nextVal = lists[listIdx][elemIdx + 1];
  currentMax = Math.max(currentMax, nextVal);
  heap.offer(new int[]{nextVal, listIdx, elemIdx + 1});

Pitfall 3: Not Stopping When a List is Exhausted

WRONG:
  → Poll minimum → list has no next element → skip and continue
  → Heap now has K-1 elements → one list uncovered
  → Range found after this is INVALID (doesn't cover all lists)

RIGHT:
  → Poll minimum → check if next element exists
  → If NOT → STOP immediately → return best range found so far

Pitfall 4: Heap Entry Without List/Element Tracking

WRONG:
  → Push only the value into the heap
  → When polling, don't know which list it came from
  → Can't advance the correct list's pointer

RIGHT:
  → Each heap entry = {value, listIndex, elementIndex}
  → After polling, use listIndex to find the list
  → Use elementIndex + 1 to get the next element

Pitfall 5: Range Comparison Tie-Breaking

Problem says: "If ties, return the range with smaller start"

→ When comparing ranges:
     → Primary: smaller size (b - a)
     → Secondary: smaller start (a)
→ If newSize < bestSize → update
→ If newSize == bestSize AND newStart < bestStart → update

→ With our algorithm: we naturally visit configurations with increasing min
→ Earlier configurations have smaller start values
→ So we use < (strictly less) for range comparison
→ Ties are automatically resolved in favor of smaller start

Min-Heap — Mental Model:
  → "The min-heap is like a tournament bracket"
  → "K contestants (current elements) compete"
  → "The winner (minimum) gets to advance to the next round"
  → "Their replacement (next element from same list) enters the bracket"
  → "Meanwhile, we separately track who's the strongest (maximum)"
  → "The range = strongest - weakest at each round"
