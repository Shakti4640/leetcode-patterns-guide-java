PROJECT 25: Connect Level Order Siblings — Next Pointer

🎯 Tree BFS → Sibling Linking Within Level → Queue Neighbor Awareness

🧩 THE HYPOTHETICAL PROBLEM

→ You are given a PERFECT or ARBITRARY binary tree
→ Each node has an additional pointer: next
→ Populate each node's next pointer to point to its RIGHT NEIGHBOR at the same level
→ If there is no right neighbor → next = null
→ The next pointer is initially null for all nodes

Example 1 (Perfect Binary Tree):
  BEFORE:
              1 → null
             / \
            2   3 → null
           / \ / \
          4  5 6  7 → null

  AFTER:
              1 → null
             / \
            2 → 3 → null
           / \ / \
          4→ 5→6→ 7 → null

Example 2 (Arbitrary Binary Tree):
  BEFORE:
              1 → null
             / \
            2   3 → null
           / \   \
          4   5   7 → null

  AFTER:
              1 → null
             / \
            2 → 3 → null
           / \   \
          4→ 5 → 7 → null

  Note: 5's right neighbor is 7 (NOT 6 — node 6 doesn't exist)
  The next pointer skips gaps → connects to next EXISTING node at same level

Example 3:
  Input: null (empty tree)
  Output: null (nothing to connect)

Example 4:
  Input: single node [1]
  Output: 1 → null (no right neighbor)

CONSTRAINT:
  → O(1) extra space is ideal (beyond the queue for BFS)
  → But queue-based BFS solution is acceptable and common

🔍 HOW TO APPROACH IT

Step 1 — Recall Projects 17 and 24: BFS Template

→ Project 17: standard level-order → queue + level-size snapshot
→ Project 24: zigzag → same template + direction flag + addFirst/addLast
→ Project 25: sibling linking → same template + node-to-node connection

→ The BFS template processes nodes LEVEL BY LEVEL
→ Within each level, nodes come out of the queue LEFT TO RIGHT
→ The node that comes out NEXT is the RIGHT NEIGHBOR of current node
→ We just need to LINK current node's next pointer to the next node in queue

→ This is the simplest conceptual modification yet:
     → Project 17: collect value
     → Project 24: collect value with direction
     → Project 25: link to neighbor

Step 2 — The Linking Logic Within BFS

During BFS, when processing level of size K:

  Node 0 → Node 1 → Node 2 → ... → Node K-1 → null

  For each node i (0 to K-2):
    node_i.next = node_{i+1}
  
  For the last node (K-1):
    node_{K-1}.next = null  (already null by default)

HOW to get node_{i+1} while processing node_i?

APPROACH A: Use queue.peek()
  → After polling node_i from queue
  → queue.peek() gives node_{i+1} (if not last in level)
  → node_i.next = queue.peek()
  → But careful: queue.peek() after adding children may give WRONG node

APPROACH B: Track previous node
  → Maintain a "prev" variable
  → For each node polled: prev.next = currentNode
  → Then: prev = currentNode
  → First node of level: prev = null → no linking
  → This is CLEANER and SAFER

APPROACH B is better because:
  → queue.peek() after enqueueing children → peek returns child, not sibling
  → prev tracking doesn't depend on queue state
  → prev tracking is explicit and easy to reason about

Step 5 — O(1) Space Solution (Perfect Tree Only)

→ For PERFECT binary trees → an O(1) space solution exists
→ Uses the ALREADY-ESTABLISHED next pointers to traverse levels
→ No queue needed!

IDEA:
→ Process level by level
→ For each node at current level → link its CHILDREN
→ Use next pointers to move across the current level

For node at current level:
  → node.left.next = node.right          (left child → right child)
  → if node.next != null:
       node.right.next = node.next.left   (right child → cousin)

This works because:
  → Current level's next pointers are ALREADY set (from previous iteration)
  → We use those next pointers to traverse current level
  → While traversing → we set NEXT level's next pointers
  → Bootstrap: level 0 has one node → no next needed

LIMITATION: Only works for PERFECT trees
  → Arbitrary trees have gaps → node.next.left may not exist
  → Need more complex logic for arbitrary trees (dummy pointer technique)
  → BFS approach works for ALL tree types → more general

🛠️ HOW CAN WE SOLVE IT

Algorithm 1: BFS with Previous Node Tracking (General — Works for ALL Trees)

connect(root):

  1. IF root is null → return root

  2. CREATE queue, add root

  3. WHILE queue is not empty:
       a. levelSize = queue.size()
       b. prev = null
       
       c. FOR i = 0 to levelSize - 1:
            → node = queue.poll()
            
            → IF prev != null:
                 prev.next = node      (link previous to current)
            
            → prev = node              (current becomes previous)
            
            → IF node.left != null: queue.offer(node.left)
            → IF node.right != null: queue.offer(node.right)
       
       d. (Last node's next is already null by default)

  4. RETURN root

Algorithm 2: O(1) Space — Level-by-Level Using Next Pointers (Perfect Tree)

connect(root):

  1. IF root is null → return root

  2. SET leftmost = root
     → leftmost tracks the first node of the CURRENT level

  3. WHILE leftmost.left != null:    (while next level exists)
       a. SET curr = leftmost
       
       b. WHILE curr != null:        (traverse current level using next)
            → curr.left.next = curr.right
            → IF curr.next != null:
                 curr.right.next = curr.next.left
            → curr = curr.next        (move to next node at same level)
       
       c. leftmost = leftmost.left   (move to next level)

  4. RETURN root

Algorithm 3: O(1) Space — Arbitrary Tree (Dummy Child Technique)

connect(root):

  1. IF root is null → return root

  2. SET curr = root

  3. WHILE curr != null:             (while current level exists)
       a. CREATE dummy = new Node(0)  (dummy head for next level's linked list)
       b. SET tail = dummy            (tail of next level's linked list)
       c. SET temp = curr             (traverse current level)
       
       d. WHILE temp != null:
            → IF temp.left != null:
                 tail.next = temp.left
                 tail = tail.next
            → IF temp.right != null:
                 tail.next = temp.right
                 tail = tail.next
            → temp = temp.next         (move across current level using next)
       
       e. curr = dummy.next            (move to next level's first node)

  4. RETURN root

✅ WHAT SHOULD YOU DO

→ ALWAYS use the "prev tracking" approach for BFS solution
     → Safer than queue.peek()
     → Works for both perfect and arbitrary trees
     → Clear and easy to reason about

→ ALWAYS reset prev to null at the START of each level
     → prev = null before the for loop
     → First node of each level has no left neighbor → no linking
     → If prev carries over from previous level → wrong cross-level links!

→ ALWAYS let the last node's next remain null
     → Default value is null → no explicit setting needed
     → Don't accidentally link last node to first node of next level

→ ALWAYS enqueue children AFTER linking
     → Link current node → then enqueue its children
     → Order within the for loop: link → update prev → enqueue
     → This keeps the queue state predictable

→ ALWAYS consider which tree type the problem specifies:
     → Perfect tree → O(1) space solution is possible and expected
     → Arbitrary tree → BFS is cleanest, O(1) space needs dummy technique
     → Read the problem carefully for tree type constraints

→ ALWAYS return root (not void)
     → The root node itself doesn't change
     → But LeetCode expects root returned for chaining

🚫 WHAT SHOULD YOU NOT DO

→ DO NOT use queue.peek() after enqueueing children
     WRONG:
       node = queue.poll()
       queue.offer(node.left)     // child added to queue
       queue.offer(node.right)    // child added to queue  
       node.next = queue.peek()   // peek returns CHILD, not sibling!
     
     → After adding children, queue.peek() gives the next level's node
     → NOT the sibling at the current level
     → This creates WRONG cross-level links

→ DO NOT forget to reset prev at each level boundary
     WRONG:
       Node prev = null;            // set once outside all loops
       while (!queue.isEmpty()) {
           int size = queue.size();
           for (int i = 0; i < size; i++) {
               node = queue.poll();
               if (prev != null) prev.next = node;  // links last of level N to first of level N+1!
               prev = node;
           }
       }
     
     → prev from level N's last node → links to level N+1's first node → WRONG
     → prev MUST be reset to null at the start of each level

→ DO NOT link nodes across levels
     → next pointer connects nodes at the SAME level only
     → Last node of each level → next = null
     → Never link level N's last to level N+1's first

→ DO NOT assume perfect tree for general solution
     → Perfect tree: every internal node has exactly 2 children
     → Arbitrary tree: nodes may have 0, 1, or 2 children
     → O(1) space perfect-tree solution BREAKS on arbitrary trees
     → If problem says "arbitrary" → use BFS or dummy technique

→ DO NOT modify the tree structure (left/right pointers)
     → Only modify the next pointer
     → left and right pointers must remain unchanged
     → Accidentally overwriting left/right → tree destroyed

→ DO NOT use extra space for level collection
     → Project 17/24: collected values into lists → needed for output
     → Project 25: we're MODIFYING the tree in-place → no lists needed
     → The tree IS the output → no separate result structure

🔀 WHEN TO USE THIS PATTERN

✓ Tree nodes have an extra "next" pointer to populate
✓ Need to connect nodes at the same level horizontally
✓ Level-order traversal is the natural fit
✓ Variants:
     → Connect siblings in perfect binary tree
     → Connect siblings in arbitrary binary tree
     → Populate right-side pointers
     → Build horizontal linked lists at each level
✓ The "prev tracking" technique works for ANY level-aware linking
✓ O(1) space with next pointers → when tree type allows traversal via next

🚫 WHEN NOT TO USE THIS PATTERN

✗ Nodes don't have a next pointer field
     → Can't add structural connections without the field
     → Would need a separate data structure (HashMap<Node, Node>)
     
✗ Need to connect nodes vertically (parent-child)
     → That's just normal tree structure → already exists via left/right
     
✗ Need to connect nodes across levels
     → next pointer is specifically for same-level connections
     → Cross-level connections would be a different data structure
     
✗ Graph instead of tree
     → Graphs may have cycles → BFS needs visited tracking
     → "Level" concept is less clear in general graphs
     
✗ Need the connections to be bidirectional
     → next is one-directional (left to right)
     → For bidirectional → need both next and prev pointers

⚙️ HOW IT ONLY WORKS — Internal Mechanics

The "prev" Tracking Technique — Deep Dive

CONCEPT:
→ "prev" always holds the MOST RECENTLY PROCESSED node at the current level
→ When we process a new node → the previous node's right neighbor IS this new node
→ So: prev.next = currentNode

STATE TRANSITIONS within one level:

  Before level starts: prev = null

  Process node_0: prev = null → skip linking → prev = node_0
  Process node_1: prev = node_0 → node_0.next = node_1 → prev = node_1
  Process node_2: prev = node_1 → node_1.next = node_2 → prev = node_2
  ...
  Process node_k: prev = node_{k-1} → node_{k-1}.next = node_k → prev = node_k

  After level ends: node_k.next = null (default)

RESULT: node_0 → node_1 → node_2 → ... → node_k → null

WHY prev = null at level start?
→ First node of each level has NO left neighbor
→ prev = null signals "don't link anything yet"
→ The if (prev != null) guard skips the first node cleanly

Why BFS Guarantees Correct Sibling Order

BFS processes nodes in LEVEL ORDER, LEFT TO RIGHT:

Tree:
          1
         / \
        2   3
       / \ / \
      4  5 6  7

Queue evolution:
  Start:          [1]
  After level 0:  [2, 3]           → left before right ✓
  After level 1:  [4, 5, 6, 7]     → all left-to-right ✓

WHY left-to-right is guaranteed:
→ For each node, we enqueue LEFT child before RIGHT child
→ Queue is FIFO → left child comes out before right child
→ Parent nodes are processed left-to-right (by induction from previous level)
→ So their children enter queue in correct left-to-right order

INDUCTION:
  Base: Level 0 has one node → trivially left-to-right
  Step: If level N is left-to-right in queue
        → Level N's nodes are processed left-to-right
        → Each node enqueues left child then right child
        → Level N+1 enters queue in left-to-right order ✓

THEREFORE: BFS always produces correct sibling ordering
→ Linking consecutive nodes = linking siblings = correct next pointers

O(1) Space Solution — How Next Pointers Bootstrap

PERFECT TREE — Level-by-level without queue:

Level 0: [1]                     → 1.next = null (only node)
Level 1: [2, 3]                  → 2.next = 3, 3.next = null
Level 2: [4, 5, 6, 7]           → 4→5→6→7→null

HOW to process level 1 to SET level 2's next pointers:

  Start at leftmost = 1
  
  WHILE leftmost has children (leftmost.left != null):
    curr = leftmost (start of current level)
    
    WHILE curr != null (traverse current level via next):
      
      Step A: Link curr's left child to curr's right child
        curr.left.next = curr.right
        → 2.left(4).next = 2.right(5) → 4→5
      
      Step B: Link curr's right child to next node's left child
        if curr.next != null:
          curr.right.next = curr.next.left
          → 2.right(5).next = 3.left(6) → 5→6
      
      curr = curr.next  → move to node 3
      
      Step A: 3.left(6).next = 3.right(7) → 6→7
      Step B: curr.next = null → skip
      
      curr = null → stop
    
    leftmost = leftmost.left → move down to level 2
  
  Level 2 result: 4→5→6→7→null ✓

THE BOOTSTRAP:
→ Level 0 is set trivially (one node)
→ Level 0's next pointers let us traverse level 0
→ While traversing level 0 → we set level 1's next pointers
→ Level 1's next pointers let us traverse level 1
→ While traversing level 1 → we set level 2's next pointers
→ ... and so on

→ Each level's next pointers are SET by the level above
→ Each level's next pointers are USED to traverse that level
→ The level above is already done → safe to use its next pointers
→ This is why it works WITHOUT a queue → O(1) extra space!

O(1) Space for Arbitrary Trees — The Dummy Child Technique

PROBLEM with arbitrary trees:
  → curr.next.left may not exist → can't link curr.right to it
  → Need to find the FIRST existing child in the next level

SOLUTION: Build a linked list of next level's nodes using a dummy head

Tree:
          1
         / \
        2   3
       / \   \
      4   5   7

Processing level 1 [2, 3] to set level 2's next pointers:

  dummy = Node(0)    → dummy head
  tail = dummy        → tail of the linked list being built
  
  curr = 2:
    2.left = 4 → tail.next = 4 → tail = 4    → dummy→4
    2.right = 5 → tail.next = 5 → tail = 5   → dummy→4→5
    curr = 2.next = 3
  
  curr = 3:
    3.left = null → skip
    3.right = 7 → tail.next = 7 → tail = 7   → dummy→4→5→7
    curr = 3.next = null → stop
  
  Level 2's next chain: 4→5→7→null ✓
  Next level starts at: dummy.next = 4

→ The dummy node serves the same purpose as in Project 23
→ It provides a uniform starting point for building the linked list
→ Works for ANY tree shape → no assumption about perfect structure

Time and Space Complexity — All Approaches

APPROACH 1: BFS with queue
  Time:  O(n) → each node enqueued and dequeued once
  Space: O(w) → queue holds at most one level → w = max width
         → Balanced tree: O(n/2) = O(n)
         → Skewed tree: O(1)

APPROACH 2: O(1) space — perfect tree
  Time:  O(n) → each node visited once
  Space: O(1) → only pointer variables → no queue
  Limit: ONLY works for perfect binary trees

APPROACH 3: O(1) space — arbitrary tree (dummy technique)
  Time:  O(n) → each node visited once
  Space: O(1) → dummy node + pointer variables → no queue
  Works: For ALL binary trees

COMPARISON:
  ┌──────────────┬─────────┬─────────┬──────────────┐
  │ Approach     │ Time    │ Space   │ Tree Type    │
  ├──────────────┼─────────┼─────────┼──────────────┤
  │ BFS + queue  │ O(n)    │ O(n)    │ Any          │
  │ Perfect tree │ O(n)    │ O(1)    │ Perfect only │
  │ Dummy child  │ O(n)    │ O(1)    │ Any          │
  └──────────────┴─────────┴─────────┴──────────────┘

→ BFS is simplest to code and understand
→ O(1) approaches are impressive in interviews
→ Know both → choose based on problem constraints

⚠️ COMMON PITFALLS AND FAILURE MODES

Pitfall 1: Not Resetting prev Between Levels

WRONG:
  ListNode prev = null;
  while (!queue.isEmpty()) {
      int size = queue.size();
      for (int i = 0; i < size; i++) {
          Node node = queue.poll();
          if (prev != null) prev.next = node;
          prev = node;
          // enqueue children...
      }
  }

What happens:
  Level 0: process node(1) → prev = node(1)
  Level 1: process node(2) → prev.next = node(2) 
           → node(1).next = node(2) → CROSS-LEVEL LINK! ✗

  Result: 1 → 2 → 3 → 4 → 5 → 6 → 7 → null
  → All nodes chained together → level boundaries lost!

RIGHT:
  while (!queue.isEmpty()) {
      int size = queue.size();
      Node prev = null;              // ← RESET for each level
      for (int i = 0; i < size; i++) {
          Node node = queue.poll();
          if (prev != null) prev.next = node;
          prev = node;
          // enqueue children...
      }
  }

  Result: 
    Level 0: 1 → null
    Level 1: 2 → 3 → null
    Level 2: 4 → 5 → 6 → 7 → null ✓

Pitfall 2: Using queue.peek() After Enqueueing Children

WRONG:
  for (int i = 0; i < size; i++) {
      Node node = queue.poll();
      if (node.left != null) queue.offer(node.left);
      if (node.right != null) queue.offer(node.right);
      
      if (i < size - 1) {
          node.next = queue.peek();   // ← WRONG!
      }
  }

What happens for node(2) in level 1:
  Poll node(2)
  Enqueue node(4) → queue = [3, 4]    (3 is still there, but...)
  Wait, actually queue = [3] → then add 4,5 → queue = [3, 4, 5]
  queue.peek() = node(3) → this is correct!
  
  BUT for node(3) in level 1:
  Poll node(3)
  Enqueue node(6), node(7) → queue = [4, 5, 6, 7]
  i = 1 = size-1 → skip peek → correct

  Hmm, in this case peek BEFORE enqueueing would work:

ACTUALLY the issue is MORE subtle. Consider:
  If we peek AFTER enqueueing:
    Level 1: [2, 3] in queue
    Poll 2 → enqueue 4, 5 → queue = [3, 4, 5]
    peek = 3 ✓ (sibling)
    
    Poll 3 → enqueue 6, 7 → queue = [4, 5, 6, 7]
    i = size-1 → don't peek ✓

  This WORKS but is FRAGILE:
  → Depends on i < size - 1 check
  → If we forget the check → peek returns next level's node
  → prev tracking is more robust and self-documenting

RECOMMENDATION: Use prev tracking → no peek ambiguity

Pitfall 3: O(1) Space on Arbitrary Tree Without Dummy

Tree:
          1
         / \
        2   3
         \   \
          5   7

O(1) perfect-tree approach:
  curr = 2:
    curr.left = null → SKIP
    curr.right = 5 → but what do we link 5 to?
    curr.right.next = curr.next.left → 3.left = null → NullPointerException!

→ The perfect-tree O(1) approach assumes ALL children exist
→ On arbitrary trees → null children → crashes or wrong links
→ MUST use dummy technique for arbitrary trees

Pitfall 4: Forgetting That next Is Already null

→ Node constructor initializes next = null
→ The LAST node of each level should have next = null
→ Since it's already null → no action needed
→ But if you accidentally set it to something else → bug

SAFE PATTERN:
  → Only set next when prev != null
  → Never explicitly set next = null (it already is)
  → This prevents accidental overwrites

Pitfall 5: Confusing Node.next With ListNode.next

→ In linked list problems (Projects 16, 20, 23): ListNode.next = next node in list
→ In this problem: TreeNode.next = right neighbor at same level
→ TreeNode ALSO has .left and .right for children

→ The "next" pointer in this problem is HORIZONTAL (same level)
→ The "left/right" pointers are VERTICAL (parent to child)
→ Don't confuse them in your code
→ Some implementations name it "nextRight" for clarity

Pitfall 6: Verifying the Solution

→ How to verify next pointers are correct?
→ Traverse each level using next pointers:
     Start at leftmost node of each level
     Follow next pointers until null
     Compare with BFS level output

→ If they match → next pointers are correct
→ This is a good testing strategy for your solution
