PROJECT 33: Minimum Window Substring

🎯 Sliding Window → HashMap Match Counting → Shrink-to-Optimal Technique

🧩 THE HYPOTHETICAL PROBLEM

→ You are given TWO strings: s (the source) and t (the target)
→ Find the SMALLEST substring of s that contains ALL characters of t
→ Including duplicates: if t has two 'a's, the window must have at least two 'a's
→ If no such window exists, return empty string ""
→ If multiple windows of same minimum size exist, return any one
→ Characters are case-sensitive: 'A' ≠ 'a'

Example 1:
  s = "ADOBECODEBANC"
  t = "ABC"
  Output: "BANC"
  → Substrings containing A, B, C:
       "ADOBEC" (length 6)
       "ADOBECODEB" (length 10)  
       "CODEBA" (length 6)
       "BANC" (length 4)  ← SMALLEST
  → Answer: "BANC"

Example 2:
  s = "a"
  t = "a"
  Output: "a"

Example 3:
  s = "a"
  t = "aa"
  Output: ""  → s has only one 'a', t needs two → impossible

Example 4:
  s = "AAABBBCCC"
  t = "ABC"
  Output: "ABBBC" or "BBBCC" → depends on implementation, both length 5
  Actually: let's trace → "ABBBC" has A,B,C → length 5
                          "BBBCC" has B,C but no A → invalid
  So: first valid window containing A,B,C → then shrink

🔍 HOW TO APPROACH IT

Step 1 — Why This Problem is the Sliding Window Boss Battle

→ In Project 5 (Max Sum Subarray Size K):
     → Fixed window size → slide and maintain sum
     → Simple: add right element, remove left element

→ In Project 6 (Smallest Subarray with Sum ≥ Target):
     → Variable window → expand right, shrink left when condition met
     → Condition: numeric sum ≥ target → easy to check

→ In Project 12 (Longest Substring with K Distinct):
     → Variable window → HashMap tracks character frequencies
     → Condition: number of distinct characters ≤ K
     → Shrink when condition VIOLATED

→ THIS PROJECT combines everything AND adds new complexity:
     → Variable window ✓ (from Project 6)
     → HashMap frequency tracking ✓ (from Project 12)
     → NEW: matching AGAINST a target string's frequencies
     → NEW: tracking how many characters are FULLY matched
     → NEW: shrinking to find MINIMUM valid window (not maximum)
     → NEW: updating the global best DURING shrinking

This is the SYNTHESIS problem for sliding window

Step 2 — The Brute Force First

→ Check every possible substring of s
→ For each substring, verify if it contains all characters of t
→ Track the shortest valid one

for i = 0 to len(s)-1:
    for j = i to len(s)-1:
        substring = s[i..j]
        if containsAll(substring, t):
            updateMinimum(substring)

→ O(n²) substrings × O(n) verification each = O(n³) time
→ WAY too slow for large inputs
→ We need O(n) — and sliding window gives us that

Step 3 — The Key Insight: Two HashMaps + Match Counter

→ We need to track TWO things:
     1. What t REQUIRES (target frequencies)
     2. What our current window HAS (window frequencies)

→ Target frequency map (built once, never changes):
     t = "ABC" → {A:1, B:1, C:1}
     t = "AABC" → {A:2, B:1, C:1}

→ Window frequency map (changes as window slides):
     Window "ADOB" → {A:1, D:1, O:1, B:1}

→ MATCH COUNTER: how many characters are FULLY satisfied
     → A character is "fully satisfied" when:
          windowFreq[char] >= targetFreq[char]
     → When matchCount == number of distinct characters in t
          → The window is VALID → contains all of t's characters

→ This match counter is the KEY OPTIMIZATION
     → Without it: check all characters each time → O(k) per step
     → With it: update only the affected character → O(1) per step

Step 4 — The Expand-Then-Shrink Strategy

→ EXPAND: move right pointer → add character to window
     → If adding this character causes its frequency to MEET the target
     → Increment matchCount

→ SHRINK: while window is VALID (matchCount == required):
     → Current window is valid → record it if it's the smallest so far
     → Try to shrink from left → remove leftmost character
     → If removing this character causes its frequency to DROP BELOW target
     → Decrement matchCount
     → Continue shrinking as long as window remains valid

→ This is the OPPOSITE of Project 12:
     Project 12: Expand until INVALID → shrink until valid → track LONGEST
     Project 33: Expand until VALID → shrink while still valid → track SHORTEST

The mental model:
  → Expand right to FIND a valid window
  → Shrink left to MINIMIZE that valid window
  → Record the minimum
  → Continue expanding right to find next valid window
  → Repeat

Step 5 — Full Walkthrough

s = "ADOBECODEBANC", t = "ABC"
Target: {A:1, B:1, C:1}, required = 3 distinct chars

Expand right:
  right=0: 'A' → window{A:1} → A meets target(1) → match=1

  Window: s[0..5] = "ADOBEC" (length 6) → record as best

  Continue expanding:

    Window: s[3..10] = "BECODEBA" (length 8) → still worse

    Stop shrinking

  Continue expanding:
  right=11: 'N' → match=2

    Window: s[9..12] = "BANC" (length 4) → NEW BEST!

  Answer: "BANC" (length 4) ✓

🛠️ HOW CAN WE SOLVE IT

The Algorithm — Precise Steps

1. BUILD target frequency map from string t
     → Count occurrences of each character in t
     → Track how many DISTINCT characters are in t → call this "required"

2. INITIALIZE window frequency map (empty)
     → Track occurrences of each character in current window
     → Initialize matchCount = 0 (no characters fully matched yet)

3. INITIALIZE pointers and result trackers:
     → left = 0, right = 0
     → minLength = infinity, minStart = 0

4. EXPAND: iterate right from 0 to s.length()-1:
     a. Add s.charAt(right) to window frequency map
     b. If this character is in target AND
        windowFreq[char] == targetFreq[char] (exactly meets requirement):
          → matchCount++

     c. SHRINK: while matchCount == required:
          → Current window [left, right] is valid
          → If (right - left + 1) < minLength:
               → Update minLength and minStart
          → Remove s.charAt(left) from window frequency map
          → If this character is in target AND
             windowFreq[char] < targetFreq[char] (just dropped below):
               → matchCount--
          → left++

5. RETURN: if minLength is still infinity → return ""
           else → return s.substring(minStart, minStart + minLength)

The Match Counter — Why It's Brilliant

WITHOUT match counter:
  → To check if window is valid:
  → For each character in target: check windowFreq[char] >= targetFreq[char]
  → This is O(k) where k = distinct characters in t
  → Done at every expansion step → total O(n × k)

WITH match counter:
  → Maintain a running count of fully-matched characters
  → When adding a character: check if THIS character just became fully matched
  → When removing a character: check if THIS character just became unmatched
  → Each check is O(1)
  → Validity check: matchCount == required → O(1)
  → Total: O(n)

The counter goes UP when:
  → We add a character and its window count EQUALS its target count
  → NOT when window count EXCEEDS target count
  → Only the exact transition from "not enough" to "enough" triggers increment

The counter goes DOWN when:
  → We remove a character and its window count drops BELOW its target count
  → The exact transition from "enough" to "not enough" triggers decrement

Why windowFreq == targetFreq (Not >=) for Incrementing Match

Consider t = "AA" → target = {A:2}

Adding characters:
  → Add first 'A': windowFreq{A:1}, targetFreq{A:2}
       → 1 == 2? NO → matchCount stays 0
       → Correct: we need 2 A's but only have 1 → not yet matched

  → Add second 'A': windowFreq{A:2}, targetFreq{A:2}
       → 2 == 2? YES → matchCount becomes 1
       → Correct: we now have enough A's

  → Add third 'A': windowFreq{A:3}, targetFreq{A:2}
       → 3 == 2? NO → matchCount stays 1
       → Correct: we already counted this match → don't double-count

Removing characters:
  → Remove 'A': windowFreq{A:2}, targetFreq{A:2}
       → 2 < 2? NO → matchCount stays 1
       → Correct: still have enough A's

  → Remove 'A': windowFreq{A:1}, targetFreq{A:2}
       → 1 < 2? YES → matchCount becomes 0
       → Correct: no longer have enough A's

KEY: The == check on increment and < check on decrement
     ensure matchCount changes EXACTLY at the transition points

✅ WHAT SHOULD YOU DO

→ ALWAYS build the target map FIRST before starting the window
     → This is a one-time O(m) operation where m = length of t
     → It never changes during the algorithm

→ ALWAYS use == (not >=) when incrementing matchCount
     → windowFreq[char] == targetFreq[char] → just reached the threshold
     → Using >= would increment matchCount every time we add an extra copy
     → matchCount would over-count → window validity check breaks

→ ALWAYS use < (strict less than) when decrementing matchCount
     → windowFreq[char] drops below targetFreq[char] → just lost the threshold
     → Must decrement AFTER reducing the frequency

→ ALWAYS update the minimum BEFORE shrinking the left pointer
     → The current window [left, right] is valid
     → Record it first, THEN try to shrink
     → Shrinking might invalidate the window

→ ALWAYS track minStart and minLength separately
     → Don't store the actual substring during the loop
     → String creation is O(n) → doing it repeatedly is wasteful
     → Store indices, create substring only at the end

→ ALWAYS handle the "no valid window" case
     → If minLength remains Integer.MAX_VALUE → return ""
     → This happens when t contains characters not in s
     → Or when t is longer than s

🚫 WHAT SHOULD YOU NOT DO

→ DO NOT increment matchCount when windowFreq > targetFreq
     → This over-counts → matchCount exceeds "required"
     → Window appears valid when it's not fully minimal
     → Leads to wrong minimum window

→ DO NOT forget to handle characters NOT in the target
     → Characters in s but not in t still enter the window
     → They get added to windowFreq → harmless
     → But they should NOT affect matchCount
     → Check "is this character in target?" before updating matchCount

→ DO NOT shrink BEFORE recording the valid window
     → Common bug: shrink first, then check
     → After shrinking, window might be invalid
     → You'd miss recording the valid state

→ DO NOT use substring creation inside the loop
     → s.substring(left, right+1) inside the while loop
     → Each call creates a new String object → O(n) per call
     → Over n iterations → O(n²) just for string creation
     → Store indices instead

→ DO NOT use a separate validity check function
     → Tempting: write boolean isValid() that compares all frequencies
     → This is O(k) per call → called O(n) times → O(nk) total
     → The match counter eliminates this → O(1) per step

→ DO NOT confuse the shrink direction
     → Project 12 (Longest with K distinct): shrink when INVALID → find longest
     → THIS project: shrink while VALID → find shortest
     → The shrink CONDITION is opposite
     → Project 12: while (map.size() > K) → shrink
     → Project 33: while (matchCount == required) → shrink

🔀 WHEN TO USE THIS PATTERN

✓ Find the SMALLEST/SHORTEST substring satisfying a character condition
     → Minimum window containing all target characters
     → Minimum window containing at least K distinct characters
     → Shortest substring with all unique characters from a set

✓ The condition involves CHARACTER FREQUENCY matching
     → Not just "contains character X" but "contains X at least N times"
     → HashMap frequency tracking is natural

✓ The condition is MONOTONIC with window size:
     → If window [left, right] is valid
     → Then window [left, right+1] is ALSO valid (adding more can't hurt)
     → This monotonicity enables the sliding window approach
     → If adding an element could INVALIDATE a valid window → can't use this

✓ You need O(n) time complexity
     → Brute force is O(n²) or O(n³)
     → Sliding window reduces to O(n) with the match counter

✓ Similar problems:
     → Minimum window containing all characters of t (this problem)
     → Find all anagrams of t in s (fixed window variant)
     → Longest substring with at most K replacements (Project 34)
     → Permutation in string (check if any permutation of t exists in s)

🚫 WHEN NOT TO USE THIS PATTERN

✗ The condition is NOT monotonic
     → Example: "find substring with EXACTLY K distinct characters"
     → Adding a character might go from valid (K distinct) to invalid (K+1)
     → Standard sliding window doesn't directly work
     → Need: atMost(K) - atMost(K-1) trick

✗ You need to find ALL valid substrings (not just the minimum)
     → Sliding window finds the optimal → not all
     → For all occurrences → need additional bookkeeping

✗ The condition involves ORDER of characters
     → "Find smallest substring where characters appear in order of t"
     → This is a subsequence problem → not a frequency problem
     → Sliding window doesn't help → need DP or two-pointer variants

✗ The source string is very small
     → For tiny inputs → brute force is fine and simpler
     → Sliding window's advantage is asymptotic → matters for large n

✗ Non-string problems where elements have dependencies
     → Sliding window works on independent elements
     → If element i's validity depends on element j's position → different approach

⚙️ HOW IT ONLY WORKS — Internal Mechanics

The Two-Phase Engine

PHASE 1 — EXPANSION (right pointer moves):
  → Goal: find a VALID window
  → Right pointer moves one step at a time
  → Each step adds one character to the window
  → Match counter tracks progress toward validity
  → When matchCount == required → transition to Phase 2

PHASE 2 — CONTRACTION (left pointer moves):
  → Goal: MINIMIZE the valid window
  → Left pointer moves inward
  → Each step removes one character from the window
  → If removal breaks validity → transition back to Phase 1
  → Before each removal → record current window as candidate minimum

The phases ALTERNATE:
  Expand → become valid → shrink → become invalid → expand → become valid → ...

This creates a "caterpillar" motion:
  → Right end stretches forward (expand)
  → Left end catches up (shrink)
  → Like an inchworm crawling along the string

Why Each Pointer Moves at Most n Times

→ Right pointer: starts at 0, moves to n-1, never moves backward
     → Total right movements: exactly n

→ Left pointer: starts at 0, moves forward only, ends at most at n
     → Total left movements: at most n

→ Total pointer movements: at most 2n
→ Each movement: O(1) work (HashMap get/put, comparison)
→ TOTAL TIME: O(2n) = O(n)

→ This is the AMORTIZED analysis:
     → Some right-moves trigger many left-moves (during shrinking)
     → But left can only move forward → bounded by n total
     → The inner while loop doesn't make it O(n²)
     → Because left's total movement across ALL outer iterations is ≤ n

HashMap vs Array for Frequency Tracking

OPTION A — HashMap<Character, Integer>:
  → Works for any character set (Unicode, etc.)
  → O(1) average per operation (hashing)
  → Slight overhead: boxing, hashing, collision handling
  → Flexible

OPTION B — int[128] or int[256] array:
  → Works for ASCII characters only
  → O(1) guaranteed per operation (direct indexing)
  → No hashing overhead → faster in practice
  → Less flexible

FOR INTERVIEWS:
  → If characters are ASCII → use int[128] → simpler and faster
  → If characters could be Unicode → use HashMap
  → Most LeetCode problems: ASCII → array is preferred

IMPLEMENTATION WITH ARRAY:
  → int[] targetFreq = new int[128];
  → int[] windowFreq = new int[128];
  → targetFreq['A']++ instead of map.put('A', map.getOrDefault('A',0)+1)
  → Cleaner and faster

The Validity Invariant

DEFINITION:
  → Window is VALID iff for every character c in target:
       windowFreq[c] >= targetFreq[c]

EQUIVALENT (via match counter):
  → Window is VALID iff matchCount == required
  → Where required = number of distinct characters in t

MAINTAINED BY:
  → Increment matchCount ONLY when windowFreq[c] transitions
    from (targetFreq[c] - 1) to targetFreq[c]
       → i.e., when windowFreq[c] BECOMES equal to targetFreq[c]

  → Decrement matchCount ONLY when windowFreq[c] transitions
    from targetFreq[c] to (targetFreq[c] - 1)
       → i.e., when windowFreq[c] DROPS BELOW targetFreq[c]

PROOF OF CORRECTNESS:
  → Each distinct character in t contributes exactly 0 or 1 to matchCount
  → It contributes 1 iff windowFreq[c] >= targetFreq[c]
  → matchCount = sum of all these contributions
  → matchCount == required iff ALL characters are satisfied
  → Therefore: matchCount == required ⟺ window is valid ✓

⚠️ COMMON PITFALLS AND FAILURE MODES

Pitfall 1: Match Counter Over-Increment

WRONG:
  windowFreq[c]++;
  if (windowFreq[c] >= targetFreq[c]) {
      matchCount++;    // ← BUG: triggers EVERY time freq ≥ target
  }

EXAMPLE:
  t = "A" → target{A:1}, required=1
  s = "AAA"

  Add 'A': windowFreq{A:1} → 1 >= 1 → matchCount=1 ✓
  Add 'A': windowFreq{A:2} → 2 >= 1 → matchCount=2 ✗ WRONG
  Add 'A': windowFreq{A:3} → 3 >= 1 → matchCount=3 ✗ WRONG

  matchCount should never exceed required (which is 1)

CORRECT:
  windowFreq[c]++;
  if (windowFreq[c] == targetFreq[c]) {  // ← EXACT equality
      matchCount++;
  }

  Add 'A': windowFreq{A:1} → 1 == 1 → matchCount=1 ✓
  Add 'A': windowFreq{A:2} → 2 == 1? NO → matchCount=1 ✓
  Add 'A': windowFreq{A:3} → 3 == 1? NO → matchCount=1 ✓

Pitfall 2: Match Counter Under-Decrement

WRONG:
  if (windowFreq[c] <= targetFreq[c]) {
      matchCount--;    // ← BUG: decrements before freq drops below
  }
  windowFreq[c]--;

CORRECT ORDER:
  windowFreq[c]--;     // ← FIRST reduce frequency
  if (windowFreq[c] < targetFreq[c]) {  // ← THEN check if dropped below
      matchCount--;
  }

WHY ORDER MATTERS:
  → windowFreq[c] was 2, targetFreq[c] is 1
  → WRONG way: 2 <= 1? NO → don't decrement → then reduce to 1
       → matchCount stays same ✓ (accidentally correct here)
  → But: windowFreq[c] was 1, targetFreq[c] is 1
  → WRONG way: 1 <= 1? YES → matchCount-- → then reduce to 0
       → matchCount decremented ✓ (correct but fragile logic)

  CORRECT way: reduce first, then check < :
  → windowFreq 2→1: 1 < 1? NO → matchCount same ✓
  → windowFreq 1→0: 0 < 1? YES → matchCount-- ✓
  
  Clean, consistent, correct

Pitfall 3: Not Checking if Character is in Target

WRONG:
  char c = s.charAt(right);
  windowFreq[c]++;
  if (windowFreq[c] == targetFreq[c]) {  // targetFreq[c] might be 0
      matchCount++;
  }

ISSUE:
  → Character 'D' is NOT in target → targetFreq['D'] = 0
  → windowFreq['D'] goes from 0 to 1
  → 1 == 0? NO → no increment → SAFE in this case

  → BUT what about when windowFreq['D'] was already 0?
  → Adding first 'D': windowFreq 0→1, target is 0, 1 == 0? NO → fine
  → This actually works because default is 0
  → BUT: for the shrink side:

  char c = s.charAt(left);
  windowFreq[c]--;
  if (windowFreq[c] < targetFreq[c]) {  // targetFreq[c] = 0
      matchCount--;  // ← BUG: -1 < 0 is false, so no decrement → actually safe
  }

  → With int arrays defaulting to 0, this is accidentally safe
  → But with HashMaps, targetFreq.get(c) might return null → NPE
  → ALWAYS guard with: if (targetFreq[c] > 0) or check containsKey

Pitfall 4: Empty Target String

→ t = "" → no characters required → every window is valid
→ Minimum valid window = "" (empty string)
→ Should return ""
→ Edge case: check if t is empty at the start

Pitfall 5: Target Longer Than Source

→ t = "ABCDEF", s = "ABC"
→ Impossible to contain all of t in any substring of s
→ Algorithm naturally handles this:
     → matchCount never reaches required
     → minLength stays Integer.MAX_VALUE
     → Return ""

📈 SCALABILITY CONSIDERATIONS

Time Complexity: O(n + m)
  → n = length of s, m = length of t
  → Building target map: O(m)
  → Sliding window: O(n) — each pointer moves at most n times
  → Total: O(n + m)

Space Complexity: O(k)
  → k = number of distinct characters in s and t combined
  → For ASCII: O(128) = O(1) with fixed-size array
  → For Unicode: O(k) with HashMap

Practical performance:
  → n = 10^5: ~100,000 operations → instant
  → n = 10^6: ~1,000,000 operations → few milliseconds
  → n = 10^7: ~10,000,000 operations → tens of milliseconds
  → The constant factor is very small (array lookups, comparisons)

vs Brute Force:
  → Brute force: O(n² × m) for checking all substrings
  → n = 10^5: 10^10 × m operations → TIMEOUT
  → Sliding window: 10^5 operations → instant
  → Speedup: ~100,000x for n=10^5

🔍 INTERPRETABILITY CONSIDERATIONS

→ The "expand then shrink" model is physically intuitive:
     → Imagine stretching a rubber band along the string (expand)
     → Once it covers all required characters (valid)
     → Tighten from the left end (shrink) to find minimum
     → When it snaps (loses a required character) → stop
     → Stretch the right end again → repeat

→ The match counter is like a checklist:
     → t = "ABC" → checklist: ☐A ☐B ☐C
     → As window expands: ☑A ☐B ☐C → ☑A ☑B ☐C → ☑A ☑B ☑C → VALID!
     → As window shrinks: ☑A ☑B ☐C → INVALID → stop

→ Interview communication:
     → "I use a sliding window with two frequency maps"
     → "A match counter tracks how many characters are fully satisfied"
     → "I expand right until valid, then shrink left to minimize"
     → "Each pointer moves at most n times → O(n) total"

→ Debugging:
     → Print window boundaries, matchCount, and current best at each step
     → Verify matchCount matches manual count of satisfied characters
     → If matchCount is wrong → check the == and < conditions

THE MATCH COUNTER PATTERN:
→ Instead of checking ALL characters each time → O(k) per step
→ Track a single integer that counts fully-matched characters
→ Increment on EXACT transition: windowFreq[c] reaches targetFreq[c]
→ Decrement on EXACT transition: windowFreq[c] drops below targetFreq[c]
→ Validity check: matchCount == required → O(1) per step

THE MINIMUM vs MAXIMUM DUALITY:
→ Finding MINIMUM valid window:
     → Expand until valid → shrink while valid → record during shrink
→ Finding MAXIMUM valid window:
     → Expand until invalid → shrink until valid → record before shrink
→ Same skeleton, opposite shrink conditions

THE EVOLUTION OF SLIDING WINDOW:
→ Project 5:  Fixed window → running sum → simple add/remove
→ Project 6:  Variable window → shrink when valid → find minimum size
→ Project 12: Variable window → HashMap → shrink when invalid → find maximum
→ Project 33: Variable window → HashMap + match counter → shrink while valid → minimum
→ Project 34: Variable window → max-frequency trick → replacement budget → maximum

PATTERN SIGNATURE:
→ "Find smallest substring containing..." ✓
→ Character frequency matching required ✓
→ Monotonic validity (bigger window stays valid) ✓
→ = SLIDING WINDOW + MATCH COUNTER + SHRINK-TO-OPTIMAL
