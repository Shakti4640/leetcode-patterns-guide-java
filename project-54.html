🧩 THE HYPOTHETICAL PROBLEM

→ You are given an array of integers representing histogram bar HEIGHTS
→ Each bar has WIDTH = 1
→ Find the area of the LARGEST rectangle that fits entirely
  within the histogram

Visual:

Heights: [2, 1, 5, 6, 2, 3]

        ┌───┐
    ┌───┤   │
    │   │   │       ┌───┐
┌───┤   │   │   ┌───┤   │
│   │   │   │   │   │   │
│   │   │   │   │   │   │
└───┴───┴───┴───┴───┴───┘
  2   1   5   6   2   3

The largest rectangle has area = 10
→ It spans bars at index 2 and 3 (heights 5 and 6)
→ The limiting height is 5 → width = 2 → area = 5 × 2 = 10

Another rectangle: height=2, spanning indices 0 through 5
→ area = 2 × 6 = 12? NO — bar at index 1 has height 1
→ Cannot span across bar with height 1 at height 2
→ Must RESPECT all bars in the range

Correct answer: 10

🔍 HOW TO APPROACH IT

Step 1 — Understand What "Rectangle in Histogram" Means

→ A rectangle in a histogram must:
     → Have a fixed HEIGHT
     → Span CONSECUTIVE bars
     → Every bar in the span must be ≥ the rectangle's height
     
→ The rectangle's height is LIMITED by the SHORTEST bar in its span
→ The rectangle's width is the NUMBER of consecutive bars it spans
→ Area = height × width

Step 2 — Think About Each Bar as the Limiting Height

→ KEY INSIGHT: For each bar i with height h[i]
→ Ask: "If THIS bar is the shortest bar in a rectangle, 
        how wide can that rectangle be?"

→ The rectangle extends LEFT until it hits a bar SHORTER than h[i]
→ The rectangle extends RIGHT until it hits a bar SHORTER than h[i]
→ Width = rightBoundary - leftBoundary - 1
→ Area = h[i] × width

Example: heights = [2, 1, 5, 6, 2, 3]

Bar 0 (h=2): extends left to edge, right stops at index 1 (h=1 < 2)
             width = 1, area = 2 × 1 = 2

Bar 1 (h=1): extends left to edge, right to edge (nothing shorter)
             width = 6, area = 1 × 6 = 6

Bar 2 (h=5): left stops at index 1 (h=1 < 5), right stops at index 4 (h=2 < 5)
             width = 2, area = 5 × 2 = 10  ← MAXIMUM

Bar 3 (h=6): left stops at index 2 (h=5 < 6), right stops at index 4 (h=2 < 6)
             width = 1, area = 6 × 1 = 6

Bar 4 (h=2): left stops at index 1 (h=1 < 2), right to edge
             width = 4, area = 2 × 4 = 8... wait let me recalculate

Actually for bar 4 (h=2):
  → Left: index 3 has h=6 ≥ 2 ✓, index 2 has h=5 ≥ 2 ✓, 
           index 1 has h=1 < 2 ✗ → left boundary = 1
  → Right: index 5 has h=3 ≥ 2 ✓, then edge → right boundary = 6
  → width = 6 - 1 - 1 = 4, area = 2 × 4 = 8

Bar 5 (h=3): left stops at index 4 (h=2 < 3), right to edge
             width = 1, area = 3 × 1 = 3

Maximum area = 10 ✓

Step 3 — The Core Question Becomes...

→ For each bar i, find:
     → LEFT BOUNDARY: index of the nearest bar to the LEFT that is SHORTER
     → RIGHT BOUNDARY: index of the nearest bar to the RIGHT that is SHORTER

→ THIS IS EXACTLY the Monotonic Stack pattern from Project 36
→ Project 36 found "Next Greater Element" using a DECREASING stack
→ Here we need "Next SMALLER Element" using an INCREASING stack
→ Same machinery — flipped comparison

Step 4 — Why Brute Force Fails

Brute Force Approach 1: Check every pair (i, j)
→ For each pair of indices, find min height in range, compute area
→ O(n²) pairs × O(n) to find min = O(n³)

Brute Force Approach 2: For each bar, expand left and right
→ For each bar i, scan left until shorter bar, scan right until shorter bar
→ O(n) per bar × O(n) bars = O(n²)

Both are too slow for n = 100,000
→ We need O(n) → Monotonic Stack

🛠️ HOW CAN WE SOLVE IT

Approach 1 — Two-Pass with Monotonic Stack (Clearest to Understand)

IDEA:
→ Pass 1: Find "Previous Smaller Element" index for each bar (left boundary)
→ Pass 2: Find "Next Smaller Element" index for each bar (right boundary)
→ Pass 3: Compute area for each bar using boundaries

Pass 1 — Previous Smaller Element (Left Boundary):
→ Maintain a stack of indices in INCREASING order of heights
→ Scan LEFT to RIGHT
→ For each bar i:
     → Pop all bars from stack that are ≥ heights[i]
     → Stack top (after popping) is the previous smaller element
     → If stack empty → left boundary is -1 (extends to left edge)
     → Push i onto stack

Pass 2 — Next Smaller Element (Right Boundary):
→ Maintain a stack of indices in INCREASING order of heights
→ Scan RIGHT to LEFT
→ For each bar i:
     → Pop all bars from stack that are ≥ heights[i]
     → Stack top (after popping) is the next smaller element
     → If stack empty → right boundary is n (extends to right edge)
     → Push i onto stack

Pass 3 — Compute Maximum Area:
→ For each bar i:
     → width = rightBoundary[i] - leftBoundary[i] - 1
     → area = heights[i] × width
     → Track maximum

Approach 2 — Single-Pass with Monotonic Stack (Optimal)

IDEA:
→ Maintain an INCREASING stack of indices
→ When we encounter a bar SHORTER than stack top
→ The stack top bar has found its RIGHT boundary (current index)
→ The stack top bar's LEFT boundary is the element below it in stack
→ We can compute its area immediately and pop it

This is more elegant but harder to grasp initially
→ I will implement BOTH approaches

Visual Walkthrough — Single Pass

Heights: [2, 1, 5, 6, 2, 3]
Stack: [] (stores INDICES, maintains increasing height order)

i=0, h=2:
  Stack empty → push 0
  Stack: [0]    (heights: [2])

i=1, h=1:
  h[1]=1 < h[stack.top]=h[0]=2 → POP index 0
    → Popped bar: index=0, height=2
    → Right boundary: i=1
    → Left boundary: stack is now empty → -1
    → Width: 1 - (-1) - 1 = 1
    → Area: 2 × 1 = 2
  Stack empty, push 1
  Stack: [1]    (heights: [1])
  maxArea = 2

i=2, h=5:
  h[2]=5 > h[stack.top]=h[1]=1 → just push
  Stack: [1, 2]    (heights: [1, 5])

i=3, h=6:
  h[3]=6 > h[stack.top]=h[2]=5 → just push
  Stack: [1, 2, 3]    (heights: [1, 5, 6])

i=4, h=2:
  h[4]=2 < h[stack.top]=h[3]=6 → POP index 3
    → Popped bar: index=3, height=6
    → Right boundary: i=4
    → Left boundary: stack.top=2
    → Width: 4 - 2 - 1 = 1
    → Area: 6 × 1 = 6
    maxArea = 6
    
  h[4]=2 < h[stack.top]=h[2]=5 → POP index 2
    → Popped bar: index=2, height=5
    → Right boundary: i=4
    → Left boundary: stack.top=1
    → Width: 4 - 1 - 1 = 2
    → Area: 5 × 2 = 10
    maxArea = 10
    
  h[4]=2 ≥ h[stack.top]=h[1]=1 → stop popping, push 4
  Stack: [1, 4]    (heights: [1, 2])

i=5, h=3:
  h[5]=3 > h[stack.top]=h[4]=2 → just push
  Stack: [1, 4, 5]    (heights: [1, 2, 3])

END OF ARRAY — Pop remaining elements (right boundary = n = 6):

  POP index 5: height=3, right=6, left=stack.top=4
    → Width: 6 - 4 - 1 = 1, Area: 3 × 1 = 3

  POP index 4: height=2, right=6, left=stack.top=1
    → Width: 6 - 1 - 1 = 4, Area: 2 × 4 = 8

  POP index 1: height=1, right=6, left=stack empty → -1
    → Width: 6 - (-1) - 1 = 6, Area: 1 × 6 = 6

FINAL maxArea = 10 ✓

✅ WHAT SHOULD YOU DO

→ ALWAYS think "for each bar, what is its maximum rectangle?"
     → This reframes the problem from "find a rectangle" to
        "find boundaries for each bar"

→ ALWAYS use an INCREASING monotonic stack
     → Project 36 used DECREASING stack for "Next Greater"
     → This problem uses INCREASING stack for "Next Smaller"
     → The FLIP is: greater→decreasing, smaller→increasing

→ ALWAYS store INDICES in the stack, not heights
     → You need indices to compute WIDTH
     → Heights can be looked up from the array using the index

→ ALWAYS handle remaining elements after the main loop
     → Bars still in the stack never found a shorter bar to their right
     → Their right boundary is n (the end of the array)

→ ALWAYS use the element BELOW the popped element as left boundary
     → The element below in an increasing stack is the 
        "previous smaller element"
     → If stack is empty after pop → left boundary is -1

→ ALWAYS compute width as: rightBoundary - leftBoundary - 1
     → The -1 accounts for the boundaries being EXCLUSIVE
     → The rectangle spans from leftBoundary+1 to rightBoundary-1

🚫 WHAT SHOULD YOU NOT DO

→ DO NOT use a DECREASING stack
     → Project 36 used decreasing for "Next Greater Element"
     → This problem needs "Next SMALLER Element"
     → Decreasing stack triggers pop on GREATER elements
     → Increasing stack triggers pop on SMALLER elements
     → Using the wrong stack type gives wrong boundaries

→ DO NOT confuse width calculation
     → Width is NOT rightIndex - leftIndex
     → Width is rightBoundary - leftBoundary - 1
     → Boundaries are EXCLUSIVE — the bars AT the boundary indices
        are shorter and NOT part of the rectangle

→ DO NOT forget to process remaining stack elements
     → After iterating through all bars, stack may still have elements
     → These bars extend all the way to the right edge
     → Forgetting this misses potentially large rectangles
     → Bar with height=1 spanning the entire array would be missed

→ DO NOT store heights in the stack instead of indices
     → If you store heights, you lose positional information
     → You cannot compute width without knowing WHERE the bar is

→ DO NOT try to find the global minimum and split
     → Divide-and-conquer on minimum works but is O(n log n)
     → Monotonic stack achieves O(n)
     → And the stack approach is more interview-friendly

→ DO NOT confuse this with "Trapping Rain Water" (Project 55)
     → Histogram rectangle: find max area UNDER bars
     → Trapping water: find volume BETWEEN bars
     → Similar stack mechanics but DIFFERENT computation logic

🔀 WHEN TO USE THIS PATTERN

✓ You need to find BOUNDARIES (nearest smaller/greater) for each element
✓ The problem involves AREA or RANGE maximization
✓ Each element's contribution depends on how far it EXTENDS
✓ The "extension" is bounded by the first element violating a condition

SIGNATURE PROBLEMS:
→ Largest Rectangle in Histogram ← THIS
→ Maximal Rectangle in Binary Matrix (2D extension of this)
→ Maximum Width Ramp
→ Sum of Subarray Minimums
→ Any problem asking "for each element, find its span/range"

🚫 WHEN NOT TO USE THIS PATTERN

✗ Boundaries depend on elements far away, not the NEAREST one
     → Monotonic stack finds the NEAREST smaller/greater
     → If you need the Kth nearest → different technique

✗ The condition is not monotonic
     → "Nearest element with same parity" → no stack ordering helps
     → Stack relies on monotonic invariant

✗ You need bidirectional relationships simultaneously
     → Stack naturally handles one direction at a time
     → Two-pass or single-pass with careful bookkeeping needed
     → (This problem handles it — but be aware of the complexity)

✗ Elements are being dynamically inserted/deleted
     → Monotonic stack is for STATIC arrays
     → Dynamic scenarios need balanced BSTs or segment trees

⚙️ HOW IT ONLY WORKS — Internal Mechanics

The Increasing Stack Invariant

INVARIANT: Heights at indices stored in the stack are in 
           STRICTLY INCREASING order from bottom to top

Why INCREASING for "Next Smaller"?
→ When a new bar is SHORTER than stack top
→ The stack top has found something shorter to its right
→ That shorter bar IS the right boundary
→ We pop and compute

Contrast with Project 36 (Next Greater Element):
→ DECREASING stack for "Next Greater"
→ When a new element is TALLER than stack top
→ Stack top has found something greater → pop and record

THE PATTERN:
→ Looking for Next GREATER → maintain DECREASING stack → pop on GREATER
→ Looking for Next SMALLER → maintain INCREASING stack → pop on SMALLER
→ The stack maintains the OPPOSITE order of what you're searching for

Why the Element Below = Left Boundary

Stack (increasing): [..., a, b, c]    where h[a] < h[b] < h[c]

When we pop c because h[i] < h[c]:
→ c's right boundary = i (the element that caused the pop)
→ c's left boundary = b (the element below c in stack)

WHY is b the left boundary?
→ b is in the stack BELOW c
→ Stack is increasing → h[b] < h[c]
→ b was pushed BEFORE c → b is to the LEFT of c
→ Everything BETWEEN b and c was already popped (they were ≥ h[c])
→ So b is the NEAREST element to the LEFT that is SHORTER than c
→ EXACTLY the "Previous Smaller Element" for c

This is the ELEGANT part:
→ Single stack gives us BOTH boundaries simultaneously
→ Right boundary = the element that triggers the pop
→ Left boundary = the element below in the stack

Width Formula Derivation

For popped element at index top:
→ Left boundary (exclusive): leftBound (index of previous smaller)
→ Right boundary (exclusive): rightBound (index of next smaller)

The rectangle spans from leftBound+1 to rightBound-1 inclusive
→ Number of bars = (rightBound - 1) - (leftBound + 1) + 1
                  = rightBound - leftBound - 1

So: width = rightBound - leftBound - 1
    area  = heights[top] × width

When stack is empty after pop:
→ Left boundary = -1 (virtual index before the array)
→ width = rightBound - (-1) - 1 = rightBound

Single-Pass vs Two-Pass Equivalence

TWO-PASS approach:
  Pass 1 → leftBoundary[i] for all i (using stack, left to right)
  Pass 2 → rightBoundary[i] for all i (using stack, right to left)
  Pass 3 → area[i] = heights[i] × (right[i] - left[i] - 1)

SINGLE-PASS approach:
  → Process left to right
  → When popping element from stack:
       → Right boundary = current index i
       → Left boundary = new stack top (or -1)
       → Compute area immediately

Both are O(n) time, O(n) space
→ Single-pass is more elegant, fewer iterations
→ Two-pass is easier to understand and debug
→ In interviews, use whichever you can explain clearly

Why Every Element is Pushed and Popped Exactly Once

→ Each of the n elements is PUSHED onto the stack exactly once
→ Each of the n elements is POPPED from the stack at most once
→ Total operations = 2n = O(n)
→ Even though there's a while loop inside the for loop
→ The TOTAL pops across ALL iterations ≤ n
→ This is AMORTIZED O(1) per element
→ Same analysis as Project 36

⚠️ COMMON PITFALLS AND FAILURE MODES

Pitfall 1: Forgetting to Drain the Stack

After processing all bars, elements remaining in the stack
never encountered a shorter bar to their right.

WRONG:
for (int i = 0; i < n; i++) {
    // ... process and pop
}
// DONE? NO! Stack still has elements!

CORRECT:
for (int i = 0; i < n; i++) {
    // ... process and pop
}
while (!stack.isEmpty()) {
    // Pop remaining elements with rightBoundary = n
}

ALTERNATIVE — Sentinel approach:
→ Append a bar of height 0 at the end
→ This forces ALL remaining bars to be popped during the main loop
→ Eliminates the need for a separate drain loop
→ Cleaner code but requires understanding WHY

Pitfall 2: Equal Heights — Strict vs Non-Strict Comparison

Heights: [5, 5, 5]

If we pop only when strictly smaller (h[i] < h[stack.top]):
→ Equal heights just get pushed
→ Stack: [0, 1, 2] → drain phase computes correctly

If we pop when ≤ (h[i] <= h[stack.top]):
→ Each equal bar pops the previous one
→ The LAST bar of a group gets the full width
→ Also correct, but intermediate pops compute SMALLER areas

BOTH work for finding the MAXIMUM area
→ But if you need EACH bar's exact boundaries, be careful
→ For this problem (just max area), either comparison works
→ Convention: use < for right boundary, the max will still be found

Pitfall 3: Width Calculation Off-by-One

Popped index: top = 3
Right boundary: i = 5
Left boundary: stack.peek() = 1

WRONG: width = i - top = 5 - 3 = 2
→ This only counts bars from index 3 to 4
→ Misses bar at index 2 which might also be tall enough

CORRECT: width = i - stack.peek() - 1 = 5 - 1 - 1 = 3
→ Counts bars at indices 2, 3, 4
→ Because everything between left boundary (1) and right boundary (5)
   was ≥ heights[3] — they were already popped or are being popped

Pitfall 4: Empty Stack During Width Calculation

When computing left boundary after a pop:
→ If stack is empty → no previous smaller element exists
→ Left boundary = -1 (bar extends to the very beginning)
→ Width = i - (-1) - 1 = i

WRONG: width = i - stack.peek() - 1  // NullPointerException!
CORRECT: 
  int leftBound = stack.isEmpty() ? -1 : stack.peek();
  int width = i - leftBound - 1;

Pitfall 5: Integer Overflow in Area Calculation

→ heights[i] can be up to 10^4 or 10^9 depending on constraints
→ width can be up to n (10^5)
→ area = height × width
→ If height = 10^9 and width = 10^5 → area = 10^14
→ OVERFLOWS int (max ~2.1 × 10^9)
→ Use long for area computation

long area = (long) heights[top] * width;

📈 SCALABILITY CONSIDERATIONS

TIME COMPLEXITY:
→ Each element pushed once, popped once → O(n) amortized
→ Two-pass approach: 3 × O(n) = O(n)
→ Single-pass approach: O(n)
→ Both are optimal — cannot do better than O(n) since we must read all bars

SPACE COMPLEXITY:
→ Stack can hold at most n elements → O(n)
→ Two-pass approach also needs leftBound[] and rightBound[] arrays → O(n)
→ Single-pass approach needs only the stack → O(n)

PRACTICAL PERFORMANCE:
→ n = 100,000 → ~200,000 operations → instant
→ n = 10,000,000 → ~20,000,000 operations → < 1 second
→ Cache-friendly: sequential array access + stack operations
→ No recursion → no stack overflow risk

COMPARISON WITH ALTERNATIVES:
→ Brute force: O(n²) → fails at n > 10,000
→ Divide and conquer: O(n log n) → works but slower
→ Segment tree: O(n log n) → overkill for this problem
→ Monotonic stack: O(n) → optimal and clean

🔍 INTERPRETABILITY CONSIDERATIONS

→ The two-pass approach is HIGHLY readable:
     → "First, find left boundary for each bar"
     → "Then, find right boundary for each bar"
     → "Then, compute area for each bar"
     → Each pass has a single clear purpose

→ The single-pass approach is more ELEGANT but harder to explain:
     → "When a bar gets popped, we know both its boundaries"
     → Requires understanding of stack invariant
     → Interviewers are impressed but may ask follow-up questions

→ For INTERVIEWS:
     → Start by explaining the two-pass approach
     → Mention that single-pass optimization exists
     → Code whichever you're more comfortable with
     → The two-pass is easier to get right under pressure

→ For DEBUGGING:
     → Print stack contents at each step
     → Print boundaries when each element is popped
     → Verify width computation matches expected span

🧠 KEY TAKEAWAY

WHAT WE LEARNED:
→ "Largest Rectangle in Histogram" = find BOUNDARIES for each bar
→ Left boundary = Previous Smaller Element
→ Right boundary = Next Smaller Element
→ Area = height × (rightBound - leftBound - 1)
→ Monotonic INCREASING stack finds "Next Smaller" (flip of Project 36)

THE MENTAL MODEL:
→ Each bar "wants" to expand as far as possible
→ It stops expanding when it meets a SHORTER bar
→ The shorter bars on both sides are its BOUNDARIES
→ Monotonic stack discovers these boundaries in O(n)

THE RELATIONSHIP TO PROJECT 36:
→ Project 36: "Next Greater Element" → DECREASING stack → pop on GREATER
→ Project 54: "Next Smaller Element" → INCREASING stack → pop on SMALLER
→ Same machinery, flipped comparison
→ But here we COMPUTE something (area) at pop time
→ Not just recording the answer — performing a calculation

THE SINGLE-PASS INSIGHT:
→ When an element is popped, it knows:
     → Right boundary = the element that triggered the pop
     → Left boundary = the element now on top of the stack
→ This dual-boundary discovery is what makes the single-pass possible
→ The stack encodes BOTH left and right boundary information simultaneously

PATTERN SIGNATURE:
→ "For each element, how far does it extend?" → Monotonic Stack
→ "Bounded by nearest smaller on both sides" → Increasing Stack
→ "Maximize area/range/span" → Compute at pop time
