PROJECT 48: Number of Connected Components in Graph

🎯 Union Find → Parent Array → Find with Path Compression → Union by Rank

🧩 THE HYPOTHETICAL PROBLEM

→ You are given N nodes labeled 0 to N-1
→ You are given a list of UNDIRECTED edges
→ edges[i] = [a, b] means node a and node b are connected
→ Find the NUMBER OF CONNECTED COMPONENTS

→ A connected component = a group of nodes where every node
  can reach every other node through some path of edges

Example 1:
  n = 5
  edges = [[0,1], [1,2], [3,4]]
  
  Groups:
    {0, 1, 2} → connected via 0-1 and 1-2
    {3, 4}    → connected via 3-4
  
  Answer: 2 components

Example 2:
  n = 5
  edges = [[0,1], [1,2], [2,3], [3,4]]
  
  Groups:
    {0, 1, 2, 3, 4} → all connected in a chain
  
  Answer: 1 component

Example 3:
  n = 4
  edges = []
  
  Groups:
    {0}, {1}, {2}, {3} → no edges → every node is its own component
  
  Answer: 4 components

Example 4:
  n = 5
  edges = [[0,1], [2,3], [0,2]]
  
  Initially: {0,1} and {2,3}
  Edge [0,2] MERGES them: {0,1,2,3}
  Node 4 remains alone: {4}
  
  Answer: 2 components

🔍 HOW TO APPROACH IT

Step 1 — Recognize This is an UNDIRECTED Graph Problem

PROJECT 47 (Course Schedule):
  → DIRECTED edges: prerequisite → course (one-way)
  → Question: "Is there a valid ordering?" → Topological Sort
  → Cycle detection = "can all nodes be processed?"

THIS Problem:
  → UNDIRECTED edges: a — b (both ways)
  → Question: "How many separate groups exist?"
  → No ordering needed → no topological sort
  → No direction → no in-degrees

KEY DIFFERENCE:
  → Directed graph → ordering/dependency → Kahn's BFS (Project 47)
  → Undirected graph → grouping/connectivity → Union Find (this project)

Step 2 — Consider Simpler Approaches First

APPROACH A — BFS/DFS from Each Unvisited Node:
  → Start from node 0 → BFS/DFS to find all reachable nodes → that's component 1
  → Find next unvisited node → BFS/DFS again → component 2
  → Repeat until all nodes visited
  → Count of BFS/DFS runs = number of components
  → Time: O(V + E), Space: O(V + E) for adjacency list
  → WORKS perfectly for this problem

APPROACH B — Union Find:
  → Initially, every node is its own component (n components)
  → For each edge [a, b]: MERGE the components containing a and b
  → After processing all edges: count remaining distinct components
  → Time: O(V + E × α(V)) ≈ O(V + E), Space: O(V)
  → α(V) = inverse Ackermann function → practically constant

WHY LEARN UNION FIND if BFS/DFS works?
  → Union Find handles DYNAMIC connectivity
       → "Add edge → are A and B now connected?"
       → BFS/DFS would need to rebuild from scratch
  → Union Find answers "Are A and B in the same group?" in O(1) amortized
  → Union Find is the FOUNDATION for:
       → Kruskal's Minimum Spanning Tree
       → Accounts Merge (Project 57)
       → Dynamic graph connectivity
       → Network connectivity monitoring
  → It's a TECHNIQUE, not just a solution to this one problem

Step 3 — The Core Idea: Representative/Parent

CONCEPT:
  → Every group has a REPRESENTATIVE (also called "root" or "leader")
  → To check if two nodes are in the same group:
       → Find the representative of each
       → Same representative → same group
       → Different representatives → different groups

ANALOGY:
  → Imagine each person points to their "boss"
  → The top boss (who points to themselves) is the representative
  → To check if two people work in the same company:
       → Follow the chain of bosses up to the top
       → Same top boss → same company

DATA STRUCTURE:
  → parent[] array where parent[i] = the node that i points to
  → Initially: parent[i] = i (everyone is their own boss)
  → FIND(x): follow parent pointers until parent[x] == x → that's the root
  → UNION(x, y): find roots of x and y → make one point to the other

Step 4 — Visualize the Process

n = 5, edges = [[0,1], [2,3], [0,2]]

INITIAL STATE:
  parent: [0, 1, 2, 3, 4]
  Each node is its own root
  Components: {0}, {1}, {2}, {3}, {4} → 5 components

PROCESS EDGE [0, 1]:
  find(0) = 0, find(1) = 1 → different roots → UNION
  Set parent[1] = 0  (1 now points to 0)
  parent: [0, 0, 2, 3, 4]
  Components: {0,1}, {2}, {3}, {4} → 4 components

PROCESS EDGE [2, 3]:
  find(2) = 2, find(3) = 3 → different roots → UNION
  Set parent[3] = 2  (3 now points to 2)
  parent: [0, 0, 2, 2, 4]
  Components: {0,1}, {2,3}, {4} → 3 components

PROCESS EDGE [0, 2]:
  find(0) = 0, find(2) = 2 → different roots → UNION
  Set parent[2] = 0  (2 now points to 0)
  parent: [0, 0, 0, 2, 4]
  Components: {0,1,2,3}, {4} → 2 components

  Wait — node 3 points to 2, and 2 now points to 0
  So find(3): 3 → parent[3]=2 → parent[2]=0 → parent[0]=0 → root is 0 ✓
  Node 3 is in the same component as 0 ✓

FINAL: 2 components

Step 5 — The Problem with Naive Find

Consider a CHAIN: 0 ← 1 ← 2 ← 3 ← 4 ← 5 ← ... ← 999

find(999): 
  999 → 998 → 997 → ... → 0
  Takes 999 steps!

find(998):
  998 → 997 → ... → 0
  Takes 998 steps!

→ Each find is O(n) in the worst case
→ n finds = O(n²) total → terrible

SOLUTION: PATH COMPRESSION
  → When you find the root, make EVERY node on the path point directly to the root
  → Next time you call find on any of those nodes → O(1)!

find(999) with path compression:
  Follow: 999 → 998 → 997 → ... → 0 (root found = 0)
  Now set: parent[999]=0, parent[998]=0, parent[997]=0, ...
  All nodes now point directly to root

find(999) again: 999 → parent[999]=0 → done! O(1)

Step 6 — The Problem with Naive Union

If we always make the FIRST root point to the SECOND root:

union(0,1): parent[0] = 1      tree height = 1
union(1,2): parent[1] = 2      tree height = 2
union(2,3): parent[2] = 3      tree height = 3
union(3,4): parent[3] = 4      tree height = 4

Result: 0 → 1 → 2 → 3 → 4    (a degenerate chain)

find(0) takes 4 steps → O(n) per find

SOLUTION: UNION BY RANK
  → Track the "rank" (approximate height) of each tree
  → Always attach the SHORTER tree under the TALLER tree
  → This keeps the tree balanced → height stays O(log n)

With union by rank:
  union(0,1): rank equal → pick one → parent[1]=0, rank[0]=1
  union(2,3): rank equal → parent[3]=2, rank[2]=1
  union(0,2): rank equal → parent[2]=0, rank[0]=2

  Result:    0 (rank 2)
           / | \
          1  2  (height = 2, not 4)
             |
             3

find(3): 3 → 2 → 0 → only 2 steps instead of 4

Step 7 — The Combined Power

PATH COMPRESSION + UNION BY RANK together:

→ Each find() operation costs O(α(n)) amortized
→ α(n) = inverse Ackermann function
→ For ALL practical purposes: α(n) ≤ 4 for n up to 10^80
→ Effectively O(1) per operation

→ This makes Union Find one of the most efficient data structures
→ Almost-constant-time group membership queries
→ Almost-constant-time group merging

🛠️ HOW CAN WE SOLVE IT

The Algorithm

1. INITIALIZE:
     → parent[i] = i for all i (each node is its own root)
     → rank[i] = 0 for all i (each tree has height 0)
     → componentCount = n (initially n separate components)

2. FOR EACH EDGE [a, b]:
     → rootA = find(a)  (with path compression)
     → rootB = find(b)  (with path compression)
     → IF rootA != rootB:
          → union(rootA, rootB) using rank
          → componentCount--  (two components merged into one)
     → IF rootA == rootB:
          → Already in same component → skip

3. RETURN componentCount

Find with Path Compression — Two Implementations

RECURSIVE (clean but risk of stack overflow for deep trees):
  find(x):
    if parent[x] != x:
      parent[x] = find(parent[x])  // point directly to root
    return parent[x]

ITERATIVE (safe for large inputs):
  find(x):
    // Phase 1: Walk to root
    root = x
    while parent[root] != root:
      root = parent[root]
    
    // Phase 2: Compress path — point all nodes to root
    while parent[x] != root:
      next = parent[x]
      parent[x] = root
      x = next
    
    return root

Union by Rank

union(rootA, rootB):
  if rank[rootA] < rank[rootB]:
    parent[rootA] = rootB          // shorter tree goes under taller
  else if rank[rootA] > rank[rootB]:
    parent[rootB] = rootA
  else:
    parent[rootB] = rootA          // same rank: pick one, increase rank
    rank[rootA]++

✅ WHAT SHOULD YOU DO

→ ALWAYS implement BOTH path compression AND union by rank
     → Path compression alone: O(log n) amortized per operation
     → Union by rank alone: O(log n) per operation
     → Both together: O(α(n)) ≈ O(1) amortized per operation
     → In interviews: implement both → shows deep understanding

→ ALWAYS initialize parent[i] = i
     → Every node starts as its own root
     → This is the "each node is its own component" state

→ ALWAYS find roots BEFORE comparing
     → Don't compare parent[a] with parent[b]
     → parent[a] might not be the root (could be an intermediate node)
     → MUST call find(a) and find(b) to get actual roots

→ ALWAYS decrement component count on successful union
     → "Successful" = the two nodes were in DIFFERENT components
     → If already in the same component → no merge → no decrement

→ ALWAYS check if rootA != rootB before union
     → If they're the same → already connected → nothing to do
     → Skipping this check won't break correctness but wastes effort
     → Also prevents incorrect rank increments

🚫 WHAT SHOULD YOU NOT DO

→ DO NOT compare parent[a] == parent[b] without finding roots
     parent: [0, 0, 1, 1]
     
     Is node 0 connected to node 3?
     parent[0] = 0, parent[3] = 1 → different! But...
     find(0) = 0, find(3) = find(1) = find(0) = 0 → SAME root!
     
     → Direct parent comparison is WRONG
     → MUST use find() which follows the chain to the root

→ DO NOT use Union Find for DIRECTED graphs
     → Union Find treats edges as UNDIRECTED (symmetric)
     → union(a, b) = union(b, a) → no direction
     → For directed dependencies → Topological Sort (Project 47)

→ DO NOT forget to handle the edge case of no edges
     → n nodes, 0 edges → n components
     → parent[i] = i for all i → each is its own component
     → componentCount = n → correct

→ DO NOT implement path compression without iterative option for large inputs
     → Recursive find can cause StackOverflowError for deep chains
     → n = 100,000 with chain structure → recursion depth 100,000
     → Java default stack ≈ 512KB → ~10,000 frames → CRASH
     → Use iterative find for production code

→ DO NOT use Union Find when simple BFS/DFS suffices and no dynamic updates
     → For a ONE-TIME "count components" query → BFS/DFS is simpler
     → Union Find shines when edges arrive DYNAMICALLY
     → Or when you need REPEATED "are X and Y connected?" queries
     → In interviews: Union Find shows deeper knowledge → preferred

→ DO NOT confuse rank with SIZE
     → Rank = approximate height of the tree
     → Size = number of nodes in the component
     → Union by rank: attach shorter tree under taller tree
     → Union by size: attach smaller tree under larger tree
     → BOTH work → both give O(α(n)) with path compression
     → We implement both variants for completeness

🔀 WHEN TO USE UNION FIND

✓ You need to track CONNECTIVITY between elements
✓ Elements are grouped and groups can MERGE
✓ You need to answer "Are X and Y in the same group?" efficiently
✓ Edges/connections arrive DYNAMICALLY (one by one)
✓ You need to count the number of DISTINCT groups

REAL-WORLD ANALOGS:
  → Social network: "Are Alice and Bob in the same friend circle?"
  → Network connectivity: "Can server A reach server B?"
  → Image processing: "Which pixels belong to the same region?"
  → Game boards: "Are these two cells connected?"
  → Account merging: "Do these accounts belong to the same person?" (Project 57)
  → Kruskal's MST: "Does adding this edge create a cycle?"

PATTERN SIGNATURE:
  → Undirected connections ✓
  → Group membership queries ✓
  → Dynamic merging of groups ✓
  → = UNION FIND

🚫 WHEN NOT TO USE UNION FIND

✗ You need to SPLIT groups (Union Find only MERGES, never splits)
     → Once two groups are merged, they cannot be un-merged
     → For split operations → use different data structures

✗ You need ORDERING of elements (use Topological Sort — Project 47)

✗ Graph is DIRECTED (use BFS/DFS or Topological Sort)

✗ You need SHORTEST PATH between nodes (use BFS or Dijkstra)

✗ You need to enumerate ALL members of a group efficiently
     → Union Find answers "same group?" in O(1)
     → But listing all members requires O(n) scan
     → If you need frequent member listing → use adjacency list + BFS

✗ Graph is static and you only need component count ONCE
     → BFS/DFS is simpler and equally efficient
     → Union Find is overkill (but still valid)

⚙️ HOW IT ONLY WORKS — Internal Mechanics

The Parent Array as a Forest

parent[] represents a FOREST of trees

→ Each tree = one connected component
→ Root of each tree = representative of that component
→ parent[root] = root (root points to itself)
→ All other nodes point toward the root (directly or indirectly)

EXAMPLE:
  parent: [0, 0, 0, 3, 3]
  
  Tree 1:    0          Tree 2:  3
           / |                   |
          1  2                   4
  
  Components: {0,1,2} and {3,4}
  Roots: 0 and 3

AFTER path compression (find(2) called):
  parent: [0, 0, 0, 3, 3]  → already flat, no change
  
  But if parent were [0, 0, 1, 3, 3]:
  Tree 1:    0          
             |          
             1          
             |          
             2          
  
  find(2): 2→1→0 (root=0) → compress: parent[2]=0, parent[1]=0
  
  Tree 1:    0
           / |
          1  2    → FLAT — future finds are O(1)

Path Compression — Detailed Mechanics

BEFORE find(5):
  parent: [0, 0, 1, 2, 3, 4]
  
  Chain: 5 → 4 → 3 → 2 → 1 → 0 (root)
  
        0
        |
        1
        |
        2
        |
        3
        |
        4
        |
        5

RECURSIVE find(5):
  find(5) → parent[5]=4, so find(4)
    find(4) → parent[4]=3, so find(3)
      find(3) → parent[3]=2, so find(2)
        find(2) → parent[2]=1, so find(1)
          find(1) → parent[1]=0, so find(0)
            find(0) → parent[0]=0 → return 0
          parent[1] = 0, return 0
        parent[2] = 0, return 0
      parent[3] = 0, return 0
    parent[4] = 0, return 0
  parent[5] = 0, return 0

AFTER:
  parent: [0, 0, 0, 0, 0, 0]
  
        0
      / | \ \ \
     1  2  3 4 5    → COMPLETELY FLAT

→ One expensive find() → all future finds are O(1)
→ Cost is "amortized" across operations

Union by Rank — Why It Works

WITHOUT union by rank:
  Always attach first to second:
  
  union(0,1): 0→1
  union(1,2): 1→2   → chain: 0→1→2
  union(2,3): 2→3   → chain: 0→1→2→3
  
  Height = n-1 = O(n) → find is O(n)

WITH union by rank:
  union(0,1): rank equal → parent[1]=0, rank[0]=1
      0
      |
      1
      
  union(2,3): rank equal → parent[3]=2, rank[2]=1
      2
      |
      3
      
  union(0,2): rank equal → parent[2]=0, rank[0]=2
      0
     / \
    1   2
        |
        3
  
  Height = 2 (not 3)

THEOREM:
  → With union by rank, tree height ≤ log₂(n)
  → Because: a tree of rank r has at least 2^r nodes
  → Proof: when two trees of rank r merge, result has rank r+1
    and at least 2^r + 2^r = 2^(r+1) nodes ✓
  → So rank ≤ log₂(n) → height ≤ log₂(n) → find is O(log n)

Combined Complexity — Inverse Ackermann

WITH both path compression AND union by rank:

→ Amortized cost per operation = O(α(n))
→ α(n) = inverse Ackermann function

What is α(n)?
  → Ackermann function A(n) grows INSANELY fast:
       A(1) = 2
       A(2) = 4
       A(3) = 16
       A(4) = 65536
       A(5) = 2^65536 (a number with ~20,000 digits)
  
  → α(n) = inverse: "how many times must I apply A to exceed n?"
  → For n = number of atoms in the observable universe (~10^80):
       α(10^80) = 4
  
  → For ANY practical input size: α(n) ≤ 4
  → Effectively O(1) per operation

→ m operations on n elements: O(m × α(n)) ≈ O(m) total
→ This makes Union Find one of the fastest data structures in existence

Time Complexity for This Problem

→ Initialize parent and rank arrays: O(n)
→ Process each edge: find + find + union = O(α(n)) per edge
→ Total for e edges: O(e × α(n)) ≈ O(e)
→ OVERALL: O(n + e)
→ Same as BFS/DFS but with different constant factors

Space Complexity

→ parent array: O(n)
→ rank array: O(n)
→ No adjacency list needed (unlike BFS/DFS)
→ TOTAL: O(n)
→ BETTER than BFS/DFS which needs O(n + e) for adjacency list

⚠️ COMMON PITFALLS AND FAILURE MODES

Pitfall 1: Forgetting to Find Roots Before Union

WRONG:
  void union(int a, int b) {
      parent[a] = b;  // ← directly modifying without finding roots
  }

  → This doesn't connect the COMPONENTS
  → It just makes a point to b
  → The rest of a's component is orphaned

RIGHT:
  void union(int a, int b) {
      int rootA = find(a);
      int rootB = find(b);
      if (rootA != rootB) {
          parent[rootA] = rootB;  // connect the ROOTS
      }
  }

Pitfall 2: Not Checking rootA != rootB Before Union

If rootA == rootB and we do parent[rootA] = rootB:
  → parent[rootA] = rootA (since rootA == rootB)
  → No harm done (already pointing to itself)
  → BUT if we decrement componentCount:
       → componentCount decreases incorrectly
       → Final answer is WRONG

ALWAYS CHECK:
  if (rootA != rootB) {
      // perform union
      componentCount--;
  }

Pitfall 3: Recursive Path Compression Stack Overflow

For n = 100,000 with a chain structure:
  → find(99999) recurses 99,999 levels deep
  → Java default stack ≈ 512KB → ~10,000 frames
  → StackOverflowError!

FIX: Use ITERATIVE path compression
  → Two-pass approach: find root, then compress
  → No recursion → no stack overflow
  → We implement both versions

Pitfall 4: Rank Update Errors

WRONG:
  union(rootA, rootB):
    parent[rootA] = rootB
    rank[rootB]++         // ← always incrementing

RIGHT:
  union(rootA, rootB):
    if rank[rootA] < rank[rootB]:
      parent[rootA] = rootB     // no rank change
    else if rank[rootA] > rank[rootB]:
      parent[rootB] = rootA     // no rank change  
    else:
      parent[rootB] = rootA
      rank[rootA]++             // only increment when ranks are EQUAL

→ Rank only increases when two trees of EQUAL rank merge
→ Incrementing in other cases corrupts the rank invariant

Pitfall 5: Confusing Node Values with Node Indices

If nodes are labeled 1 to n (not 0 to n-1):
  → parent array of size n+1
  → Initialize parent[i] = i for i = 1 to n
  → Or translate to 0-based indexing

If nodes are strings (like in Project 57 — Accounts Merge):
  → Map strings to integer indices
  → Use Union Find on the integer indices
  → Map back to strings at the end

📈 SCALABILITY CONSIDERATIONS

n = nodes, e = edges

→ n = 100, e = 200 → O(300) → instant
→ n = 100,000, e = 500,000 → O(600,000) → instant
→ n = 1,000,000, e = 5,000,000 → O(6,000,000) → fast
→ n = 10,000,000, e = 50,000,000 → O(60,000,000) → manageable

→ Union Find scales LINEARLY (with α(n) constant)
→ Space is O(n) only → no adjacency list overhead
→ Handles dynamic edge additions naturally

COMPARISON WITH BFS/DFS:
  → Static graph, one query → BFS/DFS and UF are equivalent
  → Dynamic edges (online) → UF is vastly superior
       → BFS/DFS: must rebuild graph + re-traverse = O(n+e) per query
       → UF: O(α(n)) per edge addition = practically O(1)
  → Repeated "are X and Y connected?" → UF answers in O(1) amortized

🧠 KEY TAKEAWAY

WHAT WE LEARNED:
→ Union Find is a data structure for DYNAMIC GROUP MANAGEMENT
→ parent[] array represents a forest of trees → one tree per component
→ find() follows parent pointers to the root → identifies the group
→ union() connects two roots → merges two groups
→ Path compression makes find() amortized O(1)
→ Union by rank keeps trees balanced → prevents degenerate chains
→ Combined: O(α(n)) per operation → practically constant time
→ Component count = n minus number of successful unions

THE CORE MENTAL MODEL:
→ "Everyone starts as their own leader"
→ "When two people connect, their groups merge under one leader"
→ "To check if two people are in the same group: find both their leaders"
→ "Path compression: once you find the leader, tell everyone directly"
→ "Union by rank: bigger group absorbs smaller group"

PATTERN SIGNATURE:
→ Undirected connections ✓
→ Group membership / connectivity queries ✓
→ Dynamic merging of groups ✓
→ No need to split groups ✓
→ = UNION FIND

THREE OPERATIONS TO REMEMBER:
  1. find(x) → return root of x's component (with path compression)
  2. union(x, y) → merge components of x and y (by rank)
  3. connected(x, y) → find(x) == find(y)
